# vLLM的PagedAttention核心原理是什么?它如何提升推理效率

### PagedAttention 核心原理与效率提升

**1. 核心设计思想：受操作系统虚拟内存启发**

PagedAttention 将操作系统中分页管理的思想引入到 LLM 的 KV Cache 管理中。它将每个序列的 KV Cache 切分成固定大小的块，以便在非连续的 GPU 内存空间中进行存储和计算，从而解决了传统方法中内存浪费和碎片化严重的问题。

**2. 传统 KV Cache 的痛点**

*   **静态内存预分配**：为了处理不断增长的序列长度，传统推理引擎通常需要为每个请求预分配能够容纳最大序列长度的连续内存块。
*   **内存浪费与碎片化**：
    *   由于大多数请求的实际生成长度远小于最大长度，导致大量预留内存处于闲置状态（内部碎片）。
    *   随着请求的频繁创建和销毁，连续内存空间变得支离破碎，难以容纳新的长请求（外部碎片）。
*   **GPU 利用率低**：内存资源的低效利用直接限制了 Batch Size（批大小），导致 GPU 计算单元常处于空闲等待状态，整体利用率通常仅为 20%-40%。

**3. PagedAttention 解决方案详解**

#### A. 分块管理

*   **Block 划分**：将 KV Cache 按照固定的 Token 数量（例如 16 个 Token）切分成一个个小的 Block。
*   **非连续存储**：这些 Block 在 GPU 显存中**不需要物理连续**。系统维护一个逻辑到物理的映射表（类似 CPU 的页表），记录每个序列的逻辑 Block 序号对应的实际物理内存地址。

#### B. 按需分配

*   **动态增长**：不再预分配大块连续内存。随着 Decoder 生成新的 Token，系统只按需分配新的 Block。
*   **高效利用**：内存完全根据实际序列长度使用，几乎消除了内部碎片，内存浪费可控制在 4% 以内。

#### C. 内核计算优化

PagedAttention 重新设计了 Attention 计算内核，使其能够识别和处理非连续的 Block 数据，无需在计算前进行繁琐的数据拷贝和内存整理。

#### D. 显存共享与 Copy-on-Write

*   **前缀共享**：对于具有相同前缀（如相同的 System Prompt）的多个请求，vLLM 可以让它们映射到同一个物理 Block，实现显存的只读共享。
*   **写时复制**：当某个请求需要修改共享的 Block 时，系统才会自动复制一份新的 Block 供其使用，从而极大节省了显存占用。

**4. 架构图解：逻辑视图与物理视图的映射**

```text
【逻辑序列视图】
+-----------------------+-----------------------+
| Token Block 0         | Token Block 1         |
+-----------------------+-----------------------+

           映射

【Block 表】
+------------+------------+
| Block 0    | Ptr -> P2  |  <-- 指向物理 Block 2
| Block 1    | Ptr -> P5  |  <-- 指向物理 Block 5 (非连续)
+------------+------------+

           映射

【物理显存视图】
+-----+-----+-----+-----+-----+-----+
| P1  | P2  | P3  | P4  | P5  | ... |
+-----+-----+-----+-----+-----+-----+
      ^                   ^
      |                   |
   (Block 0)            (Block 1)
```

**5. 效果总结**

*   **显存利用率**：从 30-60% 提升至 **90% 以上**。
*   **推理吞吐量**：在相同硬件条件下，吞吐量提升 **2-4 倍**。
*   **Batch Size**：由于显存压力减小，支持更大的 Batch Size，进一步发挥 GPU 并行计算能力。

**6. 实战案例与代码**

**实战案例**：在 Llama-3-70B 推理中，若设置 `max_seq_len=8192`，传统 Orca 内存预分配会导致单请求显存占用极高，A100 (80G) 只能跑 2-3 个并发。

## 易错点
1. **混淆 Continuous Batching 和 PagedAttention**：Continuous Batching 解决的是计算调度问题（不同序列混合计算），PagedAttention 解决的是显存管理问题（非连续存储），两者配合才能达到最优性能，缺一不可。
2. **忽视 COW 的开销**：虽然共享前缀能省显存，但一旦发生分支，Copy-on-Write 会产生瞬时的显存分配峰值，需预留一定的 buffer 显存。
