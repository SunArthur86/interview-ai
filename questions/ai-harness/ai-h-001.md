# LangChain的核心抽象是什么?LCEL相比传统Chain有什么优势

- **LangChain核心抽象:Runnable接口**

**所有组件都实现Runnable：** invoke/ainvoke/stream/astream/batch/abatch

- **LCEL (LangChain Expression Language):**
```python
# 声明式管道
chain = prompt | model | parser
result = chain.invoke({"input": "..."})

# 并行
parallel = RunnableParallel(a=chain_a, b=chain_b)
```

- **LCEL vs 传统Chain:**

| | 传统Chain | LCEL |
|--|-----------|------|
| 语法 | 嵌套构造 | **管道操作符** |
| 异步 | 需手动实现 | **原生支持** |
| 流式 | 需手动实现 | **内置流式** |
| 重试 | 需手动实现 | **自动重试** |
| 可观测 | 难 | **自动trace** |

- **LCEL核心能力:**
1. 自动异步/同步切换
2. 自动batching
3. 自动streaming(token级流式)
4. 内置fallback(链路容错)
5. 无缝集成LangSmith追踪

**实战案例**
在构建高并发问答服务时，使用传统 Chain 处理并发请求会导致性能瓶颈。切换到 LCEL 的 `.batch()` 方法后，请求吞吐量提升了 3 倍，且代码量从 50 行减少到 5 行。

**代码示例（自定义 Runnable + Fallback）**
```python
from langchain_core.runnables import RunnableLambda
from langchain_openai import ChatOpenAI
import json

# 自定义一个容错的 JSON 解析器
def safe_parse(text):
    try:
        return json.loads(text)
    except:
        return {"error": "parsing_failed", "raw": text}

parser = RunnableLambda(safe_parse)

# 主模型与备用模型配置
llm = ChatOpenAI(model="gpt-4")
backup_llm = ChatOpenAI(model="gpt-3.5-turbo")

# 构建带容错的链路
chain = llm | parser.with_fallbacks([backup_llm | parser])
```

## 常见考点
1. **RunnablePassthrough vs RunnableLambda**：`RunnablePassthrough` 主要用于在传递数据的同时保留或修改部分字段（常配合字典使用），而 `RunnableLambda` 用于封装任意 Python 函数使其具备流式和异步能力。
2. **流式传输的细节**：在 LCEL 中，`stream()` 默认只输出模型的 Token。如果链路中包含 `parser`，如何保证流式输出是解析后的部分结果（如流式输出 JSON 对象的片段）？需要实现自定义的 `Parser` 支持增量解析。
3. **Router 链路的实现**：LCEL 如何实现动态路由（If-Else 逻辑）？通常使用 `RunnableBranch` 或者在 `RunnableLambda` 内部根据输入返回不同的 Runnable 对象。
4. **配置传递**：`config` 参数在 LCEL 中的用途，特别是如何传递 `callbacks`（用于追踪）、`tags`（用于标记）以及自定义配置（如运行时的 temperature 覆盖）。
