# 如何选择向量数据库?Pinecone vs Milvus vs Qdrant vs pgvector

- **向量数据库对比:**

| 数据库 | 类型 | 适合规模 | 部署 | 特色 |
|--------|------|---------|------|------|
| Pinecone | SaaS | **任何** | 云 | **最省心** (全托管) |
| Milvus | 开源 | **十亿+** | 自建/云 | **最强性能** (存算分离) |
| Qdrant | 开源 | 亿级 | 自建/云 | **Rust高性能** (内存友好) |
| Weaviate | 开源 | 亿级 | 自建/云 | GraphQL API (模块化) |
| Chroma | 开源 | 百万级 | 嵌入式 | **最简单** (轻量级) |
| pgvector | 插件 | 千万级 | PostgreSQL | **SQL集成** (事务支持) |

- **选型决策:**
```
规模 < 10万 → Chroma (嵌入式,零运维)
规模 10万-1000万 → Qdrant / pgvector (性价比平衡)
规模 > 1000万 → Milvus (极致性能) / Pinecone (免运维)
不想运维 → Pinecone (贵但省心,自动扩缩容)
已有PostgreSQL → pgvector (无缝集成,ACID事务支持)
需要混合检索 → Qdrant / Weaviate (原生BM25+向量)
需要强一致性 → Milvus (依赖etcd) / pgvector
```

- **混合检索支持:**
- **Qdrant**: 原生支持 (Fusion算法: 稠密向量+Payload过滤+BM25/稀疏向量)
- **Weaviate**: 原生支持 (倒排索引+向量)
- **Milvus 2.4+**: 支持标量过滤 + 稀疏向量(BM25)
- **pgvector**: 结合PostgreSQL全文检索,通过SQL做加权

- **架构示意:**
```
┌─────────────┐
│  Application │
└──────┬──────┘
       │
┌──────▼──────────────────────────────────┐
│          Vector Database                │
├────────────┬───────────────┬──────────────┤
│ Write Path │  Index (HNSW) │ Read Path    │
│ (Insert)   │  + Ann Search │ (Query/Filter)│
└────────────┴───────────────┴──────────────┘
```

- **实战案例**：在构建企业级知识库时，我们起初使用了 Pinecone，但随着数据量达到 5000 万向量，月费用激增且无法利用公司内部的数据安全策略。后迁移至 **Milvus**，利用其存算分离架构将数据放在廉价对象存储（MinIO）上，计算节点独立扩缩容，成本降低了70%且数据私有化。

- **关键代码**：
```python
from pymilvus import connections, Collection, FieldSchema, CollectionSchema, DataType

# 定义 Schema，支持向量 + 标量过滤 (演示混合检索能力)
fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="content", dtype=DataType.VARCHAR, max_length=65535),
    FieldSchema(name="department", dtype=DataType.VARCHAR, max_length=64) # 用于过滤
]
schema = CollectionSchema(fields, description="Enterprise KB")
collection = Collection(name="kb", schema=schema)
```

- **对比表格**：

| 特性维度 | Pinecone (SaaS) | Milvus (自建) | Qdrant (自建/SaaS) |
| :--- | :--- | :--- | :--- |
| **运维复杂度** | 低 (全托管) | 高 (需K8s/运维团队) | 中 (Docker一键起，集群稍难) |
| **性能上限** | 中高 | **极高** (C++ + 存算分离) | 高 (Rust实现) |
| **数据控制权** | 低 (数据在云端) | **高** (本地/私有云) | **高** |
| **功能完整性** | 基础功能稳 | **功能最全** (支持多种索引/Scalar) | 较全 (Fusion算法是亮点) |

- **## 常见考点**
1. **CAP权衡**：Milvus和Pinecone在AP（可用性/分区容错）上优化，pgvector在CP（一致性/分区容错）上表现更好（继承PG特性）。
2. **索引类型**：HNSW（高性能）、IVF（平衡）、Flat（精确但慢）的选择场景。
3. **混合检索实现**：如何解决向量与关键词的得分对齐问题（Reciprocal Rank Fusion / Score加权）。
4. **内存与磁盘**：HNSW通常常驻内存，IVF/PQ可部分放磁盘，对成本的影响。

## 记忆要点

- Pinecone：SaaS全托管，最省心但贵，适合不想运维场景
- Milvus：开源高性能，存算分离，适合十亿级大规模数据自建
- Qdrant：Rust编写高性能，内存友好，原生支持混合检索
- pgvector：PostgreSQL插件，适合已有PG生态且需强事务/ACID的场景
