# 语义缓存如何保证安全

语义缓存通过向量相似度复用历史结果，其安全性保障机制如下：

1.  **严格的租户隔离**：
    *   缓存 Key 必须包含 `TenantID`，防止跨租户数据泄露（Tenant A 的敏感回答被 Tenant B 命中）。
2.  **Key 的版本与参数绑定**：
    *   缓存 Key 需包含 `ModelID`、`Prompt Template` 的 Hash 值。若模型升级或 Prompt 修改，必须自动失效旧缓存，防止返回过时或逻辑不符的答案。
3.  **相似度阈值保守设置**：
    *   设定较高的余弦相似度阈值（如 > 0.95），防止近似但不等义的请求命中缓存（如「转账给张三」与「转账给李四」）。对于涉及权限操作或 PII 信息的请求，建议禁用缓存。
4.  **敏感任务策略**：
    *   金融、医疗或写操作请求，建议强制标记 `Cache-Control: no-store`，或要求模型对缓存结果进行二次确认。
5.  **TTL 与主动失效**：
    *   设置合理的 TTL；对于知识库更新频繁的场景，提供「主动失效」接口，在文档更新时清除相关语义缓存。

**实战案例**：
曾出现过一个严重的 Case：用户 A 询问"我的订单号是什么"，答案被缓存；随后用户 B 发出了完全相同的语义问题（因测试脚本复用），由于缓存 Key 仅由 Query Embedding 构成，系统直接返回了用户 A 的订单号。修复方案是在 Metadata 中强制加入 `user_id` 过滤条件，检索时必须同时满足语义相似与 `user_id` 匹配。

**代码示例**：
```python
# 伪代码：带租户隔离的安全语义缓存检索
def get_cache(query_embedding, user_id, model_version):
    results = vector_store.search(
        vector=query_embedding,
        filter={
            "user_id": user_id,      # 强制硬过滤，确保租户/用户隔离
            "model_ver": model_version # 确保模型版本一致性
        },
        score_threshold=0.98,         # 提高阈值以避免语义漂移
        top_k=1
    )
    return results[0] if results else None
```

| 策略 | 传统 KV 缓存 | 语义向量缓存 |
| :--- | :--- | :--- |
| **Key 匹配方式** | 精确匹配 | 模糊匹配 (余弦相似度) |
| **主要风险** | 参数细微变化导致缓存未命中 | **语义近似导致权限/数据越界** |
| **隔离手段** | Key 前缀 | Metadata 过滤 + 高阈值 |
| **适用场景** | ID 查询、配置读取 | 通用问答、摘要生成 |
