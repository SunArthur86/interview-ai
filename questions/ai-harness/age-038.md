# Context Engineering 和 Prompt Engineering 有什么区别

这是 2025 年面试高频题,两者有本质区别:

- **Prompt Engineering:** 关注单次对话中如何写好 prompt.核心是措辞、结构、示例选择.关注点:怎么问.

- **Context Engineering:** 关注整个 Agent 运行过程中,如何管理和组装上下文窗口中的所有信息.关注点:给模型看什么.

- **Context Engineering 包含的维度:**
1. **System Prompt 组装:** 角色定义、工具描述、安全规则
2. **历史消息管理:** 压缩、截断、摘要
3. **工具结果注入:** 格式化、过滤敏感信息、控制长度
4. **外部知识注入:** RAG 检索结果、文件内容
5. **上下文预算分配:** 在有限的窗口中平衡各部分占比

- **为什么 Context Engineering 更重要:**
- 生产环境模型能力固定,真正拉开差距的是上下文质量
- 上下文管理直接影响成本、延迟、质量
- Prompt Engineering 是 Context Engineering 的子集

### 实战深化

**1. 实战案例:**
在一个文档分析 Agent 中，直接把整份 PDF 塞进去会导致模型迷失在无关的页眉页脚中。通过 Context Engineering 优化，只注入 RAG 检索出的 Top 3 相关片段 + 用户画像摘要，将任务准确率从 65% 提升至 90%，同时 Token 消耗减少 70%。

**2. 代码示例 (Python - 动态上下文构建):**
```python
def build_context_window(session: Session, query: str, max_tokens: int):
    context_budget = max_tokens - estimate_tokens(SYSTEM_PROMPT)
    
    # 1. 优先分配 RAG 检索结果 (高价值信息)
    rag_text = "\n".join([r.content for r in search(query, top_k=3)])
    context_budget -= estimate_tokens(rag_text)
    
    # 2. 剩余空间分配给最近的历史对话
    history_msgs = select_recent_messages(session.history, budget=context_budget)
    
    return SYSTEM_PROMPT + rag_text + format_history(history_msgs)
```

**3. 对比表格:**

| 维度 | Prompt Engineering | Context Engineering |
| :--- | :--- | :--- |
| **核心关注** | 指令的修辞和逻辑 | 信息的选择与组装策略 |
| **作用范围** | 单次 Prompt 字符串 | 整个对话生命周期与窗口 |
| **主要手段** | CoT, Few-Shot, Role Playing | 滑动窗口, RAG, 压缩, 优先级队列 |
| **优化目标** | 提升模型理解力 | 提升信噪比 & 控制成本 |
| **失效场景** | 模型能力不足 | 上下文过长或检索质量差 |

## 边界情况
1. **系统提示词 膨胀**：随着工具定义和规则变多，System Prompt 可能占据 50% 以上的窗口，挤压用户输入，此时需考虑“动态指令加载”或“指令压缩”。
2. **上下文截断导致的逻辑断裂**：简单地截断历史可能会导致丢失前文设定的变量或约束条件（如用户最开始说“不要提及价格”，截断后 Agent 开始报价），需设计“关键元数据持久化”机制。
3. **多模态上下文挤占**：图片或音频 Token 化后极占空间，需在预算分配中专门为多模态预留 buffer，并在超限时降级为纯文本描述。

## 面试追问
1. 上下文压缩通常有几种策略（如 Extractive vs Abstractive）？在你的实践中，使用 LLM 进行 Abstractive 摘要时，如何保证不丢失关键实体（如人名、ID）？
2. 当检索出的 RAG 内容与用户的直接指令冲突时，Context Engineering 层面该如何处理排序和展示？
3. 如何动态衡量一段历史对话的“重要性”？除了时间衰减，有没有用过基于注意力机制或对齐评分的方法来决定保留哪些历史？

## 易错点
1. **将 Context Engineering 仅等同于 RAG**：很多人认为只是检索文档，实际上历史对话摘要、工具定义的优化、错误信息的过滤都属于此范畴。
2. **忽视信噪比优化**：盲目增加上下文量（如堆砌检索结果）往往会降低模型效果（Lost in the Middle 现象），精准控制 Token 分配比单纯增加信息更关键。

## 记忆要点

- Prompt Engineering 关注单次指令措辞，Context Engineering 关注全生命周期信息组装。
- Context 包含：System Prompt、历史管理、工具结果、外部知识、预算分配。
- 核心差异：Prompt 是“怎么问”，Context 是“给模型看什么”，后者决定信噪比与成本。
