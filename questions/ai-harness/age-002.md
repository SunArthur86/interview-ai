# 熔断器和重试分别解决什么问题?一起用时要注意什么

### 熔断器与重试机制

**核心作用区别**：
- **熔断器**：主要用于**防止故障扩散**（雪崩效应）。当检测到下游服务故障率达到阈值时，熔断器打开，快速失败，保护系统资源不被耗尽。
- **重试**：主要用于**提高偶发故障的容错性**。对于网络抖动等瞬态故障，通过重试可以提升最终请求成功率。

**联合使用时的注意事项**：
1. **禁止 Open 状态重试**：当熔断器处于 Open 状态时，客户端应直接执行降级逻辑（如返回默认值、切换备用服务），**绝对禁止**发起重试，否则会加重下游负担并延缓恢复。
2. **区分错误类型**：
   - 对于**429 (Too Many Requests)**，重试必须配合**退避策略**（如 Exponential Backoff），且需配合客户端限流，避免重试风暴。
   - 对于**4xx 系列客户端错误**（如 400, 404），通常不应重试。
   - 对于**5xx 系列服务端错误**或网络超时，适合进行有限次数的重试。
3. **重试预算**：限制单位时间内的重试比例，防止因重试量过大导致系统压垮自身。

**实战案例**：
某电商大促期间，推荐服务因数据库慢查询导致响应超时，客户端开启了无限重试且未退避，导致数据库连接池瞬间被耗尽，连带拖垮了支付服务。后引入 Sentinel 熔断并配置 `maxRetries=3` 配合 `ExponentialBackoff` 解决。

**代码示例**：
```java
// Resilience4j 示例：重试与熔断组合配置
RetryConfig config = RetryConfig.custom()
  .maxAttempts(3)
  .waitDuration(Duration.ofMillis(500))
  .retryOnException(e -> e instanceof WebServiceException) // 5xx重试
  .build();
CircuitBreakerConfig cbConfig = CircuitBreakerConfig.custom()
  .failureRateThreshold(50) // 失败率50%触发
  .waitDurationInOpenState(Duration.ofSeconds(30)) // 熔断30秒
  .build();
```

**对比表格**：

| 特性 | 熔断器 | 重试机制 |
| :--- | :--- | :--- |
| **核心目标** | 保护系统，防止雪崩 | 提高成功率，容错瞬态故障 |
| **触发条件** | 达到失败率或慢调用比例 | 单次请求失败或超时 |
| **处理动作** | 直接拒绝/降级 | 重新发起调用 |
| **恢复方式** | 半开状态探测成功 | 下一次请求成功自然停止 |
| **适用错误** | 系统级故障、下游不可用 | 网络抖动、短暂资源锁 |

### 熔断器状态流转图

```text
      (失败率/慢调用比例超过阈值)
+---------+                         +---------+
|  Closed | ----------------------> |  Open   |
| (正常)  |                         | (熔断)  |
+---------+                         +---------+
    ^   ^                                 |  (经过冷却时间窗 Sleep Window)
    |   | (半开探测成功)                  | (进入半开状态，放行少量请求)
    |   |                                 v
    |   |                         +---------+
    |   +----------------------- | Half-Open|
    |       (半开探测失败)       | (探测)   |
    |                             +---------+
    +---------------------------------------+
    (请求重置计数器，恢复闭合)
```

### 常见考点
1. **熔断器的三种状态转换条件是什么？**（特别是半开状态如何探测服务恢复）
2. **重试风暴是如何产生的？如何通过指数退避和 Jitter（抖动）避免？**
3. **在 RPC 框架（如 Dubbo、Spring Cloud CircuitBreaker）中，熔断和超时配置的优先级是怎样的？**
