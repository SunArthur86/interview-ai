# Agent 的 Memory 系统如何设计?短期记忆和长期记忆怎么区分

**Agent Memory 设计** 是生产级 Agent 的核心难题之一.

**短期记忆:**
*   **存储位置:** LLM 上下文窗口或 KV Cache.
*   **包含内容:** 当前对话历史、工具调用结果、中间推理步骤.
*   **生命周期:** 单次任务或单轮对话.
*   **管理策略:** 滑动窗口截断、基于重要性的摘要、Token 预算动态分配.

**长期记忆:**
*   **存储位置:** 外部存储系统(向量数据库、关系数据库、图数据库).
*   **包含内容:** 用户偏好、历史决策、学到的知识、项目结构索引.
*   **生命周期:** 跨会话持久化.
*   **管理策略:** 语义检索、时间衰减、定期更新与归档.

**数据流向示意图:**

**设计要点:**
1.  **记忆写入:** 什么信息值得记住?通过反思机制提取关键洞察,避免存入噪声.
2.  **记忆检索:** 如何高效找到相关记忆?混合检索(关键词+向量) + 时间衰减 + 重排序.
3.  **记忆更新:** 如何处理冲突信息?基于时间戳的版本控制或置信度比较.
4.  **遗忘机制:** 如何淘汰过时记忆?基于访问频率(LRU)和信息熵值.

**常见实现:** MemGPT、LangChain Memory、Letta(前 MemGPT).

### 实战深化

**1. 实战案例:**
曾遇到一个私有知识库问答 Agent，系统错误地将用户随口说的“以后不要这样了”当成了业务规则写入长期记忆，导致后续正确查询被屏蔽。解决方案是增加“记忆审核层”，只有 LLM 判定为高置信度的事实或指令才入库，且支持用户显式“撤回”记忆。

**2. 代码示例 (Python - 反思式写入):**
```python
def write_memory(user_input: str, agent_response: str):
    # 1. 提取: 仅提取重要的结构性信息，忽略闲聊
    reflection_prompt = """
    分析以下对话，提取值得存入长期记忆的关键信息（用户偏好、事实数据）。
    如果仅为闲聊或临时指令，返回 NULL。
    对话：{user_input} -> {agent_response}
    """
    memory_content = llm.generate(reflection_prompt)
    
    if memory_content != "NULL":
        vector_db.insert(memory_content)
```

## 边界情况
1. **记忆污染与遗忘**：如果用户纠正了之前的错误认知（如“我以前喜欢咖啡，现在改喝茶了”），长期记忆必须支持版本更新而非简单追加，否则会导致检索冲突。
2. **记忆检索的递归陷阱**：Agent 可能读取了自己的记忆后产生新的幻觉，又将幻觉写入记忆，需防止“自我强化”的错误循环。
3. **隐私与权限隔离**：多租户环境下，记忆检索必须严格限制在租户范围内，且需对写入长期记忆的内容进行 PII（个人敏感信息）自动脱敏。

## 面试追问
1. 在高并发写入时，如何保证向量数据库的更新是实时的？有没有使用过“异步写入队列”或“批处理”来优化性能？
2. 当检索出的多条记忆相互矛盾时（例如用户在不同时间说了相反的话），你的重排序算法如何判定哪一条权重更高？
3. 短期记忆向长期记忆转化的具体触发条件是什么？是基于对话轮数、时间间隔，还是基于某种“重要性”评分模型？

## 易错点
1. **混淆 KV Cache 与 Memory**：KV Cache 是推理过程中的系统级缓存，无法跨会话或跨请求持久化，不能作为长期记忆的存储介质。
2. **过度依赖语义检索**：对于精确匹配的需求（如订单号、API Key），向量检索效果极差，必须结合关键词搜索或结构化查询（SQL/ES）。
