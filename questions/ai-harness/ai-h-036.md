# 如何建立完整的LLM应用评估体系?LLM-as-Judge如何设计

- **LLM评估三层体系:**

- **1. 单元测试(函数级)**
- Prompt格式验证（输出是否为JSON）
- JSON Schema验证（结构是否符合定义）
- 工具参数合法性（Function Calling参数类型正确）
- *工具*：Pydantic, pytest

- **2. 集成测试(端到端)**
- 评估数据集(100-1000样本)
- 人工标注金标准
- 自动化批量测试
- *指标*：
  - **Exact Match**: 完全匹配（主要适用于代码、数学）
  - **Semantic Similarity**: 句子向量相似度（如BERTScore）

- **3. 线上监控(实时)**
- 用户反馈(赞/踩)
  - *延迟反馈*: 用户点了“踩”可能已经很久了。
- 质量打分(LLM-as-Judge)
  - 每条请求异步送入Judge模型打分。
- 异常检测(幻觉率/延迟/成本)

- **LLM-as-Judge设计:**
使用更强模型（如GPT-4）评估弱模型（如Llama-7B）。
```python
judge_prompt = f"""
你是评估专家.请评估以下回答的质量.

问题：{question}
回答：{answer}
参考答案：{reference}

评分维度(1-5分):
1. 准确性:答案是否正确，无事实性错误
2. 完整性:是否覆盖所有要点
3. 相关性:是否切题，无废话
4. 流畅性:是否表达清晰，语言自然

输出JSON格式评分.
"""
```

- **注意事项:**
- Judge模型要比被评估模型更强（能力阶梯原则）
- 多次评估取平均(减少随机性/温度影响)
- 人工抽检验证Judge质量（校准Judge模型）
- 注意Judge的偏见(偏好长回答、特定风格)

- **实战案例**：在电商客服助手优化中，使用LLM-as-Judge评估发现模型在处理“退换货”问题时分数虚高（因为话术通顺），但人工评估发现实际政策引用错误。后来引入了“基于RAG上下文的事实核验”评估维度，强制Judge检查回答是否依据提供的知识库片段，显著降低了幻觉率。

- **代码示例**：
```python
# 基于RAG的事实核验 Prompt
def build_rag_fact_check_prompt(question, answer, retrieved_contexts):
    return f"""
    [任务]
    请检查[回答]中的事实性陈述是否严格基于[参考上下文]。如果回答包含上下文中没有的信息，视为幻觉。
    
    [参考上下文]
    {' '.join(retrieved_contexts)}
    
    [用户问题]
    {question}
    
    [模型回答]
    {answer}
    
    [输出格式]
    {{
        "has_hallucination": true/false,
        "hallucinated_parts": ["具体错误内容..."],
        "reason": "原因分析"
    }}
    """
```

- **评估体系对比表**：

| 维度 | 传统指标 (BERTScore/Rouge) | LLM-as-Judge | 人类评估 |
| :--- | :--- | :--- | :--- |
| **成本** | 极低 (CPU推理) | 中/高 (API调用) | 极高 (人力) |
| **理解能力** | 弱 (仅字面重叠/向量相似) | 强 (理解语义、逻辑、指令遵循) | 最强 (具备常识与业务背景) |
| **适用场景** | 筛选、回归测试 | 复杂任务质量评估、模型迭代 | 金标准构建、Final校验 |
| **稳定性** | 高 | 中 (受Prompt/温度影响) | 高 (排除主观差异) |

## 常见考点
1. **参考答案缺失**：在没有标准答案的开场问题中，如何设计评估指标？（使用Reference-free的评估方式，如Self-consistency或单纯评分Coherence）。
2. **评估成本**：LLM-as-Judge本身也是昂贵的LLM调用，如何在大规模生产中控制评估成本？（使用蒸馏版的小型Judge模型，或抽样评估）。
3. **幻觉检测**：如何让Judge模型有效检测“一本正经胡说八道”？（RAG场景下需检查Answer是否在Context中，Fact-Checking prompts）。

## 记忆要点

- 评估体系：单元测(格式/JSON Schema) → 集成测(端到端数据集) → 线上监控(用户反馈/质量)。
- LLM-as-Judge：用强模型(如GPT-4)评估弱模型，维度含准确性/完整性/相关性。
- 设计要点：Judge能力需强于被测模型，多次评分取平均，人工抽检校准防偏见。
- RAG核验：强制Judge检查回答是否基于Context，检测幻觉，防模型一本正经胡说。
- 成本控制：大规模评估可用蒸馏小模型或抽样评估，平衡效果与开销。
