# 评估与测试

### 评估与测试

**核心目标**：建立度量体系，确保 Agent 的准确性、鲁棒性和效率符合预期。

**原理详解**：

1. **评估维度**：
   - **准确性**：任务完成度、答案正确性（人工标注或 LLM-as-Judge）。
   - **鲁棒性**：面对噪声、对抗输入的稳定性。
   - **效率**：Token 消耗、延迟、步骤数。

2. **自动化测试框架**：
   - **单元测试**：验证路由逻辑、参数解析。
   - **集成测试**：Mock LLM 输出，验证编排逻辑正确性。
   - **端到端测试**：在沙箱环境中运行真实任务，控制成本。

3. **评估方法**：
   - **基准测试**：使用固定数据集和评分器，对比不同版本的 Prompt/模型。
   - **LLM-as-Judge**：用强模型给弱模型打分（需注意偏袒风格问题，建议多裁判投票）。

4. **回归与 A/B 测试**：
   - **回归测试**：新版本发布前必须在评估集上不低于基线。
   - **A/B 测试**：线上分流，观察真实业务指标（如用户点击率、任务成功率）。

5. **应对坑点**：
   - 注意数据泄漏（测试集混入训练）。
   - 线下评估好不代表线上一定好（分布差异）。

#### 边界情况与极端场景
- **金标准的不确定性**：某些开放性问题并没有唯一的“标准答案”，硬性比对会导致评分失真。需引入基于语义相似度的评估指标（如 BERTScore）或人工抽检。
- **对抗样本攻击**：测试集中需包含 Prompt 注入、恶意指令混淆等对抗性样本，验证 Agent 是否会泄露 System Prompt 或执行危险操作。
- **工具调用失败的评估**：不仅要评估工具“调用成功”的情况，还要专门构造工具“报错”、“超时”、“返回空值”的测试用例，评估 Agent 的错误恢复和自愈能力。

**实战案例**：在某电商客服 Agent 上线中，我们发现 F1 分数虽然提高了 5%，但“退款”意图的误判导致客服工单暴涨 20%。引入“误判导致的业务成本”作为评估指标后，我们调整了模型阈值，宁可拒绝回答也不降低精度。

**代码示例**：
```python
# 使用 Ragas 框架进行自动化评测的简单示例
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy

def run_evaluation(dataset):
    # dataset 包含 question, contexts, answer, ground_truths
    result = evaluate(
        dataset,
        metrics=[faithfulness, answer_relevancy]
    )
    # 必须通过断言，否则 CI 流水线失败
    assert result['faithfulness'] > 0.85, "Faithfulness dropped below baseline"
    return result
```

## 面试追问
1. **追问（Bad Case 分析）**：如果你的 Agent 在测试集上表现很好（Accuracy 95%），但上线后用户投诉说“听不懂人话”，你会如何定位问题？可能是发生了什么分布偏移？
2. **追问（自动化评估）**：在 LLM-as-Judge 的评估中，如果裁判模型本身也有幻觉或偏见，你有什么工程手段来减轻这种影响？
3. **追问（测试成本）**：端到端测试调用真实 LLM 成本很高，如何在保证覆盖率的前提下，利用 Mock 技术有效降低测试成本？

## 易错点
1. **误区**：认为测试集的覆盖率越高越好，盲目追求大而全的测试集。
   **纠正**：测试集的质量 > 数量。如果测试集中包含大量简单重复的样本，模型虽然分数高，但在长尾困难样本上可能表现很差。应重点关注“困难样本”的覆盖。
2. **误区**：只关注自动化的评估指标，忽视了人工反馈循环。
   **纠正**：自动化指标通常只能衡量“相关性”或“格式正确性”，很难衡量“有用性”和“安全性”。必须建立人工标注/审核通道（RLHF），将 Bad Case 持续反哺回测试集。

## 记忆要点

- 三大评估维度：准确性、鲁棒性、效率，需建立定性定量结合的自动化测试体系。
- 因为自动化指标难查安全与有用性，所以必须结合LLM-as-Judge与人工Bad Case反哺。
- 测试策略对比：集成用Mock降成本，而端到端在沙箱跑真实任务。
- 边界防范：不仅测正常流程，更要专门构造工具报错和对抗样本验证自愈能力。
