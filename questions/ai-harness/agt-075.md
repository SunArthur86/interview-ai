# 结构化输出

### 6. 结构化输出

#### 6.1 JSON 输出控制
**概念解释**
要求模型输出合法 JSON，便于 `json.loads` 解析。关键是：键名固定、类型明确、禁止尾随逗号、禁止注释（标准 JSON 无注释）。

**Prompt 示例**
```text
仅输出一个 JSON 对象，不要 Markdown 代码块，不要解释文字。
Schema 逻辑：
- summary: string
- items: array of {name: string, price: number}
若某字段未知，用 null。
```

#### 6.2 XML 标签输出
**概念解释**
用 `<field>...</field>` 包裹字段，适合人类可读或与某些旧解析器兼容；可嵌套，但需约定转义规则。

**示例**
```text
<result>
<intent>...</intent>
<slots>
  <location>...</location>
  <time>...</time>
</slots>
</result>
```

#### 6.3 Markdown 格式输出
**概念解释**
适合文档、报告、README；若给程序解析，需约定标题层级与列表符号，避免模型随意换风格。

**Prompt 示例**
```text
请根据下列数据写一份 Markdown 报告，严格使用以下结构，不要增加一级标题：
## 摘要
（2～3 句）
## 关键指标
使用表格，列：指标名 | 数值 | 环比
## 风险与建议
使用有序列表，不超过 5 条
数据：
{{metrics_json}}
```

#### 6.4 使用 Pydantic 进行输出解析
**概念**
在 Python 中用 Pydantic 定义数据结构，配合 JSON Schema 或先让模型输出 JSON 再 `model_validate`。

**代码示例**
```python
from pydantic import BaseModel, Field
from typing import List, Optional
import json

class Item(BaseModel):
    name: str = Field(..., description="商品名称")
    price: float = Field(..., ge=0)

class OrderSummary(BaseModel):
    summary: str
    items: List[Item]
    note: Optional[str] = None

def parse_llm_json(text: str) -> OrderSummary:
    # 生产环境应先剥离 ```json 代码块、修复常见 JSON 问题
    data = json.loads(text)
    return OrderSummary.model_validate(data)
```

#### 6.5 Function Calling 作为结构化输出
**概念**
让模型以 `tool_calls` 形式输出结构化动作，比「纯文本 JSON」更不易掺杂废话（取决于实现与对齐）。

**原理**
宿主根据 Schema 校验参数，再执行；适合 Agent。

**面试 Q&A**
Q：如何保证 JSON 输出绝对正确？
A：多层保障：
1. Prompt 明确要求「仅 JSON、禁止 Markdown」。
2. 用 JSON Schema 在服务端校验。
3. 失败则 repair：用第二次调用让模型根据错误信息修正。
4. 关键路径用 Function Calling + 强校验。

#### 边界情况补充
1. **字段缺失与类型错误**：模型可能漏掉必填字段或将整数返回为浮点数（如 `10` vs `10.0`）。解析代码需具备容错能力（如 Pydantic 的 `Field(default=...)` 或类型强制转换）。
2. **Markdown 代码块干扰**：即使 Prompt 明令禁止，模型仍常输出 ```json ... ``` 包裹。解析器必须包含正则清理逻辑，否则 `json.loads` 必然报错。
3. **特殊字符转义**：在文本内容包含双引号、换行符或反斜杠时，模型可能忘记转义，导致 JSON 结构破坏。特别是在生成 SQL 或代码字符串时极易发生。
4. **长列表截断**：当输出包含数组（如 List[Items]）且内容过长时，模型可能因达到 Stop Token 或上下文上限而截断输出，导致 JSON 不闭合（缺少 `]` 或 `}`）。

## 面试追问
1. 当模型输出的 JSON 缺少右括号导致解析失败时，除了重新生成，有什么自动修复策略？（考察：基于括号匹配的自动补全、Partial Parsing 技术）
2. Function Calling 和 强制 JSON Mode 在底层实现上有什么区别？为什么 Function Calling 往往更稳定？（考察：模型训练对齐差异、特殊 Token 的使用）
3. 对于超大 Schema（包含几百个字段的复杂结构），模型很难遵循。如何拆分或设计 Schema 以提高准确率？（考察：Schema 拆分、两阶段生成、Refinement 技术）

## 易错点
1. **过度依赖 Prompt 而忽视代码层校验**：仅靠 Prompt “请输出合法 JSON” 无法保证 100% 成功率。必须在代码层结合 Pydantic 或 JSON Schema 进行校验和异常捕获，并提供 Fallback 逻辑。
2. **混淆 JSON Mode 与 Grammar Constrained Sampling**：JSON Mode 通常是训练对齐或后处理；而 Grammar Constrained Sampling（如 llama.cpp 的 grammar 功能）是在推理解码层面限制字符只能由 JSON 路径生成，后者能从数学上保证合法性，但需推理框架支持。

## 记忆要点

- JSON痛点：Markdown代码块、尾随逗号、长列表截断导致不闭合
- 强校验保障：Prompt要求+Pydantic验证+代码兜底自动修复
- Function Calling：比纯文本输出结构更严，不掺杂废话
- 防截断策略：解析器必须支持基于括号匹配的自动补全机制
- 盲区：仅靠Prompt要求100%合法是误区，代码层必须有容错
