# vLLM 的 PagedAttention 和连续批处理是如何协同工作的?它解决了什么核心问题

- **vLLM 的两大核心创新**

- **实战案例**：在迁移至 vLLM 之前，我们的 70B 模型推理服务在 A100 80GB 上最大并发只能撑 32，且经常 OOM；切换到 vLLM 开启 PagedAttention 后，并发提升至 128 且显存占用稳定在 60GB 左右，吞吐量翻了 4 倍。

- **问题背景**
传统推理(HuggingFace Transformers)的问题:
1. 每个 KV Cache 预分配连续内存 → 显存碎片严重(浪费 60-80%)
2. 请求必须等同一个 batch 全部完成才能接受新请求 → 吞吐低

- **对比表格：vLLM vs HF Transformers**
| 特性 | HF Transformers (Static Batching) | vLLM (PagedAttention + CB) |
|------|----------------------------------|---------------------------|
| KV Cache 管理 | 预分配连续内存，碎片严重 | 分页管理，碎片近乎为 0 |
| Batch 调度 | 静态，需等待整批完成 | 连续批处理，实时进出 |
| 显存利用率 | ~20-40% | ~90%+ |
| 吞吐量 | 基准 | 14x-24x 提升 |

- **创新 1: PagedAttention(分页注意力)**

**类比操作系统的虚拟内存：** 
- KV Cache 不存储在连续物理内存,而是分成固定大小的 Block(如 16 个 token/block)
- Block Table 映射逻辑块到物理块
- 显存碎片 → 接近 0
- 显存利用率从 ~40% 提升到 ~96%

- **创新 2: Continuous Batching(连续批处理)**

- 传统 Static Batching:等整个 batch 完成
- Continuous Batching:
- 某个请求完成 → 立即从 batch 移除
- 新请求到来 → 立即加入当前 batch
- 每一步动态重组 batch
- 吞吐量提升 2-4x

- **协同工作**
```
1. 新请求到来 → PagedAttention 分配 KV Block
2. 加入当前 batch → Continuous Batching 统一调度
3. 某请求完成 → 回收 KV Block → 等待新请求复用
4. 全局显存管理 → 零碎片 + 零等待
```

- **效果**:vLLM 吞吐量比 HuggingFace 高 14-24x(论文数据)

- **代码示例**：vLLM 基础调用示例
```python
from vllm import LLM, SamplingParams

# 初始化 LLM 引擎 (自动启用 PagedAttention 和 Continuous Batching)
llm = LLM(model="meta-llama/Llama-2-7b-hf", tensor_parallel_size=1)

# 定义采样参数
sampling_params = SamplingParams(temperature=0.8, top_p=0.95, max_tokens=100)

# 批量推理 (自动处理不同长度输入)
prompts = ["Hello, my name is", "The future of AI is"]
outputs = llm.generate(prompts, sampling_params)
```

- **边界情况补充**：
1. **Block Fragmentation（内部碎片）**：即使使用了分页机制，如果请求的长度不是 Block Size 的整数倍（如 Block=16，请求需要 17 tokens），最后一个 Block 仍会有大部分空间浪费，vLLM 通过 Last Block Cache Sharing（如 CgGraph）尝试优化。
2. **Prefix Caching 失效**：如果使用的是 LoRA 或其他 Adapter，KV Cache 的内容会动态变化，可能导致 Prefix Cache 无法共享，需禁用相关优化。
3. **内存预分配上限**：vLLM 启动时通常指定 `gpu_memory_utilization`，如果运行中实际并发突增超过物理显存+Swap 限制，会导致崩溃，需合理的 Admission Control（准入控制）。
4. **长序列跨 Block 计算**：在 Attention 计算时，跨多个不连续物理 Block 的读取会导致显存访问模式非连续，可能轻微降低计算 Kernel 的效率。

- **## 常见考点**
1. **Block Table**: vLLM 如何管理不同 Sequence 长度的 KV Cache？（通过 Block Table 映射逻辑 Block 到物理 Block，支持非连续存储）
2. **解码迭代**: 在 Continuous Batching 中，一次 Forward Pass 后 Batch 的构成会发生什么变化？（完成的请求移除，新请求加入，物理 KV Cache 内存被复用）
3. **Cg graph**: vLLM 使用 CUDA Graphs 优化了什么？（优化了 Kernel 启动开销，尤其在 Decode 阶段的小矩阵计算中效果显著）

- ## 易错点
1. **认为 PagedAttention 没有代价**：维护 Block Table 和处理物理内存的非连续访问会引入少量的计算和逻辑开销，对于极短序列（如 <32 tokens），性能提升可能不如长序列明显。
2. **混淆 Continuous Batching 与 Orca**：Continuous Batching 是动态调整 Batch 的请求集合，而 Orca/Iterative Level Scheduling 是在生成过程中对已生成的 Token 进行迭代级的调度，二者常结合但概念不同。

- ## 面试追问

3. 如果开启了 Speculative Decoding（推测采样），vLLM 的调度策略会有什么变化？
