# 生产环境如何设计 Agent 的模型路由和 fallback 策略

**模型路由** 是根据任务特征选择不同模型,平衡效果、成本和延迟.

**路由策略:**

**1. 规则路由:** 根据任务类型硬编码.简单任务(如格式转换)用小模型,复杂任务(如架构设计)用大模型.响应最快,但灵活性差.
**2. 模型路由(LLM Router):** 用一个小模型先判断任务复杂度,再路由到对应模型.灵活但增加了一次推理延迟.
**3. 级联路由:** 先用便宜模型尝试,如果置信度低再升级到大模型.成本最优但延迟波动大.

**Fallback 策略:**
*   **模型超时** -> 切换到备用供应商
*   **API 限流** -> 排队重试 + 指数退避
*   **输出质量差** -> 自动重试 + Prompt 调整(如提高 temperature 或简化指令)
*   **完全失败** -> 返回缓存结果或人工接管

### 实战深化

**1. 实战案例:**
某 SaaS 客服系统曾因所有请求都走 GPT-4 导致月底成本爆表。引入级联路由后（先走 GPT-3.5，仅当置信度<0.8 或包含“愤怒”情绪时升级 GPT-4），成本降低了 60%，但要注意大模型偶尔会有“幻觉”，小模型验证反而更稳。

**2. 代码示例 (Python):**
```python
async def route_and_execute(query: str):
    # Step 1: 路由判断 (或直接用规则分类)
    complexity = await classify_complexity(query) 
    model = "gpt-4-turbo" if complexity > 0.8 else "gpt-3.5-turbo"

    # Step 2: 级联尝试
    try:
        response = await call_llm_with_timeout(model, query, timeout=5)
        if verify_confidence(response) < 0.7:
            raise LowConfidenceError("Fallback to stronger model")
        return response
    except (TimeoutError, LowConfidenceError) as e:
        # Fallback: 切换备用模型
        return await call_llm_with_timeout("claude-3-opus", query)
```

## 边界情况
1. **全区域宕机**：当主模型和备用模型供应商（如 OpenAI 和 Anthropic）同时不可用时，系统应降级到基于规则的硬逻辑或返回明确的错误提示，而非无限重试。
2. **语义漂移**：不同模型的 Prompt 偏好不同（如 Claude 喜欢长文本，GPT 喜欢结构化指令），路由切换时需动态调整 Prompt 模板，否则可能导致切换后质量反而下降。
3. **级联死循环**：如果小模型因能力不足产生幻觉，触发了“置信度低”而切换到大模型，但大模型因上下文不足也被判定为“置信度低”，可能导致在模型间反复横跳。

## 面试追问
1. 在高并发场景下，级联路由会增加 P99 延迟，你如何通过预热或并发请求来优化这一点？
2. 如何评估路由模型的准确性？如果路由模型把复杂任务错误地分发给小模型，导致小模型产生不可逆的错误（如误删数据），怎么处理？
3. 如果不同模型的计费方式差异很大（如有的按 Token，有的按请求次数），你的路由策略会做哪些调整以优化总成本？

## 易错点
1. **忽略备用模型的行为一致性**：在 Fallback 到另一个模型（如从 GPT 切到 Claude）时，若不调整 Prompt 格式，极大概率会触发格式解析错误。
2. **盲目重试导致雪崩**：当发生限流时，简单的指数退避如果在所有客户端同时发生，可能导致流量集中爆发引发更严重的限流，需增加“抖动”因子。

## 记忆要点

- 路由策略：规则路由最快，LLM 路由最灵活，级联路由（先小后大）成本最优。
- Fallback 机制：超时切换供应商，限流指数退避重试，质量差自动重试或人工接管。
- 关键指标：路由准确率、平均成本、P99 延迟、Fallback 触发率、首字生成时间(TTFT)。
