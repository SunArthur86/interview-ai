# LLM 应用的监控和传统 Web 应用的监控有什么不同

- **LLM 监控的独特维度**

传统 Web:CPU/内存/QPS/延迟
LLM 应用还需要:

1. **Token 消耗**:每次请求消耗的 token 数(直接关联成本)
2. **质量指标**:幻觉率、回答相关性、用户满意度
3. **LLM 延迟**:首 token 延迟 (TTFT) + 每秒生成速度
4. **工具调用成功率**:Agent 场景中工具调用的成功/失败率
5. **Prompt 变更追踪**:不同 prompt 版本的效果对比

- **实战案例:**
某客服机器人上线后，APM 显示错误率极低，但用户投诉激增。通过 LangSmith 查看 LLM 追踪日志发现，虽然模型正常返回，但**拒绝回答率**高达 40%（因为安全护栏过于敏感，拦截了正常问题）。通过调整 Prompt 中的安全指令，将拒绝率降至 5%，满意度回升。这说明传统监控"可用"不代表 AI "好用"。

- **代码示例 (Python - 手动埋点 Tracing):**
```python
import time

def monitored_llm_call(prompt):
    start_time = time.time()
    
    # 1. 记录输入 Token (估算)
    input_tokens = len(prompt.split()) 
    
    # 2. 调用 LLM
    response = llm.generate(prompt)
    
    # 3. 计算 TTFT (首字延迟) 和 总延迟
    ttft = response.first_token_time - start_time
    latency = time.time() - start_time
    
    # 4. 发送到监控系统 (如 Prometheus/LangSmith)
    metrics.log({
        "llm_latency": latency,
        "llm_ttft": ttft,
        "input_tokens": input_tokens
    })
    return response
```

- **常用工具**:LangSmith, Phoenix (Arize), Langfuse, W&B

- **一句话理解**:传统监控关注'系统是否正常',LLM 监控还要关注'AI 回答是否靠谱'.

- **监控维度对比表:**
| 维度 | 传统 Web 应用监控 | LLM 应用监控 |
|------|-----------------|-------------|
| **性能指标** | Latency, QPS, Error Rate | TTFT (首字延迟), TPOT (生成速度), Throughput |
| **资源指标** | CPU, Memory, Disk | GPU Utilization, VRAM, Token Throughput |
| **业务质量** | 转化率, 订单量 | 幻觉率, 忠实度, 相关性, 用户点赞/踩 |
| **可追踪性** | Request ID, Stack Trace | Prompt 版本, Context 摘要, 中间推理步骤 |

- **LLM 可观测性数据流**
```
用户请求
    │
    ▼
┌───────────────┐      ┌───────────────┐
│  Tracing Span │─────▶│ Trace Collection│
│ (Prompt/Ans) │      │   (LangSmith) │
└───────┬───────┘      └──────┬───────┘
        │                    │
        ▼                    ▼
┌───────────────┐      ┌───────────────┐
│   Metrics     │      │   Evaluation  │
│ (Latency/Cost)│─────▶│ (Quality/Bleu)│
└───────────────┘      └───────────────┘
```

- **边界情况补充**：
1. **长文本处理**：当输入 Prompt 超长或输出生成长度达到模型上限时，监控需额外关注“截断率”和“长文本延迟非线性增长”。
2. **流式中断**：在流式输出场景下，用户提前断开连接，需监控“废弃 Token 消耗”，防止产生大量无效成本。
3. **并发竞态**：在高并发下，模型 API 的 Rate Limit (429 错误) 并非系统错误，需在监控中单独分类统计。
4. **安全攻击**：监控需识别 Prompt Injection 或越狱尝试的频率，区分普通流量与恶意探测流量。

- **常见考点:**
1. **TTFT vs TPOT**: 解释首 Token 延迟和每 Token 生成时间的区别及其对用户体验的影响。
2. **Tracing 实现**: 在 LangChain 中如何将应用日志自动上传到 LangSmi

- ## 易错点
1. **混淆“无错误”与“高质量”**：传统监控只看 HTTP 200 状态码，但 LLM 可能 200 返回却内容胡言乱语，监控必须包含语义层面的评估。
2. **忽略 Context Window 边界**：未监控超出上下文窗口导致的历史信息丢失，这会直接导致回答质量下降但系统日志无报错。
