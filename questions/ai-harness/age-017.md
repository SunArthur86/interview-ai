# 蓝绿与金丝雀如何取舍

蓝绿部署与金丝雀发布是两种主要的发布策略，取舍取决于业务场景与资源约束：

**1. 蓝绿部署**
*   **原理**：维护两套完全相同的生产环境（蓝和绿）。在同一时刻，只有一套环境承载流量。切换时通过路由将流量瞬间从蓝切到绿。
*   **优点**：
    *   **回滚极快**：只需切回路由，秒级回滚，无代码回滚。
    *   **环境隔离**：新版本在无流量情况下预热，完全就绪后切换。
    *   **无版本共存**：避免了数据 Schema 冲突（在切换瞬间完成）。
*   **缺点**：
    *   **资源冗余**：需要双倍的服务器资源（2x Cost）。
    *   **停机感知（非平滑）**：虽然有连接排空，但长连接（WebSocket）可能受影响。
*   **适用场景**：对稳定性要求极高、可接受高成本、需要快速回滚的传统后端服务。

**2. 金丝雀发布**
*   **原理**：在生产环境中保留旧版本，部署新版本，通过灰度规则（如 5% 流量）逐步将用户引流到新版本。
*   **优点**：
    *   **资源利用率高**：只需逐步扩容新版本，最终替换旧版本，资源占用平滑。
    *   **精准验证**：可以针对特定用户（如内部员工）先发布，收集反馈。
    *   **错误受控**：如果新版本报错，只有小部分用户受影响，影响面可控。
*   **缺点**：
    *   **配置复杂**：需要精细的流量控制网关（如 Istio, Nginx）。
    *   **回滚稍慢**：需要逐步调整流量比例至 0，不如蓝绿切换干脆。
    *   **数据兼容挑战**：新旧版本同时运行，必须保证数据库 Schema 的向前/向后兼容性。
*   **适用场景**：资源敏感、高频迭代、需要验证业务逻辑或用户体验的服务（如推荐算法、前端页面）。

### 实战对比

| 特性 | 蓝绿部署 | 金丝雀发布 |
| :--- | :--- | :--- |
| **资源消耗** | 高 (需要全量备用环境) | 低 (仅需少量实例起步) |
| **回滚速度** | 秒级 (路由切换) | 分钟级 (流量逐步调整) |
| **流量控制** | 粗粒度 (全量切换) | 精细粒度 (百分比/用户特征) |
| **发布时长** | 快速 (部署后即刻切换) | 较慢 (需逐步观察放量) |
| **复杂度** | 低 (基础设施层支持即可) | 高 (需网关/Service Mesh支持) |

### 实战案例
在一次支付网关升级中，我们采用了**金丝雀发布**。起初仅 1% 流量，但新版本引入的的一个第三方库导致偶发的死锁，只影响了极少量用户。我们立即通过流量权重回滚，避免了全站资损。如果是蓝绿部署，一旦切流全量，故障面将是 100%，且回滚需重新切回，风险远高于金丝雀。

### 代码示例 (Nginx 权重配置)
```nginx
upstream backend {
    # 金丝雀：新版本 V2 占 10% 流量
    server new_v2_pod:8080 weight=1;
    # 旧版本 V1 占 90% 流量
    server old_v1_pod:8080 weight=9;
}
```

## 常见考点
1.  **在微服务架构中，金丝雀发布如何保证全链路一致性？**（追问：服务 A 调用服务 B，A 升级了 B 没升级怎么办？——Trace 驱动的灰度或 Header 透传）
2.  **数据库变更在蓝绿/金丝雀中如何处理？**（追问：先改库还是先发代码？——兼容性设计，Add Column, Rename Column 双写等）
3.  **K8s 中如何实现这两种策略？**（追问：Service Selector 切换 vs Ingress 权重调整）

### 边界情况补充
*   **有状态服务**：对于数据库或消息队列消费端，蓝绿/金丝雀需要考虑数据同步、连接断开后的重连机制以及消息重复消费的问题（如 RabbitMQ/Kafka 的 Consumer Group Rebalance）。
*   **缓存一致性问题**：新版本上线后，如果变更了缓存 Key 的结构或命名，可能导致新旧版本同时读写缓存造成“缓存穿透”或“数据不一致”，发布策略需配合缓存预热或版本隔离的 Key。
*   **DNS 缓存延迟**：蓝绿切换如果是基于 DNS 的，由于客户端和运营商 DNS 缓存存在 TTL，会导致部分用户在切换后仍访问到旧环境，造成流量“泄漏”，通常建议配合 LB 层切换而非 DNS。

## 易错点
1.  **忽略金丝雀的数据回滚**：金丝雀发布导致新版本写入新格式的数据到 DB，旧版本读这些数据会报错。错误认知是“流量切回来了就没事了”，实际上脏数据已经产生。
2.  **全链路透传缺失**：只在入口网关做了金丝雀分流，但微服务间调用未透传灰度标记，导致下游服务被旧版本随机调用，破坏了灰度验证的封闭性。

## 面试追问
1.  如果新版本引入了不可逆的数据结构变更（如删除了一个字段），既不能用蓝绿也不能用金丝雀，通常采用什么策略？（提示：维护窗口、扩缩容停机发布）
2.  在金丝雀发布中，如何自动化判断新版本是否“健康”？如果错误率只有 0.1% 但 P99 延迟飙升 50%，是否触发自动回滚？（追问：多维度的健康检查指标配置）
3.  针对 Session 粘滞 场景，蓝绿部署切换时如何保证用户 Session 不丢失？

## 记忆要点

- 蓝绿部署：双倍资源，秒级回滚，适合稳定性要求高、需快速切换的传统后端。
- 金丝雀发布：资源利用率高，灰度放量，适合高频迭代、需验证业务逻辑的服务。
- 核心取舍：蓝绿重隔离与速度，金丝雀重成本控制与精准验证。
