# 生产RAG中查询扩展的最佳实践是什么?如何平衡效果和延迟

- **查询扩展技术对比:**

| 技术 | 延迟增加 | 效果提升 | 适用 |
|------|---------|---------|------|
| Query Rewriting | +1次LLM | 中 | 模糊查询 |
| HyDE | +1次LLM | **高** | 事实型查询 |
| Multi-Query | +1次LLM | 高 | 覆盖面不足 |
| Step-Back | +1次LLM | 高 | 全局问题 |
| Query Decomposition | +1次LLM | **高** | 复杂多跳 |

- **延迟优化:**

1. **并行查询**
- Multi-Query的多个查询并行检索
- 只增加1次LLM调用延迟(生成查询)

2. **缓存**
- 相同/相似查询的结果缓存
- HyDE的假想文档缓存

3. **条件触发**
- 简单查询不做扩展
- 用分类器判断是否需要扩展

- **生产推荐组合:**
```
用户查询 → 分类(简单/复杂)
├─ 简单 → 直接向量检索
└─ 复杂 → Query Rewriting + Multi-Query(3个变体) → 并行检索 → RRF融合 → Rerank
```

**并行查询架构图:**
```text
User Query: "RAG 优化技巧"
│
├─> [LLM: Query Expansion] (耗时 T1)
│    ├─ Q1: "RAG 查询优化"
│    ├─ Q2: "RAG 分块策略"
│    └─ Q3: "RAG Reranker 模型"
│
├─> [并发 Vector Search] (耗时 T2, T2 < T1)
│    ├─ DocSet A (for Q1) ─┐
│    ├─ DocSet B (for Q2) ─┤→ Reciprocal Rank Fusion (RRF)
│    └─ DocSet C (for Q3) ─┘
│
└─> Final Ranked List
```

- **实战案例：** 某技术文档搜索系统，对于“如何部署”这类泛词，直接检索效果极差。引入 HyDE 生成一段详细的假想部署步骤后，向量匹配度大幅提升，NDCG 指标从 0.3 提升至 0.65，且通过并行化仅增加了约 300ms 延迟。

- **关键代码：**
```python
from langchain.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough

# 生产实战：HyDE + 并行检索
deployment_prompt = ChatPromptTemplate.from_template(
    "请生成一段关于用户问题的详细技术实现步骤：{question}"
)

# 生成假设文档
generate_hypothetical = deployment_prompt | llm | StrOutputParser()

# 并行执行：原始检索 + 假设文档检索
retrieval_chain = {
    "context": generate_hypothetical | retriever,
    "question": RunnablePassthrough()
}
```

## 常见考点
1. **RRF (Reciprocal Rank Fusion)**：为什么要用 RRF 而不是简单的分数相加？（答：不同向量模型或检索器的分数分布不具可比性，RRF 基于排名融合，鲁棒性更强）
2. **HyDE 缺陷**：对于幻觉严重的小模型，HyDE 为什么可能失效？（答：小模型生成的假想文档质量差，包含错误信息，导致检索方向跑偏）
3. **Step-back Prompting**：具体原理是什么？（答：先让模型抽象出更高层级的概念，再基于高层概念检索，适用于需要宏观理解的问题）
