# 如何构建一套 AI Agent 的回归测试体系？核心难点在哪里？

Agent 的回归测试旨在确保在代码或模型更新后，原有的工具调用链路和逻辑依然正确。由于 Agent 的非确定性（随机采样）和外部依赖（API、环境），测试难度远超传统软件。

**构建体系**：
1.  **场景库建设**：收集真实的用户 Query 和对应的最优工具调用序列（Golden Trace）。
2.  **可重复性模拟**：构建 Mock Server，模拟外部工具的返回结果，确保测试环境不受外部 API 变动影响。
3.  **评估指标**：除了最终答案的正确性（Exact Match 或 LLM-as-a-Judge），还要检查中间过程是否调用了正确的工具以及参数是否合规。
4.  **确定性控制**：在测试时固定 Seed 和 Temperature，减少随机性波动。

**核心难点**：
1.  **非确定性输出**：即使 Seed 固定，底层框架或模型版本的微小差异也可能导致输出路径改变。
2.  **评估成本高**：需要真实运行多步推理和外部调用，测试耗时且消耗 Token。
3.  **长尾场景覆盖**：Agent 面对复杂异常时的行为难以穷举。

## 边界情况
- **Stateful 工具处理**：当工具涉及状态变更（如“下单”、“删除文件”），简单的 Mock 回放可能不足以验证副作用，需要引入“事务回滚”或专门的沙箱环境进行验证。
- **多模态输入**：对于包含图片、文件的 Query，Mock Server 需要支持二进制数据的正确编码和传输，否则可能导致序列化错误。
- **循环与死锁**：Agent 陷入无限循环（如反复查询同一工具）是回归测试必须捕获的极端边界，需设置最大步数阈值作为硬性熔断。

### 实战案例
某订票 Agent 在回归测试中经常失败，原因是天气 API 的实时数据变动导致 Agent 无法复现历史决策。通过引入 **VCR (Video Cassette Recorder)** 模式录制真实的 API 响应并在测试时回放，将测试通过率从 60% 稳定提升至 95% 以上。

### Agent 测试方法对比
| 维度 | Golden Trace (录制回放) | LLM-as-a-Judge (模糊评估) | 传统单元测试 |
| :--- | :--- | :--- | :--- |
| **准确性** | 高（逻辑一致） | 中（主观性强） | 高（代码级） |
| **成本** | 低 | 高（消耗 Token） | 低 |
| **覆盖度** | 覆盖特定路径 | 覆盖开放式结局 | 覆盖代码分支 |

## 面试追问
1. 当工具 API 发生 Breaking Change（如参数结构变更）时，如何自动化检测并更新现有的 Mock 数据集？
2. 在 CI/CD 流水线中，如何平衡测试覆盖率与 Token 成本？是否可以使用小参数模型（如 Llama-7B）作为 Judge 进行初步筛选？
3. 如何测试 Agent 的“拒绝回答”边界（例如处理恶意 Prompt 或超出能力范围的请求）？

## 易错点
1. **过度依赖 Seed**：误以为固定了 `seed=0` 就能保证完全复现。但在不同版本的 PyTorch 或 CUDA 环境下，浮点运算顺序的改变仍会导致结果发散。
2. **混淆评估对象**：将 Agent 的推理能力与 LLM 的知识能力混淆。回归测试应侧重于工具调用逻辑的正确性，而非模型对通用知识的问答准确性。

## 记忆要点

- 核心难点：非确定性输出、评估成本高、长尾场景覆盖难。
- 构建体系：建立场景库、Mock 外部依赖、固定 Seed 控制随机性、多维度评估。
- 评估指标：除最终答案正确性外，需检查工具调用序列和参数合规性。
- 实战技巧：使用 VCR 模式录制回放 API 响应，隔离外部变动，提升测试稳定性。
