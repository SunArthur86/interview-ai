# 主流Agent框架对比?LangGraph vs CrewAI vs AutoGen vs Dify

- **Agent框架全景对比:**

| 框架 | 设计理念 | 适合 | 学习成本 | 生产就绪 |
|------|---------|------|---------|---------|
| **LangGraph** | 图结构编排 | 复杂工作流 | 高 | **高** |
| **CrewAI** | 角色协作 | 明确分工 | 低 | 中 |
| **AutoGen** | 对话式协作 | 探索任务 | 中 | 中 |
| **Dify** | 可视化平台 | 企业应用 | **低** | **高** |
| **OpenAI Swarm** | 极简Handoff | 路由分诊 | **最低** | 低(实验) |

- **核心架构差异 (LangGraph vs AutoGen):**

```text
LangGraph (有向循环图):
┌─────────────┐    State    ┌─────────────┐
│   Agent A   │────────────>│   Agent B   │
└──────┬──────┘             └──────┬──────┘
       │                         │
       └────────(Edge: Condition)────────┘
  (基于状态转移，控制精确，可回环)

AutoGen (多代理对话):
  Agent A ───[msg]───> Agent B ───[msg]───> Agent A
       <──────────────────────────────────────────
  (基于对话流，灵活但较难控制死循环)
```

- **选型决策树:**
- 零代码 → Dify / Coze
- 少量代码 → CrewAI
- 全代码控制 → 简单路由用Swarm,自由协作用AutoGen,复杂状态用LangGraph

- **实战案例：** 在金融研报生成场景中，LangGraph 能优雅处理“如果数据缺失则回退到搜索，否则继续计算”的包含回路的复杂逻辑，而 AutoGen 往往会在两个代理互相争论数据有效性时陷入死循环，需强制设置 `max_turns` 终止。

- **关键代码示例 (LangGraph 状态回环):**
```python
from langgraph.graph import StateGraph, END

def should_retry(state: dict):
    return "retry" if state["errors"] > 0 else "end"

workflow = StateGraph(AgentState)
workflow.add_node("search_agent", search_node)
workflow.add_node("writer_agent", writer_node)
# 条件边实现回环：若出错回到 search，否则结束
workflow.add_conditional_edges(
    "writer_agent",
    should_retry,
    {"retry": "search_agent", "end": END}
)
```

- **架构细节对比：**

| 特性 | LangGraph | AutoGen | CrewAI |
| :--- | :--- | :--- | :--- |
| **控制流** | 有向图 (状态机) | 对话流 (消息传递) | 顺序/层级流程 |
| **状态管理** | 共享中心化 State | 分布式在消息历史 | 隐式共享 |
| **人机协同** | 原生支持 `interrupt` | 需手动挂起 | 支持人工审核 |

- **趋势观察:**
1. **MCP标准化** - 框架逐渐支持MCP (Model Context Protocol)
2. **声明式>命令式** - DSPy模式推广，Prompt即代码
3. **可视化+代码混合** - Dify+LangGraph (Dify作为前端编排，Graph作为后端逻辑)
4. **Agent OS层** - 运行时与框架分离

- **边界情况**：
1. **图结构复杂度爆炸**：在LangGraph中，随着节点增加，边的条件判断逻辑会变得极其复杂（尤其是全连接场景），导致代码难以维护。需考虑子图或模块化封装。
2. **多模态/工具输入**：AutoGen在处理非文本（如图片、函数返回的复杂对象）传递时，序列化/反序列化容易出错，需严格定义Message schema。
3. **长时间运行的Agent**：Dify这类平台虽然有工作流，但在执行超过5分钟的复杂任务时，HTTP连接可能超时，需使用异步Webhook回调模式。

- **## 面试追问**
1. LangGraph 的 `checkpointer` 机制是如何实现“时间旅行”和暂停/恢复的？底层数据结构通常是什么？（如：基于SQLite的CheckpointSaver）
2. 在 AutoGen 的 GroupChat 中，如果发言的 Agent 选择了不发言，通常如何配置让控制权平滑流转给下一个 Agent？
3. Dify 的“工作流”和“Agent”模式在底层执行引擎上有何本质区别？（工作流是DAG，Agent是循环+推理）

- **## 易错点**
1. **混淆“路由”与“对话”**：认为 AutoGen 的 `UserProxyAgent` 只是简单的传话筒。实际上它在代码执行、安全阻断方面承担了关键责任，配置不当可能导致LLM执行任意Python代码破坏系统。
2. **LangGraph 状态覆盖风险**：在 `reducer` 函数中如果不正确处理更新逻辑（如直接赋值而非 append），可能导致多个节点的更新互相覆盖，从而丢失中间状态。
