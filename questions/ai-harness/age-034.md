# 什么是 Harness Engineering?它和传统 Agent 框架有什么区别

- **Harness Engineering** 是 2025 年兴起的概念,指围绕大模型构建的工程化外壳,让模型在真实场景中可靠工作.

- **核心理念:** 模型本身只是推理引擎,真正决定 Agent 效果的是围绕模型的工程化体系.

- **Harness 包含的关键组件:**
1. 上下文工程:组装 system prompt、工具定义、历史消息
2. 工具治理:权限控制、参数校验、结果过滤
3. 状态管理:任务状态快照、恢复机制
4. 可观测性:Trace 轨迹、工具调用日志、成本追踪
5. 安全防护:Prompt 注入检测、输出过滤

- **与传统框架的区别:** LangChain 关注链式调用编排,Harness Engineering 更关注生产级的可靠性、可观测性和安全性.

- **代表产品:** Hermes Agent、OpenClaw、Claude Code 都是 Harness Engineering 的实践.

- **Harness Engineering 架构图:**

```text
        ┌─────────────────────────────────┐
        │      用户 / 应用层 (App)         │
        └───────────────┬─────────────────┘
                        │
        ┌───────────────▼─────────────────┐
        │       Harness 外壳 (工程层)      │
        │  ┌─────────────────────────┐   │
        │  │  安全 & 治理             │   │
        │  │  (认证, 鉴权, 限流)      │   │
        │  └───────────┬─────────────┘   │
        │  ┌───────────▼─────────────┐   │
        │  │  状态 & 上下文管理       │   │
        │  │  (Session, Memory, DB)  │   │
        │  └───────────┬─────────────┘   │
        │  ┌───────────▼─────────────┐   │
        │  │  工具编排 & 调度         │   │
        │  │  (API, Workflow, DAG)   │   │
        │  └───────────┬─────────────┘   │
        │  ┌───────────▼─────────────┐   │
        │  │  可观测性               │   │
        │  │  (Logs, Metrics, Trace) │   │
        │  └───────────┬─────────────┘   │
        └───────────────┼─────────────────┘
                        │ 标准化接口
        ┌───────────────▼─────────────────┐
        │   LLM 推理引擎 (模型层)          │
        │   (GPT-4, Claude, Local LLM)    │
        └─────────────────────────────────┘
```

- **对比表格:**
| 特性 | 传统 Agent 框架 (如 LangChain) | Harness Engineering |
| :--- | :--- | :--- |
| **核心关注** | 快速原型开发、编排能力 | 生产稳定性、可观测性 |
| **模型耦合** | 较强 (直接依赖 SDK) | 弱耦合 (统一抽象层) |
| **错误处理** | 依赖手动 Try/Catch | 全局熔断、重试、降级策略 |
| **成本控制** | 缺失或简单计数 | 细粒度预算、Token 限流 |
| **适用阶段** | MVP/验证期 | 大规模生产环境 |

- **实战案例:** 在某金融风控 Agent 落地时，直接使用 LangChain 导致出现“Prompt 泄露”（将系统指令打印给用户）和“幽灵 Token”（模型返回空字符但计费）。引入 Harness 层后，通过严格输出的 Schema 校验和成本熔断机制，避免了数万元的潜在损失。

- **代码示例:**
```python
class LLMHarness:
    def __init__(self, model_provider, rate_limiter):
        self.model = model_provider
        self.limiter = rate_limiter

    async def generate(self, prompt, max_tokens=1000):
        # 工程层：统一限流与熔断
        await self.limiter.acquire()
        try:
            # 工程层：统一超时控制
            return await asyncio.wait_for(
                self.model.chat(prompt), 
                timeout=30.0
            )
        except TimeoutError:
            return FallbackResponse()
```

- **## 常见考点:**
1. **边界划分**: Harness 和 传统的 Application Framework (如Spring Boot) 的边界在哪里？哪些逻辑应该放在Harness里，哪些放在业务代码里？
2. **模型无关性**: Harness 层如何设计才能做到快速切换底座模型（如从GPT-4切换到Claude 3.5）而无需改动业务逻辑？
3. **成本控制**: Harness 如何在工程层面实现细粒度的成本控制（如Token预算告警、Prompt自动压缩）？
4. **故障恢复**: 当LLM API返回503或超时，Harness层的设计模式（如重试、降级、Fallback）有哪些最佳实践？
