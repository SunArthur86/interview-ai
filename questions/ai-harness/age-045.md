# 生产 AI 应用中,如何做 Token 成本归因和优化

- **Token 成本管理** 是生产 AI 应用的核心运营指标.

- **成本归因:**
  - **1. 请求级追踪:** 每次请求记录 input_tokens、output_tokens、模型、单价,计算实际费用. **关键细节:** 需区分 Prompt Token 和 Completion Token,因为通常 Completion Token 单价是 Input 的 2-3 倍.
  - **2. 功能级分账:** 按 API 端点或功能模块分组统计.如 RAG 检索花了多少、代码生成花了多少.
  - **3. 用户级分账:** 按用户/租户统计用量,用于计费和配额管理.
  - **4. 成本标签:** 给每个请求打标签(项目、环境、实验版本),支持多维度切片分析.

- **优化手段:**
  - **1. 模型降级:** 简单任务用小模型(GPT-4o-mini、DeepSeek-V3),复杂任务才用大模型. **关键细节:** 实施策略可使用“级联路由”,先用小模型尝试,若置信度分数低于阈值则转由大模型处理.
  - **2. Prompt 压缩:** 减少冗余上下文、精简 System Prompt. **关键细节:** 移除停用词,使用更指令化的语言;对于 Few-shot 示例,可以使用向量检索选出最相关的 K 个示例而非固定插入.
  - **3. 缓存:** Prefix Caching、语义缓存(相似请求返回缓存结果). **关键细节:** KV Cache 是推理引擎(如 vLLM)层面的优化,能复用计算过程中的 KV 状态;应用层可针对 API 响应做缓存.
  - **4. 批处理:** Batch API 通常有 50% 折扣. **关键细节:** 适用于离线、非实时任务(如夜间文档总结、批量 Embedding 生成),将多个请求打包一次性提交.
  - **5. 上下文管理:** 定期压缩对话历史、限制工具返回长度. **关键细节:** 实现“滑动窗口”机制,保留最近的 N 轮对话,或使用摘要模型将历史对话压缩为一段 Context.

### 实战深化

**实战案例:** 我们的日志分析助手最初每次都将原始的 10KB 日志片段全量放入 Context。引入“截断策略”后，先通过规则过滤掉 Trace ID 和时间戳等非关键信息，仅将错误堆栈部分送给 LLM，使单次请求成本降低了 70%。

**边界情况:**
- **超长输出截断：** 模型生成内容可能会超过 `max_tokens` 限制。若不做处理，会导致 JSON 格式截断（如缺少闭合括号），进而导致下游解析器报错。需在 Prompt 中增加“必须在 N 字内结束”的强指令，或在应用层做重试机制。
- **RAG 中的上下文碎片化：** 在 RAG 场景中，检索出的文档切片可能存在大量重叠，导致 Token 虚高且造成“迷失中间”现象。需要在送入模型前做跨切片的去重和合并，而不仅仅是截断。
- **多模态 Token 计费差异：** 图片和音频的 Token 换算通常与文本不同（如一张高清图可能折算 1000+ Tokens）。成本归因系统必须能识别 MediaType 并应用正确的计费倍率，否则会造成巨大账单误差。

## 面试追问
1. **关于 Prompt 压缩：** 如果使用小模型来压缩长文本历史，是否可能出现“信息丢失”导致后续推理错误？如何量化这种信息损失并设定压缩阈值？
2. **关于缓存一致性：** 在知识库（RAG）场景下，如果源文档被更新了，如何保证基于旧文档生成的语义缓存能被及时失效或更新？
3. **关于动态路由：** 你提到了级联路由，但如果第一层小模型判断错误（把复杂任务判为简单），会导致用户体验变差且浪费两次请求。如何平衡判断模型的准确率和整体的性价比？

## 易错点
1. **只看输入不看输出：** 很多优化策略只关注减少输入 Token（如压缩 Prompt），但忽略了输出 Token 往往是成本的 3 倍。限制输出长度（如要求“回答不超过 50 字”）是更直接的降本手段。
2. **盲目使用 Batch API：** 认为 Batch API 总是更省钱。实际上 Batch API 通常有最低起送量（如 1 小时或 1.8万 Token），且延迟极高（24小时内）。对于对实时性有要求的业务，错误使用 Batch API 会导致严重的 SLA 违约。
