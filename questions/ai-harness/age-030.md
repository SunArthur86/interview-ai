# 语义缓存和精确缓存有什么区别?如何防止缓存「串味」(不同问题返回相同缓存)

- **语义缓存 vs 精确缓存:**

| | 精确缓存 | 语义缓存 |
|--|---------|---------|
| 匹配方式 | 完全匹配(HASH) | **语义相似度匹配** |
| 实现 | Redis/HashMap | 向量检索(GPTCache) |
| 召回率 | 低(一点不同就miss) | **高** |
| 精确度 | **100%** | 可能误匹配 |
| 适合 | 固定QA | **开放问答** |

- **防止缓存「串味」的措施:**

1. **相似度阈值**
- 设定最低余弦相似度(如0.95)
- 低于阈值的不返回缓存

2. **语义验证**
- 用小LLM快速验证:缓存问题和当前问题是否真的等价
- 成本低但多一次推理

3. **分类隔离**
- 不同任务类型的缓存分开存储
- 避免跨领域误匹配

4. **TTL过期**
- 时效性内容设置短TTL
- 避免返回过时信息

5. **用户隔离**
- 用户特定的问题不做全局缓存
- 或做用户级缓存命名空间

- **语义缓存检索流程图:**

```text
   用户提问
      │
      ▼
 ┌─────────┐
 │ 生成向量 │
 └────┬────┘
      │
      ▼
 ┌─────────────┐
 │ 向量数据库检索 │
 │ (Top-1 Candidate)│
 └──────┬──────┘
      │
      ▼
 ┌─────────────────┐
 │ 余弦相似度 > 阈值? │──No──▶ 直接请求LLM
 └──────┬──────────┘
      │ Yes
      ▼
 ┌─────────────────┐
 │  LLM 语义验证层  │──No──▶ 直接请求LLM
 │ (去噪/判别)      │
 └──────┬──────────┘
      │ Yes
      ▼
   返回缓存结果
```

- **## 常见考点:**
1. **缓存Key设计**: 除了向量相似度，是否结合了Hash（如Prompt结构化指纹）来防止完全不同但向量巧合相似的问题？
2. **索引选择**: 向量检索使用哪种索引（HNSW, IVF）？如何在召回率和延迟之间权衡？
3. **预计算 vs 实时**: 缓存的Embedding是预计算好存入，还是实时计算？实时计算如何避免缓存本身成为瓶颈？
4. **缓存失效策略**: 如果知识库更新了（如RAG中的文档变动），如何批量失效受影响的语义缓存？

- **实战案例：** 某问答系统曾出现“华为手机如何”命中了“苹果手机如何”的缓存，原因是两者Embedding极度相似且仅设置了0.85的阈值。后来引入了“意图分类+用户ID”作为缓存Namespace，才彻底解决串味问题。

- **代码示例：**
```python
from gptcache import Cache
from gptcache.adapter.api import init_similar_cache

# 初始化带阈值的语义缓存
init_similar_cache(data_dir="./cache", 
    cache_obj=Cache(),
    similarity_threshold=0.95, # 关键：高阈值防止串味
    embedding_func=your_embedding_model
)

# 查询逻辑（伪代码）
answer = cache.get(question)
if answer is None:
    answer = llm.generate(question)
    cache.set(question, answer) # 自动存入向量
```

- **## 边界情况:**
1. **多模态输入**: 当问题包含图片或文件（多模态大模型），纯文本的语义缓存会失效。需将图片特征或文件Hash纳入缓存Key，或仅对文本Prompt部分做缓存。
2. **否定词干扰**: “如何关闭账号”与“如何不关闭账号”在语义空间可能高度相似（仅差一个否定词），容易导致严重“串味”。必须引入否定词检测权重或强制哈希校验。
3. **超长Prompt截断**: 如果输入Prompt超过了Embedding模型的上下文窗口，截断策略可能导致前半部分的关键指令丢失，使得语义匹配发生偏差。需对超长文本使用分段Embedding或摘要Embedding。

- **## 面试追问:**
1. 如果业务要求极高精度（如金融），只使用向量检索的语义缓存风险太大，你会设计什么样的混合缓存架构来平衡准确率与命中率？
2. 在系统重启或缓存预热阶段，如何防止大量请求穿透（Cache Stampede）导致后端LLM服务雪崩？
3. 当 Embedding 模型升级（如从 v1 升级到 v2）导致向量空间分布不一致时，如何处理存量缓存数据？是全量失效还是建立双写机制？

- **## 易错点:**
1. **缓存答案的幻觉固化**: 如果第一次LLM生成的回答包含幻觉且被存入缓存，后续所有相似用户请求都会直接得到这个错误答案。必须在缓存过期策略或验证环节引入人工反馈机制（Thumbs Down自动失效）。
2. **忽视上下文差异**: 用户追问“那价格呢？”依赖于上文对话。如果仅将单句问题作为Key进行语义缓存，会忽略上下文依赖，导致返回错误的缓存答案。应将 Sliding Window 的历史对话摘要拼接进 Embedding 输入。
