# 什么是 LLM Gateway(模型网关)?生产环境为什么需要它

- **LLM Gateway** 是位于应用和 LLM 供应商之间的代理层,提供统一的 API 接口和治理能力.

- **架构示意:**
```text
┌─────────────┐      ┌─────────────────────────────┐      ┌───────────────┐
│  应用服务    │ ───> │    LLM Gateway (中间层)     │ ───> │  LLM 供应商 A  │
│ (Client)    │      │  ┌─────────────────────┐    │      │ (OpenAI)      │
└─────────────┘      │  │ 1. 统一接口          │    │      └───────────────┘
                     │  │ 2. 路由与LB (策略)   │───>┼────>│  LLM 供应商 B  │
                     │  │ 3. 限流/熔断/重试    │    │      │ (Claude/Azure)│
                     │  │ 4. Prompt 模板       │    │      └───────────────┘
                     │  │ 5. 缓存              │    │
                     │  │ 6. 可观测性/日志     │    │      ┌───────────────┐
                     │  └─────────────────────┘    └────>│  LLM 供应商 C  │
                     │                             │      │ (开源/本地)    │
                     └─────────────────────────────┘      └───────────────┘
```

- **核心功能:**
  - **1. 统一接口:** 屏蔽不同供应商的 API 差异(OpenAI、Claude、国产模型),应用只对接 Gateway.支持协议转换(如将 OpenAI 格式转换为 Anthropic 格式).
  - **2. 负载均衡与路由:** 根据任务类型、成本、延迟自动选择模型. **关键细节:** 支持多级路由策略,如基于 Prompt 指纹的智能路由(简单问答分发给小模型,复杂推理分发给大模型),以及基于可用性的故障转移.
  - **3. 限流与熔断:** 防止下游模型服务过载,保护应用不被 API 限流影响. **关键细节:** 实现 Token Bucket 或 Leaky Bucket 算法进行速率限制;配置熔断器(如 Circuit Breaker 模式),当下游错误率超过阈值(如 50%)时自动熔断,快速失败.
  - **4. 成本控制:** Token 级别的成本追踪和归因,支持按项目/用户/功能分账. **关键细节:** 实时计算输入/输出 Token 消耗与预估费用,支持设置预算阈值并触发告警.
  - **5. 缓存:** 相同请求直接返回缓存结果,降低成本. **关键细节:** 除精确匹配外,支持语义缓存,即对语义相似度超过阈值(如余弦相似度 > 0.95)的 Query 命中缓存,大幅提升命中率.
  - **6. 安全:** Prompt 注入检测、敏感词过滤、审计日志. **关键细节:** 在请求发送前进行 PII(个人隐私信息)识别与脱敏,防止敏感数据泄露给模型.
  - **7. 可观测性:** 统一的监控面板、请求日志、质量指标. **关键细节:** 采集延迟、TTFT(首字时间)、吞吐量等指标,并支持 Trace ID 全链路追踪.

- **常见实现:** LiteLLM(开源)、Kong AI Gateway、Cloudflare AI Gateway、Portkey.

- **什么时候需要:** 日均调用量超过 1 万次、使用多个模型供应商、需要成本管控.

### 实战深化

**实战案例:** 某次 OpenAI API 大面积故障（5xx 错误），由于 Gateway 配置了熔断与自动切换，业务流量在 30ms 内自动切换到 Azure OpenAI 端点，实现了用户无感的容灾，避免了 SLA 违约。

**代码示例 (Python - 简易 Fallback 逻辑):**
```python
import openai

def llm_chat_with_fallback(prompt, model="gpt-4"):
    try:
        # 尝试主供应商
        return openai.ChatCompletion.create(model=model, messages=[prompt])
    except openai.error.APIError as e:
        print(f"Primary provider failed: {e}, switching to backup.")
        # 切换到备用供应商（配置不同的 Base URL 和 Key）
        openai.api_base = "https://api.backup-provider.com/v1"
        openai.api_key = "BACKUP_KEY"
        return openai.ChatCompletion.create(model=model, messages=[prompt])
```

**特性对比表格 (是否使用 Gateway):**

| 特性 | 直连 LLM API | 使用 LLM Gateway |
| :--- | :--- | :--- |
| **供应商切换** | 需要修改代码并重新部署 | 配置化热更新，秒级切换 |
| **统一计费** | 需手动汇总各平台账单 | 自动聚合，支持实时预算控制 |
| **缓存策略** | 需应用自己实现 Redis 缓存 | 内置语义缓存/精确缓存 |
| **可观测性** | 依赖各供应商后台（数据分散） | 统一 Dashboard，全链路 Trace |
| **安全防护** | 需自行编写中间件拦截 | 内置 PII 扫描、注入检测 |
| **流式转发** | 简单透传 | 支持流量整形、实时 Token 计费 |

## 常见考点
1. **如何处理流式响应?**
   Gateway 需要支持 SSE (Server-Sent Events) 转发,保持长连接,并将上游模型的流式数据块无损地透传给客户端,同时进行实时的 Token 计数.

2. **Fallback 机制的设计细节?**
   当主模型(如 GPT-4)超时或限流时,如何平滑切换到备用模型(如 GPT-3.5-Turbo)而保证业务逻辑不中断?
