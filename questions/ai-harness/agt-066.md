# 性能优化

### 性能优化

**核心目标**：解决 Agent 系统中的串行 LLM 调用、高延迟网络 IO 和上下文膨胀问题。

**原理详解**：

1. **异步处理**：
   - 使用 `asyncio` 或消息队列，避免 I/O 等待阻塞线程。
   - **并行工具调用**：无依赖的工具调用并发执行，大幅降低总耗时。

2. **流式输出**：
   - 利用 SSE (Server-Sent Events) 边生成边返回，降低首字延迟 (TTFB)。
   - 需处理流式中断情况，保证最终状态一致性。

3. **并发控制**：
   - 使用 `Semaphore` 限制对同一模型的并发请求数，防止触发 429 (Rate Limit)。

4. **连接复用**：
   - HTTP 客户端开启 `keep-alive`，数据库使用连接池，减少握手开销。

5. **缓存分层**：
   - **L1 (进程内)**：LRU 缓存，极低延迟。
   - **L2 (Redis)**：分布式共享，支持 TTL。
   - 读路径：L1 -> L2 -> 回源模型。

#### 边界情况与极端场景
- **空响应或超时处理**：在并行调用中，若某个工具超时，不应阻塞整个流程。需配置合理的 `timeout` 和 `fallback` 机制（如返回默认值或重试），避免“短板效应”拖死整体响应。
- **Token 溢出**：并行工具返回结果后合并上下文时，总长度可能超出模型 Context Window。需实现动态截断策略（如保留开头、结尾，摘要中间部分）或分块处理。
- **热点数据雪崩**：缓存失效瞬间（如大量重复查询同时涌来），可能导致所有请求同时回源击穿模型。需使用互斥锁防止缓存重建时的并发回源。

#### 实战案例
某旅游规划 Agent 原本串行调用“查航班”、“查天气”、“查酒店”，总耗时 15秒。通过分析 DAG（有向无环图）发现这三者无依赖关系，改为 `asyncio.gather` 并行调用后，耗时降至 5秒（由最慢的接口决定），用户体验显著提升。

#### 代码示例
```python
import asyncio

async def call_tool(tool_name: str, query: str):
    # 模拟工具调用 IO 耗时
    try:
        await asyncio.wait_for(asyncio.sleep(1), timeout=2.0) # 添加超时控制
    except asyncio.TimeoutError:
        return f"Timeout: {tool_name}"
    return f"Result for {tool_name}"

async def parallel_agent_executor():
    # Agent 决策需要调用的工具列表（并行）
    tools_to_call = ["search_flights", "check_weather", "book_hotel"]
    
    # 并行执行，无需等待上一个完成
    results = await asyncio.gather(
        *[call_tool(name, "user query") for name in tools_to_call],
        return_exceptions=True # 异常不抛出，作为结果返回
    )
    
    return results
```

#### 优化策略对比
| 优化手段 | 优化维度 | 适用场景 | 潜在风险 |
| :--- | :--- | :--- | :--- |
| **异步/并行** | 吞吐量 & 延迟 | 多步骤无依赖 Agent | 增加代码复杂度，需处理竞态条件 |
| **流式输出 (SSE)** | 首字延迟 (TTFT) | 生成式对话 | 前端需处理增量渲染，难以中途取消 |
| **L1/L2 缓存** | 吞吐量 & 成本 | 高频重复问题 | 缓存一致性难题，可能返回过期数据 |
| **连接复用** | 网络延迟 | 高并发短连接 | 占用更多长连接资源，需配置合理超时 |

## 面试追问
1. **追问（架构设计）**：如果并行的工具调用之间存在部分数据依赖（例如“查酒店”需要“查航班”的目的地时间），你如何动态构建 DAG 并优化执行顺序？
2. **追问（监控排查）**：在生产环境中，如果发现 99% 的请求延迟都很低，但 1% 的请求耗时极长（长尾延迟），你通常从哪些维度排查？
3. **追问（流式处理）**：在流式输出场景下，如果第一句话生成后发现工具调用参数错误，如何实现“回滚”或“中断”以避免用户看到无用的后续生成？

## 易错点
1. **误区**：认为只要使用了 `asyncio` 就一定能提升性能。
   **纠正**：如果 CPU 密集型操作（如复杂的正则匹配、数据清洗）在异步函数中运行，仍会阻塞事件循环。必须将 CPU 密集型任务放到单独的线程池或进程池中执行。
2. **误区**：缓存设置得越大越好，过期时间越长越好。
   **纠正**：过长的 TTL 会导致“僵尸数据”问题，即业务逻辑已变更（如退款规则修改），但用户仍收到基于旧规则缓存的答案，引发严重的客诉。
