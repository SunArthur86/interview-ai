# Pass@k 是什么指标?它如何衡量代码生成模型的质量

**Pass@k** 是评估代码生成模型的标准指标，衡量生成代码的通过率。

**定义:** 给定一个编程题，模型生成 $n$ 个样本（$n \ge k$），从中随机抽取 $k$ 个候选答案。只要这 $k$ 个样本中至少有一个通过了所有测试用例，就算通过。

**计算公式与原理细节:**
由于大模型具有随机性，我们通常采样 $n$ 个样本（如 $n=200$），其中通过测试的有 $c$ 个。公式为组合概率计算：
$$ Pass@k = 1 - \frac{\binom{n-c}{k}}{\binom{n}{k}} $$
- **$n$ (总采样数)**：为了保证统计显著性，通常取较大的值（如 HumanEval 标准中 $n=200$）。
- **$c$ (通过数)**：在 $n$ 个样本中，能跑通所有单元测试的样本数量。
- **$k$ (评估候选数)**：模拟用户允许模型生成的尝试次数。
- **边界条件**：当 $c=0$ 时，Pass@k 为 0；当 $c=n$ 时，Pass@k 为 1。

**常见取值与场景:**
- **Pass@1**: 只生成一次，要求"一击即中"。这是最严格的指标，直接反映用户在 Copilot 等工具中的实际体验。
- **Pass@10**: 生成 10 个，允许用户从列表中选择。反映模型在代码补全列表中的可用性。
- **Pass@100**: 更宽松的学术评估指标，常用于衡量模型的知识覆盖率和极限潜力。

**评估基准:**
- **HumanEval (OpenAI)**: 164 道手写 Python 编程题，包含函数签名和文档字符串。
- **MBPP (Google)**: 974 道入门级 Python 编程题，由众包编写。
- **HumanEval+**: HumanEval 的增强版，通过自动生成更多"脆弱"的测试用例，防止模型过拟合或仅通过简单测试。
- **SWE-bench**: 真实 GitHub 仓库的 Issue 修复任务，考验长上下文和 repo 级理解能力。

**面试要点:**
- Pass@1 最重要，因为用户通常只接受第一个建议；
- Pass@10 反映模型的上限潜力，两者差距大说明模型稳定性差（方差大）。

**实战案例:**
在通过 SWE-bench 评估 Agent 时，我们曾发现模型在修复 Bug 时 Pass@1 很低，但 Pass@10 很高。排查发现是因为生成的单测代码中**随机种子**未固定，导致偶发性失败。在 Prompt 中强制要求"Fix seed=42"后，Pass@1 提升了 15%，说明有时 Pass@k 低并非模型逻辑错误，而是环境不稳定导致的。

**代码示例 (Python):**
```python
import math

def calculate_pass_at_k(n, c, k):
    """计算 Pass@k 指标
    Args:
        n: 总样本数
        c: 通过测试的样本数
        k: 候选答案抽取数量
    """
    if c == 0: return 0.0
    if n - c < k: return 1.0  # 错误样本不足 k 个，必中
    
    # 组合数计算：C(n, k) = n! / (k! * (n-k)!)
    def comb(n, k): 
        return math.comb(n, k) # Python 3.8+

    return 1.0 - (comb(n - c, k) / comb(n, k))

# 示例：采样200个，通过5个，看取1次命中的概率
print(f"Pass@1: {calculate_pass_at_k(200, 5, 1):.4f}")
```

**常见考点:**
1. **组合数学含义**：为什么分母是 $C(n, k)$？它表示从总样本 $n$ 中选 $k$ 个的所有可能情况；分子 $C(n-c, k)$ 表示选出的 $k$ 个全都是错误样本的情况。
2. **n 和 k 的关系**：为什么评估时 $n$ 通常远大于 $k$？（为了保证无偏估计，模拟从模型分布中采样的过程）。
3. **HumanEval vs HumanEval+**：为什么原版指标可能虚高？（因为模型可能通过过拟合训练数据或仅仅猜对了简单的边界条件，HumanEval+ 引入了更严格测试来过滤这种情况）。
