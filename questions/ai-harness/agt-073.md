# Chain-of-Thought（CoT）思维链

### 4. Chain-of-Thought（CoT）思维链

#### 4.1 原理与概念
**概念解释**
Chain-of-Thought（思维链）指让模型在给出最终答案前，显式写出中间推理步骤（自然语言），从而改善多步推理、数学、逻辑题的表现。

**原理详解**
通过增加「推理 token」，模型在潜在空间中更易进行多步分解；对复杂任务相当于把单跳预测拆成多跳。代价是更长输出、更高延迟与成本，且有时会出现「看似合理但错误」的推理链。

#### 4.2 Zero-shot CoT
**方法**：不手写示例推理过程，仅在问题末尾加一句触发语。
- 英文：「Let's think step by step」
- 中文：「请逐步推理」

#### 4.3 Manual CoT（手写推理步骤）
**方法**：在 Few-shot 里直接写出「推理过程 + 答案」，模型模仿该模式。

**示例片段**
```text
Q: 一个商店先涨价 20% 再降价 20%，最终价格相对原价如何变化？
推理：设原价 100。涨价 20% 后为 120。降价 20% 是在 120 基础上降，即 120 * 0.8 = 96。96 < 100。
最终答案：价格下降了 4%。
```

#### 4.4 Auto-CoT
**概念**
一类自动化方法：用程序从数据集中采样问题，用模型生成多条推理链，再筛选高质量链作为示例，减少人工写 CoT 的成本。

**流程**：
1. 聚类或抽样得到代表性问题。
2. 对每个问题用模型生成 k 条推理链。
3. 用规则或评分模型（如答案是否匹配标准、步骤是否自洽）过滤。
4. 将优质「问题 + 推理链」写入 Few-shot 库。

#### 4.5 CoT 在 Agent 中的应用
**场景**：Agent 在规划、工具选择、异常处理时常用 CoT：先让模型写出「当前状态、目标、下一步动作理由」，再执行工具，降低盲目调用。

**注意**：生产环境可对用户隐藏推理链，仅内部记录；对外只给结论，避免泄露敏感中间信息。

**面试 Q&A**
Q：CoT 为什么能提升推理题正确率？
A：它把任务分解成显式步骤，降低一步到位的难度；对 Transformer 而言，更多相关中间 token 有助于后续 token 的条件预测。但并非万能，错误链也会误导最终答案。

#### 边界情况补充
1. **计算错误累积**：在多步数学推理中，模型若在中间步骤出现微小计算偏差，后续所有步骤可能均基于该错误假设，导致最终答案谬以千里（误差放大效应）。
2. **幻觉推理**：模型可能为了强行凑出合理的逻辑链，编造不存在的前提或引用，尤其是在知识密集型问答中。推理链看似通顺，但事实依据完全错误。
3. **不适用场景**：对于语义分类、情感分析等直觉型任务（Pattern Matching），强制使用 CoT 反而会降低性能，增加模型「想太多」导致的随机性。

## 面试追问
1. 如何量化评估 CoT 推理链的质量？如果最终答案错误但推理过程部分正确，该如何反馈给模型？（考察：过程监督 vs 结果监督、奖励模型设计）
2. CoT 会显著增加推理延迟和成本，在实时性要求高的场景下（如实时客服），有哪些优化手段？（考察：知识蒸馏、Speculative Decoding、Early Exit）
3. 对于非逻辑类任务（如创意写作），CoT 是否有其他变体形式或应用价值？（考察：思维发散、隐性 CoT）

## 易错点
1. **混淆思维链与思维树**：CoT 是线性的单链推理，而 Tree of Thoughts (ToT) 包含探索、回溯和多路径评估，适用于更复杂的规划问题，不能混为一谈。
2. **忽视置信度校准**：CoT 推理步骤的流畅程度并不等同于答案的正确率。模型可能以极高的自信写出一个完全错误的推理过程。

## 记忆要点

- 核心机制：通过增加推理Token，将单跳预测拆解为多跳降低难度
- Zero-shot CoT：仅加触发语（如逐步推理）即可激活模型推理
- Auto-CoT：聚类问题自动生成并过滤推理链，免去人工编写成本
- Agent应用：先写状态目标再执行工具，防盲目调用
- 局限：误差会放大，直觉类任务（如分类）强用CoT反而增加随机性
