# 自我反思 Prompt

### 5. 自我反思 Prompt（Self-Reflection）

#### 5.1 Self-Reflection（自我反思）
**概念解释**
让模型在初稿之后再生成一轮：检查错误、遗漏、与约束的一致性，并输出修订版或自检清单。

**原理详解**
相当于同一上下文内的二次采样，但角色从「生成者」切到「评审者」，对部分任务能降幻觉、补约束。

#### 5.2 Self-Consistency（自洽性）
**概念解释**
对同一问题采样多条推理路径（不同温度或多候选），对最终答案做投票或一致性检查，选多数派或最可信路径。

**原理详解**
用计算换可靠性；适合答案空间有限（选择题、结构化结论）的任务，对开放写作类任务投票意义较小。

#### 5.3 Self-Critique（自我批判）
**概念解释**
显式要求模型指出初版答案的问题列表（事实、逻辑、格式），再据此改写。

**Prompt 模板示例**
```text
【材料】
{{context}}
【初版答案】
{{draft_answer}}
【任务】
你是严格审稿人。请只做两件事：
1) 列出初版答案中可能存在的问题（事实是否可由材料支持、逻辑跳跃、格式是否符合要求），最多 5 条；若没有明显问题，写「未发现明显问题」。
2) 给出修订后的答案；若初版已足够好，第二项原样重复初版答案并注明「维持不变」。
输出两段，标题分别为「问题清单」「修订答案」。
```

#### 5.4 反思在 Agent 中的作用
**场景**：在工具返回异常、空结果或冲突时，反思 Prompt 引导模型重试策略（换查询词、换工具、向用户澄清），而不是硬编答案。

#### 5.5 成本考量
**面试 Q&A**
Q：Self-Reflection 会增加多少成本？值得吗？
A：通常增加约一倍或更高延迟与 token；对高风险、高价值输出（医疗、法律、财务摘要）或格式极易错的场景值得；对低价值闲聊往往不值得。

#### 边界情况补充
1. **无限循环/反复横跳**：在 Self-Critique 中，如果模型基于错误的理由去修正一个原本正确的答案，可能导致答案变差。在多轮反思中，可能会出现 A->B->A->B 的摇摆，需要设置最大迭代次数（Max Iterations）作为熔断机制。
2. **幻觉放大**：在评审阶段，模型可能凭空捏造原文不存在的错误，导致“为了改错而改错”。这通常发生在模型缺乏足够的原始上下文（Context）进行回溯验证时。
3. **置信度缺失**：在 Self-Consistency 投票中，如果多个路径各不相同且分布分散（如 10 次采样得到 5 个不同答案），简单的多数投票无法给出可靠结果，此时应触发“拒绝回答”机制。

## 面试追问
1. 在多轮反思循环中，如何防止模型出现“死循环”或“越改越差”？你需要设计什么样的停止条件？（考察：算法稳定性、Convergence 策略）
2. Self-Consistency 需要多次采样，成本较高。如果只允许一次采样，你如何通过 Prompt Engineering 模拟自洽性检查？（考察：Internal Monologue、反向思考）
3. 如何评估 Self-Critique 环节本身的质量？如果模型指出的“错误”其实是“正确”的，怎么处理？（考察：Golden Set 构建、Evaluator 模型）

## 易错点
1. **混淆 Self-Correction 与 Re-generation**：Self-Correction 通常是基于前一次结果的局部修补；而 Re-generation 是丢弃前文重新生成。单纯的“再生成一次”并不是有效的 Self-Correction，必须显式要求模型指出错误点。
2. **忽视上下文长度限制**：在多轮反思中，原始上下文、初稿、批判意见、修正稿会迅速消耗 Token。若不加截断或摘要策略，容易超出 Context Window 导致信息丢失。

## 记忆要点

- Self-Critique：身份切换为审稿人，先列问题清单再出修订版
- Self-Consistency：多次采样投票，以计算换可靠性适用答案空间小任务
- Agent反思：工具报错时反思换策略重试，而非硬编答案
- 防死循环：多轮反思易反复横跳，必须设最大迭代次数熔断
- 成本权衡：反思翻倍耗时与Token，仅适用高价值或易错场景
