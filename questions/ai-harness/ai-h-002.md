# LlamaIndex和LangChain的定位有什么区别?各自擅长什么场景

- **定位差异:**

| | LangChain | LlamaIndex |
|--|-----------|------------|
| 核心定位 | **通用LLM应用框架** | **数据+LLM框架** |
| 擅长 | Agent/链式调用/工具 | **RAG/索引/查询** |
| 抽象 | Runnable/Chain | **Document/Node/Index** |
| 数据处理 | 基础 | **深度优化** |
| 查询引擎 | 基础 | **多种查询模式** |

- **LlamaIndex核心概念:**
1. **Document** - 原始文档(PDF/HTML/Markdown)
2. **Node** - 文档分块后的最小单元
3. **Index** - 节点的索引结构(向量/树/关键词)
4. **Query Engine** - 查询引擎(路由/子问题/多步)
5. **Retriever** - 检索器

- **LlamaIndex高级查询模式:**
- **Sub-Question Query** - 拆分复杂问题
- **Multi-Document** - 跨文档推理
- **Recursive Retrieval** - 分层检索
- **Router Query** - 动态选择检索策略

- **选择建议:**
- 纯RAG → LlamaIndex(数据处理更细)
- Agent/工作流 → LangChain/LangGraph
- 通用 → 两者结合

**实战案例**
在处理百万级 PDF 文档库时，直接用 LangChain 的基础向量检索导致准确率极低（大量跨章节上下文丢失）。改用 LlamaIndex 的 `Recursive Retrieval`（先检索摘要块，再回溯原文块）后，召回相关度从 0.6 提升至 0.85。

**代码示例（LlamaIndex QueryEngine）**
```python
from llama_index.core import VectorStoreIndex, Document
from llama_index.core.query_engine import RouterQueryEngine
from llama_index.core.selectors import LLMMultiSelector

# 创建索引
documents = [Document(text="...")]
index = VectorStoreIndex.from_documents(documents)

# 配置查询引擎（支持混合检索）
query_engine = index.as_query_engine(
    similarity_top_k=5,
    response_mode="tree_summarize"  # 针对大量上下文的总结模式
)

response = query_engine.query("项目的核心风险是什么？")
```

## 常见考点
1. **Nodes vs Documents**：为什么 LlamaIndex 引入 Node 概念？因为 Vector Store 通常针对固定大小的 Chunk 进行检索，而 Document 可能非常长。Node 允许携带更丰富的元数据（如父文档引用、关系），实现「检索到小节点，返回大上下文」。
2. **Recursive Retrieval 原理**：如何实现从粗到细的检索？通常先检索摘要或元数据，再根据结果检索对应的具体块，常用于处理大规模文档库。
3. **LangChain 与 LlamaIndex 的互操作性**：能否在 LangChain 中使用 LlamaIndex 的 Retriever？可以，LlamaIndex 提供了 `LangChainRetriever` 包装器，反之亦然。
4. **Post-processing（后处理）**：LlamaIndex 在检索后、生成前支持哪些操作？如 Rerank（重排序）、Filters（元数据过滤）和 Fusion（多路查询结果合并）。
