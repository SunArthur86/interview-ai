# 幻觉问题的工程解决方案

### 幻觉问题的工程解决方案

**核心概念**：幻觉是模型生成看似合理但事实错误的内容。需通过“事前-事中-事后”全流程治理。

**原理详解**：

1. **事前预防**：
   - **RAG (检索增强)**：引入外部知识库，要求模型“仅依据上下文回答”。
   - **结构化 Prompt**：明确指令“不知道请直接说不知道”。
   - **知识边界**：限定模型的回答领域，避免跨界瞎编。

2. **事中控制**：
   - **置信度过滤**：利用 Logprobs 或自评分数过滤低置信度回答（需校准）。
   - **工具验证**：涉及计算、事实查询时，强制调用工具或代码执行，以结果为准。

3. **事后校验**：
   - **事实核查**：生成后进行二次检索交叉验证。
   - **引用标注**：要求答案每一句话都标注来源 [doc_id]，便于人工审核。

4. **常见误区**：
   - RAG 不一定能消除幻觉：检索质量差或模型忽略上下文时仍会出错。
   - 仅靠“诚实指令”无效：必须配合外部知识约束和校验机制。

#### 边界情况与极端场景
- **检索上下文冲突**：当 RAG 检索到的多个文档片段之间存在事实冲突（例如不同时间点的政策变化），模型容易产生混淆或融合错误。需在 Prompt 中明确指示“优先采信最新日期的信息”或“列出不同观点”。
- **否定约束失效**：模型往往难以处理否定逻辑（如“不要提竞争对手名字”）。即便 Prompt 明确禁止，模型仍可能在生成复杂长文本时“遗忘”该约束。需结合关键词过滤层作为最后防线。
- **虚假引用**：模型可能会编造一个看起来很像真的文档 ID（如 `[doc:999]`）。工程上必须校验返回的 Citation ID 是否确实存在于本次检索的上下文列表中。

**实战案例**：在构建金融研报助手时，GPT-4 经常编造不存在的股票代码。我们实施了“工具强制调用”策略，凡是涉及“当前股价”或“市盈率”的查询，强制挂载 Python 解释器去调用真实 API，而不是依赖模型的参数记忆。

**代码示例**：
```python
# 强制模型引用来源并校验置信度
def hallucination_check(response, retrieved_ids, threshold=0.1):
    # 1. 置信度校验
    if response.logprobs and response.logprobs.token_logprobs:
        min_conf = min(response.logprobs.token_logprobs)
        if min_conf < threshold:
            return None  # 拒绝回答
    
    content = response.choices[0].message.content
    
    # 2. 引用完整性校验（伪代码逻辑）
    import re
    citations = re.findall(r'\[citation:(\d+)\]', content)
    for cite_id in citations:
        if cite_id not in retrieved_ids:
            return "Error: 引用了不存在的文档来源"
            
    # 3. 检查是否包含引用标记
    if not citations:
        return "Error: 缺少引用来源"
        
    return content
```

## 面试追问
1. **追问（知识时效性）**：对于实时性要求极高的场景（如体育比分、突发新闻），RAG 的数据库往往有延迟。除了实时流式写入外，还有没有其他工程手段来减少“时效性幻觉”？
2. **追问（黑盒模型）**：如果使用的是无法访问 Logprobs 的黑盒模型 API（如早期版本或某些国产模型），你如何实现“置信度过滤”或“事中控制”？
3. **追问（评估幻觉）**：如何自动化地量化评估幻觉率？单纯用 LLM-as-Judge 去判断事实性是否可靠？

## 易错点
1. **误区**：认为 RAG 只要检索到了相关文档，模型就一定能正确回答。
   **纠正**：存在“忽略上下文”现象，即模型更倾向于使用其内部训练数据而非提供的检索内容。解决方法包括在 Prompt 中加强指令（“必须仅基于以下文本”）或使用微调过的模型来增强 RAG 遵循度。
2. **误区**：过度的过滤机制会导致用户体验下降。
   **纠正**：如果置信度阈值设置过高，会导致大量合理回答被拦截，产生“复读机”或频繁报错。需要在“准确性”和“可用性”之间寻找平衡点，可以通过分析 Bad Case 动态调整阈值。
