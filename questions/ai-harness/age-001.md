# 为什么要做模型路由,而不是全量用一个最强模型

**核心原因**：最强模型往往更贵、更慢，且并非所有子任务都需要。路由能在质量、成本、延迟、可用性、合规之间做权衡；同时多供应商提高容灾能力，避免单点故障。

**深入分析**：

1.  **成本效益**
    *   **场景**：简单问答（如"法国首都"） vs 复杂推理（如"代码审计"）。
    *   **细节**：GPT-4 可能是 $0.03/1K tokens，而 Llama-3-8B 自建可能仅需 $0.0003/1K tokens。对于简单任务，小模型能力足够，路由可降低 90%+ 的成本。

2.  **延迟优化**
    *   **场景**：实时对话 vs 离线报告生成。
    *   **细节**：大模型推理通常首字延迟（TTFT）高。路由可根据 SLA 要求，将实时请求分发给参数量小、量化程度高的模型，以保证用户体验。

3.  **任务专精**
    *   **场景**：特定领域（如医学、法律、代码）。
    *   **细节**：通用最强模型（如 GPT-4）可能在特定细分领域不如微调过的 7B 模型（如 Med-PaLM）。路由器能识别任务域并分发到领域专家模型。

4.  **容错与合规**
    *   **场景**：企业私有数据。
    *   **细节**：部分数据因隐私合规不能出域，需路由到本地私有模型；非敏感数据才路由到云端最强模型。

### 路由策略分类

1.  **基于规则**：
    *   示例：如果 Prompt 包含"代码"，路由至 CodeLlama；如果包含"总结"，路由至 GPT-3.5。
    *   优缺点：简单可控，但维护成本高，无法泛化。

2.  **基于 ML 分类器**：
    *   训练一个 BERT 或 Logistic Regression 模型，预测 Query 的难度级别（Easy/Medium/Hard），然后映射到对应模型。

3.  **基于 LLM 路由**：
    *   使用一个轻量级 LLM 分析 Query 并输出 JSON 路由决策。
    *   优缺点：灵活性极高，但增加了一跳的延迟。

4.  **基于效用的强化学习**：
    *   定义奖励函数 = Accuracy - Cost*lambda。训练 Agent 学习最优路由策略。

### 实战案例
**路由模型自身的"瓶颈"效应**：在引入路由层后，路由本身成为了新的单点和性能瓶颈。如果路由逻辑（如使用 LLM 进行分类）耗时超过 200ms，则对于简单任务的整体响应延迟反而劣化。解决方案是使用极小参数模型（如 1B 参数量）或基于 Bert 的分类器进行路由，将路由延迟控制在 10-20ms 以内。

## 边界情况
1.  **路由决策失效时的兜底**：当路由器错误地将高难度任务分配给小模型时，必须设计"逃生舱"机制。例如小模型输出置信度低或结果长度异常时，系统应能自动触发重试并将请求升级至大模型。
2.  **并发洪峰**：当大模型服务达到 Rate Limit 时，路由器应能动态降级策略，临时将原本应发往大模型的流量溢出到中/小模型，避免直接报错，牺牲部分质量换取可用性。
3.  **长上下文差异**：不同模型的 Context Window 长度不一致（如 8k vs 128k）。路由时不仅需考虑难度，还需校验 Prompt 长度是否超过目标模型的承受上限，否则需进行截断或摘要。

## 面试追问
1.  **如何评估路由策略的效果？**
    *   追问点：考察是否能提出除成本之外的评估指标，如"端到端成功率"、"用户满意度（CSAT）"以及"路由准确率"（人工标注路由决策是否正确）。
2.  **如果某个大模型服务商突然宕机，路由系统如何做到秒级切换？**
    *   追问点：考察健康检查机制和故障转移策略。例如心跳检测、被动失败探测，以及是否需要保持多厂商模型的 Prompt 格式兼容层。
3.  **如何解决路由器本身带来的延迟增加？**
    *   追问点：考察工程落地能力。答案可包括：使用 BERT 等小模型替代 LLM 进行分类、本地部署路由服务、以及使用异步非阻塞架构。

## 易错点
1.  **忽略 Token 消耗的隐性成本**：有些人认为小模型便宜就随便用，但实际上小模型输出啰嗦可能导致总 Token 数比大模型更高。路由策略应综合考虑"输入成本"和"输出长度预期"。
2.  **过度依赖路由模型的"黑盒"判断**：直接让大模型自己决定"这个任务难不难"往往不稳定。更好的做法是结合用户历史行为（如用户过去对复杂度的偏好）或显式的业务标签（如"涉及金额"则强制走强模型）进行硬性干预。
