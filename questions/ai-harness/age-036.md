# Agent Drift(任务漂移)是什么?如何识别和应对

**Agent Drift** 是指 Agent 在执行长任务时偏离原始目标的现象.

**三种漂移类型:**

**1. 任务漂移:** Agent 逐渐偏离用户的原始意图.例如用户要求写排序算法,Agent 跑去优化完全无关的文件.
**2. 上下文幻觉:** Agent 在长对话中产生与事实不符的上下文.例如虚构不存在的文件路径或函数名.
**3. 注意力稀释:** 随着上下文变长,Agent 对早期指令的注意力减弱.

**识别方法:**
*   **Trace 轨迹分析:** 检查每轮工具调用是否在合理范围内,计算当前工具调用与初始指令的语义距离.
*   **上下文窗口监控:** 关注 Token 使用量增长曲线,异常增长可能意味着陷入循环或冗余思考.
*   **中间结果验证:** 检查工具返回值是否被正确使用,是否存在“无视错误继续执行”的情况.
*   **目标对齐评分:** 在每一轮推理结束前,要求模型自评当前行动与原始目标的关联度(0-1分).

**应对策略:**
1.  **定期上下文压缩:** 保留关键信息(如 System Prompt、关键决策路径),丢弃冗余的中间步骤.
2.  **任务状态快照:** 每 N 轮检查是否仍在正确轨道,引入“决策点”机制确认是否继续.
3.  **人工审批:** 关键操作(如文件删除、资金交易)需用户确认,形成“人机回环”.
4.  **工具调用约束:** 限制可用工具集,根据当前阶段动态开启/关闭特定工具权限,减少分心可能.
5.  **目标重注入:** 在长对话的中段,自动将用户的原始指令作为 System Message 强制注入一次.

- **实战案例:** 在日志分析 Agent 的开发中，曾发生过 Agent 陷入“错误归因死循环”：为了解释一个 Error，不断读取之前的日志，产生新的幻觉日志解释，导致单次对话消耗 150k Token 且无结果。后来增加了“工具调用去重”和“负面约束”检测才解决。

- **代码示例:**
```python
def check_drift(current_action, original_goal, history):
    # 1. 语义距离检查 (实际中可用 Embedding Cosine Similarity)
    similarity = semantic_similarity(current_action, original_goal)
    if similarity < 0.3:
        return True # 严重漂移
    
    # 2. 循环检查
    if len(history) > 3:
        last_3 = [h.action for h in history[-3:]]
        if len(set(last_3)) == 1:
            return True # 陷入死循环
    
    return False
```
