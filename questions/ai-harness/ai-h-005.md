# LangGraph解决了LangChain的什么问题?什么时候应该用LangGraph

- **LangChain LCEL的局限:**

LCEL是DAG(有向无环图),不支持:
1. **循环** - 无法实现「执行→检查→重试」循环
2. **条件分支** - 只能线性管道
3. **状态管理** - 无全局状态
4. **中断/恢复** - 无法暂停

- **实战案例**：构建代码审查 Agent 时，必须使用 LangGraph 实现“Linter 生成修复建议 -> 单元测试 -> (测试失败?) -> 回退到 Linter 重试”的闭环，LCEL 无法处理这种带状态反馈的循环。

- **LangGraph解决方案:StateGraph**

```python
# 定义全局状态
class State(TypedDict):
    messages: list
    current_node: str
    retry_count: int

# 构建图
graph = StateGraph(State)
graph.add_node("agent", agent_node)
graph.add_node("tools", tool_node)
graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", should_continue,
{"continue": "tools", "end": END})
graph.add_edge("tools", "agent")  # 循环!

app = graph.compile(checkpointer=MemorySaver())
```

**架构示意图:**
```
      (Start)
        |
        v
  +---------+      (条件判断)
  | Agent   | <------------------+
  +---------+                     |
        |                        |
   continue?                      |
        | Yes                     |
        v                        |
  +---------+                     |
  |  Tools  | ---------------------+
  +---------+      (Action Done)
```

- **什么时候用LangGraph:**
1. 需要循环(Agent重试/反思/多轮)
2. 需要条件分支(路由/审批)
3. 需要全局状态管理
4. 需要中断/恢复
5. 多Agent协作编排

- **LangGraph vs LangChain LCEL 对比:**

| 特性 | LangChain LCEL | LangGraph |
|------|----------------|-----------|
| 流程控制 | 线性 DAG | **Cyclic Graph (循环/分支)** |
| 状态管理 | 隐式 | **显式 StateGraph** |
| 人机协作 | 较弱 | **强大的中断/恢复支持** |
| 适用场景 | 简单链式任务 | **复杂 Agent 编排** |

- **什么时候不用:**
- 简单的prompt→model→parser → 用LCEL
- 无状态的API调用 → 用普通函数

**## 边界情况**
1. **死循环与最大步数限制**：在 Agent 自主循环中（如工具调用失败不断重试），必须配置 `recursion_limit` 或自定义终止条件，否则会导致无限循环和资源耗尽。
2. **状态冲突处理**：当多个分支同时更新 State 中的同一个字段（如 `messages`）时，如果 Reducer 定义不当，可能会导致数据覆盖或丢失，需明确 Reducer 的合并策略（如 append 或 replace）。

**## 面试追问**
1. LangGraph 的 `checkpointer` 是如何实现跨会话状态持久化的？如果后端存储挂了，有什么容灾机制？
2. 在多 Agent 协作中，如何避免不同 Agent 之间产生“死锁”或“争吵不休”的情况？
3. LangGraph 的 `subgraph`（子图）是如何工作的？它如何与父图的状态进行交互？

**## 易错点**
1. **混淆节点与 Runnable**：LangGraph 中的 Node 是一个函数 `(state) -> new_state`，而 LangChain 的 Runnable 是对象。虽然 Node 内部可以调用 Runnable，但两者概念不同。
2. **状态传递误解**：在 `add_conditional_edges` 中，条件函数是基于当前的 State 做判断，而不是基于 Node 的返回值（除非 Node 的返回值直接覆盖了 State 的关键字段）。

---

## 常见考点
1. **State 的传递机制**：LangGraph 中的每个 Node 都接收完整的 State 对象，并返回 State 的更新部分（Partial Update）。Reducer 函数（如 `add_messages`）如何处理状态更新冲突（如多个节点同时更新 messages 列表）？
2. **CheckpointSaver 的作用**：`MemorySaver` 如何实现 Human-in-the-Loop？它将每一步的状态快照持久化（存到 Redis 或内存），允许用户通过 `thread_id` 恢复之前的对话状态并在特定节点暂停。
3. **Node 的输入输出约定**：Node 函数的签名通常是什么？`(state: State) -> Dict[str, Any]`。返回的字典会自动合并到全局 State 中。
4. **LangGraph vs State Machine**：LangGraph 本质上是一个 Pregel-inspired 的状态机实现
