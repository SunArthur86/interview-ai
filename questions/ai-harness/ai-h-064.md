# 设计一个生产级多模型网关:如何实现模型路由、熔断和自动降级

- **多模型网关架构**

```
用户请求
→ 路由决策(基于 query 复杂度/成本/延迟)
→ 模型调用(GPT-4 / Claude / GLM / 本地 7B)
→ 熔断检查(如果模型连续失败 → 降级)
→ 降级链(大模型 → 小模型 → 缓存 → 兜底回复)
→ 响应
```

- **实战案例**：某次 OpenAI API 宕机期间，我们的网关层检测到连续超时错误，自动将流量切换至本地部署的 Llama-2-70B，虽然响应稍慢，但保证了核心业务完全中断，SLA 合规率维持在 99.9%。

- **1. 模型路由策略**
- **规则路由**:代码问题 → Code模型,简单FAQ → 小模型
- **LLM 路由**:用轻量模型(如 BERT)分类 query 难度
- **成本路由**:Token 预算 < 500 → 小模型,> 2000 → 大模型
- **延迟路由**:SLA 要求 < 1s → 本地模型

- **对比表格：降级策略设计**
| Level | 处理逻辑 | 可靠性 | 成本 | 体验 |
|------|---------|-------|------|------|
| L1 (异构Fallback) | GPT-4 → Claude | 高 | 高 | 几乎无变化 |
| L2 (云端降本地) | Claude → 本地 7B | 高 | 低 | 能力可能下降 |
| L3 (缓存兜底) | 本地 → 语义缓存 | 中 | 最低 | 极速但可能旧 |
| L4 (静态回复) | 缓存 → 固定文案 | 低 | 0 | 业务降级，体验差 |

- **2. 熔断器设计(三态)**
- **Closed(正常)**:请求正常通过
- **Open(熔断)**:连续失败超阈值 → 直接拒绝(快速失败)
- **Half-Open(探测)**:放少量请求试探是否恢复

```python
class CircuitBreaker:
    def __init__(self, threshold=5, timeout=60):
        self.failures = 0
        self.state = 'closed'

    async def call(self, model, prompt):
        if self.state == 'open':
            return await self.fallback(prompt)
        try:
            result = await model.generate(prompt)
            self.failures = 0  # 重置
            return result
        except Exception:
            self.failures += 1
            if self.failures >= self.threshold:
                self.state = 'open'
                schedule_reset()  # 60s 后转 half-open
            return await self.fallback(prompt)
```

- **3. 降级链**
- Level 1: GPT-4 → Claude(不同厂商 fallback)
- Level 2: Claude → 本地 7B 模型(云 → 本地)
- Level 3: 本地模型 → 语义缓存(历史相似回答)
- Level 4: 缓存 → 固定兜底回复('服务繁忙,请稍后重试')

- **4. 监控**
- 每个模型的 QPS / 延迟 / 成功率 / 成本
- 路由分布(多少去了大模型 vs 小模型)
- 熔断触发频率

- **## 常见考点**
1. **路由算法**: 如何实现低延迟的路由决策？（在网关本地维护规则引擎或轻量级模型，避免远程调用）
2. **降级一致性**: 当从 GPT-4 降级到小模型时，如何保证 Prompt 格式兼容？（网关层做 Prompt 转换和模板适配）
3. **幂等性**: 如果客户端超时重试，网关如何避免重复扣费或重复调用？（使用全局唯一 Request ID 进行去重处理）

- **边界情况**
- **降级雪崩**：当主模型宕机触发熔断，流量全部打到备用模型（如本地 7B）时，可能导致备用模型因负载过高也崩溃。解决方案是在备用模型前也进行限流，或引入多级备胎。
- **状态同步**：如果主模型（如 GPT-4）是有状态的（如涉及 Session 记忆），降级到无状态的本地模型时，上下文信息可能丢失。网关需负责提取并注入必要的 Context。
- **故障恢复后的回流**：当 GPT-4 恢复正常后，是立即切回还是逐步切回？（通常采用「金丝雀发布」策略，先切 5% 流量验证，防止故障震荡）。

- **## 面试追问**
1. 如何实现细粒度的限流？例如不仅要限制总 QPS，还要限制每个用户的 Token 消耗速率？（需要实现 Token Bucket 算法并结合 Tokenizer 预估输入长度）
2. 在多租户场景下，如何保证租户 A 的高流量不会挤占租户 B 的模型请求资源？（需实现基于权重的公平调度或租户隔离队列）
3. 网关自身如何保证高可用？如果网关变成了单点瓶颈（SPOF），架构上如何解耦？（采用无状态设计 + 多副本部署 + Load Balancer）

- **## 易错点**
1. **熔断状态泄露**：忘记实现 Half-Open 到 Closed 的自动恢复逻辑，导致服务恢复后熔断器一直开启，人为制造了服务不可用。
2. **混淆重试与熔断**：对于非幂等操作（如写操作），降级时的重试可能会导致副作用。网关需区分「可重试异常」（如超时）和「不可重试异常」（如 4xx 参数错误）。
