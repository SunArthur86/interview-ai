# Prefill-Decode 分离(PD Separation)是什么?为什么大规模推理服务必须做

- **PD 分离** 将推理的 Prefill(首 token 计算)和 Decode(逐 token 生成)分配到不同的 GPU 集群.

- **实战案例**：在双十一大促期间，我们测试发现如果不做 PD 分离，长 Prompt 导致的排队阻塞会让 P99 延迟飙升至 5s+；分离后，Prefill 集群快速消化计算压力，Decode 集群维持低延迟流式输出，用户体验极大改善。

- **为什么需要分离?**

Prefill 和 Decode 的计算特征完全不同:

| 维度 | Prefill | Decode |
|------|---------|--------|
| 计算特征 | Compute-bound | Memory-bound |
| Batch 友好 | ✅ 可并行 | ❌ 难以批量 |
| GPU 利用率 | 高(大量矩阵乘) | 低(逐 token) |
| 延迟来源 | 模型计算 | KV Cache 读取 |

- **分离架构**

```
用户请求 → Prefill 集群(高算力 GPU)
↓ 计算 KV Cache
↓ 传给 Decode 集群
Decode 集群(大显存 GPU)
↓ 逐 token 生成
↓ 流式返回
```

- **对比表格：架构选型对比**
| 特性 | 单集群架构 | PD 分离架构 |
|------|-----------|------------|
| 硬件利用率 | 受限于两种阶段的折中 | 各自最优硬件配置 |
| 长 Prompt 延迟 | 受排队影响大 | 受 Decode 侧影响小 |
| 工程复杂度 | 低 | 高 (需跨节点通信) |
| 适用场景 | 中低并发 | 高并发 / 长文本 |

- **优势**
1. Prefill 集群用 H100(高 FLOPS)→ 快速算完 prompt
2. Decode 集群用 A100 80GB(大显存)→ 支撑更多并发
3. 两个集群独立扩缩容
4. KV Cache 通过高速网络(RDMA)传输

- **挑战**
1. KV Cache 传输延迟(需要 RDMA/InfiniBand)
2. 负载均衡更复杂
3. 故障恢复需要处理跨集群状态

- **实际案例**
- DeepSeek:PD 分离 + 每日处理千亿 token
- Google:GBoard 的 LLM 服务
- 大厂趋势:PD 分离是大规模推理的必选项

- **边界情况补充**：
1. **首字抖动**：跨节点传输 KV Cache 会引入额外的网络 RTT，可能导致 TTFT（Time To First Token）出现偶发尖峰，需监控网络 P99 延迟。
2. **Decode 请求饥饿**：如果流量突发导致 Prefill 压力过大，可能会导致新请求的 KV Cache 生成过慢，进而阻塞 Decode 集群请求进入，需做背压机制。
3. **多轮对话**：同一 Session 的多轮对话，必须保证路由到同一个 Decode 节点以利用 Local KV Cache，否则需频繁跨节点传输历史 KV，代价极高。
4. **弹性伸缩滞后**：Decode 集群扩容通常涉及显存状态的迁移或预热，无法像无状态 Web 服务那样秒级扩容。

- **## 常见考点**
1. **资源瓶颈**: 为什么说 Decode 阶段是 Memory-bound？（因为每个 Token 生成都需要读取完整的历史 KV Cache，显存带宽成为瓶颈，而非计算能力）
2. ** speculative decoding**: 推测采样技术通常在哪个阶段使用？（Decode 阶段，用小模型提前生成，大模型并行验证）
3. **架构选型**: 如果网络带宽不足，PD 分离还能带来性能提升吗？（不能，KV Cache 传输开销可能抵消计算收益）

- ## 易错点
1. **认为分离一定能省钱**：PD 分离主要是为了提升吞吐和稳定性，由于引入了双倍维护成本和网络开销，对于低并发短文本场景，成本反而可能更高。
2. **忽视耦合性**：认为 Prefill 和 Decode 可以完全解耦，实际上它们在 KV Cache 格式、采样参数上必须严格兼容，升级模型版本需同步升级两个集群。

- ## 面试追问
1. 在 PD 分离架构下，如何处理 Decode 节点宕机导致的正在生成的请求丢失？
2. 除了 Prefill/Decode 分离，还有哪些细粒度的算子分离策略（如 Attention 与 MLP 分离）？
3. 如何设计调度算法以防止长 Prompt 任务“饿死”短文本任务？
