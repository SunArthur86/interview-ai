# 部署与运维

### 性能优化

**核心目标**：解决 Agent 系统中的串行 LLM 调用、高延迟网络 IO 和上下文膨胀问题。

#### 原理详解

1.  **异步处理**
    *   使用 `asyncio` 或消息队列，避免 I/O 等待阻塞线程。
    *   **并行工具调用**：无依赖的工具调用并发执行，大幅降低总耗时。

2.  **流式输出**
    *   利用 SSE (Server-Sent Events) 边生成边返回，降低首字延迟 (TTFB)。
    *   **细节**：需处理流式中断情况，保证最终状态一致性。

3.  **并发控制**
    *   使用 `Semaphore` 限制对同一模型的并发请求数，防止触发 429 (Rate Limit)。

4.  **连接复用**
    *   HTTP 客户端开启 `keep-alive`，数据库使用连接池，减少握手开销。

5.  **缓存分层**
    *   **L1 (进程内)**：LRU 缓存，极低延迟。
    *   **L2 (Redis)**：分布式共享，支持 TTL。
    *   **读路径**：L1 -> L2 -> 回源模型。

#### 边界情况补充

*   **冷启动与 JIT 编译**：在 Serverless 或容器缩容场景下，新实例启动时可能没有 GPU 预热或模型加载时间，导致首请求极慢（可达数秒）。需通过“预热探针”发送空请求来初始化模型加载。
*   **流式输出的 Token 延迟累积**：在 Agent 编排复杂（如多 Agent 对话）时，流式输出如果在每层都做完整处理再转发，会累积显著延迟。需要实现“透传流式”，即上游一产生 Token 立即推送到下游，而非等待整段生成完毕。
*   **死锁与超时级联**：当某个工具服务挂掉时，如果未设置合理的超时时间，会导致 Agent 协程永久挂起，进而耗尽连接池。必须为所有外部调用设置熔断和超时机制。

#### 并行调用架构图

```text
LLM generates Plan: [Tool A, Tool B, Tool C]
      │
      ├─ Tool A (Search) ──────┐
      │                        │
      ├─ Tool B (Calc) ────────┤──> Gather Results ──> Final LLM Synthesis
      │                        │
      └─ Tool C (Lookup) ──────┘

(Serial Time = A + B + C)
(Parallel Time = Max(A, B, C))
```

#### ## 常见考点
1.  **流式传输中的 JSON 解析**：流式输出 JSON 时可能被截断，如何保证前端能正确解析？（提示：使用缓冲区或特定的流式 JSON 解析库，确保字段完整性）。
2.  **并发限制策略**：当面对不同 LLM 厂商（如 OpenAI, Anthropic）不同的 RPM/TPM 限制时，如何设计限流器？（提示：令牌桶算法，按租户或 API Key 分隔）。
3.  **冷启动优化**：如何减少模型首次加载或容器冷启动带来的延迟？（提示：保持连接预热、使用预留实例/GPU）。
4.  **Asyncio 死锁**：在复杂的 Agent 编排中，如何避免异步任务中的死锁或资源竞争？

## 面试追问
1.  **优先级队列**：当系统负载极高，Token 生成速率受限于供应商限制时，如何设计调度策略以保证“付费用户”或“高优先级任务”的请求不被饿死？（提示：实现多级优先级队列，而非简单的 FIFO）。
2.  **流式中断补偿**：如果用户在流式输出中途关闭了连接（WebSocket 断开），后端如何感知并立即停止 LLM 的生成操作以节省 Token 成本？（提示：利用连接关闭事件回调，调用 LLM 提供的 `cancel` 接口或直接中断流读取）。
3.  **Burst 流量应对**：面对突发流量（如营销活动），除了水平扩容，在应用层有哪些快速手段防止系统崩溃？（提示：开启快速失败模式，降级非核心模型推理功能，返回缓存的历史结果）。

## 易错点
1.  **伪并发**：代码虽然使用了 `asyncio`，但在等待多个工具结果时，错误地使用了串行的 `await`（如 `res1 = await tool_a(); res2 = await tool_b()`），而非 `await asyncio.gather()`，导致性能未能提升。
2.  **忽略 OpenAI SDK 的异步版本**：直接在异步环境中使用同步的 OpenAI SDK（如 `openai.ChatCompletion.create`），这会阻塞整个事件循环，导致并发能力急剧下降。必须使用 `openai.AsyncOpenAI`。
