# 全链路可观测性

### 全链路可观测性

**核心概念**：Agent 路径动态且复杂，可观测性不仅是记录“错了”，更要还原“错在哪一步、哪个工具、哪个模型”。

**原理详解**：

1. **结构化日志**：
   - 使用 JSON 格式，包含关键字段：`trace_id`、`span_id`、`timestamp`、`level`、`model`、`token_usage`、`latency_ms`。
   - 避免：记录不可解析的自然语言段落。

2. **链路追踪**：
   - **Trace**：一次用户请求的全生命周期。
   - **Span**：基本单元（一次 LLM 调用、一次工具执行），通过 `parent_span_id` 构成调用树。
   - 能还原 Agent 的思考链、并行分支和耗时瓶颈。

3. **关键步骤记录**：
   - 记录每一步的 `thought`（思考摘要）、`tool_input`（参数）、`observation`（结果摘要）、`retry_count`。

4. **工具集成**：
   - **LangSmith**：与 LangChain 深度集成，便于调试和 Prompt 版本对比。
   - **LangFuse**：开源，侧重成本、质量分析。

5. **实现思路**：
   - 使用 `contextvars` 传递 Trace ID。
   - 集成 OpenTelemetry 协议导出数据。

#### 边界情况补充

*   **流式输出的 Trace 关联**：在流式响应场景下，如果发生异常或超时，确保 Trace 能够正确记录“部分成功”或“流中断”的状态，避免 Trace 状态丢失或仅记录到超时而未记录已产生的 Token 消耗。
*   **异步/并行任务的上下文传递**：Agent 往往会并行调用工具。在异步环境中（如 Python asyncio），必须确保 TraceID 能够自动传播到子线程/协程中，否则日志会断裂，无法还原完整的调用树。
*   **敏感信息过滤**：Agent 可能会处理包含 PII（个人隐私信息）的数据。在将 Prompt 和 Tool Input 写入可观测性平台（如 LangSmith 或自建日志库）前，必须配置动态脱敏策略，避免违反合规要求。

#### 实战案例
在某金融 Agent 调试中，系统偶尔返回金额错误。通过追踪 `trace_id` 发现是 Agent 调用了“汇率查询”工具，但因超时未正确处理 `observation`，导致 Agent 陷入幻觉自行编造了汇率值，最终在日志中定位到异常的 Span 根因。

#### 代码示例
```python
from contextvars import ContextVar
import time
import uuid

# 用于在协程间传递 Trace ID
TRACE_ID: ContextVar[str] = ContextVar('trace_id')

def log_llm_span(prompt: str, model: str):
    trace_id = TRACE_ID.get()
    span_id = str(uuid.uuid4())
    start_time = time.time()
    
    # 模拟 LLM 调用
    # completion = client.chat.completions.create(...)
    
    duration = time.time() - start_time
    
    # 结构化日志输出 (实际工程应发送至 OTLP Collector)
    log_entry = {
        "trace_id": trace_id,
        "span_id": span_id,
        "event_type": "llm_call",
        "model": model,
        "prompt_len": len(prompt),
        "latency_ms": round(duration * 1000, 2),
        "timestamp": time.time()
    }
    print(log_entry) # Replace with logger.info(json.dumps(log_entry))
```

## 面试追问
1.  **Trace ID 透传**：在微服务架构下，如果 Agent 调用的工具是一个独立的 HTTP 服务，如何保证 TraceID 能够透传到下游服务，以便在日志系统中聚合全链路日志？（提示：使用 HTTP Headers 如 `traceparent` 或自定义 Header 传递）。
2.  **采样策略**：高并发场景下，全量记录 Trace 成本过高。如何设计采样策略？例如，对于错误请求或高延迟请求是否应该 100% 采样？（提示：概率采样 + 错误/慢请求动态兜底采样）。
3.  **非结构化日志分析**：如果现有的 Agent 系统产生的日志是半结构化的（如包含 Thought 的自然语言），如何利用 LLM 对这些历史日志进行自动化的根因分析？（提示：利用 LLM 批量分析 Trace 日志，提取常见失败模式）。

## 易错点
1.  **混淆 Logging 与 Tracing**：只打印普通的文本日志而没有建立 Span 之间的父子关系，导致当并发量大时，无法分辨哪次 LLM 调用属于哪个用户请求，造成“日志孤岛”。
2.  **忽略 Input/Output 的体量**：记录 Prompt 和 Response 的完整内容对 Debug 极其有用，但这会极大增加存储成本和网络带宽消耗。常见错误是不加区分地全量记录，导致可观测性系统本身成为瓶颈。应策略性地记录（如：只记录 Error 级别的完整 Payload，Success 级别只记录 Meta）。

## 记忆要点

- 可观测性核心：Trace 记录全生命周期，Span 记录单步调用（LLM/工具）。
- 日志结构化：JSON 格式，包含 trace_id、token_usage、latency_ms。
- 工具集成：LangSmith 强调试，LangFuse 重成本分析，自建用 OTLP。
- 关键点：异步环境需透传 Trace ID，防止日志链路断裂。
- 边界：流式输出需记录部分成功状态，敏感信息需脱敏后再记录。
