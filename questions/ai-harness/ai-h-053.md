# Cursor 的核心技术是什么?它和 GitHub Copilot 有什么区别

- **Cursor** 是基于 VS Code fork 的 AI 原生 IDE,核心技术包括:

- **1. 全仓库上下文理解:**
- Codebase Indexing:将整个代码仓库索引为向量
- 检索时用 RAG 找到相关代码片段
- 支持 @file、@function、@web 精确引用

- **2. Agent 模式:**
- 类似 Claude Code 的 Agent Loop
- 可以读写文件、执行命令、运行测试
- 支持多文件编辑

- **3. Inline 补全 + Chat:**
- Tab 补全(类似 Copilot)
- Cmd+K 行内编辑
- 侧边栏 Chat 带全仓库上下文

---

### 实战案例
在重构一个旧项目的 API 结构时，我使用了 Cursor 的 **Composer (Agent)** 功能。我只输入了“将所有 REST API 迁移到 GraphQL”，它检索了整个后端目录，自动创建了 Schema 定义，并逐一修改了 10+ 个 Controller 文件。虽然第一次生成了个别类型不匹配的错误，但我只需在 Diff 视图中选中错误部分并输入“Fix type error”，它便自动完成了后续的修正。

### Cursor vs GitHub Copilot 详细对比

| 维度 | Cursor | GitHub Copilot |
| :--- | :--- | :--- |
| **上下文范围** | **全仓库语义索引** (Codebase Index) | **局部** (当前文件 + 少量引用文件) |
| **核心交互** | Chat (Cmd+L) + Inline Edit (Cmd+K) + **Agent (Composer)** | Inline (Tab 补全) + Chat (Copilot Chat) |
| **多文件编辑** | **原生支持** (跨文件引用、批量修改) | 较弱 (主要依赖插件或手动操作) |
| **模型选择** | **灵活** (Claude 3.5 Sonnet, GPT-4o, 自定义 Local) | **固定** (OpenAI GPT-4/GPT-3.5) |
| **隐私性** | 支持纯本地模型 (隐私模式) | 默认需发送代码至 GitHub/Microsoft |
| **Agent 能力** | **强** (内置终端执行、自愈、多步规划) | **弱** (主要是建议，较少自动操作文件系统) |

Cursor 的核心优势是 Agent 级别的全仓库编辑能力.

## 记忆要点

- 核心差异：Cursor是全仓库语义索引+Agent模式，Copilot是局部补全。
- Cursor技术：Codebase Indexing(向量化)、Agent(多文件读写/自愈)、灵活模型。
- 交互方式：Tab补全、Cmd+K行内编辑、Cmd+L全库Chat、Composer批量重构。
- 优势场景：跨文件重构、复杂逻辑迁移，Copilot适合单文件片段补全。
