# 如何选择LLM应用框架?LangChain vs LlamaIndex vs DSPy vs 原生API

- **选型决策矩阵:**

```
你的需求是什么?
├─ 简单API调用 → 原生API/OpenAI SDK
├─ RAG系统
│   ├─ 快速原型 → LlamaIndex
│   └─ 生产级 → Haystack/LangChain
├─ Agent
│   ├─ 单Agent工具调用 → OpenAI Assistants API
│   ├─ 自定义Agent循环 → LangChain
│   └─ 复杂状态管理 → LangGraph
├─ Prompt优化 → DSPy
├─ 结构化输出 → Instructor/Outlines
└─ 全栈应用 → LangChain生态
```

- **实战案例**：在一个金融合规项目初期，我们使用 LangChain 快速验证了 Agent 思路；但在上线前，由于 LangChain 的稳定性问题和链路追踪困难，我们将核心检索逻辑迁移到 LlamaIndex 以利用其高级查询优化，同时将整体编排迁移到 Haystack 以满足企业级日志和监控要求。

- **详细对比:**

| 维度 | 原生API | LangChain | LlamaIndex | DSPy | Haystack |
|------|---------|-----------|------------|------|----------|
| 学习成本 | **最低** | 高 | 中 | 高 | 中 |
| 灵活性 | **最高** | 中 | 中 | 中 | 中 |
| 抽象层级 | 无 | 高 | 中 | 高 | 高 |
| RAG能力 | 需自建 | 强 | **最强** | 弱 | 强 |
| Agent能力 | 需自建 | **强** | 中 | 弱 | 中 |
| Prompt优化 | 需自建 | 弱 | 弱 | **最强** | 弱 |
| 生产就绪 | **是** | 中 | 中 | 低 | **高** |

- **个人建议:**
- 快速验证:原生API + 几个辅助函数
- 中等复杂度:LangChain
- RAG为主:LlamaIndex
- 追求效果:DSPy
- 追求稳定:Haystack

## 常见考点
1. **Vendor Lock-in（厂商锁定）**：如果过度依赖 LangChain/LangGraph 等重框架，未来迁移底层模型（如从 OpenAI 切到 Llama 3）或框架本身的难度如何？回答：抽象层越高，切换成本通常越低（只需更换底层 LLM Runnable），但框架本身的 API 变动（如 LangChain v0.1 -> v0.2 迁移）可能带来维护成本。
2. **原生 API 的适用边界**：所谓「简单 API 调用」具体指什么？通常指单轮对话、简单的 System Prompt + User Prompt 组合。一旦涉及 Prompt Template 管理、多轮对话历史维护、工具调用等，手写原生 API 的代码量会指数级上升，且难以维护。
3. **框架的「黑盒」风险**：使用 LangChain 或 LlamaIndex 时，开发者可能不清楚底层的 Prompt 拼接逻辑或 Token 计算方式。面试时强调：生产环境需要深入阅读源码或使用 LangSmith 等观测工具，避免因为框架的默认行为导致不可控的成本或效果问题。
4. **DSPy 的冷启动问题**：DSPy 需要训练集来优化 Prompt。在冷启动（没有数据）阶段，如何选择？通常先用 LangChain 或原生 API 快速跑通流程，收集日志数据后，再转用 DSPy 进行效果优化。
