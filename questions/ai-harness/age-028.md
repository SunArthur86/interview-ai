# Agent系统的金丝雀发布需要关注哪些指标?蓝绿和金丝雀如何取舍

- **金丝雀发布关键指标:**

| 指标类型 | 具体指标 | 阈值 |
|---------|---------|------|
| **质量** | 任务成功率 | >95% |
| **质量** | 幻觉率 | <5% |
| **延迟** | P50/P99延迟 | <旧版本±10% |
| **成本** | 每请求Token消耗 | <预算 |
| **安全** | Prompt注入触发率 | <1% |
| **用户** | 负反馈率(踩) | <5% |

- **蓝绿 vs 金丝雀:**

| | 蓝绿 | 金丝雀 |
|--|------|--------|
| 切换速度 | **快(秒级)** | 慢(渐进) |
| 风险 | 高(全量切换) | **低(小流量)** |
| 回滚 | **快(切回)** | 需停止新版本 |
| 资源 | 2x(两套环境) | 1x+少量额外 |
| 适合 | 低风险变更 | **高风险变更** |

- **Agent长任务的部署不中断:**
1. 滚动部署:新实例逐个上线
2. 旧实例等正在执行的任务完成再下线
3. Session粘性:用户请求保持到同一实例

- **金丝雀流量架构图:**

```text
        用户请求
           │
           ▼
    ┌─────────────┐
    │  流量网关   │
    └──────┬──────┘
           │
      ┌────┴─────┐
      │  权重分流 │
      └────┬─────┘
           │
     ┌─────┴──────┐
     ▼            ▼
┌─────────┐   ┌─────────┐
│ 旧版本  │   │ 新版本  │
│ (95%)   │   │ (5%)    │
└─────────┘   └─────────┘
```

- **## 常见考点:**
1. **灰度策略**: 如何根据用户ID或Session ID进行灰度，确保同一用户体验一致？
2. **自动化回滚**: 当检测到P99延迟突增或错误率超过阈值时，如何实现秒级自动回滚？
3. **数据一致性**: 蓝绿部署切换时，数据库Schema变更或缓存预热如何处理？
4. **监控延迟**: 指标采集本身是否有延迟？如何保证金丝雀问题在扩大前被发现（Prometheus scrape interval）？

- **实战案例：** 在发布Agent新工具调用能力时，金丝雀1%流量导致特定领域（如金融术语）的工具参数错误率飙升，由于配置了Prometheus Alert监控Tool.ErrorRate，系统在3分钟内自动切流，避免了全量事故。

- **代码示例（Istio流量配置）：**
```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: agent-service
spec:
  http:
  - match:
    - headers:
        x-canary:
          exact: "true"
    route:
    - destination:
        host: agent-service
        subset: v2  # 新版本
  - route:
    - destination:
        host: agent-service
        subset: v1  # 默认旧版本
      weight: 95
    - destination:
        host: agent-service
        subset: v2
      weight: 5
```
