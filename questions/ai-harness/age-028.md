# Agent系统的金丝雀发布需要关注哪些指标?蓝绿和金丝雀如何取舍

- **金丝雀发布关键指标:**

| 指标类型 | 具体指标 | 阈值 |
|---------|---------|------|
| **质量** | 任务成功率 | >95% |
| **质量** | 幻觉率 | <5% |
| **延迟** | P50/P99延迟 | <旧版本±10% |
| **成本** | 每请求Token消耗 | <预算 |
| **安全** | Prompt注入触发率 | <1% |
| **用户** | 负反馈率(踩) | <5% |

- **蓝绿 vs 金丝雀:**

| | 蓝绿 | 金丝雀 |
|--|------|--------|
| 切换速度 | **快(秒级)** | 慢(渐进) |
| 风险 | 高(全量切换) | **低(小流量)** |
| 回滚 | **快(切回)** | 需停止新版本 |
| 资源 | 2x(两套环境) | 1x+少量额外 |
| 适合 | 低风险变更 | **高风险变更** |

- **Agent长任务的部署不中断:**
1. 滚动部署:新实例逐个上线
2. 旧实例等正在执行的任务完成再下线
3. Session粘性:用户请求保持到同一实例

- **金丝雀流量架构图:**

```text
        用户请求
           │
           ▼
    ┌─────────────┐
    │  流量网关   │
    └──────┬──────┘
           │
      ┌────┴─────┐
      │  权重分流 │
      └────┬─────┘
           │
     ┌─────┴──────┐
     ▼            ▼
┌─────────┐   ┌─────────┐
│ 旧版本  │   │ 新版本  │
│ (95%)   │   │ (5%)    │
└─────────┘   └─────────┘
```

- **## 常见考点:**
1. **灰度策略**: 如何根据用户ID或Session ID进行灰度，确保同一用户体验一致？
2. **自动化回滚**: 当检测到P99延迟突增或错误率超过阈值时，如何实现秒级自动回滚？
3. **数据一致性**: 蓝绿部署切换时，数据库Schema变更或缓存预热如何处理？
4. **监控延迟**: 指标采集本身是否有延迟？如何保证金丝雀问题在扩大前被发现（Prometheus scrape interval）？

- **实战案例：** 在发布Agent新工具调用能力时，金丝雀1%流量导致特定领域（如金融术语）的工具参数错误率飙升，由于配置了Prometheus Alert监控Tool.ErrorRate，系统在3分钟内自动切流，避免了全量事故。

- **代码示例（Istio流量配置）：**
```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: agent-service
spec:
  http:
  - match:
    - headers:
        x-canary:
          exact: "true"
    route:
    - destination:
        host: agent-service
        subset: v2  # 新版本
  - route:
    - destination:
        host: agent-service
        subset: v1  # 默认旧版本
      weight: 95
    - destination:
        host: agent-service
        subset: v2
      weight: 5
```

- **## 边界情况:**
1. **冷启动效应**: 新版本刚上线时，连接池、JIT编译或缓存未热身可能导致首包延迟（P99）瞬间飙升，需在金丝雀分析时区分“性能问题”与“冷启动开销”，通常设置预热期。
2. **数据漂移**: LLM对特定Prompt的输出概率分布可能随时间变化，或上游知识库更新，导致新旧版本在相同输入下表现差异，需监控Input Hash分布的一致性。
3. **依赖服务洪峰**: 蓝绿部署瞬间全量切换可能对下游数据库或向量库造成连接数洪峰，金丝雀需关注下游服务的错误率是否随流量比例线性增长。

- **## 面试追问:**
1. 如果新版本的Token成本高出10%，但用户满意度显著提升，你如何设定自动化回滚的止损策略？
2. 在多Agent协作场景下，如果只升级了Planner Agent的金丝雀版本，而Executor Agent保持旧版，如何处理版本不兼容导致的协议解析失败？
3. 对于长轮询或流式响应的Agent，金丝雀发布期间如何保证连接不中断且监控指标准确归因？

- **## 易错点:**
1. **平均值陷阱**: 仅关注平均延迟或平均成功率，而忽略了长尾请求（P99/P999）的极端失败情况，这在Agent系统中尤为致命，因为少数用户的糟糕体验可能掩盖在整体高成功率中。
2. **流量放大的误判**: 在Agent系统中，一次用户请求可能触发多次工具调用或内部LLM请求，直接将“用户请求数”作为金丝雀流量控制指标，可能导致底层服务承受数倍压力，应使用“后端实际调用次数”进行限流。
