# Agent 服务的蓝绿部署和金丝雀部署如何选择?Agent 长任务怎么做部署不中断

- **部署策略对比**

| 维度 | 蓝绿部署 | 金丝雀部署 |
|------|----------|------------|
| 流量切换 | 瞬时全量切换 | 渐进式 (5% -> 100%) |
| 回滚速度 | 极快 (切回流量) | 较快 (需逐步调整权重) |
| 资源消耗 | 高 (2倍全量资源) | 低 (只需扩容少量副本) |
| 风险控制 | 低 (全量验证后切) | 中 (需实时监控) |
| 适用场景 | 紧急修复, 短任务 Agent | 常规迭代, 验证 Prompt 效果 |

- **蓝绿部署**
- 两套环境(蓝/绿),切换流量
- 优势:秒级回滚
- 劣势:双倍资源
- 适合:短期请求(<30s)

- **金丝雀部署**
- 新版本只放 5% 流量 → 观察 → 逐步扩大
- 优势:低风险,渐进
- 劣势:需要精细的监控指标
- 适合:迭代频繁

- **Agent 的特殊挑战:长任务**

Agent 任务可能运行几分钟到几小时(如多步研究、代码生成)

- **实战案例**：某数据分析 Agent 执行长 SQL 查询耗时 10 分钟。在发布新版本时，旧 Pod 被直接 Kill，导致用户查询中断且状态丢失，最终需人工介入恢复。改进后采用了 Drain 模式 + Checkpoint 恢复机制。

- **长任务部署不中断方案**

1. **Drain 模式**
- 旧版本标记为 'draining'
- 不接受新请求,但继续处理进行中的
- 新请求路由到新版本
- 等所有旧请求完成 → 关闭旧版本

2. **Checkpoint 迁移**
- 每步保存 Agent 状态(checkpoint)
- 旧版本 drain 时,进行中的任务保存 checkpoint
- 新版本从 checkpoint 恢复继续
- 需要 Agent 架构支持状态序列化

3. **版本亲和**
- 新请求绑定到新版本
- 旧请求继续用旧版本直到完成
- 通过 sticky session 实现

- **金丝雀关注指标**
- 任务完成率(不能下降)
- 平均步数(不能增加太多)
- Token 消耗(不能暴增)
- 幻觉率(人工抽检)
- 用户满意度(点赞率)

- **代码示例**：
```yaml
# Kubernetes: PreStop Hook 配置实现优雅停止
apiVersion: apps/v1
kind: Deployment
metadata:
  name: agent-service
spec:
  template:
    spec:
      containers:
      - name: agent
        image: agent:v2
        lifecycle:
          preStop:
            exec:
              # 调用内部接口进入 Draining 模式，并休眠等待长任务结束
              command: ["/bin/sh", "-c", "curl -X POST http://localhost:8080/admin/drain && sleep 300"]
```

- **推荐**
- 短任务 Agent:金丝雀(5%→20%→100%)
- 长任务 Agent:Drain + Checkpoint 迁移
- 紧急回滚:蓝绿

- **Agent 版本管理**
- Prompt 版本化:每次修改记录 diff
- 模型版本绑定:prompt_v3 + GPT-4-turbo-2024-04
- 工具版本绑定:tool_v2 + 新版本 schema
- 组合签名:`agent:prompt_v3+model_gpt4+tools_v2`

- **补充：有状态服务的迁移细节**
- **状态序列化**：Agent 的 Memory（对话历史、工具返回结果）需要存入外部存储（如 Redis），而非仅保存在 Pod 内存中，以便 Checkpoint 恢复。
- **优雅关闭**：Kubernetes `preStop` Hook 需预留足够时间（如 Task 平均时长的 P99）让长任务完成或迁移。

## 常见考点
1. **如何验证新版本 Agent 的正确性？**
   - 使用“对抗性数据集”进行影子测试，对比新旧版本输出差异，不直接响应用户。
2. **多步任务中间状态出错如何回滚？**
   - 设计事务性 Agent，每步原子操作，失败时回滚到上一个 Checkpoint 或直接告知用户失败。
3. **金丝雀发布时的 Session Sticky 如何实现？**
   - 通过网关（如 Nginx/Istio）基于 Cookie 或 Header 中的 `session_id` 进行一致性哈希路由。

## 记忆要点

- 策略选择：短任务用金丝雀（渐进式验证），长任务用蓝绿或 Drain 模式保状态。
- 长任务挑战：直接 Kill Pod 会导致运行中任务中断，需优雅停止。
- 不中断方案：Drain 模式（旧请求做完不接新）+ Checkpoint（状态序列化恢复）。
- 版本亲和：同一 Session 请求路由到同一版本，避免状态混乱。
- 监控重点：金丝雀期间关注任务完成率和 Token 消耗，防止 Prompt 恶化导致成本暴增。
