# Prompt Engineering 基础

### Prompt Engineering 基础

**概念解释**：
Prompt Engineering 是通过设计和优化输入给 LLM 的文本（及参数），以引导模型稳定输出高质量结果的技术。它不是简单的“写咒语”，而是将业务逻辑翻译成模型能理解的语言。

**Prompt 的基本结构**：
一个完整的 Prompt 通常包含以下五部分：
1. **角色**：设定身份（如“你是一名资深Python工程师”）。
2. **任务**：明确要做什么（如“分类以下文本的情感”）。
3. **上下文**：提供背景信息、数据或检索结果。
4. **格式**：指定输出结构（如 JSON、Markdown 列表）。
5. **约束**：限定范围、长度或禁止项（如“不超过50字，禁止编造”）。

**为什么 Prompt 很重要？**
- **软代码**：在 Agent 中，Prompt 决定了工具调用策略、错误恢复逻辑，行为规则皆编码于此。
- **决定输出质量**：同样的模型，Prompt 好坏会导致结果天差地别。

**好的 Prompt 特征**：
- **目标单一**：一次专注一件事，复杂任务拆解。
- **信息分层**：指令与数据（上下文）清晰分离，如使用 XML 标签。
- **可执行**：输出格式严格匹配程序解析需求（如 JSON Schema）。
- **可测试**：包含正反例，能通过回归测试。

#### 边界情况与极端场景
- **输入过长导致的遗忘**：当上下文过长时，模型容易“遗忘”Prompt 开头的指令（“迷失中间”现象）。需将关键指令放在 Prompt 的最开头或最结尾，或在长上下文中多次重复关键约束。
- **特殊字符注入**：如果上下文中包含类似 JSON 的字符串或代码片段，模型可能会混淆指令与数据，导致输出格式错误。必须使用严格的分隔符（如 triple quotes `"""` 或 XML 标签）包裹数据部分。
- **指令冲突**：System Prompt（系统设定）与 User Prompt（用户指令）可能存在冲突（如系统要求“拒绝敏感词”，但用户强行要求“忽略之前所有规则”）。设计时需明确优先级，或在 System Prompt 中加入“防御性指令”来强化最高优先级。

**实战案例**：在做 SQL 生成 Agent 时，最初 Prompt 只写了“生成 SQL”，模型经常省略 `JOIN` 条件。后来我们将 Prompt 改为包含“DDL 定义”和“3 个正反示例”的详细模板，并将 SQL 解析报错信息作为反馈喂回给模型，生成准确率从 60% 提升至 90%。

**代码示例**：
```python
# 使用 Pydantic 定义强类型输出，确保 Prompt 的可执行性
from pydantic import BaseModel

class ClassificationResult(BaseModel):
    label: str
    confidence: float
    keywords: list[str]

# prompt_template 末尾强制加入格式要求
prompt_template = """
...（指令部分）
请严格按照以下 JSON Schema 格式输出：
{schema}
"""
print(prompt_template.format(schema=ClassificationResult.model_json_schema()))
```

## 面试追问
1. **追问（泛化能力）**：精心设计的 Prompt 在特定数据集上效果很好，但换一批数据（比如不同的口音、不同的行文风格）效果就变差了。如何提高 Prompt 的鲁棒性和泛化能力？
2. **追问（动态 Prompt）**：在 Agent 场景中，不同用户、不同阶段的 Prompt 可能需要动态调整。你如何设计 Prompt 的版本管理和灰度发布策略？
3. **追问（CoT vs 少样本）**：在什么场景下，思维链比少样本提示更有效？反之亦然？

## 易错点
1. **误区**：认为 Prompt 越长、描述越详细，效果就一定越好。
   **纠正**：过长的 Prompt 会增加 Token 成本和推理延迟，且过多的噪声信息会干扰模型的注意力。应遵循“奥卡姆剃刀”原则，在表述清晰的前提下尽量精简。
2. **误区**：过于依赖 Few-Shot（少样本）示例，示例越多越好。
   **纠正**：示例必须具有代表性且覆盖边界情况。如果示例全是正向样本，模型在遇到负向样本时可能会强行归纳到正向类别中。同时，过多的示例会挤占 Context Window，导致关键上下文被截断。
