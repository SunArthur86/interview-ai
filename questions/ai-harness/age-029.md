# OpenTelemetry在Agent系统中一般打哪些Span?如何设计Trace结构

- **Agent Trace 结构:**
```
Trace (一次完整Agent任务)
├─ Span: agent.run (根Span)
│   ├─ Span: llm.call (LLM调用)
│   │   ├─ Span: llm.tokenize
│   │   ├─ Span: llm.inference
│   │   └─ Span: llm.detokenize
│   ├─ Span: tool.call (工具调用)
│   │   ├─ Span: tool.search (搜索)
│   │   └─ Span: tool.execute (执行)
│   ├─ Span: rag.retrieve (RAG检索)
│   │   ├─ Span: embedding
│   │   ├─ Span: vector_search
│   │   └─ Span: rerank
│   └─ Span: memory.update (记忆更新)
```

- **每个Span记录:**
- 服务名(service.name)
- 操作名(operation.name)
- 开始/结束时间(duration)
- 属性(attributes): model, tokens, cost
- 事件(events): 错误、重试
- 状态(status): OK/ERROR

- **关键Span属性:**
```python
span.set_attribute("llm.model", "gpt-4")
span.set_attribute("llm.input_tokens", 1500)
span.set_attribute("llm.output_tokens", 300)
span.set_attribute("llm.cost_usd", 0.012)
span.set_attribute("tool.name", "search")
span.set_attribute("tool.success", True)
```

- **## 常见考点:**
1. **Trace传播**: 在微服务调用链中，Trace Context（如traceparent header）如何通过HTTP或RPC传播到LLM Provider或Vector DB？
2. **采样率**: 高并发下为了控制成本，如何设置动态采样率（如优先采样错误请求或高Token消耗的请求）？
3. **Span链接**: 异步场景下（如流式输出），如何将生成过程中的多个事件链接到同一个Span？
4. **Baggage传递**: 如何利用Baggage在Span间传递业务上下文（如 UserID、TenantID），而无需修改每个函数的参数？

- **实战案例：** 排查某Agent响应慢的问题时，通过Trace发现 `vector_search` Span 耗时2秒，进一步查看Attributes发现检索了Top-100个向量，将其优化为Top-10后延迟下降70%。

- **代码示例：**
```python
from opentelemetry import trace

tracer = trace.get_tracer("agent_service")

with tracer.start_as_current_span("agent.run") as parent_span:
    parent_span.set_attribute("user.id", "12345")
    
    with tracer.start_as_current_span("tool.call") as child_span:
        child_span.set_attribute("tool.name", "weather_api")
        try:
            result = call_weather_api()
        except Exception as e:
            child_span.record_exception(e)
            child_span.set_status(trace.Status(trace.StatusCode.ERROR, str(e)))
            raise
```

- **## 边界情况:**
1. **超长上下文截断**: 当Prompt超过LLM最大Token限制时，Trace应记录被截断的文本内容或哈希，以便排查为何关键指令丢失，同时需记录实际发送的Token数与模型上限的比值。
2. **流式输出的Span生命周期**: LLM流式响应可能持续数十秒，Span的Duration会非常长。如果在这期间发生网络中断或客户端取消，必须确保Span被正确关闭并标记为Canceled状态，避免产生僵尸Span。
3. **并发工具调用的Trace关联**: Agent并行调用多个工具时，Trace结构会呈现“扇出”形状。需注意并发子Span的结束时间可能晚于父Span（如果父Span未Wait），此时应使用Links关联而非严格的父子嵌套。

- **## 面试追问:**
1. 在高并发场景下，如果OpenTelemetry Collector成为了性能瓶颈，你会如何优化Sidecar的资源配额或采集策略？
2. 对于基于Function Calling的Agent，如何将LLM返回的JSON字符串解析过程中的失败映射到Span的Attributes中？
3. 当Trace跨越多个独立的SaaS服务（如使用OpenAI API和Pinecone）时，如何保证这些异构系统的Trace ID能够统一关联？（注：通常需依赖W3C tracecontext透传）

- **## 易错点:**
1. **敏感信息泄露**: 将用户的Prompt、PII（个人隐私信息）或API Key直接记录在Span的Attributes中，导致日志中心存储敏感数据。必须设置过滤器或对敏感字段进行脱敏（Hash或掩码）。
2. **高基数基数陷阱**: 将具有无限可能性的值（如 UserID、生成的文本内容）作为Tag Key而非Attribute，导致时序数据库内存暴涨。在Otel中，应将高基数数据放入Attributes或Events，避免作为索引键。

## 记忆要点

- Trace结构：根Span为agent.run，子Span覆盖llm.call、tool.call、rag.retrieve。
- 关键属性：记录model、tokens、cost、tool.name等，用events记录重试和错误。
- 并发关联：并行工具调用呈扇出结构，用Links关联而非严格父子嵌套。
- 实战排查：通过Span耗时定位瓶颈，如vector_search耗时过长需优化Top-K。
