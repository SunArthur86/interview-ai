# LiteLLM和模型网关的作用是什么?如何管理多供应商LLM

- **LiteLLM核心:统一API接口**

用一个接口调用所有LLM供应商(OpenAI/Anthropic/Zhipu/DeepSeek/本地Ollama等).

- **网关架构图:**
```text
Client Application
      │
      │ (Standard OpenAI Format: POST /v1/chat/completions)
      ▼
┌──────────────────────────────────────┐
│         LiteLLM Proxy Server         │
│  ┌────────────┐   ┌──────────────┐   │
│  │   Router   │──>│ Load Balancer│   │
│  └────────────┘   └──────────────┘   │
│       │                                  │
│       │  (Model Name Mapping)            │
│       ▼                                  │
│  ┌──────────┐  ┌──────────┐  ┌────────┐│
│  │ OpenAI   │  │ Anthropic│  │  Local ││
│  │ Provider │  │ Provider │  │  Ollama││
│  └──────────┘  └──────────┘  └────────┘│
└───────────────────────────────────────┘
```

- **实战案例**：
某企业初期使用 GPT-4，成本过高。引入 LiteLLM 网关后，配置了路由策略：简单逻辑类问题分流给 DeepSeek（成本低），复杂推理类问题分流给 Claude 3（能力强）。在未改动业务代码的情况下，整体调用成本降低了 60%。

- **核心功能:**
1. **统一接口** - 100+供应商,一套SDK (`completion(model="provider/model")`)
2. **负载均衡** - 多个 API Key 轮询,支持权重设置 (`litellm.set_weight`) 
3. **Fallback** - 主供应商失败自动切换 (配置 `fallbacks` 列表)
4. **成本追踪** - 自动计算 Token 费用,支持导出至 LangSmith/Datadog
5. **限流** - 按用户/项目限流 (`rpm_limit`, `tpm_limit`)
6. **缓存** - 相同请求缓存结果 (支持 Redis/S3)

- **使用方式:**
```python
from litellm import completion

# 统一接口,只需改model名
response = completion(model="gpt-4", messages=[...])
response = completion(model="claude-3-5-sonnet", messages=[...])
response = completion(model="zhipu/glm-4", messages=[...])
```

- **关键代码 (配置文件 config.yml):**
```yaml
model_list:
  - model_name: gpt-4-turbo
    litellm_params:
      model: openai/gpt-4-turbo
      api_key: os.environ/OPENAI_API_KEY
  - model_name: gpt-4-turbo
    litellm_params:
      model: azure/gpt-4-turbo # Azure作为备用
      api_key: os.environ/AZURE_API_KEY
      api_base: os.environ/AZURE_API_BASE

router_settings:
  routing_strategy: "usage-based-routing" # 按预算自动路由
  fallbacks: [{"model_group": "gpt-4-turbo"}]
```

- **模型网关:**
- 部署为独立服务 (`litellm --config /path/to/config.yaml`)

- **## 易错点**
1. **超时传播问题**：网关作为中间层，其超时设置必须小于客户端的超时设置。如果网关设置了 60s 超时但客户端是 90s，客户端会收到网关的 504 错误而非上游模型的直接错误，这会导致排查困难。通常建议网关超时略短于客户端预期。
2. **Failover 的副作用**：配置 Fallback 自动切换时，如果不同模型（如从 GPT-4 切换到 Claude 3）的 Output 格式（如 JSON Schema 或 Tool Calling 格式）不完全一致，会导致下游解析代码报错。切换时应尽量保证功能对等，或在网关层做格式统一。

- **## 面试追问**
1. 当某个供应商 API 出现大面积故障（如 500 错误），网关的 Fallback 机制会导致瞬间巨大的流量冲击到备用供应商，如何设计熔断机制防止备用服务被击穿？
2. 多租户场景下，如何利用网关实现基于租户级别的配额管理和路由隔离（例如 VIP 用户始终走高优模型通道）？
3. LiteLLM 或其他网关如何处理流式输出的缓冲？在网关层做流式聚合是否会增加首字延迟（TTFT）？
