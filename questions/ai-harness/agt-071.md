# Prompt 设计原则

### 2. Prompt 设计原则

#### 2.1 清晰性
**概念解释**
用词准确、无歧义，避免「它、这个、上面」指代不明；关键术语与业务定义需写清楚。

**原理详解**
歧义会放大模型的随机性；清晰指令缩小「合法输出」的空间，提高一致性。模型是基于概率预测下一个 Token，模糊的指令会导致概率分布过于分散，引入随机噪声。

**正面示例**
```text
【角色】你是电商售后客服质检助手，只根据给定规则做分类，不编造订单信息。
【任务】阅读用户留言，输出唯一标签：退货 / 换货 / 物流查询 / 其他。
【上下文】
用户留言：{{user_message}}
（若提供）订单号：{{order_id}}
【格式】
仅输出一行 JSON：{"label":"...", "confidence":0.0-1.0, "reason":"一句话理由"}
【约束】
- 若信息不足以判断，label 填 "其他"，confidence 降低并在 reason 说明缺失信息。
- 不要输出 JSON 以外的任何文字。
```

**反面示例**
```text
把日期改一下格式。
```

#### 2.2 具体性
**概念解释**
说明粒度（多长、几个要点）、判定标准（何为「高优先级」）、反例（什么情况算错误）。

**原理详解**
边界模糊会导致模型自行脑补规则，具体性通过正反例和量化指标锁定输出范围。这是为了解决模型对「常识」和「业务规则」的混淆，例如「总结」默认可能侧重主要信息，但业务可能需要包含「免责声明」等看似次要但关键的信息。

**正面示例**
```text
用三条要点总结下文，每条不超过 25 个汉字；不得引入文中未出现的人名或数字。
```

#### 2.3 结构化
**概念解释**
使用标题、编号、XML 标签、Markdown 小节、分隔线，使模型与人类都易于解析「哪部分是规则、哪部分是数据」。

**原理详解**
标签化（如 `<document>...</document>`）能降低「把用户数据当指令」的风险，也便于多段材料拼接。Transformer 的注意力机制对特殊分隔符敏感，结构化的 Prompt 能帮助模型更好地建立长距离依赖，区分指令区与数据区。

**Prompt 结构架构图**
```text
┌──────────────────────────────────────────────────────────────┐
│ System Prompt / Meta Instructions (角色设定、全局目标)       │
├──────────────────────────────────────────────────────────────┤
│ Delimiters & Rules (分隔符、约束条件、输出格式定义)          │
│  e.g., "Use XML tags below", "Output JSON only"              │
├──────────────────────────────────────────────────────────────┤
│ Few-Shot Examples (少样本示例)                               │
│  ┌─ Example 1 ─┐  ┌─ Example 2 ─┐  ┌─ Example 3 ─┐          │
│  │ Input: ...   │  │ Input: ...   │  │ Input: ...   │          │
│  │ Output: ...  │  │ Output: ...  │  │ Output: ...  │          │
│  └─────────────┘  └─────────────┘  └─────────────┘          │
├──────────────────────────────────────────────────────────────┤
│ Current Input (当前待处理数据)                               │
│  {{user_input}} or <context>...</context>                    │
├──────────────────────────────────────────────────────────────┤
│ Trigger Cue (触发提示，如 "Answer:" 或 "### Response")      │
└──────────────────────────────────────────────────────────────┘
```

#### 边界情况与极端场景
- **分隔符注入攻击**：用户输入的内容中如果包含与 Prompt 相同的分隔符（如用户输入 `###` 结尾），可能会打断模型对指令的解析，导致 Prompt Leak 或指令覆盖。设计时应使用随机生成的唯一 UUID 字符串作为分隔符。
- **多语言混合**：当 Prompt 中英文混合，或者包含代码、Latex 公式等多种语言时，模型的注意力分配可能不均。建议尽量保持 Prompt 语言的一致性，或明确告知模型“请使用中文回复”。
- **指令过载**：在一个 Prompt 中塞入过多的约束条件和格式要求，可能导致模型“顾此失彼”，特别是某些低参数量的模型。若任务极度复杂，应考虑拆分为多次交互（链式调用）。

## 面试追问
1. **追问（负面约束）**：模型通常很难严格遵守“不要做某事”的负面约束。如果你必须要求模型“绝对不要输出代码片段”，你会在 Prompt 设计上做哪些特殊处理？
2. **追问（少样本选择）**：在构建 Few-Shot 示例时，你是如何选择示例的？是随机采样还是基于语义相似度检索？各有何优劣？
3. **追问（迭代优化）**：如果你发现一个 Prompt 在 10% 的 Corner Case 下失效，你会如何系统地进行迭代和修复？

## 易错点
1. **误区**：只关注 Prompt 的文本内容，忽视了参数设置。
   **纠正**：Prompt Engineering 的核心不仅在于文本，还在于与超参数（如 `temperature`, `top_p`）的配合。高精度的分类任务应设置 `temperature=0` 以保证确定性，而创意生成任务则适当调高。
2. **误区**：认为使用了结构化标签（如 XML）就绝对安全。
   **纠正**：虽然 XML 标签能提升效果，但如果模型训练数据中包含大量类似格式的错误数据（例如爬取的网页），模型可能会模仿错误的格式。必须通过 E2E 测试验证输出的实际解析成功率。
