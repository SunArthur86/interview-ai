# 什么是 AI Harness(AI 工程化框架)?它解决什么问题

- **AI Harness** 指围绕大模型构建应用所需的工程化框架和工具链。

- **解决的问题**
1. 直接调 API 不够:缺乏记忆、工具、安全等能力
2. 生产化困难:监控、限流、降级、版本管理
3. 成本控制:token 消耗、缓存策略

- **核心组件**
- 模型路由:多模型选择 + 熔断降级
- RAG 引擎:检索 + 生成
- Agent 编排:工具调用 + 任务规划
- 记忆系统:短期/长期记忆
- 可观测性:日志/Trace/Metrics
- 安全护栏:Prompt 注入防护

- **实战案例:**
在构建企业级知识库助手时，直接使用 LangChain 的 `VectorStoreRouterChain` 导致冷启动极慢（需加载所有向量库索引）。我们自定义了 **Lazy-loading Harness**：仅在 Agent 识别到特定领域意图时，才初始化对应的 VectorStore 连接，将冷启动时间从 30s 降低到 2s，同时节省了 40% 的内存占用。

- **代码示例 (Python - 模拟 Router Harness):**
```python
from typing import Literal

def model_router(prompt: str, complexity: str) -> str:
    """根据任务复杂度路由到不同模型"""
    if complexity == "high":
        # 调用 GPT-4o/Claude 3.5
        return call_llm("gpt-4o", prompt)
    else:
        # 降级到小模型 Haiku 或 Llama-3-8B
        return call_llm("haiku", prompt)

# 简单的 Agent 编排模拟
def run_agent(task):
    intent = classify_intent(task)
    if intent == "search":
        return search_tool(task)
    elif intent == "code":
        return model_router(task, "high")
```

- **代表框架对比:**
| 框架 | 核心定位 | 优势 | 劣势 | 适用场景 |
|------|---------|------|------|----------|
| LangChain | "LLM 界的 Spring" | 生态最丰富，集成广 | 代码臃肿，性能损耗大 | 快速原型验证，复杂流程编排 |
| LlamaIndex | "RAG 专家" | 数据索引强，高级索引多 | 非 RAG 场景一般 | 企业级知识库，文档问答 |
| Semantic Kernel | "微软亲儿子" | 企业级集成，原生 C# 支持 | 社区较封闭，Python 包一般 | .NET 技术栈，Office 365 Copilot |
| AutoGPT / BabyAGI | "自主 Agent" | 高度自动化，自主规划 | 不可控，成本高，易死循环 | 任务分解研究，自动化脚本 |
| Haystack | "深度学习友好" | 支持自定义 Pipeline | 文档不如 LangChain 丰富 | 混合检索，需要高度定制的 NLP 流程 |

- **边界情况**
- **超长上下文处理**：当 Prompt 超过模型上下文窗口时，Harness 需自动启用 Sliding Window 或 Context Compression，而非直接报错。
- **并发死锁**：在 Agent 编排中，若工具 A 依赖工具 B 的结果，而工具 B 反向调用 Agent，需设置调用深度限制（如 `max_steps=10`）防止死循环。
- **非结构化输出**：当 LLM 返回 JSON 格式错误（如缺少闭合括号），Harness 需具备自动修复（如 Output Fixing Parser）或重试机制。

- **## 面试追问**
1. 在 LangChain 中，Chain 和 RunnableLambda 的核心区别是什么？为什么推荐使用 LCEL 表达式？
2. RAG 系统中，如果召回的相关性很差，除了调整 Embedding 模型，在 Harness 层面还能做哪些优化（如重排序 Rerank、查询改写 Query Transformation）？
3. 如何保证 Agent 调用工具时的安全性？如果工具包含删除数据库高危操作，如何进行权限校验？

- **## 易错点**
1. **过度设计**：并非所有简单的问答应用都需要引入 LangChain 这种重型框架。对于单次 Prompt-Response 任务，直接调用 API 维护成本更低且性能更好。
2. **忽略 Prompt 版本管理**：很多人将 Prompt 硬编码在代码里。生产级 Harness 应配合 Prompt 管理平台（如 LangSmith 或自定义 DB）实现 Prompt 的 A/B 测试和灰度发布。
