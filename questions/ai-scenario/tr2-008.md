# 在构建 RAG 系统时，为什么 BM25（关键词检索）与 向量检索 的混合检索通常优于单一的向量检索？

向量检索基于语义相似度，擅长处理同义词和概念匹配，但对专有名词、ID 号等精确匹配关键词往往表现不佳（例如搜索“GPT-4”时可能匹配到“大语言模型”但漏掉具体型号）。BM25 基于精确词频匹配，擅长处理关键词和稀疏特征，能够弥补语义检索的“模糊”短板。混合检索通过结合两者（通常使用 RRF - Reciprocal Rank Fusion 融合排序），既能召回语义相关的文档，又能保证关键词的精确匹配，从而显著提升召回率和准确性，特别是在用户包含特定实体或专业术语的查询场景中。

### 实战案例
在金融合规问答系统中，单纯使用向量检索经常漏检特定的交易代码或合同编号，导致严重的“幻觉”风险。引入 BM25 混合检索后，精准召回率提升了约 40%，确保了关键实体的严格匹配。

### 对比表格
| 特性 | 向量检索 | BM25 关键词检索 | 混合检索 |
| :--- | :--- | :--- | :--- |
| **匹配原理** | 语义相似度 | 词频/稀疏词匹配 | 语义 + 关键词双重匹配 |
| **长尾词表现** | 强（理解意图） | 弱（词频低） | 强（互补） |
| **专有名词/ID** | 弱（模糊泛化） | 强（精确字符） | 强（覆盖全面） |
| **计算开销** | 高（需 Embedding） | 低（倒排索引） | 较高（双路计算+融合） |

### 代码示例 (Python)
```python
# 使用 RRF (Reciprocal Rank Fusion) 融合 BM25 和向量检索结果
def reciprocal_rank_fusion(results_dict, k=60):
    fused_scores = {}
    for system, docs in results_dict.items():
        for rank, doc in enumerate(docs):
            doc_id = doc['id']
            fused_scores[doc_id] = fused_scores.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(fused_scores.items(), key=lambda x: x[1], reverse=True)

# 假设 bm25_results 和 vector_results 是已排序的文档列表
final_ranking = reciprocal_rank_fusion({
    "bm25": bm25_results, 
    "vector": vector_results
})
```
