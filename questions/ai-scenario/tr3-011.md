# 代码生成Agent的评测指标有哪些？除了Pass@k，还有哪些维度的指标至关重要？

Pass@k（生成k个代码样本中至少有一个通过测试集的概率）是核心指标，但不足够。其他关键维度包括：1) Executability（可执行率）：代码是否能直接运行，无语法错误；2) Latency（延迟）：生成代码的时间长短，对IDE体验至关重要；3) Robustness（鲁棒性）：对输入微小扰动的抵抗能力；4) Security（安全性）：是否包含漏洞或恶意代码（如SQL注入）；5) Similarity：生成的代码与参考代码的编辑距离，反映是否“过度拟合”参考答案而非学会逻辑；6) 依赖管理：生成的代码是否引用了不存在的库。综合评测需要结合单元测试和静态代码分析工具。

**实战案例**：在优化内部Copilot时，模型Pass@1很高，但开发人员抱怨大量生成代码依赖企业内部私有包却未import，导致直接报错；引入“Import Validity”指标检测后，针对性微调了模型指令，代码可用性大幅提升。

**代码示例 (Python)**：
```python
import ast

def check_executability(code: str) -> bool:
    try:
        ast.parse(code)  # 仅检查语法正确性
        return True
    except SyntaxError:
        return False
```

**对比表格**：

| 指标维度 | 评估目标 | 典型工具/方法 | 业务价值 |
| :--- | :--- | :--- | :--- |
| **Pass@k** | 功能正确性 | UnitTests (HumanEval/MBPP) | 决定代码能否最终解决问题 |
| **Executability** | 语法完整性 | AST Parser, Linter | 减少Debug时间，提升初体验 |
| **Security** | 代码安全性 | Bandit, Snyk, Linting | 防止生产环境安全漏洞 |
| **Latency (TTFT)** | 响应速度 | Performance Benchmarks | 影响开发者的心流和体验 |
| **Similarity** | 创新度与抄袭 | BLEU/Edit Distance | 判断是否只是记忆训练数据 |
| **Robustness** | 边界稳定性 | Fuzzing, Mutation Testing | 保证代码在异常输入下不崩 |

## 边界情况
- **超长上下文生成**：当Prompt超过模型上下文窗口或生成的代码需要跨多个文件时，模型往往会截断代码或丢失中间逻辑，需评估“上下文完整性”。
- **无限循环与资源泄露**：生成的代码逻辑可能正确但包含死循环或未关闭的文件/数据库连接，评测需引入沙箱执行超时机制和资源监控（如内存/CPU限制）。
- **多语言混合**：在Polyglot编程场景下，评测需覆盖不同语言（如Python调用C++）的互操作性边界。

## 面试追问
1. **关于覆盖率**：如果测试用例本身覆盖率不足（HumanEval上的测试通常比较简单），Pass@k很高但实际代码质量很差，如何构建更可信的评测集？
2. **关于相似度**：在计算Similarity时，如果生成的代码逻辑正确但变量命名和结构风格与参考答案完全不同，BLEU分数会很低，你会用什么指标来修正这种误判？
3. **关于安全性**：除了静态扫描，如何设计动态评测机制来捕捉生成代码中的“后门”或潜在的Prompt注入攻击？

## 易错点
- **混淆可执行性与正确性**：通过了AST语法检查并不代表代码逻辑正确，模型常生成“虽然能跑但结果错误”的幻觉代码，Pass@k仍是逻辑正确性的金标准。
- **过度依赖单一指标**：仅优化Pass@1可能导致模型过于保守，只生成最常见但可能不是最优的解；仅优化Similarity可能导致模型仅仅是在“背诵”训练集而非生成逻辑。

## 记忆要点

- Pass@k测功能正确性，但需结合可执行率、延迟、安全性等多维指标。
- 可执行性用AST查语法，安全性用静态扫描防漏洞，延迟影响IDE体验。
- 相似度防过度拟合，依赖管理防引用不存在的库，鲁棒性抗输入扰动。
- 核心口诀：功能Pass@k，语法AST查，安全防漏洞，依赖要合法。
