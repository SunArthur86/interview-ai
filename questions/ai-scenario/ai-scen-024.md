# 如何设计LLM模型的灰度发布系统？安全地上线新模型或更新Prompt，支持快速回滚。

【场景分析】
LLM模型更新风险高（输出行为可能突变），灰度发布是降低风险的关键策略。

**【实战案例】**
某翻译应用升级模型版本（v1->v2）时，虽然整体BLEU分数提升，但未进行灰度直接全量发布，导致v2模型在特定技术术语翻译上出现“幻觉”（乱造词），引发大量用户投诉。后改进流程：先对1%的VIP白名单用户开启v2，并通过埋点监控“重试率”和“用户修改率”指标，发现异常后立即回滚。第二次发布引入了影子测试，在不影响真实响应的情况下用v2跑完所有请求，对比无误后再开启流量。

【灰度发布流程】
1. 发布前验证：
   - Golden Set回归：核心评测集全量通过
   - 对比测试：新旧模型在相同输入上的输出对比
   - 安全扫描：有害输出检测、PII泄露检测
2. 灰度策略：
   - 流量切分：5% → 10% → 25% → 50% → 100%
   - 用户分桶：按用户ID Hash分桶，同一用户始终命中同一版本
   - 场景灰度：先低风险场景（如闲聊），再高风险场景（如金融建议）
3. 监控指标：
   - 质量指标：用户满意度、点踩率、人工转接率
   - 性能指标：延迟、吞吐、错误率
   - 安全指标：有害内容率、Prompt注入成功率
   - 成本指标：平均Token消耗
4. 回滚机制：
   - 自动回滚：关键指标恶化超过阈值 → 自动切回旧版本
   - 一键回滚：运维人员可随时100%切回
   - 灰度窗口：每个阶段至少观察24小时

【版本管理】
- 模型版本：每次发布关联commit hash + 模型权重版本
- Prompt版本：Prompt模板独立版本管理，与模型版本解耦
- 配置版本：温度、top_p等参数也纳入版本管理
- 实验追踪：MLflow / W&B记录每次发布的配置和效果

【A/B测试设计】
- 假设：新模型在XX场景下的回答质量优于旧模型
- 指标：主要指标（满意度）+ 次要指标（延迟、成本）
- 样本量：基于统计显著性计算所需样本量
- 持续时间：至少7天覆盖不同时段和用户群体
- 决策标准：主要指标显著提升 + 次要指标不恶化

**【代码示例：灰度路由逻辑】**
```python
import hashlib

def route_model(user_id: str, config: dict):
    # 基于User ID Hash分桶，保证同一用户体验一致
    bucket = int(hashlib.md5(user_id.encode()).hexdigest(), 16) % 100
    
    # 累加权重判断落在哪个区间
    cumulative = 0
    for version, weight in config["weights"].items():
        cumulative += weight
        if bucket < cumulative:
            return version
    return config["default"]

# 配置示例：v2占10%，其余v1
config = {"weights": {"v2": 10, "v1": 90}, "default": "v1"}
model_version = route_model("user_12345", config)
```

**【灰度策略对比】**

| 策略 | 实现难度 | 用户体验一致性 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **随机灰度** | 低 | 差（同一用户可能看到不同版本） | 无状态推荐系统、后端计算任务 |
| **用户分桶** | 中 | **高（同一用户固定版本）** | **对话类应用、个性化推荐** |
| **特征规则** | 高 | 高 | 针对特定人群（如付费会员、新用户）测试 |
| **流量染色** | 高 | 中 | 影子测试、压测 |

## 记忆要点

- 发布流程：Golden Set回归 → 影子测试 → 灰度放量（5%→100%）→ 全量。
- 灰度策略：按用户ID Hash分桶，保证同一用户体验一致，支持场景灰度。
- 监控指标：质量（满意度/重试率）、性能（延迟）、安全（有害率）。
- 回滚机制：关键指标恶化自动回滚，配置版本化管理（模型/Prompt/参数）。
- 实战教训：未灰度直接全量导致术语翻译幻觉，需监控“用户修改率”。
