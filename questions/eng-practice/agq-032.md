# 怎么做的异步并行处理

**Situation：** Agent 执行过程中有很多 I/O 密集型操作（LLM API 调用、数据库查询、外部 API 调用），同步执行会导致大量等待时间。

**Task：** 识别可并行化的操作，用异步编程提升系统吞吐量。

**Action：** 
1. **异步框架选择：**
   - 基于 Python `asyncio` + FastAPI。
   - 所有 I/O 操作使用 `async/await`。
   - 使用 `httpx.AsyncClient` 替代 `requests` 做 HTTP 调用。

2. **并行化场景：**
   - **多路检索并行：** 向量检索、关键词检索、知识图谱查询同时发起。
   - **多工具并行调用：** 当 Agent 需要调用多个独立工具时，并行执行。
   - **批量 embedding 计算：** 文档入库时批量并行计算 embedding。

3. **并发控制：**
   - 使用 `asyncio.Semaphore` 控制最大并发数。
   - LLM API 调用并发限制为 50（受 API 限流限制）。
   - 数据库连接池大小 20（避免打爆数据库）。

4. **异步队列处理：**
   - 非实时任务（如文档索引、日志分析）通过 Celery 异步任务队列处理。
   - 避免阻塞实时请求的处理。

**Result：** 
- 多路检索并行后，检索总延迟从 500ms（串行）降低到 200ms（并行）。
- 系统整体 QPS 提升 2.5 倍。
- 文档批量入库速度提升 4 倍。

### 深化补充

**实战案例：**
在“多工具并行调用”场景中，曾遇到某个外部工具 API 响应极慢（超过 30s），导致整体 `gather` 任务被阻塞。通过引入 `asyncio.wait_for` 为每个子任务设置独立的超时时间（如 10s），确保个别慢请求不会拖垮整个 Agent 的响应速度，超时的工具自动降级处理。

**代码示例（Python）：**
```python
import asyncio

async def call_tool_with_timeout(tool_func, *args, timeout=5):
    try:
        return await asyncio.wait_for(tool_func(*args), timeout=timeout)
    except asyncio.TimeoutError:
        return {"error": "tool timeout", "data": None}

# 并行执行并带超时控制
tasks = [
    call_tool_with_timeout(api_search, query),
    call_tool_with_timeout(db_lookup, query)
]
results = await asyncio.gather(*tasks, return_exceptions=True)
```

**对比表格（同步 vs 异步 I/O）：**

| 维度 | 同步模式 | 异步模式 |
| :--- | :--- | :--- |
| **等待机制** | 阻塞线程，线程闲置等待 | 挂起协程，线程可处理其他任务 |
| **资源消耗** | 高并发时需大量线程（内存高） | 单线程即可处理高并发（内存低） |
| **适用场景** | CPU 密集型或简单的脚本 | I/O 密集型（高并发网络请求） |
| **编码复杂度** | 简单直观 | 需处理事件循环、异步安全 |

## 常见考点
1. **并行依赖处理**：如果 Agent 的下一步动作依赖上一步工具调用的结果，还能并行吗？（答：不能。并行仅适用于无依赖的任务；有依赖时只能串行，但可以通过 Pipeline 机制流水线化）
2. **异步中的 CPU 密集型任务**：Asyncio 是单线程的，如果运行复杂的计算任务会阻塞事件循环，如何解决？（答：使用 `run_in_executor` 将计算任务提交到线程池或进程池执行，或使用支持并发的 C 扩展库）
3. **超时控制**：并行发起多个请求，如何设置整体超时时间？（答：使用 `asyncio.wait_for` 包装 `gather`，或者使用 `asyncio.wait(return_when=FIRST_EXCEPTION)` 来快速失败）

## 记忆要点

- 框架选择：Python asyncio + FastAPI，IO 密集型任务全异步。
- 并行场景：多路检索并行、多工具调用并行、批量 Embedding 计算。
- 并发控制：Semaphore 限制最大并发数，防止打爆 API 或数据库。
- 容错机制：wait_for 设置超时，个别慢请求降级，不阻塞整体。
