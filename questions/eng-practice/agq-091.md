# 什么是 Agent 的工具学习(Tool Learning)

**回答：** 
1. **定义**: 
让 LLM 学会识别何时需要调用外部工具、选择哪个工具、生成正确的参数。
**本质**: 将自然语言意图映射为结构化的 API 调用。

2. **实现方式:**
- **Prompt 方式 (主流/原生):**
  - 在 Prompt 中描述可用工具的名称、功能、参数格式 (JSON Schema)。
  - LLM 通过 in-context learning 学会调用工具。
  - **优点：** 灵活、无需训练、适配任何 LLM。
  - **缺点:** 受 Prompt 长度限制，工具多时上下文拥挤；参数格式生成可能出错。

- **Fine-tuning 方式:**
  - 使用工具调用的标注数据微调模型。
  - 代表工作: Gorilla、ToolLLM 等。
  - **优点：** 更准确，能处理更复杂的 API 文档。
  - **缺点:** 需要训练数据，更新工具需重新训练或持续预训练。

- **Function Calling (API 级别):**
  - OpenAI、Anthropic 等厂商在 API 层面强制约束输出格式。
  - 模型经过专门训练以支持 function calling（特殊 token 强制输出 JSON）。
  - **优点：** 最可靠，解析容错率高。
  - **缺点:** 依赖特定厂商，不仅限于 Prompt，涉及模型底层微调。

3. **挑战:**
- **工具选择准确性:** 工具数量多时容易选错，特别是功能相似的工具。
- **参数生成:** 需要正确理解参数的语义、类型（int, str, enum）和格式（日期格式等）。
- **多工具协同:** 需要理解工具间的依赖关系（如先 A 后 B，B 的参数依赖 A 的结果）。
- **错误恢复:** 工具调用失败（如 API 500、参数校验失败）后的重试和替代策略。

4. **最佳实践:**
- **描述优化**: 工具描述要精确且简洁，包含参数的业务含义，而不仅仅是技术定义。
- **Few-shot**: 提供调用示例，特别是复杂参数组合的示例。
- **数量控制**: 限制单次 Context 中可选工具数量（建议 ≤ 8 个），使用路由层先筛选工具集。
- **容错机制**: 实现 Tool 的输入校验，如果 LLM 生成格式错误，返回 Error Message 给 LLM 进行 Self-Correction。

### 深化补充

**实战案例**：
在构建股票查询 Agent 时，直接让 LLM 输出 JSON 参数经常出现字段名拼写错误（如将 `symbol` 写成 `symbols`）。**解决方案**：引入 ReAct 模式（推理+行动）后，让模型先思考“我需要什么参数”，再生成工具调用，并配合 Pydantic 进行强类型校验，将格式错误率从 15% 降至 0.1%。

**代码示例 (Python - OpenAI Function Calling)**：
```pythonnimport openai

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get the current weather in a given location",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {"type": "string", "description": "City and state, e.g. San Francisco, CA"},
                    "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
                },
                "required": ["location"]
            }
        }
    }
]

response = openai.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "What's the weather in Boston?"}],
    tools=tools,
    tool_choice="auto"  # 让模型自动决定是否调用工具
)
```

**对比表格 (实现方式对比)**：

| 特性 | Prompt Engineering | Fine-tuning | Function Calling (厂商特有) |
| :--- | :--- | :--- | :--- |
| **开发成本** | 低，直接写 Prompt | 高，需构建数据集并训练 | 中，依赖 API 能力 |
| **通用性** | 高，适配所有模型 | 低，需针对性训练模型 | 低，绑定特定模型 API |
| **格式稳定性** | 中，易产生 JSON 格式错误 | 高，模型已学会特定格式 | 高，底层约束输出 |
| **更新难度** | 极低，修改 Prompt 即可 | 高，需重新训练 | 低，修改 Schema 定义 |
| **准确率** | 依赖模型 IQ | 在特定 API 上表现极优 | 综合表现最好 |
