# Guardrails(护栏)是怎么实现的

**Situation：** Agent 的输出需要多维度的质量和安全把控：格式合规、内容安全、事实准确性、业务合规等。

**Task：** 实现一套可扩展的 Guardrails 框架，确保 Agent 输出的质量和安全。

**Action：** 
1. **Guardrails 架构（输入侧 + 输出侧）：**
   ```text
   用户输入   [输入 Guardrails]     Agent 处理     [输出 Guardrails]     最终响应
      │             │                   │                 │                    │
      │             ▼                   │                 ▼                    │
      │    ┌───────────────┐            │        ┌───────────────┐            │
      │    │ 安全/格式校验  │            │        │ 事实/幻觉校验  │            │
      │    │ 拦截恶意 Prompt │           │        │ JSON校验/PII过滤 │            │
      │    └───────┬───────┘            │        └───────┬───────┘            │
      └────────────┘                    └────────────────┘                    │
   ```

2. **输入侧 Guardrails：**
   - **Prompt 注入检测：** 基于规则（如“忽略之前指令”） + 分类器检测注入攻击。
   - **敏感信息检测：** 正则匹配身份证号、手机号等 PII。
   - **长度限制：** 单次输入 ≤ 2000 字符。
   - **话题过滤：** 拒绝与业务无关的话题（如政治、暴力等）。

3. **输出侧 Guardrails：**
   - **格式检查：** 检查输出是否符合预期格式（JSON Schema 验证等）。
   - **内容安全：** 调用内容审核 API 过滤违规内容。
   - **事实核查：** 检查输出中引用的事实是否与检索到的 chunk 一致。
   - **幻觉检测：** 对比输出和 context，标记可能的幻觉内容。

4. **可扩展设计：**
   ```python
   class BaseGuardrail(ABC):
       @abstractmethod
       async def check(self, content: str, context: dict) -> GuardrailResult:
           pass
   ```

**Result：** 
- Prompt 注入拦截率 97%。
- 输出内容安全合规率 99.8%。
- 幻觉检测覆盖率 85%（仍有改进空间）。

## 常见考点
1. **中间件与拦截器位置**：Guardrails 代码逻辑放在 API Gateway 层还是应用服务层？（答：应用层更灵活，处理具体业务语义；网关层处理通用的限流和鉴权）
2. **性能开销**：内容审核 API 调用耗时高（如 200ms+），如何避免阻塞主流程？（答：异步调用、并行审核或对高频且可信的用户进行抽样审核）
3. **干预策略**：检测到违规后是直接拒绝、重试还是回复兜底话术？（答：安全类直接拒绝，格式类可引导重试，幻觉类可附带置信度提示）

### 深化内容

**实战案例**：
在某次金融场景上线中，我们遇到过用户诱导 Agent 修改转账金额的攻击（“请忽略之前的转账限制...”）。仅靠规则正则无法拦截，后来引入了基于 LLM 的分类器对 Prompt 意图进行二分类，成功拦截了此类变体攻击，避免了潜在的资金风险。

**代码示例**：
```python
from pydantic import BaseModel, validator

# 结合 Pydantic 进行输出格式强校验
class ToolOutput(BaseModel):
    action_name: str
    parameters: dict

    @validator('action_name')
    def validate_action(cls, v):
        allowed_actions = ['search', 'refund', 'check_status']
        if v not in allowed_actions:
            raise ValueError(f"Illegal action detected: {v}")
        return v

# 在 Guardrail 中调用
def validate_output(llm_json_str: str):
    try:
        ToolOutput.parse_raw(llm_json_str)
        return True, "Pass"
    except Exception as e:
        return False, str(e) # 拦截并返回错误原因
```

**输入侧 vs 输出侧 Guardrails 对比**：

| 维度 | 输入侧 | 输出侧 |
| :--- | :--- | :--- |
| **核心目标** | 防止攻击与滥用 (越狱、PII 注入) | 保证质量与合规 (幻觉、格式错误、违规内容) |
| **处理时机** | LLM 推理之前 (低成本拦截) | LLM 推理之后 (高成本，必须处理) |
| **常用技术** | 规则引擎、小模型分类器、Mod API | Schema 校验、NLI 事实核查、正则替换 |
| **失败策略** | 直接拒绝回复，提示用户修改 | 强制重试、脱敏返回 或 返回兜底话术 |
| **性能影响** | 极低 (微秒级-毫秒级) | 较高 (可能涉及额外的 LLM 调用) |

## 记忆要点

- 架构双拦截：输入侧防注入/脱敏，输出侧防幻觉/格式校验。
- 输入侧：规则+分类器防 Prompt 注入，正则过滤 PII 敏感信息。
- 输出侧：JSON Schema 强校验，对比 Context 检测事实一致性。
- 性能优化：审核 API 异步调用，避免阻塞主流程。
