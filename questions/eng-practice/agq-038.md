# 延迟优化做了哪些措施

**Situation：** 系统初期端到端延迟（用户提问到收到回答）平均 6.5 秒，用户体验不佳，目标是降低到 3 秒以内。

**Task：** 全链路延迟优化，将 P50 延迟降到 3 秒以内。

**Action：** 
1. **链路分析（各环节耗时）：**
   - 意图识别：300ms
   - 检索（串行）：500ms → 并行后 200ms
   - Reranking：130ms
   - LLM 生成（主要瓶颈）：4500ms → 优化后 2500ms
   - 后处理：100ms
   - **总计：** 5530ms → 3130ms

2. **优化措施：**
   - **检索并行化：** 多路检索从串行改为并行（-300ms）。
   - **模型路由：** 简单查询走快速模型（延迟 800ms vs 3000ms）。
   - **流式输出：** TTFT 500ms，用户感知到的等待时间大幅缩短。
   - **预加载：** 意图识别和检索并行执行（检索不需要等意图确定后再开始，可以先用原始 query 检索）。
   - **连接池复用：** HTTP 长连接、数据库连接池，减少连接建立时间（-50ms）。
   - **结果缓存：** 语义缓存命中时直接返回（延迟 < 100ms）。

3. **LLM 调用优化：**
   - Prompt 精简减少输入 token，LLM 处理时间缩短。
   - 使用 `max_tokens` 限制输出长度。
   - 选择低延迟的 API 端点（如 Azure 中国区）。

4. **深度工程实战（补充）：**
   - **实战案例**：在流式传输场景中，曾遇到因下游 BaaS 服务反爬虫导致的偶发超时。通过引入带抖动的指数退避重试机制，并将 LLM 推理请求与普通业务请求在 K8s 网络层做 QoS 隔离，有效消除了因网络拥塞造成的 P99 延迟毛刺。
   - **代码示例（Python 异步并发）：**
     ```python
     import asyncio
     # 并行执行检索和意图识别，消除串行等待
     async def process_query(query):
         retrieval_task = asyncio.create_task(search_engine.search(query))
         intent_task = asyncio.create_task(classifier.predict(query))
         
         # 等待两个任务完成
         docs, intent = await asyncio.gather(retrieval_task, intent_task)
         return docs, intent
     ```

**Result：** 
- P50 延迟从 6.5s 降低到 2.8s。
- P99 延迟从 15s 降低到 5.2s。
- 流式输出的 TTFT 为 500ms，用户满意度提升 22%。

## 常见考点
1. **P99 延迟为何高？如何进一步优化？** 
   *追问点*：关注长尾请求，通常由 GC 停顿、网络抖动或冷启动导致。可考虑熔断机制或独立资源池隔离长尾任务。
2. **预加载策略的副作用是什么？** 
   *追问点*：如果意图识别结果与预加载的检索结果不匹配，会导致无效的检索请求，浪费资源。需要评估命中率与资源浪费的权衡。
3. **Prompt 压缩的具体手段？** 
   *追问点*：是否使用了 System Prompt 截断、历史摘要压缩，或者去除了无用的标点符号和空格。

## 记忆要点

- 并行化：多路检索并行，意图识别与检索预加载并行。
- 模型路由：简单查询走小模型，复杂查询走大模型，降低平均延迟。
- 流式优化：TTFT 降至 500ms，用户感知响应大幅提升。
- 工程手段：连接池复用，Prompt 精简，语义缓存命中直返。
