# 线上用户问法和离线评测数据分布不一致时怎么办

这是 RAG/QA 系统上线后最常见的问题--离线指标很好,线上效果差.

**根本原因**
1. **分布偏移**: 离线评测集是人工构造的,问法规范、意图明确; 线上用户输入口语化、模糊、含错别字、混合意图.
2. **长尾缺失**: 长尾问题在离线集中覆盖不足.
3. **噪声差异**: 线上环境更复杂,含有多余信息或对抗性攻击.

**数据闭环与优化流程图：**
```text
  线上用户 Query
      │
      ├─────────────┐
      ▼             ▼
   实时系统    日志收集
      │             │
      │      ┌──────┴──────┐
      │      │   数据清洗  │
      │      └──────┬──────┘
      │             ▼
      │      ┌──────┴──────┐
      │      │   人工/自动 │
      │      │   标注/打分 │
      │      └──────┬──────┘
      │             ▼
      │      离线评测集更新
      │             │
      └─────> 模型微调/评估
```

**解决方案**
1. **线上日志回流**: 收集真实用户 query → 标注 → 补充评测集,形成闭环.
2. **Query 改写**: 对线上 query 做纠错、扩展、标准化,使其对齐离线训练语料.
3. **意图分类前置**: 先分类意图再路由到不同处理逻辑,减少模型混淆.
4. **数据增强**: 用 LLM 生成多种问法变体扩充评测集,模拟真实分布.
5. **分层数据集**: 按 query 类型 (事实型/推理型/多跳型) 分别评测,定位短板.
6. **线上 A/B 测试**: 以线上真实指标 (如点赞率、采纳率) 为准,不迷信离线指标.
7. **持续监控**: 建立 query 分布漂移检测机制.

**经验法则**: 离线评测集应该至少包含 20% 的'真实线上日志'子集.

**实战案例：** 
在客服机器人上线后，发现大量"查不到"的 Bad Case 是因为用户口语化（如"货啥时候到"）。我们通过收集一周的线上 Bad Case，利用 LLM 改写成标准查询（"预计发货时间"），构建了一个 5k 条的口语化对数据集进行微调，将意图识别准确率从 82% 提升至 94%。

**代码示例：** 
```python
# Pseudo-python: 简单的分布漂移检测
import numpy as np
from scipy.spatial.distance import jensenshannon

# 1. 计算基准分布 (离线集关键词频)
p_baseline = np.array([0.1, 0.2, 0.7]) 

# 2. 计算线上分布 (实时日志关键词频)
p_online = np.array([0.05, 0.15, 0.8])

# 3. 计算 JS 散度 (JS Divergence)
divergence = jensenshannon(p_baseline, p_online)

if divergence > 0.1:
    alert("数据分布漂移严重，触发模型重训流程")
```

**数据来源对比表：**

| 维度 | 离线黄金评测集 | 线上回流日志集 |
| :--- | :--- | :--- |
| **数据质量** | 高（人工精修） | 低（含噪声、错别字） |
| **意图分布** | 均匀（覆盖全面） | 长尾（头部意图占比极高） |
| **问法风格** | 规范、书面语 | 口语化、简短、甚至中英混杂 |
| **主要用途** | 模型选型、效果基准线 | 发现边缘Case、数据闭环、SFT |
| **获取成本** | 高（需人工标注） | 低（自动化收集） |
| **时效性** | 静态（需定期更新） | 实时（反映当前热点） |

## 记忆要点

- 根因：离线数据规范，线上口语化、含噪、长尾分布
- 闭环：线上日志回流 -> 清洗标注 -> 补充评测集 -> 微调
- 优化：Query改写对齐分布，数据增强模拟真实问法
- 监控：检测JS散度，建立分布漂移预警机制
