# 多 Agent 协作和单 Agent 有什么区别?你怎么选择

**Situation：** 随着业务复杂度增加,单个 Agent 的 Prompt 越来越长,工具数量越来越多,开始出现能力退化和维护困难的问题.

**Task：** 评估是否需要从单 Agent 架构迁移到多 Agent 协作架构,以及如何设计协作机制.

**Action：** 
1. 单 Agent 的瓶颈分析:
单 Agent 的系统 Prompt 超过 4000 tokens 后,指令遵循能力明显下降.
工具数量超过 15 个时,LLM 选择工具的准确率从 95% 下降到 78%.
所有逻辑耦合在一个 Prompt 中,修改一个功能可能影响其他功能.

2. 多 Agent 架构设计:
Supervisor Agent(主管 Agent): 负责意图识别和任务路由,不直接执行任务.
RAG Agent(检索 Agent): 专注于知识检索和答案生成.
Tool Agent(工具 Agent): 专注于外部 API 调用和数据处理.
Summary Agent(摘要 Agent): 专注于长文本摘要和对话总结.

3. 协作机制:
**通信方式：** Agent 之间通过结构化消息传递(JSON Schema 定义),避免自然语言传递导致的信息损失.
**执行策略：** Supervisor 判断任务类型后,将任务分派给专业 Agent;支持顺序执行和并行执行.
**结果汇总：** 由 Supervisor Agent 整合各子 Agent 结果,生成最终回答.

**架构流程图：**
```text
┌─────────────────────────────────────────────────────────┐
│                  用户请求                                │
└─────────────────────────┬───────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│              Supervisor Agent (Router)                   │
│  (解析意图，拆解任务，分发指令)                            │
└─────┬───────────────────────┬───────────────────────┬────┘
      │                       │                       │
      ▼                       ▼                       ▼
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  RAG Agent   │      │ Tool Agent   │      │Code Agent    │
│ (知识问答)    │      │ (API调用)     │      │ (代码执行)    │
└──────┬───────┘      └──────┬───────┘      └──────┬───────┘
```

4. 选择标准:
工具数 ≤ 8 且业务逻辑简单 → 单 Agent.
工具数 > 8 或需要专业分工 → 多 Agent.

**实战案例：**
在金融研报生成场景中，单 Agent 经常混淆 "数据采集"（需高精度）和 "摘要生成"（需文采）。拆分为 DataAgent（使用 Python Tool）和 WriterAgent（使用 GPT-4）后，代码执行错误率降低了 40%，同时文本风格的一致性大幅提升。

**代码示例（Python - 伪代码路由逻辑）：**
```python
def supervisor_router(query: str) -> str:
    """根据用户查询决定路由给哪个 Agent"""
    routing_prompt = f"""
    User Query: {query}
    Available Agents: 
    1. RAG_AGENT: For knowledge base questions.
    2. TOOL_AGENT: For data calculation, weather, APIs.
    Return only the AGENT_NAME.
    """
    # 使用轻量级模型快速路由
    response = llm_lite.predict(routing_prompt) 
    return response.strip() # e.g., "TOOL_AGENT"
```

**对比表格：**

| 特性 | 单 Agent | 多 Agent 协作 |
| :--- | :--- | :--- |
| **Prompt 复杂度** | 高，包含所有逻辑 | 低，每个 Agent 专注特定领域 |
| **Context 窗口压力** | 极大，需携带所有工具描述 | 较小，只携带当前 Agent 相关工具 |
| **调试难度** | 难，无法定位是哪个环节出错 | 易，可单独追踪每个 Agent 的输入输出 |
| **Token 成本** | 低（单次调用） | 高（多次调用 + 路由开销） |
| **延迟** | 较低 | 较高（串行调用时） |
| **适用场景** | 简单任务、低频工具调用 | 复杂工作流、需并发处理不同模态任务 |

## 记忆要点

- 单Agent瓶颈：Prompt超长能力降，工具过多选择难，逻辑耦合维护差。
- 多Agent架构：Supervisor路由，RAG/Tool/Code等专业Agent分工执行。
- 通信用结构化JSON，执行策略支持并行，Supervisor汇总结果。
- 选择标准：工具≤8且逻辑简单用单Agent，反之用多Agent协作。
