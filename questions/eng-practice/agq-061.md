# 怎么做的测试?覆盖率多少

**Situation：** AI Agent 系统的测试比传统软件更复杂：LLM 输出不确定（非确定性）、工具调用涉及外部服务、端到端流程长，且评估标准主观性强。

**Task：** 建立多层次的测试体系，确保系统质量与效果。

**Action：** 
1. **测试金字塔 (改良版):**
   **E2E 测试 (10%):** 完整的用户交互流程，模拟真实用户行为。
   **集成测试 (30%):** 组件间的交互测试，特别是 LLM 与 Tool 的交互。
   **单元测试 (60%):** 单个函数/类的测试，覆盖确定性逻辑。

   ```text
          ┌─────────────────────────────────┐
          │      E2E Tests (Agent Flow)     │  ← 少量，慢，贵，模拟真实场景
          ├─────────────────────────────────┤
          │   Integration Tests (RAG/Tools) │  ← 中等，Mock 外部依赖，验证逻辑组合
          ├─────────────────────────────────┤
          │      Unit Tests (Utils/Parsers) │  ← 大量，快，确定性高，Pytest
          └─────────────────────────────────┘
   ```

2. **单元测试 (覆盖率 85%):**
   - **对象:** 工具函数、Prompt 模板渲染、数据预处理/后处理、JSON 解析器、路由逻辑。
   - **Mock:** 使用 `unittest.mock` 或 `pytest-mock` 模拟 LLM 调用，返回固定的 `choices[0].message.content`。
   - **框架:** `pytest` + `pytest-asyncio` (处理异步调用) + `pytest-cov`。

3. **集成测试:**
   - **检索管道测试:** 给定 query，验证 Milvus/Elasticsearch 返回的结果相关性（通过 ID 验证）。
   - **工具调用测试:** 验证 Function Calling 的参数序列化和反序列化是否正确，异常处理是否生效。
   - **环境:** 使用 `testcontainers` 在本地 Docker 中启动真实的 Milvus、Postgres 实例进行测试，避免使用不稳定的 Mock 数据库。

4. **评估测试 - AI 特有:**
   - **数据集:** 建立「黄金数据集」（Golden Dataset），包含 500+ 条标注的 `query-answer` 对或 `query-context-relevance` 标注。
   - **自动化指标:**
     - **检索质量:** Recall@10 (召回率)、Precision@5 (准确率)、NDCG@5 (排序质量)、MRR (平均倒数排名)。
     - **生成质量:** BERTScore (语义相似度)、ROUGE-L (重叠度)、Faithfulness (忠实度 - RAGAS 框架)。
     - **端到端:** 任务完成率（结果解析是否成功）、User Satisfaction (基于打分模型)。
   - **CI 集成:** 使用 `RAGAS` 或 `DeepEval` 框架，在 CI 流水线中每次提交自动跑评估（抽样运行），防止核心能力回归。

5. **压力测试:**
   - 每月一次全面压测（使用 JMeter/Locust/K6），模拟 2x-5x 峰值流量。
   - 重点监控 LLM 限流、数据库连接池耗尽、Redis 超时等瓶颈。
   - 每次重大 Prompt 变更或模型升级后做回归压测。

**实战案例：** 在一次 Prompt 迭代中，为了提升回答流畅性微调了 System Prompt，导致回归测试中 Faithfulness（忠实度）指标下降了 15%（模型开始幻觉编造文档中不存在的事实）。由于集成了自动化 RAGAS 评估，CI 流水线直接报错，拦截了上线。后续修正了 Prompt，增加了“仅基于上下文回答”的强约束。

**代码示例（Python - Pytest Mock）：**
```python
import pytest
from unittest.mock import patch
from my_agent import agent_service

@pytest.mark.asyncio
async def test_agent_tool_calling():
    # 模拟 LLM 返回特定的工具调用参数
    mock_response = {
        "choices": [{
            "message": {
                "tool_calls": [{"function": {"arguments": '{"query": "test"}'}}]
            }
        }]
    }
    
    # Patch OpenAI 客户端
    with patch('my_agent.client.chat.completions.create') as mock_create:
        mock_create.return_value = mock_response
        result = await agent_service.handle_query("execute search")
        
        # 断言工具被正确调用
        assert result["tool_used"] == "search_tool"
        assert result["args"] == {"query": "test"}
```

**对比表格（传统测试 vs AI 测试）：**

| 维度 | 传统软件测试 | AI Agent/LLM 测试 |
| :--- | :--- | :--- |
| **核心目标** | 验证功能逻辑正确性 | 验证输出质量和安全性（非确定性） |
| **断言方式** | Assert A == B (确定性) | 相似度评分、模型判别、人工审核 |
| **测试数据** | 边界值、异常值 | Golden Dataset、对抗样本 |
| **执行速度** | 毫秒级 | 秒级（涉及推理）甚至分钟级（评估） |
| **环境依赖** | Mock 外部 API 即可 | 需真实或 Mock LLM、Vector DB |

**Result：** 
- 单元测试覆盖率 85%，核心业务逻辑模块（如解析器、路由）达到 95%。
- 评估测试在 CI 中 100% 自动化运行（核心集），回归测试耗时控制在 15 分钟内。
- 上线后的严重 bug 数量降低 70%，检索相关性问题减少 40%。

## 常见考点
1. **LLM 测试确定性**：如何解决 LLM 输出不稳定导致单元测试经常失败的问题？（答：Mock 返回值，或对于生成类测试使用模糊匹配而非完全相等断言，或在测试中设置 `temperature=0`）。
2. **评估集维护**：如何保证评估集的时效性和覆盖度？（答：定期从线上 Bad Case（负向反馈）中清洗数据注入评估集，保持数据分布与线上真实场景一致）。
