# 网络传输是怎么优化的

**Situation：** 知识库初始化需要对大量文档计算 embedding(800 万个 chunk),耗时过长.日常增量更新也需要高效的 embedding 计算.
**Task：** 优化 embedding 计算的吞吐量和效率.
**Action：** 
1. 批量计算优化:
  *   **大 Batch**：`batch_size=256`,利用 GPU 并行计算能力，最大化利用率.
  *   **流水线重叠**：使用 DataLoader 预加载数据到 CPU 内存,使用 `pin_memory=True` 加速 CPU->GPU 传输,避免 GPU 等待 CPU 数据处理.

2. 增量更新策略:
  *   **变更检测**：对比 MD5 hash,只对变更的文档重新计算 embedding.
  *   **实时入库**：新增文档实时计算并写入向量库,不需要全量重建.

3. 多 GPU 并行:
  *   使用 `DataParallel` 或 `DistributedDataParallel (DDP)` 在多张 GPU 上并行计算.
  *   4 张 T4 GPU 并行,吞吐量提升 3.6 倍 (略低于 4 倍是因为通信开销).

4. 维度压缩(可选):
  *   **PCA**：对于存储和检索速度敏感的场景,使用 PCA 将 1024 维降到 512 维.
  *   **权衡**：召回率仅下降 1.2%,存储减少 50%,索引构建速度提升约 40%.

**数据处理流水线：**
```text
Disk/DB (Raw Docs)
    │
    ▼
[CPU Process] (Parse -> Split -> Hash Check)
    │ (Pin Memory Queue)
    ▼
[GPU 0] [GPU 1] [GPU 2] [GPU 3]
    │       │       │       │
    └───────┴───────┴───────┘
            │ (Infer)
            ▼
     [Vector DB Insert]
```

**实战案例：** 在处理混合长度的文档时，固定的 Batch Size 导致显存利用率波动极大（长文本 OOM，短文本闲置）。引入了动态打包算法，将不等长的序列打包到同一个 Batch，通过 Padding Mask 忽略填充位置，显存利用率提升 40%，吞吐量翻倍。

**代码示例（PyTorch DataLoader）：**
```python
from torch.utils.data import DataLoader

# 启用 pin_memory 加速 CPU->GPU 传输
# num_workers > 0 实现 IO 和计算并行
dataloader = DataLoader(
    dataset, 
    batch_size=256, 
    shuffle=False, 
    num_workers=8,
    pin_memory=True,  # 关键优化：固定内存页
    collate_fn=custom_collate # 自定义函数处理变长 padding
)

for batch in dataloader:
    input_ids = batch['input_ids'].to('cuda', non_blocking=True) # non_blocking 异步传输
    with torch.no_grad():
        outputs = model(input_ids)
```

**对比表格（并行策略对比）：**
| 策略 | 实现方式 | 通信开销 | 进程模型 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **DataParallel (DP)** | 单机多卡，单进程 | 高 (GIL锁，梯度复制) | 单进程多线程 | 快速原型调试，单机 < 4卡 |
| **DistributedDataParallel (DDP)** | 单机/多机，多进程 | 低 (Ring-AllReduce) | 多进程 (每卡1进程) | 生产环境，大规模训练/推理 |
| **Pipeline Parallel** | 按模型层切分 | 中 (输入输出传递) | 多进程 | 超大模型无法放入单卡显存 |

**Result：** 
*   800 万 chunk 的全量 embedding 计算:从 48 小时降到 12 小时.
*   **日常增量更新**：1000 条新文档的 embedding 计算 < 30s.
*   GPU 利用率稳定在 80%+.

## 常见考点
1.  **DDP vs DP**：PyTorch 中 DistributedDataParallel 和 DataParallel 的区别？（DP 是单进程多线程，受限于 GIL；DDP 是多进程，效率更高）。
2.  **Product Quantization (PQ)**：除了 PCA，向量库常用的压缩技术是什么？（PQ，将向量切分成子向量分别量化，极大减少内存）。
3.  **OOM 处理**：Batch size 调大时遇到显存不足（OOM）怎么办？（Gradient Checkpointing，混合精度训练 FP16）。

## 记忆要点

- 批量计算：大Batch(256)利用GPU并行，DataLoader预加载+pin_memory加速传输。
- 增量更新：MD5对比只算变更文档，新增实时入库不重建。
- 多卡并行：DDP多进程通信低，吞吐量提升近4倍。
- 维度压缩：PCA降维(1024->512)，召回率略降但存储和索引速度大幅提升。
