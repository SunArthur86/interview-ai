# 如何做的性能基准测试

**Situation：** 需要建立性能基准,作为优化的参照和回归检测的标尺.
**Task：** 建立可重复的性能基准测试体系.
**Action：** 
1. **基准测试场景:**
**单组件基准：** 向量检索延迟（P99）、LLM 调用延迟（TTFT/TBT）、Reranking 延迟.
**端到端基准：** 简单问答延迟、RAG 问答延迟、工具调用延迟.
**压力基准：** 最大 QPS、最大并发数、错误率阈值.
2. **基准测试工具:**
自研基准测试脚本(Python pytest-benchmark)。
**压力测试：** Locust 模拟用户行为。
**结果存储和对比：** 自建 Dashboard 或 Prometheus + Grafana。
3. **基准测试执行:**
每次重大变更前后各跑一次（集成在 CI pipeline 中）。
结果存入数据库,支持历史对比。
**设置性能回归阈值：** 延迟增加 > 10% 或吞吐量下降 > 5% 触发告警，阻断合并。
4. **基准数据集:**
固定的 100 条测试 query(覆盖各种类型，长短不一)。
每次基准测试使用相同的数据集,确保可比性；Mock 外部依赖（如 LLM API）以消除网络波动影响。

**基准测试流程图：**
```text
┌─────────────┐    ┌─────────────┐    ┌──────────────┐
│ Code Commit │───>│ Benchmarks  │───>│  Report &    │
└─────────────┘    │ (Locust/    │    │  Compare     │
                   │  pytest)    │    └──────┬───────┘
                   └──────┬──────┘           │
                          │                 ▼
                   ┌──────▼──────┐    ┌──────────┐
                   │ Mocked Data │───>│ Pass/Fail│
                   └─────────────┘    └──────────┘
```

**实战案例：** 
某次重构检索逻辑，P50 延迟降低了 20ms，但 P99 延迟因为偶发的 GC 卡顿增加了 200ms。基准测试 Dashboard 清晰显示了长尾恶化，我们在上线前及时回滚了该改动。

**代码示例：** 
```python
import pytest

@pytest.mark.benchmark(group="rag_retrieval")
def test_rag_end_to_end(benchmark, mock_llm_api):
    query = "How to optimize python performance?"
    # 预热：解决冷启动问题
    run_rag_pipeline(query) 
    
    # 正式基准测试
    result = benchmark(run_rag_pipeline, query)
    assert result["score"] > 0.8 # 确保优化未损伤准确性
```
