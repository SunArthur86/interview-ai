# 为什么采用多路检索而不是单一向量检索

**Situation：** 企业知识库包含大量结构化和非结构化数据,单一向量检索在某些场景下召回率不足.例如,精确的编号查询(如“工单号 WO-20240315-001”)在纯语义检索中表现很差.

**Task：** 设计一个兼顾语义理解和精确匹配的检索架构,提升知识检索的准确率和召回率.

**Action：** 
1. 多路检索架构设计:
向量检索(语义通道): 使用 Milvus 存储文档向量,基于 embedding 相似度做语义召回, top_k=20.
关键词检索(精确通道): 使用 Elasticsearch 做 BM25 检索,处理精确查询、编号匹配等场景, top_k=20.
知识图谱检索(关系通道): 针对实体关系类问题(如“张三负责哪些项目”),从 Neo4j 中检索实体关系.

2. 结果融合策略:
采用 RRF(Reciprocal Rank Fusion)算法融合多路结果.
**公式：** RRF_score = Σ 1/(k + rank_i) ,其中 k=60 是平滑常数.
每路检索结果独立排序后,通过 RRF 算法统一排名.

**架构流程图：**
```text
用户查询: "工单 WO-20240315-001 的状态？"
    │
    ▼
┌───────────────────────────────┐
│         Query Router           │
│ (意图识别 + 查询改写/扩展)      │
└───────┬─────────┬───────────────┘
        │         │
        ▼         ▼
┌──────────────┐  ┌──────────────┐
│   关键词通道   │  │   向量通道    │
│(Elasticsearch)│  │   (Milvus)   │
│ BM25 算法     │  │ Cosine 相似度│
└──────┬───────┘  └──────┬───────┘
       │                 │
       ▼                 ▼
  [Doc A, Doc C]    [Doc B, Doc A]
       │                 │
       └────────┬────────┘
                ▼
       ┌─────────────────┐
       │  RRF 融合排序     │
       │ (倒数排名融合)    │
       └────────┬────────┘
                ▼
         [Doc A, Doc B]
                │
                ▼
           重排/返回
```

3. 查询路由策略:
意图识别模块判断查询类型,动态激活不同检索通道.
精确查询(含编号、代码等)→ 优先走关键词通道.
概念性问题(“什么是微服务”)→ 优先走向量通道.
关系查询(“谁负责某项目”)→ 启用知识图谱通道.

**Result：** 
整体检索准确率从单一向量检索的 72% 提升到多路融合的 89%.
召回率从 65% 提升到 93%.
精确编号类查询的命中率从 30% 提升到 98%.

**实战案例：**
在某政务客服场景中，单纯向量检索无法区分同名政策（如新旧“个税扣除政策”），导致引用过期条款。引入时间标量过滤+关键词召回后，成功过滤掉 90% 的过期文档引用。

**代码示例（Python）：**
```python
def rrf_merge(results_dict, k=60):
    """融合多路检索结果
    Args:
        results_dict: {"es": [doc_id1, ...], "milvus": [doc_id2, ...]}
    """
    score_map = {}
    for list_name, doc_list in results_dict.items():
        for rank, doc_id in enumerate(doc_list):
            score_map[doc_id] = score_map.get(doc_id, 0) + 1.0 / (k + rank + 1)
    # 按分数倒序排列
    sorted_docs = sorted(score_map.items(), key=lambda x: x[1], reverse=True)
    return [item[0] for item in sorted_docs]
```

## 边界情况
- **单路检索延迟不可控**：如果某一路检索（如ES查询）因慢查询或网络抖动超时，不应阻塞整体响应。需设计超时熔断机制，返回空集或部分结果，保证系统的可用性。
- **融合后的结果去重**：同一文档可能在 ES 和 Milvus 中都被召回且 ID 相同，RRF 能处理分数叠加，但在最终展示给 LLM 前，必须对 Content 进行去重，以免浪费 Token。
- **噪声放大**：当查询本身有歧义时，多路检索可能召回完全不相关的结果混合在一起，干扰 LLM 判断。如果各路检索结果的分数方差极大，需考虑降级或丢弃低分通道。

## 面试追问
1. **权重调整**：RRF 假设各通道重要性相同，但在业务上我们可能更信任 ES 的精确匹配。如何动态调整不同通道的权重，而不是用固定的 k 值？
2. **路由失效**：如果 Query Router 分类错误（例如把“工单号”识别为语义查询），导致走了错误的通道，如何设计一个快速反馈闭环来纠正这个错误？
3. **召回数量设定**：为什么选择 Top 20 而不是 Top 5 或 Top 100？在融合前后的召回数量是如何通过实验确定的？

## 易错点
- **加权平均融合的陷阱**：初学者常尝试对不同通道的归一化分数（如 Cosine 0.8 和 BM25 Score 5.2）进行加权平均，但这通常效果很差，因为不同检索器的分数分布不具可比性。RRF 通过“排名”融合巧妙地绕过了分数归一化的问题，是更鲁棒的做法。
- **重排的重要性**：仅靠 RRF 融合往往不够，特别是当多个相关文档挤在一起时。很多系统忽略了融合后的 Cross-Encoder 重排步骤，这往往是精准度提升的关键一步。
