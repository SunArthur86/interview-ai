# 检索结果不准确怎么办

**Situation：** 检索结果不准确是 RAG 系统最常见的问题,表现为:召回的 chunk 与问题无关、关键信息未被召回、排序不准确.

**Task：** 建立检索质量问题的诊断和优化流程.

**Action：** 
1. **问题诊断流程:**
   用户反馈 → 复现查询 → 查看检索结果 → 定位问题环节
   ├── 分块问题:关键信息被切断 → 优化分块策略
   ├── Embedding 质量:语义表示不准 → 评估/更换模型
   ├── 检索参数:top_k 太小 → 调整参数
   ├── 数据质量:源文档有误 → 数据清洗
   └── 查询理解:用户表述模糊 → query 重写

2. **Query 重写:**
   - 使用 LLM 将用户的口语化查询重写为更精确的检索查询.
   - **示例：** "这个东西怎么退" → "退货退款流程和规则"
   - **多查询扩展：** 将一个模糊查询扩展为多个精确子查询,分别检索后融合.

3. **HyDE(Hypothetical Document Embedding):**
   - 让 LLM 生成一个"假设的理想回答".
   - 对假设回答计算 embedding 来检索(而不是直接用 query 的 embedding).
   - 在专业领域查询中效果提升显著.

4. **检索后处理:**
   - Reranker 精排(详见技术实现 Q6).
   - **上下文扩展：** 检索到的 chunk 自动扩展前后相邻 chunk,补充上下文.

5. **持续优化闭环:**
   - 每周分析"回答不满意"的反馈,追溯检索环节的问题.
   - 定期更新评估集,跟踪检索质量指标.

**高级检索增强流水线：**
```text
User Query ("怎么退")
     │
     ▼
[Query Understanding]
     ├── Query Rewriting ───────┐
     └── HyDE (Gen Fake Answer) ┘
               │
               ▼
[Vector Search] (Semantic Match)
     │
     ▼ (Top K Documents)
[Reranking Model] (Cross-Encoder / BGE-Reranker)
     │ (Re-score and trim)
     ▼
[Context Augmentation]
     ├── Window Expansion (Fetch neighbors)
     └── Merge into Prompt
```

**补充原理细节：**
- **检索 vs 重排**：双路召回（向量+关键词）能覆盖语义和字面匹配。Reranker 计算量大，只对向量检索回来的 Top-K（如 50-100）进行精排，效果和成本的最佳平衡。
- **窗口扩展**：切分时容易丢失边界信息，通过 `chunk_id +/- 1` 将前后文本拼入 Prompt，补全语义。

**Result：** 
- Query 重写使检索准确率提升 15%.
- HyDE 在专业术语查询场景提升 22%.
- 检索相关的用户投诉减少 60%.

**实战案例**：某工业维助 RAG 系统中，用户报错代码 "E-104"，纯向量检索找不到原因。后引入 BM25 关键词检索作为补充（"E-104" 完全匹配），成功召回隐含在错误码手册中的具体维修步骤。这证明了混合检索在专业术语场景的必要性。

**代码示例**：
```python
# 混合检索权重调整示例
def hybrid_search(query_vector, query_text, k=10):
    # 向量检索 (语义)
    vec_results = vector_db.search(query_vector, k=20)
    # 关键词检索 (字面)
    kw_results = bm25_db.search(query_text, k=20)
    
    # 倒序融合加权 (RRF 算法)
    final_scores = {}
    for rank, doc in enumerate(vec_results):
        final_scores[doc.id] = final_scores.get(doc.id, 0) + 1.0/(rank+1+60)
    for rank, doc in enumerate(kw_results):
        final_scores[doc.id] = final_scores.get(doc.id, 0) + 1.0/(rank+1+60)
        
    return sorted(final_scores.items(), key=lambda x: x[1], reverse=True)[:k]
```

**常见考点**
1. **HyDE 的局限性**：什么时候 HyDE 效果不好？（答案：事实性查询，模型编造的假文档如果包含错误事实，会检索到错误内容）
2. **混合检索**：除了向量检索，为什么还需要关键词检索（BM25）？（答案：解决专有名词、ID号等向量难以表示的精确匹配问题）
3. **Reranker 性能瓶颈**：重排模型很慢，如何优化？（答案：异步化、使用蒸馏后的小模型、仅在高价值查询中使用）
4. **分块策略**：怎么切分最好？（答案：没有标准答案，通常按语义切分比固定字符切分好，但代码、表格需特殊处理）

- 检索后处理：使用Reranker精排Top-K结果，并进行窗口扩展补充上下文。
