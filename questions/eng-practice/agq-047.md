# Embedding 计算是怎么优化的

**Situation：** 系统负载随业务高峰期波动很大(如促销期间请求量暴增 5 倍),固定资源配置要么浪费要么不足.
**Task：** 实现基于业务指标的自动扩缩容.
**Action：** 
1. 监控指标:
  *   **基础指标**：CPU、内存、网络 I/O.
  *   **业务指标**：QPS、P99 延迟、错误率、LLM API 并发数.
  *   **自定义指标**：队列积压数(如 Kafka Lag)、缓存命中率.

2. HPA(水平 Pod 自动扩缩容):
  *   使用 Kubernetes Metrics Adapter 获取自定义指标.
  *   **配置示例**：
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Pods
    pods:
      metric:
        name: requests_per_second
      target:
        type: AverageValue
        averageValue: "50"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0 # 快速扩容
      policies:
      - type: Percent
        value: 100 # 每次最多翻倍
        periodSeconds: 15
```
  *   策略：QPS > 50/Pod 时扩容; QPS < 20/Pod 持续 5 分钟时缩容.

3. 预测性扩容:
  *   基于历史数据预测高峰时段(如每日上午 9 点),通过 CronHPA 提前扩容.
  *   促销等已知高峰事件手动预扩容.

**自动扩缩容闭环图：**
```text
User Traffic Surge
        │
        ▼
[Metrics Server] (CPU/Mem/QPS)
        │
        ▼
[Kubernetes HPA Controller]
        │ (Compare Metric > Target)
        ▼
[Scale Up Event] (Increase Pod Count)
        │
        ▼
[New Pod Start] (Cold Start)
        │
        ▼
[Service Ready] (Traffic Rebalanced)
```

**实战案例：** 某次突发流量导致 HPA 扩容的 Pod 还在拉取镜像，流量就已经打进来，导致大量 502。引入了 Image Puller DaemonSet 预热镜像，并配合 `readinessProbe` 的 `initialDelaySeconds` 调整，确保新 Pod 只有在应用完全启动后才挂载到 Service，解决了冷启动期间的流量丢失问题。

**代码示例（YAML 策略）：**
```yaml
behavior:
  scaleDown:
    stabilizationWindowSeconds: 300 # 缩容冷却窗口，防止抖动
    policies:
    - type: Percent
      value: 10 # 每次最多缩容 10%，避免陡降
      periodSeconds: 60
  scaleUp:
    stabilizationWindowSeconds: 0 # 允许立即扩容
    policies:
    - type: Pods
      value: 2 # 每次最多增加 2 个 Pod，防止过度扩容
      periodSeconds: 15
```

**对比表格（扩缩容策略对比）：**
| 策略 | 触发机制 | 响应速度 | 适用场景 | 缺点 |
| :--- | :--- | :--- | :--- | :--- |
| **Reactive HPA** | 指标(CPU/QPS)超阈值 | 中 (~15s - 1min) | 通用 Web 服务，不可预测流量 | 滞后性，存在冷启动缺口 |
| **CronHPA** | 定时任务 | 快速 (精确到分) | 周期性明显业务(如打卡) | 无法应对突发异常流量 |
| **Predictive KEDA** | 算法预测历史+当前 | 慢 (需学习数据) | 周期规律强，启动慢的任务 | 预测不准可能导致资源浪费 |
| **Manual** | 人工/脚本介入 | 最快 | 已知的大促活动 | 运维成本高，需人工值守 |

**Result：** 
*   高峰期自动扩容到 8 个 Pod,低谷期缩容到 2 个 Pod.
*   资源成本降低 35%(对比固定 8 Pod 配置).
*   高峰期 P99 延迟始终保持 < 5s.

## 常见考点
1.  **Cool Down Period**：为什么扩容容易，缩容难？（缩容需要冷却期防止指标抖动反复销毁创建；扩容通常希望越快越好）。
2.  **Cluster Autoscaler**：HPP（Pod 扩容）和 Cluster Autoscaler（节点扩容）的配合顺序？（先扩 Pod，Node 资源不足时再触发 Node 扩容）。
3.  **指标延迟**：监控数据的采集和上报延迟对 HPA 有什么影响？（可能导致扩容滞后，造成雪崩）。

## 记忆要点

- 监控指标：CPU/内存基础指标+QPS/P99延迟业务指标+队列Lag自定义指标。
- HPA策略：QPS>50扩容，<20持续5分钟缩容，扩快缩慢防抖动。
- 预测扩容：基于历史数据CronHPA提前扩容，应对已知高峰。
- 冷启动保护：新Pod就绪后才挂载Service，配合镜像预热。
