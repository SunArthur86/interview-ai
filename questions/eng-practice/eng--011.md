# 请设计一个企业级智能客服 Agent 平台的完整架构,支持多租户和水平扩展.

- **架构设计**

  - **接入层**
    - API Gateway: 认证、限流、路由
    - WebSocket: 支持流式输出

  - **核心层**
    - **意图识别**: BERT 分类器(8类意图) → 路由到对应处理链
    - **Agent 编排器**(状态机 + DAG 混合)
    - ReAct Agent: 通用问答
    - RAG Agent: 知识库检索
    - SQL Agent: 数据查询
    - 反思 Agent: 质量校验

  - **服务层**
    - **多路检索引擎**: 向量 + BM25 + 混合 RRF
    - **记忆系统**: Redis(短期) + Milvus(长期)
    - **工具系统**: 搜索/计算器/DB 查询 + MCP
    - **模型路由**: 多模型候选 + 三态熔断器

  - **支撑层**
    - **文档 ETL**: PDF/Word → 分块 → 向量化 → 入库
    - **全链路追踪**: OpenTelemetry
    - **配置中心**: Prompt/工具/模型版本管理

- **多租户方案**
  - 数据隔离: 每租户独立 Collection(Milvus)
  - Prompt 隔离: 每户可自定义 system prompt
  - 权限隔离: RBAC + API Key
  - 资源配额: 每租户 QPS/Token 限制

- **水平扩展**
  - 无状态 API(状态存 Redis)
  - Milvus 集群分片
  - 模型推理用 vLLM + Tensor Parallel

- **关键指标**
  - P95 延迟 < 3s
  - 幻觉率 < 5%
  - 可用性 99.9%
  - 支持 1000+ 并发

  **实战案例**: 在双11大促期间，某电商客服租户的流量突增 10 倍，导致整体 API 响应超时。**排查**: 发现该租户的知识库检索未做分片隔离，占用了集群大量 IO 资源，影响了其他租户。**优化**: 引入租户级 `priority_weight`，在 Milvus 查询层和 vLLM 推理层均实施基于权重的队列调度，确保 VIP 租户不被饿死。

  **核心代码片段**:
  ```python
  # 租户感知的模型路由器伪代码
n  class TenantAwareRouter:
      def route(self, tenant_id: str, query: str):
          # 1. 获取租户配置 (模型版本、优先级)
          config = ConfigCenter.get(tenant_id) 
          
          # 2. 根据租户配额检查限流
          RateLimiter.check(tenant_id, config.quota)
          
          # 3. 选择推理实例 (优先调度到负载低的节点或专用节点)
          if config.premium:
              instance = ModelCluster.get_instance(tier="high")
          else:
              instance = ModelCluster.get_instance(tier="standard")
              
          # 4. 携带 Trace 上下文调用推理
          with tracer.start_as_current_span("llm_inference") as span:
              span.set_attribute("tenant_id", tenant_id)
              return instance.generate(query, model=config.model_name)
  ```
