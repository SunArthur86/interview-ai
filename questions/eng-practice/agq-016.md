# 如何设计 Agent 的错误恢复和重试机制

**Situation：** 生产环境中 Agent 执行过程会遇到各种错误:LLM API 超时、工具调用失败、检索无结果等.简单的重试可能导致资源浪费或无限循环.
**Task：** 设计一套智能的错误恢复机制,让 Agent 能优雅地处理各种异常情况.
**Action：** 
1. **分层错误处理**:
   - **可恢复错误**: 网络超时、API 限流 → 指数退避重试(最多 3 次).
   - **可降级错误**: 主模型不可用 → 切换到备用模型.
   - **不可恢复错误**: 权限不足、参数错误 → 向用户报告错误原因.
2. **指数退避重试**:
   ```python
   retry_delay = base_delay * (2 ^ attempt) + random_jitter
   base_delay = 1s, max_delay = 30s, max_attempts = 3
   ```
   - **Jitter (抖动)**：增加随机性，避免多个客户端同时重试导致「惊群效应」，加重服务端负担。
3. **Agent 级别的自愈**:
   工具调用失败时,Agent 会收到错误信息,由 LLM 决定下一步:
   - 换一种方式调用工具(修改参数).
   - 尝试使用替代工具.
   - 告知用户无法完成并解释原因.
4. **断点续传**:
   复杂任务的中间状态持久化到 Redis.
   系统崩溃后可以从最近的检查点恢复执行.
   - **State Management**：不仅保存最终状态，还保存当前的 Thought、Action 序列，便于 ReAct 模式的回溯。
5. **兜底策略**:
   - 设置最大步数限制，防止陷入死循环。
   - 引入 "Human in the Loop"，当 Agent 连续 3 次尝试失败后，自动请求人工辅助。

**实战案例：**
在一个自动化的数据报表生成任务中，Agent 因为 SQL 查询结果集过大导致 OOM（内存溢出）崩溃。由于没有做断点保存，每次重试都要从头开始跑一遍聚合查询，耗时极长。优化后，我们在每一步 Tool Execution 后将 Checkpoint 存入 Redis，失败恢复时直接跳过已完成的计算步骤，大幅提升了容错效率。

**代码示例（Python - Tenacity 实现指数退避重试）：**
```python
from tenacity import (
    retry,
    stop_after_attempt,
    wait_exponential,
    retry_if_exception_type
)
import requests

# 实战：针对网络抖动和 5xx 错误进行智能重试
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=1, max=10),
    retry=retry_if_exception_type((requests.Timeout, requests.ConnectionError)),
    reraise=True
)
def call_llm_api(prompt: str):
    response = requests.post("https://api.example.com/v1/chat", json={"prompt": prompt}, timeout=5)
    response.raise_for_status()
    return response.json()
```

**错误类型处理对比：**
| 错误类型 | 判断依据 | 处理策略 | 人工介入 | 恢复成本 |
| :--- | :--- | :--- | :--- | :--- |
| **瞬时故障** | Timeout, 503/504, Rate Limit | 指数退避重试 | 否 | 低 (秒级) |
| **参数/逻辑错误** | 400 Bad Request, Validation Error | 终止任务，返回具体错误提示 | 是 (需修正输入) | 中 (需重启) |
| **服务降级** | 503 Service Unavailable (持续) | 切换备用 LLM / 简化工具逻辑 | 否 | 中 (有精度损失) |
| **权限/鉴权失败** | 401/403 Forbidden | 停止执行，触发告警 | 是 (需授权) | 高 (需运维) |

**Result：** 
- 因 API 超时导致的用户可见错误从 5% 降低到 0.3%.
- 模型降级机制确保了 99.5% 的可用性.
- 断点续传避免了复杂任务的重复执行,节省了约 15% 的 token 成本.

## 记忆要点

- 分层处理：可恢复(退避重试)、可降级(切模型)、不可恢复(报错)分类处理。
- 退避策略：指数退避+随机Jitter，避免惊群效应，最多重试3次。
- Agent自愈：LLM根据错误信息决定换参数、换工具或终止，具备推理能力。
- 断点续传：中间状态存Redis，崩溃后从最近Checkpoint恢复，避免重复执行。
