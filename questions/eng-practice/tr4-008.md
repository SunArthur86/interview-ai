# 在推理服务网关设计中，如何实现基于 Token 的限流与请求优先级调度？

在 LLM 推理服务中，传统的 QPS 限流不够精细，因为每个请求的长度和生成 Token 数差异巨大。更合理的策略是基于**计算量（如总 Token 数）**或**KV Cache 占用量**进行限流。实现上，网关需要维护每个连接的资源使用计数。对于优先级调度（如 SGLang 或 vLLM 的某些策略），通常采用**两级队列**：高优先级请求（如 VIP 用户）可以抢占低优先级请求的计算资源，或者进入“快车道”优先处理解码。常用的调度算法有**Maximal Batching**（尽可能打包）和**Continuous Batching**（迭代级调度）。此外，还需要实现**Predicted Forward Time** 机制，预测请求完成时间以便更公平地分片 GPU 时间片。
