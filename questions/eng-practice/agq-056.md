# 高并发下的限流降级策略是什么

**Situation：** 突发流量(如营销活动、批量查询)可能导致系统过载,需要保护核心服务不被打垮.

**Task：** 设计分级限流和优雅降级策略.

**Action：** 
1. **限流层次:**
   - **API 网关层:** 全局限流(总 QPS 上限 500).
   - **租户级限流：** 每个租户有独立的 QPS 配额(如 50 QPS).
   - **用户级限流：** 每个用户每分钟最多 20 次请求.

2. **限流算法:**
   - 使用 Redis + 令牌桶算法(Token Bucket).
   - 支持突发流量(桶容量 = 2倍的 QPS 配额).

3. **降级策略(分四个等级):**
| 等级 | 策略 |
| :--- | :--- |
| L0(正常) | 所有功能正常运行 |
| L1(轻微) | 关闭非核心功能(如推荐、个性化) |
| L2(中度) | 模型降级(复杂查询也走快速模型) |
| L3(严重) | 只返回缓存和FAQ回答,不调用LLM |

4. **自动降级触发条件:**
   - QPS > 阈值的 80% → L1.
   - LLM API 错误率 > 10% → L2.
   - 系统 CPU > 90% 或内存 > 85% → L3.

5. **实战案例（新增）：**
   - 双11大促期间，某电商客服 Agent 问答量激增 10 倍。触发 L2 降级，系统将复杂的“多轮推理+检索”任务自动降级为“意图识别+匹配 FAQ 模板”，将平均响应时间从 3s 压降至 150ms，成功承接了全部流量，未发生漏单或客诉。

6. **关键代码示例（新增）：**
   ```python
   import redis
   
   r = redis.Redis()
   
   def is_allowed(user_id, limit=20):
       key = f"rate_limit:{user_id}"
       # Token bucket logic simplified using counter + TTL for sliding window
       current = r.incr(key)
       if current == 1:
           r.expire(key, 60) # Window of 60s
       return current <= limit
   ```

7. **限流算法对比（新增）：**
| 算法 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- |
| 令牌桶 | 允许突发流量，灵活性高 | 实现相对复杂 | 保护后端服务，应对瞬时高峰 |
| 漏桶 | 强制平滑流量，极度稳定 | 无法应对突发，可能有延迟 | 保护数据库，Nginx限流 |
| 固定窗口 | 实现简单，计数器即可 | 存在临界突变问题（边界突刺） | 简单的业务统计 |
| 滑动窗口 | 精度高，无边界问题 | 内存占用较高，实现较复杂 | 需要极高精度的API限流 |

**Result：** 
- 在 5 倍突发流量下,核心功能仍然可用.
- 降级响应延迟 < 200ms(缓存/FAQ).
- 零系统崩溃事故.

## 记忆要点

- 分层限流：网关总控、租户配额、用户限流，使用令牌桶算法允许突发。
- 四级降级策略：L0正常、L1关非核心、L2降模型、L3仅缓存/FAQ，逐级保护。
- 自动触发条件：QPS超阈值、LLM错误率高、资源占用高时自动切换降级等级。
- 核心目标：保核心功能可用，牺牲非必要体验，防止系统雪崩。
