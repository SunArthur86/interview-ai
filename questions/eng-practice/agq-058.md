# 模型输出格式异常怎么处理

**Situation：** LLM 有时不按预期格式输出(如应输出 JSON 但输出了纯文本,或 ReAct 格式解析失败). 这种非确定性输出是 Agent 系统不稳定的主要来源之一。

**Task：** 建立鲁棒的输出解析和纠正机制，确保系统在模型输出不规范时仍能稳定运行。

**Action：** 
1. **多层解析策略:**
   采用「漏斗式」处理链路，逐步放宽解析条件，确保兜底：
   **严格正则解析 → 宽松正则解析 → LLM 重格式化 → 降级处理**

   ```text
   用户请求 → LLM 输出 ───────────────────────────────────────────────────────┐
                │                                                            │
                ▼                                                            │
   ┌──────────────────────┐  成功      ┌──────────────────────┐               │
   │  Layer 1: 严格正则   │───────────▶│ 返回结构化数据       │               │
   └──────────┬───────────┘            └──────────────────────┘               │
              │ 失败                                                    │
              ▼                                                          │
   ┌──────────────────────┐  成功      ┌──────────────────────┐               │
   │  Layer 2: 宽松正则   │───────────▶│ 返回结构化数据       │               │
   │ (容错: 空格/大小写)   │            └──────────────────────┘               │
   └──────────┬───────────┘                                                    │
              │ 失败                                                    │
              ▼                                                          │
   ┌──────────────────────┐  成功      ┌──────────────────────┐               │
   │  Layer 3: LLM 修复   │───────────▶│ 返回结构化数据       │               │
   │ (基于 Prompt 修正)   │            └──────────────────────┘               │
   └──────────┬───────────┘                                                    │
              │ 失败                                                    │
              ▼                                                          │
   ┌──────────────────────┐                                               │
   │  Layer 4: 降级处理   │◀───────────────────────────────────────────────┘
   │ (纯文本/默认回复)    │
   └──────────────────────┘
   ```

2. **具体实现细节:**
   - **第一层(严格正则):** 尝试直接解析为 JSON 对象。如果抛出 `JSONDecodeError`，进入第二层。
   - **第二层(文本清洗 + 正则):** 使用正则提取 `{...}` 或 ```json ... ``` 块，去除 Markdown 标记，再尝试解析。
   - **第三层(LLM 矫正):** 将模型输出和错误信息喂给 LLM，Prompt: "The output is invalid JSON, please fix it..."

3. **实战案例（新增）：**
   - 在 Tool Use 场景中，LLM 经常在 JSON 参数前添加“好的，我会调用这个工具：”等废话。由于 Layer 2 正则专门提取 JSON 块，成功拦截了 95% 的此类非标输出，无需消耗额外 Token 重新生成，节省了约 15% 的推理成本。

4. **关键代码示例（新增）：**
   ```python
   import json
   import re

n   def safe_parse_json(response: str):
       try:
           return json.loads(response)
       except json.JSONDecodeError:
           # Layer 2: Regex extraction
           match = re.search(r'```json(.*?)```', response, re.DOTALL)
           if not match:
               match = re.search(r'({.*})', response, re.DOTALL)
           if match:
               try:
                   return json.loads(match.group(1))
               except json.JSONDecodeError:
                   pass
           # Layer 3: Fallback or LLM correction call
           return None # Trigger fallback
   ```

5. **解析策略对比（新增）：**
| 策略 | 成功率 | 延迟 | 成本 | 适用说明 |
| :--- | :--- | :--- | :--- | :--- |
| 严格解析 | ~60% | 极低 | 0 | 对 Prompt 工程要求极高，通常无法满足生产 |
| 正则清洗 | ~90% | 低 | 0 | **推荐首选**，解决 Markdown 和废话问题 |
| LLM 矫正 | ~99% | 高 | 高 | 作为中间兜底，消耗额外 Token 和时间 |
| 降级回复 | 100% | 最低 | 0 | 保底策略，牺牲功能换取可用性 |

**Result：** 
- 解析成功率从 65% 提升至 99.2%.
- 因格式错误导致的重试率降低 80%.
- 系统整体响应稳定性大幅提升。

## 记忆要点

- 漏斗式解析：严格正则 → 宽松正则(清洗Markdown) → LLM修正 → 降级处理。
- 文本清洗技巧：使用正则提取JSON块，去除模型生成的废话和标记符号。
- LLM自修复：将解析错误反馈给模型，要求其基于错误信息重新格式化输出。
- 兜底策略：解析彻底失败时，返回纯文本或默认回复，保证系统不崩溃。
