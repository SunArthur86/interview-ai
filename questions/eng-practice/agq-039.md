# 缓存策略是什么?命中率多少

**Situation：** 企业客服场景存在大量重复和相似的问题，每次都调用 LLM 浪费资源。

**Task：** 设计多层缓存策略，提升系统响应速度并降低成本。

**Action：** 
1. **多层缓存架构：**
   - **L1 -- 精确匹配缓存：**
     - Key：query 文本的 MD5 hash。
     - 适用场景：完全相同的问题。
     - TTL：24 小时。
     - 命中率：约 8%。
   - **L2 -- 语义相似缓存（Redis + Milvus）：**
     - 对 query 计算 embedding，在缓存向量库中检索。
     - 相似度 > 0.95 时视为命中，直接返回缓存的回答。
     - TTL：12 小时。
     - 命中率：约 7%。
   - **L3 -- FAQ 知识缓存（预计算）：**
     - 对 Top-200 高频问题预计算标准答案。
     - 通过意图匹配命中 FAQ。
     - 长期有效（手动更新）。
     - 命中率：约 5%。

2. **缓存更新策略：**
   - **被动失效：** TTL 过期自动失效。
   - **主动失效：** 知识库更新时，清除相关问题的缓存。
   - **预热策略：** 系统启动时预加载 FAQ 缓存和高频问题缓存。

3. **缓存一致性：**
   - 知识库文档更新时，触发缓存清理事件。
   - 使用事件总线通知缓存模块刷新。

**缓存架构流程图：**
```text
Query Request
      │
      ▼
┌───────────────────┐
│ L1: Redis Hash    │─────[Hit]───▶ Return Answer
│   (Exact Match)   │
└───────┬───────────┘
        │ Miss
        ▼
┌───────────────────┐
│ L2: Vector Search │─────[Sim>0.95]───▶ Return Cached Answer
│   (Milvus/Redis)  │
└───────┬───────────┘
        │ Miss
        ▼
┌───────────────────┐
│ L3: FAQ Intent    │─────[Hit]───▶ Return Standard Answer
│   (Top-N Pre-calc)│
└───────┬───────────┘
        │ Miss
        ▼
┌───────────────────┐
│  Full RAG Pipeline│───────────────▶ Generate & Update Cache
│  (LLM Inference)  │
└───────────────────┘
```

**Result：** 
- 总缓存命中率约 20%（L1 8% + L2 7% + L3 5%）。
- 缓存命中时延迟 < 100ms（对比 LLM 调用的 3s）。
- 节省约 20% 的 LLM API 调用成本。

### 深化内容

**实战案例：**
在某次大促期间，大量用户咨询“退款政策”，导致 LLM API 调用量激增。由于未设置随机 TTL 偏移，缓存集体失效，导致数据库和 LLM 服务瞬间压力过大。事后引入 **TTL 随机偏移（Base TTL ± 10%）** 策略，彻底解决了 Key 集体过期的雪崩问题。

**代码示例：**
```python
import hashlib
import random

def get_cache_key(query: str) -> str:
    # L1 精确匹配缓存 Key
    return f"chat:v1:{hashlib.md5(query.encode()).hexdigest()}"

def get_cache_ttl(base_ttl: int) -> int:
    # 增加 TTL 随机抖动，防止雪崩
    jitter = int(base_ttl * 0.1)
    return base_ttl + random.randint(-jitter, jitter)
```

**对比表格：**

| 维度 | L1 精确匹配 | L2 语义向量匹配 |
| :--- | :--- | :--- |
| **存储介质** | Redis String | Milvus / Redis Search |
| **计算成本** | 极低 (Hash 计算) | 高 (Embedding 推理 + ANN 检索) |
| **召回能力** | 仅限完全一致文本 | 可容忍同义改写 / 错别字 |
| **适用场景** | FAQ 标准问、高频重复问 | 长尾问题、口语化表达 |
| **空间消耗** | 小 (仅存 Key-Value) | 大 (需存浮点向量) |

## 常见考点
1. **如何解决“缓存幻觉”问题？** 
   *追问点*：如果知识库内容更新了，但旧缓存仍命中，会导致回答错误。除了主动失效，是否考虑了版本号控制或熵值校验。
2. **语义缓存（L2）的性能瓶颈在哪里？** 
   *追问点*：Embedding 计算耗时和向量检索的延迟。如何保证 L2 的查询时间远低于 LLM 调用时间（例如控制在 50ms 内）。
3. **缓存击穿/雪崩如何防护？** 
   *追问点*：对于热点 Key 过期瞬间，是否有互斥锁更新机制；对于大规模 Key 同时过期，是否设置了随机 TTL 偏移量。

## 记忆要点

- 三层架构：L1精确匹配(8%)、L2语义向量(7%)、L3预计算FAQ(5%)，总命中率约20%。
- L2语义缓存：计算Embedding检索，相似度>0.95命中，容忍同义改写。
- 一致性保障：知识库更新触发主动失效，配合TTL随机偏移防止雪崩。
- 性能收益：命中延迟<100ms，节省20% LLM调用成本。
