# 知识图谱是怎么构建和使用的

**Situation：** 纯向量检索难以处理实体关系类查询（如“张三负责哪些项目”、“A 产品和 B 产品有什么关联”），需要知识图谱补充。

**Task：** 构建领域知识图谱，并与向量检索系统协同工作。

**Action：** 
1. **知识图谱构建流程：**
   - **实体抽取：** 使用 LLM 从文档中抽取实体（人名、产品、部门、项目等）。
   - **关系抽取：** 使用 LLM 识别实体间关系（负责、属于、依赖等）。
   - **Schema 设计：**
     - 节点类型：Person, Product, Department, Project, Document
     - 关系类型：MANAGES, BELONGS_TO, DEPENDS_ON, AUTHORED, RELATED_TO
   - **存储：** Neo4j 图数据库。

2. **自动化构建管道：**
   文档 → 分段 → LLM 实体抽取 → LLM 关系抽取 → 去重合并 → 写入 Neo4j
   - 使用 Prompt 模板约束输出格式。
   - **人工审核：** 抽取结果的置信度 < 0.8 时进入审核队列。

3. **查询集成：**
   - 意图识别模块判断是否需要图谱查询。
   - 生成 Cypher 查询语句。
   - 图谱查询结果与向量检索结果融合后送入 LLM。

**Result：** 
- 知识图谱包含 5 万个实体、12 万条关系。
- 实体关系类查询的准确率从纯向量检索的 45% 提升到 87%。
- 知识图谱月均查询量约 2000 次。

### 深化补充

**实战案例：**
在使用 LLM 抽取实体时，曾出现严重的“实体歧义”问题（例如将“苹果”分别识别为“公司”和“水果”）。通过在 Prompt 中注入领域 Ontology（本体定义），明确限定实体的 Type 和 Scope，将同义实体统一归一化到标准 ID，解决了召回混乱的问题。

**代码示例（Cypher）：**
```cypher
// MATCH 查询示例：查询某项目的所有依赖组件及其负责人
MATCH (p:Project {name: $project_name})-[:DEPENDS_ON]->(c:Component)<-[:RESPONSIBLE_FOR]-(person:Person)
RETURN c.name AS component, person.name AS owner, c.version
```

**对比表格（向量检索 vs 知识图谱）：**

| 维度 | 向量检索 | 知识图谱 |
| :--- | :--- | :--- |
| **核心能力** | 语义相似性匹配，适合模糊搜索 | 精确实体关系推理，适合结构化查询 |
| **查询方式** | “找类似的内容” | “找A的关系网” |
| **优势** | 处理非结构化文本，泛化能力强 | 提供可解释的路径，处理多跳关系 |
| **劣势** | 难以处理复杂逻辑推理，可能有幻觉 | 构建成本高，Schema 变更困难 |

## 常见考点
1. **GraphRAG 与 RAG 结合方式**：是先查图再查向量，还是同时查？（答：通常根据意图路由，或者两者并行查询后进行重排序融合；也可以利用子图作为向量检索的过滤条件）
2. **图谱更新滞后性**：非结构化文档更新后，图谱如何实时更新？（答：通过监听数据变更消息触发增量抽取流程，或建立定时全量/增量校验任务）
3. **幻觉扩散**：LLM 抽取实体和关系时产生幻觉怎么办？（答：加入约束规则，只抽取文档中明确提到的关系，利用置信度阈值过滤，并支持人工修正闭环）

## 记忆要点

- 构建流程：LLM 抽取实体与关系，去重合并后存入 Neo4j。
- Schema 设计：定义节点(人/物)与关系(负责/依赖)，解决实体歧义。
- 查询集成：意图识别路由，生成 Cypher 语句，结果与向量检索融合。
- 适用场景：图谱处理实体关系推理，向量处理语义模糊搜索。
