# 熔断器是怎么实现的?三个状态是什么

**Situation：** LLM API 和外部服务经常出现间歇性故障,如果持续重试会浪费资源并增加延迟.需要一个熔断机制来快速失败和自动恢复.
**Task：** 实现 Circuit Breaker 模式,保护系统在下游服务异常时的稳定性.
**Action：** 
1. 三个核心状态:
   - **CLOSED(闭合/正常):** 请求正常通过,记录失败率。此时维护一个滑动窗口计数器。
   - **OPEN(断开/熔断):** 所有请求直接失败(返回降级响应),不再调用下游，保护系统不阻塞。
   - **HALF_OPEN(半开/探测):** 熔断一段时间后,允许少量探测请求通过,检测下游是否恢复。如果成功则认为服务已恢复，失败则重新熔断。

2. 状态转换规则:
   - **CLOSED → OPEN:** 最近 60s 内失败率超过 50%(至少 10 次请求)。防止偶发抖动导致误判。
   - **OPEN → HALF_OPEN:** 熔断 30s 后自动进入半开状态（冷却时间 Cooldown）。
   - **HALF_OPEN → CLOSED:** 连续 3 次探测成功 → 恢复正常。
   - **HALF_OPEN → OPEN:** 任何一次探测失败 → 重新熔断（无需再等 30s，立即快速失败）。

3. 实现细节:
```python
class CircuitBreaker:
    def __init__(self, failure_threshold=0.5, recovery_timeout=30, min_requests=10, probe_success_threshold=3):
        self.state = State.CLOSED
        self.failure_count = 0
        self.success_count = 0
        self.last_failure_time = None
        self.probe_success_count = 0
        self._window = deque(maxlen=100)  # 滑动窗口，记录最近请求结果

    async def call(self, func, *args, **kwargs):
        if self.state == State.OPEN:
            # 检查是否到了冷却时间，进入半开探测
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = State.HALF_OPEN
                self.probe_success_count = 0 # 重置探测计数
            else:
                raise CircuitOpenError("Circuit is open")
        try:
            result = await func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise
```

4. 降级策略:
   - **熔断时返回缓存的结果**：如果有旧缓存，返回并标记为“可能过期”。
   - **切换到备用服务**：如主模型熔断后切换到备用模型（降级服务）。
   - **返回兜底话术**：("系统繁忙,请稍后重试")。

**熔断器状态流转图：**
```text
      Failure Rate > 50%
      + Min Requests > 10
            │
            ▼
┌───────────────────────┐
│      CLOSED           │─────────────────┐
│ (Requests Pass)       │                 │
└───────────┬───────────┘                 │
            │                             │
            │  Success                   │  Timeout Reached
            │                             │
            ▼                             │
┌───────────────────────┐                 │
│    HALF_OPEN          │◀────────────────┘
│ (Probe Requests)      │
└───────────┬───────────┘
            │
      ┌─────┴─────┐
      │ Probe Fail│ Probe Success (Continuous)
      │           │ (x3)
      ▼           ▼
┌───────────┐ ┌───────────┐
│    OPEN   │ │  CLOSED   │
└───────────┘ └───────────┘
```

**实战案例：**
去年双十一期间，我们的 OpenAI 接入层因上游流量限制突发 502 错误。由于没有设置合理的“最小请求数”阈值，熔断器仅因前 2 个请求失败就直接熔断，导致全站服务不可用。修复后增加了 min_requests=5 的配置，有效避免了偶发网络抖动导致的误熔断。

**降级策略对比：**

| 策略 | 实现复杂度 | 用户体验 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **快速失败** | 低 | 差（直接报错） | 非核心链路，强一致性要求场景 |
| **返回缓存** | 中 | 中（数据可能旧） | 读多写少、对实时性要求不高的场景（如推荐列表） |
| **切换备用** | 高 | 好（无感知） | 核心业务，有多模型或多厂商冗余的场景 |
| **限流排队** | 中 | 差（等待时间长） | 系统过载保护，防止服务雪崩 |

**Result：** 熔断机制上线后，在外部服务不可用的情况下，系统 99% 的请求能在 10ms 内快速失败或降级返回，不再出现线程池耗尽导致的级联故障。
