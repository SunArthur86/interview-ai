# 代码质量是怎么保证的

**Situation：** AI Agent 项目涉及多个技术领域(LLM、检索、后端、DevOps)，代码质量参差不齐。
**Task：** 建立代码质量保障体系。
**Action：** 
1. **代码规范：**
*   **Python：** 遵循 PEP 8 + 内部补充规范。
*   **类型注解：** 所有函数参数和返回值必须有类型注解。
*   使用 ruff(linter + formatter)统一代码风格，替代 flake8 和 black 以提升速度。
2. **代码审查：**
所有 PR 必须至少 1 人 review。
核心模块(编排器、安全模块)的 PR 需要 2 人 review。
Review 重点：逻辑正确性、异常处理、性能影响、安全性。
3. **静态分析：**
mypy 类型检查(strict 模式)，禁止使用 `Any` 类型，确保类型安全。
bandit 安全扫描，检测常见漏洞（如 SQL 注入、硬编码密码）。
圈复杂度限制(单函数 ≤ 10)，复杂逻辑强制拆分。
4. **架构守护：**
模块依赖关系检查，防止循环依赖。
接口版本管理，防止破坏性变更。

**架构流程图：**
```text
┌─────────────┐    ┌──────────────┐    ┌──────────────┐
│  Local Dev  │───>│  CI/CD Pipe  │───>│  Merge Main  │
└─────────────┘    └──────┬───────┘    └──────┬───────┘
   Pre-commit             │                  │
(Ruff + Mypy)        ┌────▼─────┐        ┌────▼─────┐
                     │  SonarQ │        │  deploy  │
                     │ Scan/CC │        └──────────┘
                     └──────────┘
```

**实战案例**：某次代码合入后，Python 运行时因 `int` 和 `str` 隐式拼接导致服务崩溃。我们在 CI 流程中引入了 Mypy 的 `--strict` 模式，并在 Pre-commit hook 中强制执行，从此杜绝了此类低级类型错误。

**代码示例**：Pre-commit 配置 + 类型检查脚本
```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.1.9
    hooks:
      - id: ruff
        args: [ --fix ]
      - id: ruff-format
  - repo: local
    hooks:
      - id: mypy-strict
        name: mypy strict check
        entry: mypy . --strict
        language: system
        pass_filenames: false
```

**对比表格**：Python 静态分析工具对比
| 工具 | Ruff | Flake8 + Black | Mypy |
| :--- | :--- | :--- | :--- |
| **主要功能** | Linting + Formatting | Linting + Formatting (分开) | 静态类型检查 |
| **执行速度** | 极快 (Rust) | 较慢 | 较慢 |
| **配置复杂度** | 低 (统一配置) | 高 (需配置两套工具) | 中 (类型标注) |
| **发现Bug类型** | 风格、简单逻辑错误 | 风格、简单逻辑错误 | 类型不匹配、空指针 |

**Result：** 
代码审查覆盖率 100%。
类型检查覆盖率 90%。
线上 bug 密度降低 50%。

## 记忆要点

- 规范工具：Ruff统一Lint和Format，Mypy开启Strict模式，Pre-commit Hook强制检查
- 代码审查：核心模块2人Review，SonarQube查圈复杂度(≤10)和安全漏洞
- 架构守护：禁止循环依赖，接口版本化管理，破坏性变更必须升级Major版本
