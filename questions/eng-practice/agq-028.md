# 嵌入模型用的什么?维度多少?怎么选的

**Situation：** 嵌入模型的选择直接影响向量检索质量.需要在效果、速度、成本之间找到平衡.

**Task：** 选择最适合中文企业场景的嵌入模型.

**Action：**
1. 候选模型对比:
   | 模型 | 维度 | 中文能力 | MTEB 排名 | 速度 |
   | :--- | :--- | :--- | :--- | :--- |
   | BGE-M3 | 1024 | 优秀 | 前 5 | 中 |
   | text-embedding-3-large | 3072 | 良好 | 前 3 | 快(API) |
   | BGE-large-zh | 1024 | 优秀 | 前 10 | 中 |
   | m3e-large | 1024 | 优秀 | 前 15 | 中 |
   | Cohere embed-v3 | 1024 | 良好 | 前 5 | 快(API) |

2. 评估维度:
   - **中文效果：** 使用内部评估集(300 条中文 query-doc pair),计算 Recall@10.
   - **多语言支持：** 部分文档是英文,需要跨语言检索能力.
   - **部署成本：** 本地部署 vs API 调用的成本对比.
   - **维度和存储：** 维度越高存储越大,检索越慢.

3. 最终选择 -- BGE-M3:
   - 中文检索 Recall@10 最高(94.3%).
   - 原生支持中英日韩多语言和跨语言检索.
   - 支持稠密向量、稀疏向量、多向量三种表示.
   - 1024 维在效果和效率间取得平衡. 可以本地部署,无数据出境风险.

4. 部署方案:
   - 使用 ONNX Runtime 加速推理.
   - GPU 部署(T4): 单条 embedding 延迟 15ms, batch 1000 条约 3s.
   - 预计算 + 增量更新: 新文档入库时实时计算 embedding.

**Result：**
- 中文检索 Recall@10 达到 94.3%,比 text-embedding-3-large 高 2.1%.
- 嵌入维度 1024,单条向量占 4KB,800 万文档的索引大小约 32GB.
- 本地部署避免了 API 调用成本和延迟.

### 细节补充与参数说明
1. **维度选择权衡**：
   - 768维（如 BGE-base）：精度略低，索引构建快，内存占用小。
   - 1024维（BGE-M3）：精度与性能的黄金平衡点，支持 Matryoshka Representation Learning（可以动态截断维度，如取前 256 维用于快速粗排）。
   - 3072维（OpenAI large）：精度最高，但存储成本高且检索延迟显著增加。

### 深化内容

**实战案例**：
在一次电商 FAQ 升级中，我们曾直接复用开源的 `m3e-base` 模型，但在处理“已发货但未收到货”这种相似语义的不同表述时，召回率极低。切换到 `BGE-M3` 并针对内部数据微调后，长尾问题的解决率提升了 40%。

**代码示例**：
```python
from FlagEmbedding import BGEM3FlagModel

# 初始化模型(使用 ONNX 以获得更快的推理速度)
model = BGEM3FlagModel('BAAI/bge-m3', use_fp16=True)

# 批量生成 Embedding (支持混合精度)
embeddings = model.encode(
    ["用户申请退款流程", "如何退款"], 
    batch_size=12, 
    max_length=512,
    return_dense=True,   # 稠密向量
    return_sparse=True,  # 稀疏向量(用于增强关键词匹配)
    return_colbert_vecs=False
)['dense_vecs']
```

**BGE-M3 与 OpenAI text-embedding-3-large 对比**：

| 特性 | BGE-M3 | OpenAI text-embedding-3-large |
| :--- | :--- | :--- |
| **部署方式** | 本地部署 (数据不出域) | 公有云 API (数据出境风险) |
| **向量维度** | 1024 (支持 Matryoshka 截断) | 3072 (可截断至 256/1024) |
| **中文能力** | 针对中文优化，行业术语强 | 通用能力强，但在垂直领域略弱 |
| **多语言检索** | 原生支持 (跨语言效果好) | 支持 (依赖翻译对齐) |
| **成本结构** | 硬件一次性投入 + 运维 | 按调用次数付费 (高频调用成本高) |
| **延迟** | ~15ms (本地 GPU) | ~200-500ms (网络 + 排队) |

## 记忆要点

- 首选 BGE-M3：1024 维平衡效果与成本，支持多语言及本地部署。
- 选型核心：中文 Recall@10 最优，支持稠密/稀疏/多向量三种表示。
- 维度权衡：768 维省空间，1024 维是黄金点，3072 维精度高但成本高。
- 部署优化：ONNX 加速，单条延迟 15ms，避免数据出境风险。
