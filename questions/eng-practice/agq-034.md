# 怎么做的 A/B 测试和 Prompt 版本管理

**Situation：** Prompt 的微小改动可能导致输出质量的显著变化，需要科学的方法来管理和评估 Prompt 变更。

**Task：** 建立 Prompt 版本管理和 A/B 测试机制。

**Action：** 
1. **版本管理：**
   - 所有 Prompt 模板存储在数据库中（不硬编码）。
   - 每个模板有版本号、创建时间、变更说明。
   - 支持按百分比灰度发布（如新版本只对 10% 的流量生效）。

2. **A/B 测试框架：**
   - 请求进来时，根据用户 ID hash 分配实验组。
   - 同时记录两个版本的回答和用户反馈。
   - 每个实验持续 7 天或累计 1000 次对话后出结论。

3. **评估指标：**
   - **自动化指标：** 回答相关性（BERTScore）、格式遵循率、工具调用准确率。
   - **人工评估：** 随机抽取 100 条对话，3 位评估者打分（1-5 分）。
   - **业务指标：** 用户满意度评分、问题解决率、平均对话轮次。

4. **决策标准：**
   - 新版本在所有指标上至少不低于旧版本。
   - 核心指标（回答相关性）提升 ≥ 2%，才接受新版本。
   - **统计显著性：** p-value < 0.05。

**Result：** 
- 通过 A/B 测试发现了 3 个看似改进实际退步的 Prompt 修改。
- Prompt 迭代周期从“凭感觉改”变为“数据驱动改”。
- 系统回答质量在 6 个月内稳步提升 18%。

### 深化补充

**实战案例：**
曾尝试通过 A/B 测试优化 RAG 检索后的“生成摘要”Prompt。新版本在 BERTScore 上略有提升，但用户点击率却下降了。分析发现是因为新版本生成的内容过于冗长。因此在后续优化中，增加了“输出长度控制在 200 字以内”的强约束指令，平衡了质量与体验。

**代码示例（Python - 伪代码）：**
```python
def get_prompt_template(user_id, template_name):
    # MurmurHash 确保分配均匀且稳定
    bucket = murmurhash3_32(user_id) % 100
    
    experiment = db.get_experiment(template_name)
    if bucket < experiment.traffic_percent:
        return experiment.prompt_b # 实验组
    else:
        return experiment.prompt_a # 对照组
```

**对比表格（离线评估 vs 在线 A/B 测试）：**

| 维度 | 离线评估 | 在线 A/B 测试 |
| :--- | :--- | :--- |
| **数据来源** | Golden Dataset（金标准集） | 真实用户流量 |
| **评估速度** | 快，即时反馈 | 慢，需积累足够样本 |
| **评估指标** | BERTScore, Rouge, 准确率 | 点击率, 转化率, 停留时长 |
| **局限性** | 可能与真实用户感知不符 | 实施成本高，存在短期风险 |

## 常见考点
1. **流量分配算法**：如何确保同一用户在同一会话中始终看到同一个 Prompt 版本？（答：基于 User ID 或 Session ID 的 Hash 分桶，并保证桶的稳定性）
2. **冷启动问题**：新 Prompt 版本没有数据时如何评估？（答：先使用 Golden Dataset（金标准数据集）进行离线评估，通过后再上线小流量灰度）
3. **多变量测试**：如果同时改了 System Prompt 和 Few-shot Examples，如何区分是哪个变量的影响？（答：使用多变量测试，但交互影响难以完全排除；通常建议控制变量法，单次实验只改动一个因子）

## 记忆要点

- 版本管理：Prompt 存库，版本号控制，支持按流量百分比灰度。
- A/B 测试：用户 ID Hash 分桶，记录双版本回答及反馈。
- 评估指标：自动化(BERTScore) + 人工打分 + 业务指标(解决率)。
- 决策标准：核心指标提升 ≥2% 且 p-value < 0.05 才上线。
