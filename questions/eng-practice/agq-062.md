# 怎么做的监控和告警

**Situation：** AI Agent 系统在生产环境中需要全方位监控：不仅是传统的系统指标，还包括 AI 特有的指标(幻觉率、工具调用成功率等)。
**Task：** 建立涵盖系统、业务、AI 三个维度的监控告警体系。
**Action：**
1.  **监控维度：**
    -   **系统层：** CPU、内存、磁盘、网络 I/O。Pod 状态、容器重启次数。数据库连接数、查询延迟。
    -   **业务层：** QPS、P50/P95/P99 延迟。错误率、超时率。用户活跃数、对话轮次分布。
    -   **AI 层(特有)：** LLM 调用延迟和成功率。Token 消耗趋势。检索命中率和 rerank 分数分布。幻觉率(基于 NLI 检测)。Agent 迭代次数分布(检测死循环趋势，若迭代次数 > N 则告警)。工具调用成功率和延迟。
2.  **监控工具栈：**
    -   **指标收集：** Prometheus (配置了 `/metrics` 抓取间隔为 15s)
    -   **可视化：** Grafana(定制了 AI Agent 专属 Dashboard)
    -   **日志：** ELK Stack(Elasticsearch + Logstash + Kibana)
    -   **链路追踪：** Jaeger (采用透传 TraceID 实现 Request Span 和 LLM Span 的关联)
    -   **AI 追踪：** LangFuse(记录每次 LLM 调用的输入输出)
3.  **告警规则：**
    -   **P1 (电话+钉钉)：** P99 延迟 > 10s，错误率 > 5% (持续 2 分钟)
    -   **P2 (钉钉)：** 幻觉率 > 5%，检索命中率 < 70%
    -   **P3 (邮件)：** Token 日消耗 > 200万
4.  **架构与数据流：**
```text
用户请求 (Agent Service)
   │
   ├─> 系统指标 (Node Exporter/ cAdvisor) ──> Prometheus ──> Alertmanager ──> 告警通道
   │
   ├─> 业务指标 (埋点 SDK) ─────────────────> Prometheus ──┘
   │
   └─> AI 上下文
         │
         ├─> 中间件/LLM Call (OpenTelemetry) ──> Jaeger ──> 链路分析
         │
         └─> 推理链路 ─────────────────────────> LangFuse ──> 可视化分析
```

**实战案例：** 某次上线后，发现 LangFuse 中 Agent 的平均步数从 3.2 突然激增至 8.5，触发 P2 告警。排查发现是向量库索引损坏导致检索失败，Agent 不断重试 ReAct 循环。如果不监控“思考步数”，仅看 QPS 和 Latency 可能会被误判为“用户问了复杂问题”，从而掩盖故障。

**代码示例（Python - Prometheus Metrics 埋点）：**
```python
from prometheus_client import Counter, Histogram, Gauge

# 定义 AI 特有指标
llm_tokens_total = Counter('llm_tokens_total', 'Total tokens consumed', ['model', 'status'])
agent_steps = Histogram('agent_execution_steps', 'Distribution of agent reasoning steps')
vector_db_similarity = Gauge('vector_db_similarity_score', 'Last retrieved doc similarity')

# 在业务代码中使用
def log_llm_call(model, tokens, success):
    status = "success" if success else "error"
    llm_tokens_total.labels(model=model, status=status).inc(tokens)

# 记录 Agent 步数
@agent_steps.time()
def run_agent_loop():
    # ... Agent logic ...
    pass
```

**对比表格（监控工具选型）：**

| 工具 | 主要用途 | 在 AI Agent 中的应用 | 优势 | 劣势 |
| :--- | :--- | :--- | :--- | :--- |
| **Prometheus/Grafana** | 系统与业务指标监控 | 展示 QPS、Latency、GPU 使用率、Token 耗量 | 生态成熟，告警强大 | 无法直接存储非结构化的 Prompt/Response |
| **Jaeger/Zipkin** | 分布式链路追踪 | 追踪请求经过服务、向量库、LLM 的全链路耗时 | 快速定位性能瓶颈节点 | 配置复杂，对 LLM 内部推理步骤追踪能力弱 |
| **LangFuse / Arize** | LLM 专用观测 | 记录每次对话的 Prompt、Response、Metadata，支持版本对比 | 专为 LLM 设计，支持 Trace 和 语义分析 | 需额外集成，非传统运维栈，存储成本较高 |

**Result：**
问题发现时间从“用户投诉后”提前到“问题发生后 3 分钟”。
监控覆盖率 100%(所有核心链路都有指标)。
月均误报率 < 5%(告警阈值经过多次调优)。

## 常见考点
1. **Grafana 面板配置**：你们针对 AI Agent 设定了哪些核心 Key Performance Indicators (KPIs)？（答：Token 效率、端到端成功率、平均检索分数、首字生成时间 TTFT）。
2. **幻觉率检测**：在生产环境中，你们是如何自动化检测“幻觉”的？（追问：是用了模型判别还是关键词规则？）（答：主要采用 NLI 模型判别生成内容与上下文的一致性，辅以关键词黑名单；成本允许时引入更强的 LLM 作为 Judge）。
3. **链路追踪**：在 LLM 多轮调用中，如何保证 TraceID 的透传一致性，特别是在涉及异步工具调用时？（答：使用 OpenTelemetry 的 Context API，将 TraceID 注入到线程局部变量或异步上下文中，确保跨进程/线程传递）。
4. **告警降噪**：面对 Agent 调用链路长导致的告警风暴，你们是如何做收敛或告警分级的？（答：使用 Alertmanager 的抑制规则，上游 P0 告警触发时自动抑制下游相关告警；设置告警聚合窗口，避免短时间内重复告警）。

## 记忆要点

- 三维监控：系统层(CPU/内存)、业务层(QPS/延迟)、AI层(幻觉率/Token消耗/工具成功率)
- 工具栈：Prometheus采集指标，Jaeger追踪链路，LangFuse记录LLM推理上下文
- 关键告警：P1报错率>5%，P2幻觉率>5%或步数激增，防止Agent死循环掩盖故障
