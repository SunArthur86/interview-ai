# 系统容量规划是怎么做的

**Situation：** 需要预估系统未来的资源需求，提前规划容量，避免资源不足导致 P0 故障或过度浪费导致成本失控。特别是 Agent 系统涉及昂贵的 GPU/Token 成本。

**Task：** 建立系统容量规划方法论。

**Action：** 
1. **容量模型与计算:**
    *   **关键资源评估:**
        *   **API Pod (网关/逻辑层):** 基于压测数据，单 Pod 峰值处理 25 QPS。设定安全系数 0.7（即 CPU 利用率不超过 70%）。公式：`Pods = (Peak_QPS * Buffer) / (25 * 0.7)`。
        *   **LLM Inference (GPU):** 核心成本点。根据 Prompt+Completion 平均 Token 数计算吞吐量（TPS）。例如 A100 80G 在 FP16 下推理 7B 模型可达 2000+ TPS。需预留显存用于 KV Cache。
        *   **Milvus (向量库):** 1000 万向量（768维）约需 48GB 内存。规划 3 节点（每节点 16GB，副本数为 1）。若需高可用，需增加副本数。
        *   **Redis:** 缓存热点数据约 10GB，会话状态约 5GB。规划 16GB 集群（主从+哨兵），预留 50% 内存用于 fork 操作和碎片。
        *   **PostgreSQL:** 对话记录约 500GB/年。提前规划 SSD 存储扩展和分区表策略。

    *   **架构依赖链与容量影响:**
        ```text
        用户流量 ──▶ [API Gateway] (CPU/Mem) ──▶ [Agent Logic] (CPU/Mem) ──▶ [Vector DB] (Mem/Disk)
                                                  │                         │
                                                  ▼                         ▼
                                             [LLM Service] (GPU) ◀── [Cache] (Mem)
        ```

2. **增长预测:**
   - **历史数据:** 基于过去 3-6 个月的 QPS、DAU、Token 消耗量。
   - **模型:** 采用线性回归或 Holt-Winters 时间序列模型预测未来 3-6 个月的峰值。
   - **业务驱动:** 结合市场活动（如双11、新功能上线）调整预测系数（通常增加 30%-50% Buffer）。
   - **评审机制:** 每季度评审一次容量规划，调整资源配置。

3. **成本优化:**
   - **实例策略:** 使用预留实例降低云服务成本（降低 30-40%）；Spot 实例用于离线批处理任务。
   - **弹性伸缩:** K8s HPA 基于 CPU/自定义指标（如 Queue Length）自动扩容；非高峰时段（夜间）自动缩容节省资源。
   - **分级算力:** 简单任务路由至小模型（如 Llama-3-8B），复杂任务路由至大模型（如 GPT-4），混合部署降低成本。

**实战案例：** 在某次“开学季”活动前夕，按照常规 QPS 预测预留了 GPU 资源，但因推广力度加大，长文本问答请求激增，导致 KV Cache 显存占用远超预期（长文本消耗是短文本的 4 倍），引发 OOM。后续修正了容量公式，引入“Token 吞吐量”而非单纯的“请求数”作为 GPU 扩容指标。

**代码示例（Python - 显存估算）：**
```python
import math

# 估算 LLM 推理显存占用 (简化模型)
# 模型参数 + KV Cache (Batch * Layers * Hidden * SeqLen * Bytes)
def estimate_gpu_memory(model_params_billion: int, batch_size: int, context_len: int, layers: int, hidden_size: int):
    # 模型权重显存 (FP16)
    model_mem = model_params_billion * 2 
    # KV Cache 显存 (估算值)
    kv_cache_mem = (batch_size * layers * hidden_size * context_len * 2 * 2) / (1024**3) 
    total_mem = model_mem + kv_cache_mem
    return math.ceil(total_mem * 1.2) # 预留 20% 碎片和操作显存

print(f"Required VRAM: {estimate_gpu_memory(7, 16, 4096, 32, 4096)} GB")
```

**对比表格（实例选型策略）：**

| 策略 | 适用场景 | 优势 | 劣势 | 成本对比 |
| :--- | :--- | :--- | :--- | :--- |
| **On-Demand (按需)** | 生产核心服务，负载波动小 | 稳定性高，数据不丢失 | 价格最高 | 1.0x (基准) |
| **Reserved (预留)** | 长期稳定运行的基础服务 (如 DB) | 折扣力度大 (约 30-50%) | 灵活性差，需承诺 1/3 年 | ~0.6x |
| **Spot (抢占)** | 离线任务、批处理、无状态 Worker | 成本极低 (可达 10%) | 可能随时被回收 | ~0.1x |

**Result：** 
系统在业务增长 300% 时平滑扩容，无服务中断。资源利用率保持在 60-80% 的健康区间。月度基础设施成本控制在预算范围内，Token 成本通过模型路由优化降低 25%。

## 常见考点
1. **GPU 估算细节**：如何计算 LLM 服务的并发能力？（答：需考虑 `Batch Size`、`KV Cache` 显存占用、`Time to First Token (TTFT)` 和 `Time Per Output Token (TPOT)`。公式：`并发数 = 显存大小 / (模型权重 + KV_Cache_Per_Request)`）。
2. **突发流量**：如果预测失败，瞬间流量翻倍怎么办？（答：启用限流保护核心服务；利用云服务的弹性能力快速扩容；对于无状态服务可快速水平扩容，对于有状态/GPU 服务通常需要预热时间，此时需降级服务）。

- 容量计算：API按QPS/单机能力算，GPU按Token吞吐量和KV Cache显存估算，预留30% Buffer

- 预测模型：基于历史数据用Holt-Winters预测峰值，结合业务活动调整系数，季度评审
