# 异步处理在哪些地方使用了

**Situation：** 系统中大量 I/O 密集型操作,同步处理会严重限制吞吐量.

**Task：** 全面采用异步编程,提升系统并发处理能力.

**Action：** 
1. **实时路径(async/await):**
   - 场景：LLM API 调用、向量数据库检索、Elasticsearch 查询.
   - 实现：用户请求处理(FastAPI 全异步框架)
   - **细节**：在 Python 中使用 `asyncio` 库，遇到 IO 阻塞时自动挂起协程，事件循环调度其他任务执行。

2. **准实时路径(异步任务队列):**
   - 场景：对话记录入库、用户行为日志写入、实时告警推送。
   - **作用**：将非核心路径的耗时操作剥离主线程，直接返回响应给用户。

3. **离线路径(Celery 任务队列):**
   - 场景：文档解析和索引构建、知识图谱更新、对话历史摘要归档、报表生成。
   - **策略**：使用 Broker（如 Redis/RabbitMQ）和 Worker 进程消费任务。

4. **并行编排:**
   - 对于无依赖的步骤（如多路检索），使用 `asyncio.gather` 并行触发。

**异步处理架构图：**
```text
User Request
      │
      ▼
┌───────────────────────┐
│   API Gateway / Web   │ (FastAPI Event Loop)
│   (Async Entry Point) │
└───────────┬───────────┘
            │
            ├───────────────┐
            │               │
            ▼               ▼
    ┌───────────────┐  ┌───────────────┐
    │ Core Sync     │  │ Offload       │
    │ (LLM/Search)  │  │ (Log/DB Save) │
    │ (async/await) │  │ (Fire & Forget)│
    └───────┬───────┘  └───────┬───────┘
            │                  │
            │            ┌─────▼─────┐
            │            │  Message  │
            │            │  Queue    │ (Redis/RabbitMQ)
            │            └─────┬─────┘
            │                  │

    ┌───────────────┐    │  Celery   │
    │  Response     │    │  Workers  │ (Offline Processing)
    │  to User      │    │  (Heavy)  │
    └───────────────┘    └───────────┘
```

**Result：** 
- 系统 QPS 从 30 提升到 85(2.8 倍).
- 文档索引速度提升 4 倍.
- 内存使用率更平稳(避免了同步阻塞导致的内存堆积).

### 深化内容

**实战案例：**
在接入 OCR 服务时，发现使用的 Python SDK 仅支持同步调用，导致 FastAPI 整个 Event Loop 被阻塞，系统吞吐量骤降。解决方案是使用 `loop.run_in_executor` 将同步的 OCR 调用扔进线程池执行，确保主线程不被阻塞，保持了高并发下的响应能力。

**代码示例：**
```python
import asyncio
from concurrent.futures import ThreadPoolExecutor

# 模拟一个不支持异步的阻塞 IO 函数
def blocking_io_task(file_path: str):
    # 例如：调用同步的 PDF 解析库
    return parse_pdf_sync(file_path) 

async def process_document(file_path: str):
    loop = asyncio.get_running_loop()
    # 使用线程池执行阻塞任务，释放 Event Loop
    result = await loop.run_in_executor(
        None, 
        blocking_io_task, 
        file_path
    )
    return result
```

**对比表格：**

| 维度 | Fire & Forget (MQ) | Async/Await (协程) |
| :--- | :--- | :--- |
| **执行时机** | 立即返回，后台消费 | 等待结果返回后才继续 |
| **可靠性** | 高 (支持持久化、重试) | 低 (进程重启则丢失) |
| **实时性** | 低 (毫秒级延迟) | 极高 (微秒级上下文切换) |
| **适用场景** | 日志记录、报表生成、邮件发送 | API 响应、数据库查询、LLM 调用 |

## 常见考点
1. **FastAPI 中的异步陷阱？** 
   *追问点*：如果调用的第三方库不支持异步（如标准的 `requests` 或某些 ORM 的同步操作），在 FastAPI 中使用会阻塞整个事件循环。必须使用支持异步的库（如 `httpx` 或 `SQLAlchemy 2.0 async`）或在线程池中运行同步代码（`loop.run_in_executor`）。
2. **如何处理异步任务中的失败重试？** 
   *追问点*：Celery 任务失败了怎么办？需要配置 `autoretry_for`，设置指数退避策略，并定义最大重试次数，避免无限重试耗尽资源。
3. **消息队列的顺序性保证？** 
   *追问点*：如果对日志入库的顺序有强要求，简单的多 Worker 消费会导致乱序。解决方案是使用单个分区/Consumer Group，或者在业务层增加时间戳/序列号进行重排。

## 记忆要点

- 实时路径：FastAPI全异步，LLM/向量库查使用async/await释放线程。
- 准实时：日志/入库等非核心操作Fire & Forget，主线程直接返回。
- 离线路径：重任务(文档解析/报表)扔进Celery队列异步消费。
- 同步陷阱：第三方库如不支持异步，用run_in_executor扔进线程池。
