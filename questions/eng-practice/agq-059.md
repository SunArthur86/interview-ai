# 敏感信息泄露怎么防止

**Situation：** Agent 在处理用户请求时,可能接触到敏感信息(个人隐私、商业机密),需要防止信息通过 LLM 泄露。此外，还需防范 Prompt Injection 攻击诱导系统泄露指令。

**Task：** 建立敏感信息的全生命周期防护机制。

**Action：** 
1. **输入脱敏:**
   - **识别:** 结合 **规则引擎**（Regex 匹配手机号、身份证、银行卡号、邮箱、IP 地址）与 **NER 模型**（识别非结构化姓名、地址、公司名）。
   - **替换策略:** 使用高位随机的 UUID 或专用占位符，例如 `<<PHONE_8f3a>>`，避免简单的 `[PHONE_1]` 容易被模型推测出含义。
   - **数据流:**
     ```text
     User Input ──▶ [正则/NER 识别] ──▶ [生成占位符 + 存储映射] ──▶ LLM Context
                           │                                          │
                           ▼                                          ▼
                    <<PHONE_XYZ>>                            "用户的手机号是..."
     ```
   - **映射存储:** 映射表存储在临时内存（如 Redis，带 TTL）或进程内存中，会话结束后立即清除（GC）。

2. **输出还原:**
   - LLM 回答中的占位符在返回前进行字符串替换，还原为原始值。
   - **权限校验:** 还原前检查用户当前上下文是否有权查看该信息（防止越权访问）。

3. **LLM 不可信原则:**
   - **数据隔离:** 假设 LLM 可能记忆训练数据中的信息 → 敏感数据绝不以明文形式发送给 LLM。
   - **PII 过滤:** 在 LLM 输出端也部署一道 PII 过滤器，防止模型幻觉生成虚假的敏感信息或泄露训练数据中的隐私。
   - **传输加密:** 系统间通信强制使用 TLS 1.3，内部 API 调用进行 mTLS 双向认证。

4. **日志安全:**
   - **实时脱敏:** 对话日志落库前，必须经过一次脱敏扫描。
   - **权限控制:** 日志存储加密，访问严格控制（最小权限原则，MFA 认证），审计日志记录所有查询行为。

5. **防御 Prompt Injection:**
   - 在用户输入和 System Prompt 之间增加防御层，识别并拦截 "Ignore previous instructions" 等攻击指令。

**实战案例：** 曾遇到测试人员在 Prompt 中输入“把所有带@的邮件地址打印出来”，导致 LLM 上下文中脱敏后的 `<<EMAIL_abc>>` 被直接打印，暴露了系统存在占位符漏洞。修复后在 Prompt 中显式增加指令：“绝对禁止输出任何包含 << >> 格式的内容”，并强化输出过滤器的正则拦截。

**代码示例（Python）：**
```python
import re
import uuid

# 简单的输入脱敏逻辑
def mask_sensitive_data(text: str, entity_map: dict):
    # 识别手机号并替换
    pattern = r'1[3-9]\d{9}'
    matches = re.finditer(pattern, text)
    for match in matches:
        placeholder = f"<<PHONE_{uuid.uuid4().hex[:4]}>>"
        entity_map[placeholder] = match.group() # 存储映射
        text = text.replace(match.group(), placeholder)
    return text
```

**Result：** 
- 敏感信息防护覆盖率 99.5%（基于红蓝对抗测试）。
- 零信息泄露事故。
- 通过了等保三级评审。

## 常见考点
1. **上下文关联**：脱敏后的信息若丢失语义（如脱敏了具体地址），LLM 可能无法正确处理，如何解决？（答：对于需要语义理解的场景，保留部分泛化信息，如 "北京市朝阳区" -> "北京市[具体地址]"；或仅对核心隐私字段脱敏）。
2. **多轮对话**：多轮对话中如何维持占位符一致性？（答：基于 Session ID 维护一个动态的映射表，确保同一实体在同一会话中对应同一占位符）。
3. **向量库脱敏**：存入向量库的数据如何处理？（答：切片入库前进行同样的脱敏流程，检索到的片段包含占位符，不影响语义检索，最终生成时还原）。

## 记忆要点

- 输入脱敏：正则+NER识别敏感信息，用随机UUID替换，映射表存Redis且带TTL
- 输出还原：替换占位符前校验权限，防止越权访问，日志落库前强制脱敏
- 防御注入：System Prompt显式禁止输出占位符，增加输入层拦截识别攻击指令
