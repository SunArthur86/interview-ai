# 大模型部署时,瓶颈通常在算力、显存、带宽还是网络?如何排查

- **四大瓶颈及排查方法**

1. **显存 (VRAM)** - 最常见瓶颈
- **症状**: OOM (Out of Memory)、无法扩大 Batch Size、甚至模型加载失败。
- **原理细节**: 显存主要由三部分占用：
  - **模型权重**: 静态占用，FP16 约占 2Bytes/param。
  - **KV Cache**: 动态占用，与 `batch_size * seq_len * num_layers * hidden_dim` 成正比。这是推理时显存激增的主因。
  - **激活值**: 前向传播时的中间特征图，训练时占用大，推理时可通过 recompute 换算力。
- **实战案例**: 在部署 Llama-3-70B 服务时，曾遇到长文本请求导致 OOM，排查发现是因为默认配置下 KV Cache 占用随上下文长度线性增长，即使模型能载入，并发处理长请求时也会瞬间撑满显存。
- **代码示例**:
```python
# 检查 PyTorch 显存分配详情
import torch
print(torch.cuda.memory_summary())
# 或使用 Nsight Systems 采样命令行
# nsys profile --stats=true python inference.py
```
- **排查**: 
  - `nvidia-smi` 观察 `Memory-Usage` 是否接近上限。
  - 使用 PyTorch 显存分析器 (`torch.cuda.memory_summary()`) 定位具体算子的显存分配。
- **解决**: 量化 (INT8/INT4/FP8)、PagedAttention (如 vLLM，将 KV Cache 视为操作系统的内存页管理)、模型并行 (TP/PP)、FlashAttention (减少激活值显存)。

2. **算力 (Compute/SM)**
- **症状**: `nvidia-smi` 显示 GPU 利用率 (`Volatile GPU-Util`) 持续接近 100%，但吞吐量 (tokens/s) 依然上不去，称为 **Compute Bound**。
- **原理细节**: 
  - 计算密集型场景（如 Prefill 阶段），受限于 GPU 的 Tensor Core 性能。
  - 如果算子实现未优化，或者数据类型不支持 Tensor Core (如 FP32)，会导致 SM (Streaming Multiprocessor) 空转等待。
- **实战案例**: 在迁移旧的 HuggingFace BERT 模型到新 GPU 时，发现 GPU 利用率 100% 但 TPS 很低，原因是未开启 `torch.bfloat16` 且使用了未优化的算子，导致 Tensor Core 无法发挥作用。
- **代码示例**:
```python
# 开启半精度以利用 Tensor Core
model = AutoModel.from_pretrained("...", torch_dtype=torch.bfloat16).cuda()
input_ids = input_ids.to(torch.bfloat16)
```
- **排查**: 
  - 使用 `NvProf` 或 `Nsight Systems` 查看 Kernel 执行时间占比，检查是否在 GEMM (矩阵乘) 上耗时过久，或者是否存在大量 Kernel Launch 开销。
  - 检查是否使用了半精度 (FP16/BF16) 以启用 Tensor Core。
- **解决**: FlashAttention-2/3 (融合 Kernel 减少 HBM 读写)、Tensor Core 优化、编译优化 (torch.compile/Triton)、增大 Batch Size 以提高并行度。

3. **访存带宽** - ** Decode 阶段核心瓶颈**
- **症状**: Decode 阶段 GPU 利用率低（例如 30%-60%），但延迟很高。这是因为算力跑不满，在等数据从显存搬运到计算单元。
- **原理细节**: 
  - Decode 是逐 token 生成，计算量小 (`O(d)`) 但需读取全部权重 (`O(d^2)`)，导致 **Arithmetic Intensity (计算密度/显存访问比)** 极低。
  - 显存带宽利用率受限，取决于 GPU 卡的显存位宽和频率 (如 A100 1.5TB/s, H100 3.35TB/s)。
- **实战案例**: 在低并发高吞吐的 A100 实例上测试 7B 模型，发现 TTFT（首字延迟）很快，但 TPOT（输出 Token 时间）受限于 PCIe 带宽或显存带宽，通过启用 4-bit 量化显著提升了 Decode 阶段的 TPS。
- **代码示例**:
```python
# Nsight Compute 监控显存带宽
# ncu --set full --page replay python inference.py
# 查看 DRAM Throughput 指标
```
- **排查**: 
  - `nvidia-smi dmon -s u` 观察 `fb util` (显存带宽利用率)。如果 `pcie` 利用率高而 `fb` 低，可能是数据传输瓶颈。
- **解决**: 增大 Batch Size (摊薄单 token 的权重读取开销)、模型量化 (减少权重的显存读取量)、使用显存带宽更高的显卡 (H100 > A100 > V100)。

4. **网络**
- **症状**: 分布式推理/训练时，网卡带宽打满 (`ibstat` 或 `ifconfig`)，GPU 存在空转等待 (`NCCL Watchdog`)，延迟波动大。
- **原理细节**: 
  - **Tensor Parallel (TP)**: 每个 Transformer Layer 需频繁做 `All-Reduce` 通信，对延迟极度敏感。
  - **Pipeline Parallel (PP)**: 微批次传输存在 bubble (空泡)。
  - 通信协议开销（TCP vs RoCE vs InfiniBand）。
- **实战案例**: 在多卡推理 Qwen-72B 时，使用 Tensor Parallel 并行，发现某张卡经常处于 NCCL Wait 状态，原因是交换机拥塞导致 All-Reduce 延迟抖动，改用 NVLink 服务器后问题消失。
- **代码示例**:
```python
# 设置 NCCL 调试日志排查网络问题
import os
os.environ["NCCL_DEBUG"] = "INFO"
os.environ["NCCL_IB_DISABLE"] = "0" # 确保开启 InfiniBand
```
- **排查**: 
  - 使用 `nccl-tests` (如 `all_reduce_perf`) 测试多卡间的实际带宽。
  - 监控 `nccl-debug` 日志或 `DCGM` 指标，查看网络重传或拥塞情况。
- **解决**: 使用 NVLink/NVSwitch、选择高性能网络 (InfiniBand > RoCE v2 > TCP)、通信与计算重叠 (Overlap)、减小 TP 度改用 PP。

**四种瓶颈特征对比表**

| 瓶颈类型 | 典型阶段 | GPU 利用率 (Volatile GPU-Util) | 显存带宽利用率 (fb util) | 核心矛盾 | 常见排查命令 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **显存 (VRAM)** | 初始加载 / 长上下文 | 不可用或 0% (OOM) | 不可用 | 容量不足 | `nvidia-smi`, `torch.cuda.memory_summary` |
| **算力** | Prefill (填坑) | **~100%** (算满了但吞吐低) | 中/高 | 计算速度跟不上 | `nsys profile`, `nvprof` |
| **访存带宽** | Decode (生成) | **低 (~30-60%)** (算力等数据) | **~100%** (搬运数据占满) | 数据搬运速度慢 | `nvidia-smi dmon -s u`, `ncu` |
| **网络** | 分布式/多卡 | 低或波动大 | 低 | 节点间通信慢 | `ibstat`, `nccl-tests`, `watch nvidia-smi` |

## 记忆要点

- 显存瓶颈：OOM，KV Cache是主因，用量化/分页解决
- 算力瓶颈：GPU利用率100%吞吐低，用Flash Attention/半精度优化
- 带宽瓶颈：Decode阶段利用率低，受限于显存带宽，用量化/增大Batch优化
- 网络瓶颈：分布式训练网卡打满，GPU空转等待数据传输
