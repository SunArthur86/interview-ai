# 你的系统支持多租户吗?怎么实现的

**Situation：** 系统需要同时服务多个企业客户,每个客户有自己的知识库、配置和数据隔离要求.
**Task：** 设计多租户架构,确保数据隔离、配置独立、性能互不影响.
**Action：** 
1. **数据隔离策略(三层隔离)**:
   - **应用层(入口)：** 网关层通过 JWT 或 Header 解析 `tenant_id`，并透传给所有下游服务，确保上下文不丢失。利用 ThreadLocal 或 AsyncContext 在服务内部传递租户信息。
   - **数据层(存储)**：
     - **知识库隔离：** 每个租户在 Milvus 中使用独立的 Collection。在大数据量下， Partition 方案虽然物理上共用存储，但逻辑隔离，若数据量极不均衡（如某租户占 90% 数据）可能会影响查询性能，故采用独立 Collection 物理隔离，单租户故障不影响其他租户索引。
     - **对话数据隔离：** 每个租户的对话记录存储在独立的数据库 Schema 中（或 PostgreSQL 的 Schema 级别）。在 SQL 执行层面强制拼接 `WHERE tenant_id = ?`，利用数据库中间件（如 ShardingSphere）进行 SQL 拦截和重写，防止开发人员漏写过滤条件导致越权。
     - **缓存隔离：** Redis 中使用 `tenant_id:` 作为 key 前缀，并利用 Redis 的 DB 编号或 KeySpace 通知进行逻辑隔离，避免 Key 冲突。
2. **配置独立**:
   - **每个租户可以自定义：** System Prompt、工具集合、模型选择、检索策略。
   - 配置中心化存储，支持热更新。通过发布/订阅机制通知服务节点刷新本地缓存，确保无需重启服务。
3. **资源限制**:
   - **配额管理：** 每个租户有独立的 QPS 限制和 token 配额。
   - **限流算法：** 使用 Redis + Lua 脚本实现令牌桶或滑动窗口算法，保证分布式环境下限流的准确性。
   - **降级策略：** 超出配额时返回 429 Too Many Requests，并携带 Retry-After 头信息。

**实战案例：**
曾遇到 A 租户进行大批量数据导入（ETL 任务），占用了 Milvus 90% 的 IO 资源，导致同机器上的 B 租户查询超时。我们将“共享 Collection + Partition 模式”迁移为“每个大租户独立 Collection”，并为小租户保留共享 Collection。虽然增加了元数据管理复杂度，但彻底解决了由于资源竞争导致的 Noisy Neighbor 问题。

**代码示例（Go - Gin 中间件注入租户上下文）：**
```go
import (
    "github.com/gin-gonic/gin"
    "context"
)

type TenantContext struct {
    TenantID string
}

// 实战：中间件解析 Header 并注入 Context
func TenantMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        tenantID := c.GetHeader("X-Tenant-ID")
        if tenantID == "" {
            c.AbortWithStatusJSON(401, gin.H{"error": "Missing Tenant ID"})
            return
        }
        // 设置请求上下文，传递给后续 Handler
        c.Request = c.Request.WithContext(context.WithValue(c.Request.Context(), "tenant", tenantID))
        c.Next()
    }
}
```

**数据隔离方案对比：**
| 维度 | 独立 Database/Collection | 共享 Database + Schema/Filter | 行级安全 (Row-Level Security) |
| :--- | :--- | :--- | :--- |
| **隔离级别** | 物理（完全隔离） | 逻辑（应用/SQL层面） | 逻辑（数据库引擎层面） |
| **数据安全性** | 最高 | 中（依赖代码规范） | 高（数据库强制） |
| **运维成本** | 高（备份、迁移繁琐） | 中 | 低 |
| **扩展性** | 差（跨租户查询难） | 好 | 好 |
| **适用场景** | 金融/核心数据、超大租户 | 中小租户、SaaS 初始阶段 | 需要跨租户统计、高合规要求 |

## 记忆要点

- 三层隔离：网关解析TenantID透传，数据层独立Collection/Schema，缓存Key加前缀。
- 存储策略：大租户独立Collection防Noisy Neighbor，小租户共享降低成本。
- 配置独立：租户自定义Prompt/模型/工具，配置中心热更新。
- 资源限制：Redis+Lua实现分布式限流，配额管理防资源抢占。
- 安全加固：SQL拦截强制拼接TenantID，防止开发漏写导致越权。
