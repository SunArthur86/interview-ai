# 记忆系统的短期和长期记忆是怎么配合的

**Situation：** Agent 需要在单次对话中维护上下文(短期记忆),同时在跨会话中记住用户偏好和历史交互(长期记忆).

**Task：** 设计分层记忆系统,实现短期和长期记忆的协同工作.

**Action：** 
1. 短期记忆:
   - **存储位置：** 内存(进程内 + Redis 备份).
   - **内容：** 当前会话的对话历史、中间推理结果、工具调用结果.
   - **管理策略：**
     - **滑动窗口：** 保留最近 10 轮完整对话.
     - 超出窗口的对话自动摘要压缩.
     - 会话结束后 30 分钟过期清理.

2. 长期记忆:
   - **存储位置：** PostgreSQL(结构化信息)+ Milvus(语义检索).
   - **内容：** 
     - **用户画像：** 偏好、常用术语、业务领域.
     - **历史摘要：** 过往会话的关键信息摘要.
     - **FAQ 记忆: 用户高频问的问题和答案对.
   - **更新策略：** 
     - 会话结束时,提取关键信息更新用户画像.
     - 每日批量处理,对历史对话做摘要归档.

3. 短期 → 长期记忆的转化:
   - 会话结束时,LLM 自动提取本次对话的关键信息:
     - 用户提到的新偏好.
     - 解决的问题和方案.
     - 未解决的问题.
   - 提取结果写入长期记忆.

4. 长期 → 短期记忆的召回:
   - 新会话开始时,根据用户 ID 加载用户画像.
   - 根据当前话题,从长期记忆中检索相关历史信息.
   - 将召回的信息注入到当前会话的 context 中.

**实战案例：** 
在一次客服场景中，用户曾提到“我是急性子，直接给结果”。通过短期→长期的转化，该偏好被写入Postgres用户画像。两周后用户再次咨询时，系统在RAG检索前先从长期记忆召回该画像，Prompt动态调整为“请直接给出结论，不要长篇大论”，显著提升了用户体验。

**代码示例 (Python - 记忆写入伪代码)：** 
```python
async def update_long_term_memory(session_id: str, history: list):
    # 提取关键信息
    extraction_prompt = f"从对话中提取用户偏好和关键事实: {history}"
    insights = await llm.generate(extraction_prompt)
    
    # 并行写入结构化库和向量库
    await asyncio.gather(
        pg.execute("UPDATE users SET preferences = $1 WHERE id = $2", insights, session_id),
        milvus.insert(data=[insights], metadata={"type": "preference", "sid": session_id})
    )
```

## 记忆要点

- 分工明确：短期记忆存当前会话(Redis)，长期记忆存画像与摘要(Postgres+Milvus)。
- 转化机制：会话结束由LLM提取关键偏好和事实，写入长期记忆库。
- 召回机制：新会话启动时，根据用户ID和话题检索相关历史，注入Context。
- 管理策略：短期记忆滑动窗口保留最近10轮，超限摘要压缩；长期记忆定期批量归档。
