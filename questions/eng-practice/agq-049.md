# Agent 死循环怎么检测和防止的

**Situation：** ReAct 模式下,Agent 可能陷入"思考-行动-观察"的无限循环.例如检索结果不满意时反复重新检索,或反复调用同一工具但参数不变.
**Task：** 设计检测和防止 Agent 死循环的机制.
**Action：** 
1. 硬性限制:
  *   **最大迭代次数**：`max_iterations = 10`,超过强制结束.
  *   **最大执行时间**：`max_execution_time = 60s`,超时强制结束.
  *   最大 token 消耗:单次请求总 token ≤ 30000,防止成本失控.

2. 智能检测:
  *   **重复动作检测**：记录最近 N 步的工具调用指纹. 如果连续 2 次调用相同工具且参数 Levenshtein 相似度 > 90%,判定为循环.
  *   **进度检测**：每 3 轮迭代后检查是否有实质性进展(新信息增量、状态变化).
  *   **回路检测**：维护一个状态集合，如果当前状态（Thought+Obs的摘要）已存在于集合中，判定为循环.

3. 处理策略:
  *   **软阻断**：检测到循环后,在下一次 Thought 的 Prompt 中注入系统提示：“检测到重复动作，请尝试换一种思路或直接结束。”
  *   **硬阻断**：如果注入提示后仍然循环 → 强制进入 Final Answer 阶段.
  *   **兜底回答**：强制结束时,基于已有信息生成最好的回答,并标注“信息可能不完整,因执行超时终止”。

4. 根因分析和预防:
  *   所有死循环事件记录日志,定期分析根因.
  *   **常见根因**：工具描述不清导致选错工具、Prompt 指令矛盾、检索结果质量差.
  *   针对性优化 Prompt 和工具描述.

**Agent 状态检测流程图：**
```text
[Start Loop]
    │
    ▼
[Generate Thought & Action]
    │
    ├─> [Check History State] ──(Match)──> [Inject Hint]
    │                                      │
    │                                      ▼
    ├─> [Check Global Limits] ──(Exceed)─> [Force Final Answer]
    │        (Time / Steps / Token)
    │
    ▼
[Execute Tool]
    │
    ▼
[Update History & State]
    │
    └──────> [Next Loop]
```

**Result：** 
*   死循环发生率从初始的 3% 降低到 0.2%.
*   99.8% 的请求在 5 轮迭代内完成.
*   死循环导致的用户投诉降低 95%.

**实战案例**：在电商查询 Agent 中，曾因 SKU 库存接口返回“暂无数据”未明确告知无货，导致 Agent 误以为网络抖动，连续 10 次重试查询同一商品，消耗了单用户 $0.5 的 Token 成本。后引入“无数据”作为终态检测，解决该问题。

**代码示例**：
```python
def detect_repetition(history: list, max_repeats=2):
    if len(history) < max_repeats: return False
    last_actions = [step['action'] for step in history[-max_repeats:]]
    # 检查工具名和参数是否完全一致
    if len(set(last_actions)) == 1:
        return True
    return False
```

**常见考点**
1.  **LangChain 中实现**：LangChain 的 AgentExecutor 是如何限制步数的？（`max_iterations` 参数）。
2.  **状态哈希**：如何高效地检测“回到了之前的状态”？（对 Observation 进行摘要或 Hash，避免全量文本比对）。
3.  **子任务拆解**：从架构层面如何减少死循环？（Plan-and-Solve 模式，先生成计划再执行，减少盲目尝试）。
