### 实战案例（踩坑经验）
在从 LangChain 迁移到自研架构时，我们曾遇到 LangChain 内部不可见的异步等待问题，导致在高并发下线程池耗尽。通过剥离核心编排逻辑并实现自定义的异步非阻塞 I/O 模型，我们解决了这个瓶颈。

### 关键代码实现（性能优化：Prompt 模板缓存）
```python
# Python: 使用 LRU Cache 缓存 Jinja2 模板渲染结果以减少 CPU 开销
from functools import lru_cache
from jinja2 import Template

# 模板对象通常可复用，避免重复编译
_template_cache = {}

def get_template(template_str: str) -> Template:
    if template_str not in _template_cache:
        _template_cache[template_str] = Template(template_str)
    return _template_cache[template_str]

@lru_cache(maxsize=1000)
def render_prompt_cached(template_key: str, context_hash: str):
    # 实际渲染逻辑，利用 hash 加速缓存命中
    return get_template(template_key).render(**decode_context(context_hash))
```

### 技术选型对比（自研 vs LangChain）

| 维度 | LangChain | 自研架构 |
| :--- | :--- | :--- |
| **开发速度** | ⭐⭐⭐⭐⭐ (开箱即用) | ⭐⭐ (需从零构建) |
| **调试难度** | ⭐ (过度封装，链路追踪难) | ⭐⭐⭐⭐⭐ (完全可控，日志清晰) |

### 常见考点
1. **技术选型理由**：为什么最终没有使用 LangChain，而是选择自研部分组件？（提示：LangChain 过度封装导致调试困难、性能损耗、企业定制需求）
2. **性能优化细节**：P50 从 6.5s 降到 2.8s，优化点具体在哪里？（提示：并发检索、Prompt 模板缓存、流式传输、模型量化推理加速）
3. **模型路由策略**：在模型路由中，如何根据 Question 的难度动态选择使用 7B 参数模型还是 GPT-4？（提示：基于语义向量的简单分类器或 Bert 微调模型进行预判）
4. **熔断机制**：当 LLM 接口超时或限流时，Agent 的降级策略是什么？
