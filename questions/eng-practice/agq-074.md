# 用户反馈怎么样?怎么收集和处理的

**Situation：** AI 系统的效果需要持续基于用户反馈优化,但用户不一定愿意主动给反馈.
**Task：** 建立有效的用户反馈收集和处理机制.
**Action：** 
1. 反馈收集渠道:
**显式反馈：** 每个回答后提供"有帮助/无帮助"按钮 + 可选的文字反馈.
**隐式反馈：** 
用户重复提问(说明上一次回答不满意).
用户转人工(AI 未能解决问题).
对话轮次过多(可能在反复沟通而未解决).
**定期调研：** 每月向活跃用户发送满意度问卷.
2. 反馈处理流程:
用户反馈 → 自动分类(满意/不满意/建议)→ 不满意反馈人工审核
→ 根因分析(检索问题/生成问题/知识缺失/其他)
3. 数据驱动的优化:
每周分析不满意反馈 Top 10 问题类型.
跟踪优化后的指标变化.
建立"问题-优化-效果"的闭环.

### 数据闭环流程图

**实战案例：** 初期用户点“踩”后我们不知道具体原因，后续引入了“负反馈理由归类”，让用户勾选“答案错误/态度不好/未听清”，发现 30% 的差评源于 AI 语速过快，于是我们在 TTS 模块增加了语速自适应调节功能，差评率下降明显。

**关键代码示例：**
```python
# 伪代码：异步捕获用户点踩事件，附带当前 Session 上下文
def handle_thumbs_down(session_id, msg_id, reason):
    # 从 Redis 获取最近 3 轮对话上下文，保留现场
    context = redis_client.get(f"ctx:{session_id}")
    
    feedback_event = {
        "type": "negative",
        "reason": reason,
        "llm_output": context[-1]['content'],
        "retrieved_docs": context[-1]['source_ids']
    }
    # 发送至 Kafka 供后续离线分析
    kafka_producer.send('feedback_topic', feedback_event)
```

**反馈机制对比：**
| 维度 | 显式反馈 (点赞/踩) | 隐式反馈 (行为数据) |
| :--- | :--- | :--- |
| **数据准确性** | 高 (直接表达意愿) | 中 (需结合上下文去噪) |
| **数据规模** | 小 (通常 <5%) | 大 (覆盖所有交互) |
| **获取成本** | 低 (UI集成) | 中 (需埋点与计算) |
| **主要用途** | 验证模型回答质量 | 发现流程体验断点 |

**Result：** 
显式反馈收集率 15%(行业平均 5%).
基于反馈的优化使满意度从 82% 提升到 88%.
平均反馈处理周期 3 天.

## 常见考点
1. **如何处理“沉默的大多数”**：只有极少数用户会给反馈，如何利用 A/B 测试来间接验证效果？
2. **隐式反馈的准确性**：用户重试不一定是因为回答错了，可能是因为他表述不清，如何结合上下文时长综合判断？
3. **反馈延迟的处理**：如果用户点“踩”后 5 分钟才提交原因，如何关联之前的对话上下文？
4. **隐私合规**：收集隐式反馈（如对话录音）时，如何确保符合 GDPR 或数据安全法？

## 记忆要点

- 收集：显式点赞/踩，隐式重试/转人工，定期问卷，保留 Session 上下文。
- 处理：自动分类归因，分析 Top 10 问题，驱动知识库更新与 Prompt 调优。
- 闭环：反馈数据入仓，LLM 辅助标注，形成“问题-优化-效果”闭环。
- 案例：差评源于语速过快，增加负反馈理由归类，TTS 自适应调节。
