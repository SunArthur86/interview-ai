# 怎么设计的对话管理和上下文维护

**Situation：** 企业客服场景中多轮对话非常普遍,需要在多轮交互中维护一致的上下文,同时控制 token 消耗.
**Task：** 设计对话管理系统,在有限的 context window 内高效利用上下文信息.
**Action：** 
1. **分层上下文管理**:
   - **系统层：** System Prompt(固定,约 800 tokens),定义 Agent 角色和行为规范.
   - **会话层：** 对话历史(动态,滑动窗口),最近 N 轮对话.
   - **检索层：** RAG 检索到的知识上下文(动态,按相关性排序).
   - **工具层：** 工具调用结果(临时,用完即清).

2. **窗口管理策略**:
   维护最近 10 轮对话的完整记录.
   超过 10 轮时,前面的对话自动摘要压缩.
   **摘要保留关键信息：** 用户意图、关键决策、未解决的问题.

3. **Token 预算分配**:
   假设总 context window 为 8K tokens:
   - System Prompt: 800 tokens (10%)
   - 对话历史 + 摘要: 2400 tokens (30%)
   - **检索上下文：** 3200 tokens (40%)
   - **预留输出：** 1600 tokens (20%)

**实战案例**：在长对话中，系统曾因未压缩历史，导致上下文超出 Token 限制截断了最新的用户问题。优化后，当历史 Token > 2000 时触发异步摘要，确保新问题始终在 Prompt 内。

**关键代码：**
```python
# 滑动窗口与摘要逻辑
MAX_HISTORY_TOKENS = 2000

def build_context(history):
    current_tokens = count_tokens(history)
    if current_tokens <= MAX_HISTORY_TOKENS:
        return history
    
    # 超出预算，保留最近 3 轮，其余摘要
    recent_msgs = history[-6:] # 3轮 = user+assistant * 3
    old_msgs = history[:-6]
    
    summary = llm_summary("Summarize key info:", old_msgs)
    
    return [{"role": "system", "content": f"History Summary: {summary}"}] + recent_msgs
```

**Result：** 
- 多轮对话的上下文一致性从 76% 提升到 94%(基于人工评估).
- token 消耗控制在预算内,无超限导致的截断问题.
- 支持最长 50 轮对话不丢失关键上下文.

## 常见考点
- **摘要的信息丢失**：压缩后的摘要是否会丢失用户的个性化语气或细节？（回答：通常会保留事实性信息，可能丢失语气，但在摘要 Prompt 中可以要求保留关键风格）
- **滑动窗口 vs 摘要**：为什么选择摘要而不是仅仅丢弃最早的对话？（回答：摘要保留了早期的关键意图，而丢弃可能导致上下文断裂）

## 记忆要点

- 分层管理：System(10%)、History(30%)、RAG(40%)、Output(20%)分配Token预算。
