# KV Cache是什么？它如何加速LLM推理？有哪些优化方法？

🎯 **本质**：KV Cache是LLM推理核心优化技术，通过缓存已计算过的Key和Value矩阵避免重复计算，将自回归生成的复杂度从O(n²)降到O(n)。

📊 **为什么需要KV Cache**：
Transformer自回归生成时，每生成一个新token需要计算Attention：Attention(Q,K,V) = softmax(QKᵀ/√d_k)·V。
- **不缓存**：第n步需重新计算前n-1个token的K和V -> O(n²)。
- **缓存KV**：前n-1个token的K,V已缓存，只需计算第n个 -> O(n)。

🏗️ **数据流原理图**：
```text
Step 1 (Token A):  Calc K_A, V_A -> Cache -> Output Token B
                  ┌───────────────────────────────┐
Step 2 (Token B):  │ Cache: [K_A, V_A]              │ <- 从缓存读取
                  └───────────────────────────────┘
                  Calc K_B, V_B -> Append -> Cache -> Output Token C
                  ┌─────────────────────────────────────┐
Step 3 (Token C):  │ Cache: [K_A, V_A, K_B, V_B]         │ <- 复用历史KV
                  └─────────────────────────────────────┘
                  Calc K_C, V_C -> Append -> Cache -> Output Token D
```

💾 **KV Cache内存开销计算**：
计算公式：`Memory = 2 * num_layers * num_heads * seq_len * head_dim * sizeof(dtype)`
*注：2表示K和V两个矩阵*

以Llama-7B为例（32层, 32头, head_dim=128, fp16）：
- 单token KV Cache = 2 * 32 * 32 * 128 * 2 bytes ≈ 0.5 MB
- 长度4096时 = 0.5MB * 4096 ≈ **2GB**。
注意：当batch size > 1时，显存占用成倍增加，KV Cache可能远超模型权重本身。

⚡ **KV Cache优化方法**：
1.  **PagedAttention（vLLM）**：类似OS虚拟内存分页，将KV Cache分为固定大小block（如16个token/block），按需分配管理，解决碎片化问题，显存利用率提升至90%+。
2.  **KV Cache量化**：从fp16（2 bytes）量化到int8（1 byte）甚至int4（0.5 byte），减少50-75%显存，配合SmoothQuant等算法可保持精度。
3.  **KV Cache压缩/淘汰**：
    - **Sliding Window**：只保留最近N个token（如4096）的KV，丢弃更早的。
    - **StreamingLLM**：引入Attention Sink（保留初始少量token如首句）+ 滑动窗口，支持无限长度生成而不崩溃。
    - **H2O**：基于 Heavy-Hitter Oracle 保留重要token。
4.  **Prefix Caching**：多请求共享相同system prompt或prompt前缀的KV Cache，减少重复计算（常用于Chat服务）。
5.  **MQA/GQA**：减少KV头的数量，将MHA中多组KV合并为一组或少数几组。Llama 2/3使用GQA。

## 常见考点
1.  **Prefix Caching的实现难点**：面试官可能问如何存储共享前缀（使用RadixTree/前缀树结构），以及多轮对话中如何处理变动的Cache。
2.  **Flash Attention与KV Cache的关系**：Flash Attention优化的是Attention Score的计算过程（减少HBM读写），虽然输出包含K/V，但它本身并不改变KV Cache的存储逻辑，两者通常是配合使用。
3.  **KV Cache量化后的计算**：量化后的KV Cache在Attention计算前可能需要De-quantization（反量化），或者使用支持Int8计算的算子，这可能会引入额外的计算开销，是典型的Trade-off。
4.  **显存边界条件**：在最大Sequence Length限制下，如何计算Batch Size的最大值？（显存总量 - 模型权重 - 激活值）/ 单请求KV Cache大小）。
