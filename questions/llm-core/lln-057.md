# RAG(检索增强生成)的完整pipeline是什么?每个环节有哪些优化方法

RAG = 检索 + 生成,通过外部知识库增强LLM的回答能力.

- **完整Pipeline:**
用户Query → [Query改写] → [检索] → [重排] → [上下文组装] → [LLM生成] → 回答

```text
        RAG 系统 Pipeline 详解

┌──────────┐      ┌───────────┐      ┌─────────────┐
│  User    │─────►│  Query    │─────►│   Dense &   │
│  Query   │      │ Transform │      │ Sparse Retr │
└──────────┘      └───────────┘      └──────┬──────┘
                                            │
                                            ▼
                                    ┌─────────────┐
                                    │   Rerank    │◄── Top-K Docs
                                    └──────┬──────┘
                                           │
                                           ▼
                                    ┌─────────────┐      ┌───────────┐
                                    │  Context    │─────►│    LLM     │
                                    │  Assembly   │      │ Generation │
                                    └─────────────┘      └─────┬─────┘
                                                                │
                                                                ▼
                                                          ┌───────────┐
                                                          │  Answer   │
                                                          └───────────┘
```

- **各环节优化:**

- **1. Query处理(检索前):**
- Query改写:用LLM将模糊query改写为更精确的检索query
- Query扩展:添加同义词/相关词扩大召回
- Query2Doc:用LLM生成伪文档扩展query
- HyDE:生成伪文档用于向量检索
- 多query融合:生成多个子query分别检索

- **2. 索引构建:**
- 文档切分策略:固定长度/语义切分/递归切分
- Embedding模型选择:bge-large, text-embedding-3等
- 多级索引:摘要索引 + 细粒度索引
- 元数据过滤:按时间/类型/权限预过滤

- **3. 检索:**
- 向量检索(Dense):语义相似度,适合模糊匹配
- 关键词检索(Sparse/BM25):精确匹配,适合专业术语
- 混合检索:Dense + Sparse融合,效果最好

- **4. 重排(Rerank):**
- Cross-encoder模型:bge-reranker, Cohere Rerank
- LLM打分:让LLM对检索结果相关性评分
- 多信号融合:相关性 + 新鲜度 + 权威性

- **5. 上下文组装与生成:**
- 长上下文处理:Lost-in-the-middle问题(中间位置信息容易丢失)
- 文档压缩:RECOMP/LangChain的stuff/map_reduce策略
- 引用标注:让LLM标注信息来源

## 常见考点
1. **混合检索的优势**：解决了纯向量检索对专有名词（如型号、人名）匹配不准，而纯关键词检索无法理解语义的问题。

**【实战案例】**
在构建法律问答RAG时，发现直接检索“合同违约金”无法匹配到“罚金”相关的法条。引入 **Hybrid Search (BM25 + Vector)** 后，关键词召回匹配了“罚金”，向量召回匹配了“赔偿责任”，通过 Rerank 融合后，检索准确率从 65% 提升至 89%。此外，针对长文档，切分策略采用“父文档检索（索引子文档，返回父文档）”，有效解决了上下文碎片化问题。

**【代码示例】(Python/LangChain)**
```python
from langchain.retrievers import ParentDocumentRetriever, BM25Retriever
from langchain.vectorstores import Chroma

# 混合检索 + Rerank 示意图
bm25_retriever = BM25Retriever.from_documents(docs)
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 10})

# 简单的加权融合或使用 EnsembleRetriever
from langchain.retrievers import EnsembleRetriever
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.5, 0.5] # 实战中建议调参，通常向量权重略高
)
```

## 记忆要点

- 核心链路记忆口诀：改写 → 索引 → 检索 → 重排 → 组装 → 生成。
- 检索前：用HyDE生成伪文档或改写扩展Query，提升召回率。
- 索引与检索：混合检索（Dense语义+Sparse/BM25关键词）解决纯向量的专有名词盲区。
- 重排：用Cross-encoder对Top-K文档精排，解决向量召回的噪声问题。
- 生成端：警惕Lost-in-the-middle现象，采用Parent Document防上下文碎片化。
