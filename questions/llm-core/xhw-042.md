# 训练时数据长度不一致怎么办

## 问题核心
在 NLP 任务中，一个 Batch 内的句子长度往往不同。但 GPU 计算要求 Tensor 形状规则（矩阵运算），因此必须将短句子补齐到和长句子一样的长度。

## 解决方案详解

### 1. Padding + Mask (填充 + 掩码)
最基础的方法。通常在句子后补 0（Padding Token）。
*   **Padding**：将 Batch 中所有序列补齐到 `batch_max_len` 或全局 `max_len`。
*   **Attention Mask**：在 Transformer 的 Attention 计算中，将 Padding 位置的 Score 设为负无穷（`-1e9`），使其在 Softmax 后概率为 0。
*   **Loss Mask**：在计算 Loss 时，只计算非 Padding 部分的 Loss。

**实战案例**：在做长文档摘要时，若按全局最大长度（如 4k）进行 Padding，短文本（200 token）会浪费 95% 的显存，导致 Batch Size 只能设为 1，BN 层失效且训练极慢。改用动态 Padding 后，效率提升 5 倍以上。

**代码示例**：
```python
def collate_fn(batch):
    # batch 是 list of dicts, 每个 dict 包含 'input_ids'
    inputs = [item['input_ids'] for item in batch]
    # 动态获取当前 batch 的最大长度，而非全局最大
    max_len = max(len(x) for x in inputs)
    # Pad 到 batch 内最大长度
    padded_inputs = torch.nn.utils.rnn.pad_sequence(inputs, batch_first=True, padding_value=0)
    # 生成 attention mask (1为真实token, 0为padding)
    attention_mask = (padded_inputs != 0).long()
    return {'input_ids': padded_inputs, 'attention_mask': attention_mask}
```

### 2. 动态 Padding (Smart Batching)
为了减少 Padding 带来的无效计算（大量 0 参与矩阵乘法），采用**按长度排序 + 分桶**的策略。

**流程图**：
```text
原始数据集:
[A: len=10] [B: len=512] [C: len=15] [D: len=505] [E: len=12]
       
       1. 排序
       ↓
[A: 10] [E: 12] [C: 15] ... [D: 505] [B: 512]
       
       2. 分桶/组Batch (Batch Size=2)
       ↓
Batch 1: [A: 10], [E: 12] -> Pad 到 12  (浪费很少)
Batch 2: [C: 15], [X: 18] -> Pad 到 18
...
Batch N: [D: 505], [B: 512] -> Pad 到 512 (浪费主要在此，但 unavoidable)
```

**优化点**：使用 `collate_fn` 在 DataLoader 内部动态对每个 Batch 进行 Pad，而不是全局 Pad 到最大长度。HuggingFace 的 `default_data_collator` 或 `DataCollatorWithPadding` 即为此设计。

### 3. 截断
对于过长的序列（如超过 512 或 1024），必须截断。
*   **Head-only**：只保留头部（对于某些特定任务可能有用，但较少见）。
*   **Tail-only**：只保留尾部（最常用，保留最新的上下文）。
*   **Head + Tail**：保留头部的 128 个 token 和尾部的 382 个 token（保留开头主语和结尾谓语），中间截断。常用于长文本分类。

**对比表格**：
| 方案 | 显存利用率 | 计算效率 | 实现复杂度 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **全局 Padding** | 低 | 低 | 低 | 数据长度差异极小，推理场景 |
| **动态 Padding** | 中 | 高 | 中 | 长度参差不齐的 SFT 训练 |
| **Packing** | 极高 (~100%) | 极高 | 高 | 预训练，长文本上下文构建 |
| **截断** | 高 | 高 | 低 | 对上下文长度不敏感的任务，如分类 |

### 4. Packing (打包) - 进阶
不使用 Padding，而是将多个短句子拼接成一个长序列。
*   **原理**：`[句子A EOS] [句子B EOS] [句子C EOS]` 
*   **实现**：需要配合 Attention Mask 机制区分不同句子（或者不区分，只要计算 Loss 时忽略填充位即可，这里填充位其实是下一个句子的内容）。LLaMA 等模型预训练时常用此策略，GPU 利用率接近 100%。

**代码示例**：
```python
# 逻辑示意：将多个短样本拼成一个长序列
# 假设 max_len=1024, samples=[len=50, len=60, len=20, ...]
packed_input_ids = []
attention_mask = [] # 或者 position_ids
current_bucket = []
for s in samples:
    if len(current_bucket) + len(s) > 1024:
        # 桶满了，送去 training，重置
        packed_input_ids.append(current_bucket)
        current_bucket = []
    current_bucket.extend(s.input_ids + [eos_token])
# 注意：Transformer CUDA kernel 需要处理因果掩码和位置 ID
# position_ids 需要在每个句子交界处重置为 0，或者全局累加
```

## 常见考点
1.  **Padding 位置**：一般 Post-padding（右补齐）还是 Pre-padding（左补齐）？Transformer 解码器通常使用 Left Padding（补在左边），以保证生成时位置编码的正确性和因果性（Causal Mask）。
2.  **梯度累积与 Padding**：Padding 部分虽然不贡献 Loss，但梯度累积时如果不除以实际有效 Token 数，会导致梯度估计不准。需使用 `loss = masked_loss.sum() / actual_token_count`。
3.  **Packing 的实现**：如何处理 Attention Mask？打包后的 Mask 通常是全 1 矩阵（因为没有 Padding），但需要 Position IDs 来指示句子的边界或位置重置。
