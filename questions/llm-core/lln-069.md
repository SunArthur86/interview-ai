# 什么场景该用 RAG,什么场景该用 Fine-tuning?如何做架构决策

这是面试超高频题，决策框架如下：

**1. 选 RAG 的场景:**
- **知识频繁更新**：新闻、股价、私有文档（无需重新训练模型，只需更新向量库）。
- **需要引用来源**：合规审计、法律咨询、客服问答（可解释性强，能给出参考链接）。
- **知识量大但深度理解要求一般**：百科全书、产品手册。
- **多租户场景**：不同用户数据隔离，检索时带用户 ID 过滤。
- **快速验证**：原型开发阶段，低成本接入新知识。

**2. 选 Fine-tuning 的场景:**
- **改变模型行为或风格**：代码生成（特定语言风格）、方言口吻模仿、JSON 格式强约束输出。
- **特定领域深度推理**：医学诊断、法律判决逻辑（需要模型内化领域特有的因果推理链）。
- **延迟敏感**：RAG 有检索延迟（~50-200ms），FT 纯推理更快。
- **数据隐私要求高**：数据不能离域，必须在本地私有化训练（如金融核心数据）。
- **降低 Token 成本**：通过 FT 学会缩写，减少 Prompt 中的少样本示例长度。

**3. 两者结合最佳实践（C-RAG / RAG+FT）:**
先 Fine-tune 让模型理解领域术语（缩写、行话）和推理方式，再用 RAG 提供最新事实性知识。这是生产系统最常见的架构，兼顾了“懂行”和“懂新知”。

**4. 关键维度对比:**
- **知识更新**：RAG 秒级（更新向量库）；FT 慢且贵（重新训练）。
- **幻觉率**：RAG 较低（受限于检索内容）；FT 较高（模型可能会生成错误记忆）。
- **可解释性**：RAG 高（引用来源）；FT 低（黑盒）。
- **部署成本**：RAG 需维护向量数据库；FT 需维护多个 LoRA 权重或基础模型。

#### 对比表格

| 维度 | RAG (检索增强生成) | Fine-tuning (微调) |
| :--- | :--- | :--- |
| **核心能力** | 事实准确性、外部知识接入 | 风格模仿、深层逻辑内化 |
| **数据更新** | 实时（更新索引） | 离线（需重新训练） |
| **幻觉控制** | 较强（有据可依） | 较弱（可能产生错误记忆） |
| **部署成本** | 中（需维护向量库+LLM） | 高（需存储多个模型权重/LoRA） |
| **延迟** | 高（含检索时间 + LLM推理） | 低（纯 LLM推理） |
| **数据隐私** | 低（需上传至检索端） | 高（可本地训练） |
| **最佳适用** | 问答、知识库、即时新闻 | 对话风格、代码生成、特定格式输出 |

```text
+------------------+      +-------------------------+      +------------------+
│    用户 Question │ ---> │     检索 (RAG)           │ ---> │   检索到的 Docs   │
+------------------+      +-------------------------+      +------------------+
                                   │                                   │
                                   v                                   v
+------------------+      +-------------------------+      +------------------+
│    最终 Answer   │ <--- │   生成 (LLM / SFT)      │ <--- │  领域知识 (FT)   │
+------------------+      +-------------------------+      +------------------+

架构说明：
1. SFT 负责将 LLM 调教为“领域专家”，理解术语、语气和思维链。
2. RAG 负责向专家提供“参考资料”，确保答案基于最新事实。
```

#### 实战案例
在构建企业内部知识问答系统时，初期仅使用 RAG，发现模型对内部缩写（如“OGM”指“运营增长组”）理解困难。通过 1000 条内部对话数据对模型进行 SFT 后，模型学会了内部术语；再结合 RAG 获取最新的会议纪要，最终实现了既“懂行话”又“懂新事”的智能助手。

#### 关键代码 (LangChain/Router逻辑)
```python
from langchain.chains import RetrievalQA
from transformers import pipeline

# 混合架构：先检索，再由微调过的模型生成
retriever = vector_db.as_retriever()
ft_model = pipeline("text-generation", model="./my-domain-model")

def hybrid_chain(question):
    # 1. RAG 获取上下文
    docs = retriever.get_relevant_documents(question)
    context = "\n".join([d.page_content for d in docs])
    
    # 2. 构造 Prompt (包含检索到的内容)
    prompt = f"""Context: {context}
Question: {question}
Answer:"""
    
    # 3. 使用经过 SFT 的模型生成 (风格更专业)
    return ft_model(prompt, max_length=200)[0]['generated_text']
```

## 常见考点
1. **RAG 的检索精度问题**：如果检索不到正确文档，RAG 效果会不如通用模型，怎么解决？（答：混合检索、重排序 Rerank、Query 改写）。
2. **Catastrophic Forgetting（灾难性遗忘）**：Fine-tuning 后模型通用能力下降，怎么缓解？（答：正则化、多任务混合训练、LoRA 只训少量参数）。
3. **数据配比**：RAG 和 FT 结合时，SFT 数据应该包含 RAG 的检索上下文吗？（答：应该包含，让模型学会利用检索到的上下文，而非依赖记忆）。

## 记忆要点

- 核心对比：RAG 解决外部知识接入，而 Fine-tuning 内化深层逻辑与输出风格。

- 选 FT：因为侧重改变行为格式或追求极低推理延迟，所以适用特定代码生成与方言模仿。
