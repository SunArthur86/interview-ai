# LLM推理中有哪些关键加速技术？vLLM的核心原理是什么？

🎯 **本质**：LLM推理优化核心是最大化GPU利用率，主要围绕**KV Cache管理**、**批处理策略**和**计算内核优化**。

📊 **核心优化技术详解**：

1.  **Continuous Batching（连续批处理/Iteration Level Scheduling）**
    -   **痛点**：静态Batching中，如果一个请求生成10个token，另一个生成1000个，短的必须等长的完成，导致大量GPU空转。
    -   **方案**：在一个Iteration结束后，立刻将完成的请求移出，加入新请求。Batch内的每个序列独立计算，通过Padding或Index select处理。
    -   **效果**：吞吐量提升5-10倍，是现代推理引擎（如vLLM, TGI）的标配。

2.  **PagedAttention（vLLM核心）**
    -   **痛点**：KV Cache大小随序列长度动态增长，预分配最大长度（如4k）会导致平均利用率极低（<20%）。
    -   **方案**：借鉴操作系统虚拟内存。
        -   将KV Cache切分为固定大小的 **Block**（如16个Token为一个Block）。
        -   通过 **Block Manager** 管理物理显存（GPU）和逻辑映射。
        -   序列生成时，按需申请Block，不预分配。
    -   **效果**：显存利用率从20-40%提升到接近95%，支持更高的并发和Batch Size。

3.  **Kernel Optimizations (计算内核优化)**
    -   **Flash Attention**: 通过Tiling技术将Attention计算分块在SRAM中进行，大幅减少HBM（显存）读写次数，虽然是训练技术，但对推理长序列加速显著。
    -   **FlashDecoding**: 针对Batch Size大但序列短的场景优化，将Batch维度的计算并行化。

4.  **Prefix Caching (前缀缓存)**
    -   **方案**：自动识别请求的公共前缀（如System Prompt），将计算过的KV Cache Block存入缓存。新请求命中时直接复用。
    -   **效果**：TTFT（首Token延迟）降低50-90%，特别适合多轮对话和RAG场景。

5.  **Tensor Parallelism (TP, 张量并行)**
    -   **方案**：将模型权重矩阵切分到多张GPU上。计算矩阵乘法时，各卡算一部分结果后通过All-Reduce通信合并。
    -   **适用**：单卡显存放不下模型时。推理时通常TP优于Pipeline Parallelism（PP），因为PP会有Bubble（气泡）空转。

🏗️ **vLLM 架构简图**：
```text
┌───────────────────────────────────────────────┐
│              LLM Engine (vLLM)                 │
├───────────────┬───────────────┬────────────────┤
│   Scheduler   │ Block Manager │   Cache Engine │
│  (优先级队列) │  (PagedKV)    │  (Prefix Cache)│
└───────┬───────┴───────┬───────┴────────┬───────┘
        │               │                │
   ┌────▼────┐     ┌────▼────┐      ┌────▼────┐
   │ GPU 0   │ ... │ GPU N   │      │ CPU RAM │
   │(TP/Model)│     │(TP/Model)│      │ (Swapping)│
   └─────────┘     └─────────┘      └─────────┘
```

💥 **实战案例**：在高峰期，vLLM曾因OOM崩溃，原因是GPU显存碎片化严重。开启vLLM的`gpu_memory_utilization=0.9`并启用CPU Swap（虽然慢但防崩溃）后，服务稳定性显著提升，避免了因长请求挤占显存导致的500错误。

💻 **代码示例**：
```python
from vllm import LLM, SamplingParams

# 关键配置：启用前缀缓存，设置块大小
llm = LLM(
    model="meta-llama/Llama-2-7b-hf",
    enable_prefix_caching=True,  # 开启Prefix缓存复用
    block_size=16,               # 优化Block大小，适配大多数序列长度
    gpu_memory_utilization=0.95  # 激进显存分配，需留余量给CUDA kernels
)
```

|------|------|--------------|-------------------|

| 适用场景 | 高并发、多轮对话 | 极致性能、固定模型 | 通用部署、HF模型生态 |
