# GRPO算法是什么？DeepSeek为什么用它？

GRPO（Group Relative Policy Optimization）是DeepSeek提出的强化学习算法，旨在简化PPO的训练流程并降低资源消耗。

**与PPO的核心区别：**

*   **PPO**：需要一个独立的Critic Model来拟合Value Function V(s)，以计算Advantage。
*   **GRPO**：**不需要Critic Model**。它利用“群体”的输出结果来作为Baseline。

**算法原理（Group Sampling）：**
对于同一个Prompt，GRPO不进行单次采样，而是让Actor采样G个Outputs。

1.  **Group Sampling**：Actor为同一个Prompt生成 G 个 Responses: {y_1, y_2, ..., y_G}。
2.  **Outcome Scoring**：对每个Response计算Outcome（可以是Reward Model打分，也可以是规则结果，如数学题答案对错）。得到 Rewards: {r_1, r_2, ..., r_G}。
3.  **Baseline Calculation**：计算该Group的均值 μ 和标准差 σ。
4.  **Advantage Estimation**：
    A_i = (r_i - μ) / σ
    这相当于利用同一个Prompt下的其他样本作为控制变量，计算相对优势。

**优势分析：**
1.  **显存减半**：省去了Critic Model及其相关优化器状态，显著降低VRAM占用。
2.  **训练更稳定**：Critic本身就是一个拟合问题，容易不准；GRPO的Baseline基于当前Batch的实际Reward，理论上更贴合当前分布。
3.  **适合推理任务**：对于数学、代码等有明确“对/错”结果的任务，可以直接使用规则作为Reward，无需训练Reward Model。

**DeepSeek-R1的应用：**
- DeepSeek-R1使用GRPO实现推理能力涌现。
- **纯RL**：在SFT模型基础上直接进行RL（主要依靠GRPO），模型自发产生了思维链。
- **AHA moment**：在训练过程中，模型突然学会了自我反思和验证，正确率大幅跃升。
- **效果**：在Math、Code等Benchmark上超越了当时的OpenAI o1-preview等模型。

### 实战深化

**实战案例**：
在数学推理任务微调中，我们曾尝试用GRPO替代PPO。对于逻辑推理题，我们将“是否通过Python执行器”作为Reward信号。由于省去了Critic训练，单卡显存占用降低了约40%，使得在消费级显卡（如4090）上进行大模型RL成为可能。

**代码示例 (Python)**：
```python
# GRPO Advantage 计算伪代码
def compute_grpo_advantage(group_rewards):
    # group_rewards: Tensor of shape (batch_size, group_size)
    mean = group_rewards.mean(dim=1, keepdim=True) # 组内均值作为 Baseline
    std = group_rewards.std(dim=1, keepdim=True)   # 组内标准差用于归一化
    
    # 防止除以0
    std = torch.clamp(std, min=1e-8)
    
    # 相对优势：当前样本相对于组内平均水平的表现
    advantages = (group_rewards - mean) / std 
    return advantages
```

**对比表格**：

| 特性 | PPO (Proximal Policy Optimization) | GRPO (Group Relative Policy Optimization) |
| :--- | :--- | :--- |
| **价值评估** | 需要 Critic Model 拟合 V(s) | 无需 Critic，使用 Group 内均值 |
| **显存占用** | 极高 (4个模型：Actor, Ref, Reward, Critic) | 较低 (3个模型：无Critic，甚至可无RM) |
| **计算复杂度** | 需额外训练Critic，计算量大 | 仅需多次前向传播采样，逻辑简单 |
| **优势函数** | A(s,a) = Q(s,a) - V(s) (基于估计值) | A = (r - mean(group)) / std (基于实际统计) |
| **适用场景** | 通用RLHF，通用任务 | 推理、代码、数学等可验证结果的场景 |
| **训练稳定性** | 依赖Critic收敛情况，初期不稳 | 基于统计量，收敛更平滑稳定 |
