# 讲讲工业界RAG具体如何优化

工业界 RAG (Retrieval-Augmented Generation) 优化是一个系统工程，涉及 Index、Retriever、Reranker、Generator 全链路。

**1. 检索阶段优化**
*   **Query 理解与改写**：
    *   *Query Expansion*：利用 LLM 生成同义词、子问题，弥补语义匹配不足。
    *   *HyDE (Hypothetical Document Embeddings)*：让 LLM 先生成一个「假设性回答」，再用该回答的向量去检索，解决 Query 和 Document 语义空间不匹配问题。
    *   *Step-back Prompting*：让模型先退一步抽象出高层概念，再进行检索，适合复杂推理问题。

**2. 排序与过滤优化**
*   **Rerank (精排)**：Retriever（如BM25/Dense）召回 Top-50/100 后，使用 Cross-encoder（如 BERT-Rerank, Cohere Rerank）进行精准打分，截取 Top-5/10。
*   **去重**：对检索结果进行语义去重，避免上下文中充满重复信息。

**3. 生成阶段优化**
*   **上下文压缩**：检索出的文档可能包含噪声，使用 LLMLingua 等模型压缩 Context，去除非关键信息，保留 Prompt 的语义完整性。
*   **引用归因**：在 Prompt 中要求模型回答时标注来源 ID，提升可信度并便于 Debug。
*   **矛盾检测**：如果检索结果间存在冲突，训练模型或设计 Prompt 识别并指出矛盾，而非胡乱拼凑。

**4. 微调优化**
*   **RAFT (Retrieval-Augmented Fine Tuning)**：训练 LLM 在检索到的「干扰文档」中识别出「金标准文档」并据此回答。增强模型的抗噪和检索辨别能力。
*   **AAR (Attributed to Retrieved)**：微调模型使其生成内容严格基于检索内容，减少幻觉。

**实战案例**：
在构建企业内部知识库问答时，我们发现BM25对“行话”（如“618大促玩法”）检索效果极差，因为文档里全是“年中大促”。实战中，我们采用了“查询重写+混合检索”策略：先用LLM将用户Query重写为标准术语，再结合BGE-M3向量和BM25进行加权召回。此外，为了防止模型在找不到答案时瞎编，我们在Prompt中强制注入“若检索内容不相关，请直接回答不知道”，并配合Rerank模型将低相关性文档的分数置零，将幻觉率降低了40%。

**链路组件对比**：
| 组件 | 常用技术/模型 | 核心作用 | 优势 | 劣势 |
| :--- | :--- | :--- | :--- | :--- |
| **Retriever (召回)** | BM25, DPR, BGE-M3 | 从海量Doc中快速筛选Top-K | 速度快，覆盖面广 | 精度有限，语义鸿沟 |
| **Reranker (精排)** | BGE-Reranker, Cohere, T5 | 对Top-K进行精准打分排序 | 显著提升相关性 | 计算慢，增加Latency |
| **Context Comp** | LLMLingua, Selective Context | 压除噪声，保留关键信息 | 节省Token，降低干扰 | 可能损失细节信息 |
| **Generator** | Llama-3, Qwen (w/ SFT) | 基于Context生成答案 | 理解能力强，流畅度高 | 仍可能产生幻觉 |

## 记忆要点

- 检索阶段：用HyDE先造假设性文档再向量化，有效弥补Query与Doc语义空间鸿沟
- 精排去噪：双编码器广召回Top-K后，必接Cross-encoder精排做准确重打分
- 上下文处理：用LLMLingua压缩长文本，去除冗余降低大模型输入干扰
- 进阶微调：通过RAFT机制，强迫模型在干扰文档中精准识别金标准内容
