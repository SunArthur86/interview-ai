# 如何评估微调效果？有哪些关键指标和注意事项？

🎯 **本质**：LLM微调评估需要多维度衡量——不能只看loss，要全面评估能力保持、领域提升和潜在退化。

📊 **评估维度**：

**1. 自动评估**
  - **领域任务**：
    - 选择该领域的标准基准（如医学: MedQA, 法律: LawBench, 代码: HumanEval）。
    - 与基座模型对比，确认微调有效。
  
  - **通用能力保持**：
    - **MMLU**（多任务语言理解）：考察知识广度。
    - **GSM8K**（数学推理）：考察逻辑推理。
    - **C-Eval/CMMLU**：中文场景下的通用能力。
    - 确保微调没有导致“灾难性遗忘”。
  
  - **安全评估**：
    - 对齐性：是否产生有害内容。
    - 鲁棒性：对抗Prompt测试（如“忽略以上指令”）。

**2. 人工评估**
  - **盲评**：
    - 同时展示基座模型和微调模型的回答，打乱顺序。
    - 标注者选择更好的一个（Win/Tie/Lose），计算 Elo Rating。
    - 优点是贴近人类偏好，缺点是昂贵且慢。
  
  - **维度评分**：
    - 准确性：事实是否正确。
    - 完整性：回答是否全面。
    - 流畅性：语言是否自然。
    - 相关性：是否切题。
    - 风格：是否符合人设（如礼貌、专业）。

**3. LLM-as-Judge**
 用GPT-4等强模型评估微调模型输出。
  - **方法**：
    - **Pairwise comparison**：两两比较，输出 A>B, B>A, A=B。
    - **Pointwise scoring**：单独打分（1-10分）。
    - **Reference-based**：与标准答案对比。
  - **注意**：Judge模型不应与被评估模型相同（避免自我偏见），通常需评估Judge与人类的一致性。

**4. 训练指标**
  - **Loss曲线**：
    - Train loss 持续下降，Val loss 也下降 -> 正常。
    - Train loss 下降，Val loss 上升 -> 过拟合，应停止训练。
    - Loss 震荡不降 -> 学习率过大或数据有问题。
  - **评估频率**：
    - 每 N 步（如500步）在验证集上跑一次评估。
    - 保存最佳checkpoint（基于验证集指标，而非最后一个step）。

**实战案例**：某次代码生成模型微调中，Train Loss平稳下降，但在HumanEval上Pass@1反而掉了15%。排查发现训练数据中大量包含“解释代码”的低质量样本，导致模型学会了废话而非生成代码，剔除这些样本后效果回升。

**代码示例**：
```python
# 使用Judge模型进行Pairwise评估的简易实现
def pairwise_evaluation(judge_model, prompt, output_a, output_b):
    eval_prompt = f"""
    User: {prompt}
    
    Model A: {output_a}
    Model B: {output_b}
    
    请对比Model A和Model B的回答。输出格式：
    如果 A > B，输出 "A"
    如果 B > A，输出 "B"
    如果 A = B，输出 "Tie"
    只输出结果。
    """
    response = judge_model.generate(eval_prompt)
    return response.strip() # 返回 "A", "B" 或 "Tie"
```

**对比表格（评估策略）**：
| 评估方法 | 成本 | 速度 | 客观性 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **Loss/Perplexity** | 极低 | 极快 | 弱 (不一定与效果正相关) | 训练阶段监控、早期筛选 |
| **自动基准** | 低 | 快 | 强 (标准答案) | 准确性要求高的任务 (代码/数学) |
| **LLM-as-Judge** | 中 | 中 | 中 (存在Bias) | 通用QA、创意写作、格式化任务 |
| **人工评估** | 高 | 慢 | 强 (人类偏好) | 上线前的最终验收、Golden Set测试 |

**常见问题诊断**：
| 问题 | 症状 | 原因 | 解决 |
|------|------|------|------|
| 过拟合 | Train loss降，Val loss升，泛化差 | 数据太少/epochs太多 | 减少epochs/增加数据/正则化 |
| 灾难性遗忘 | 通用能力（MMLU）下降 | 只用领域数据 | 混入10%-20%通用数据 |
| 格式不稳定 | 输出JSON格式不对/跑出ChatML | 数据格式不统一 | 统一数据格式，增加格式正则约束 |
| 模式坍缩 | 总是输出相似内容/复读 | 数据多样性不足(Temperature低) | 增加数据多样性，调整采样参数 |
| 指令遵循差 | 不听指挥（如“请总结”） | Instruction数据质量差 | 优化Prompt，清洗指令数据 |

## 常见考点
1. **微调时 Loss 下降但人工评估效果变差是怎么回事？**：这通常被称为“目标不匹配”。模型可能学会了过拟合训练集的特定模式（如语气或格式），而牺牲了通用性或事实准确性，或者训练数据存在错误的标签。
2. **如何解决灾难性遗忘？**：数据混合（Mix 10% 通用数据）、参数高效微调（如只训练 LoRA 适配器）、多任务学习。
3. **LLM-as-a-Judge 的局限性**：Judge 模型可能存在 Position Bias（偏向第一个答案）或 Length Bias（偏向更长的答案），可以通过交换位置或提示词缓解。

## 记忆要点

- 评估维度：结合自动基准（准确率）、人工盲评（偏好）和LLM-as-Judge（性价比）。
- 训练监控：关注Loss曲线，Val loss上升意味着过拟合，需保存最佳Checkpoint。
- 能力保持：在领域任务测试外，必须测MMLU/GSM8K，确保未发生灾难性遗忘。
- 常见问题：Loss降但效果差通常是目标不匹配或数据标签错误，需检查数据质量。
- 评估策略：Loss监控训练，基准测能力，人工做验收，Judge做中间筛选。
