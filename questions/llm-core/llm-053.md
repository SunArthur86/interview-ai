# 介绍一下Qwen通义千问的结构

通义千问是阿里巴巴研发的大型语言模型系列。其核心架构基于 Transformer，主要包含以下关键技术点：

**1. 整体架构**
*   **基础组件**：采用 Decoder-only 的 Transformer 架构。
*   **位置编码**：使用 RoPE (Rotary Positional Embeddings) 增强位置信息的感知。
*   **激活函数**：使用 SwiGLU 替代传统的 ReLU 或 GeLU，提升模型表达能力。
*   **归一化**：采用 RMSNorm (Root Mean Square Layer Normalization)，提升训练稳定性。

**2. 词汇表与分词**
*   **词汇表大小**：约 152K，包含大量中文字符和多语言词汇，提高了中文和多语言的压缩效率，降低了推理成本。
*   **分词器**：基于 BPE (Byte Pair Encoding)。

**3. 训练方法**
*   **预训练**：使用了高达 3T tokens 的高质量多语言数据。
*   **长上下文**：通过 NTK-aware 插值、LogN-Scaling 等技术扩展上下文长度，支持长文本处理。
*   **对齐**：采用了 SFT (监督微调) 和 RLHF (人类反馈强化学习) 技术，特别是使用了 DPO (直接偏好优化) 进行对齐训练。

**4. 特殊设计**
*   **解耦嵌入**：输入嵌入和输出投影的权重不共享。
*   **QKV Bias**：在注意力的 QKV 层中添加偏置，增强模型外推能力。

**## 常见考点**
*   **RoPE 的原理**：为什么 RoPE 比绝对位置编码更适合外推？
*   **SwiGLU 的优势**：相比 GeLU，SwiGLU 多了什么参数量，带来的性能提升来源是什么？
*   **RMSNorm vs LayerNorm**：Qwen 为什么选择 RMSNorm？计算量减少了多少？

**实战深化**

*   **实战案例**：在使用 Qwen-14B 进行长文档摘要时，我们发现当输入长度超过 8k 后，模型开始出现“复读机”现象或逻辑断裂。通过引入 **NTK-aware Scaling** 动态调整 RoPE 的 base 值（从 10000 提升至 500000），在不重新训练的情况下，成功将有效上下文窗口从 8k 推理至 32k，虽然精度略降，但满足了 RAG 应用的长文档检索需求。

*   **代码示例 (RMSNorm 实现)**：
    ```python
    import torch
    
    class RMSNorm(torch.nn.Module):
        def __init__(self, dim: int, eps: float = 1e-6):
            super().__init__()
            self.eps = eps
            self.weight = torch.nn.Parameter(torch.ones(dim))
    
        def _norm(self, x):
            # Qwen 使用 RMSNorm 替代 LayerNorm，去掉了 mean 减法计算，加快推理速度
            return x * torch.rsqrt(x.pow(2).mean(-1, keepdim=True) + self.eps)
    
        def forward(self, x):
            output = self._norm(x.float()).type_as(x)
            return output * self.weight
    ```

## 记忆要点

- 基础架构：Decoder-only Transformer，使用RoPE位置编码和SwiGLU激活函数。
- 归一化：采用RMSNorm，去掉了均值计算，提升训练稳定性与速度。
- 分词器：基于BPE，词汇表约152K，优化中文和多语言压缩效率。
- 特殊设计：解耦嵌入（输入输出权重不共享），QKV层加Bias增强外推。
