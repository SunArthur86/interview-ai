# 为什么推理速度受限于显存带宽

**🫐理想情形：** 计算 VS 内存带宽
当我们在进行矩阵乘法（如多头注意力中的 $QK^T$、全连接层等）时，如果计算单元（CUDA 核心、张量核心等）的浮点性能（FLOPS）远大于显存可以提供的数据吞吐量，就会出现「带宽受限」的情况：

*   计算单元几乎处于「空转」或「等待数据」的状态，无法一直满负荷运转。
*   因为参数和中间激活值都需要从显存读取，当 L2/L1 缓存不够大或命中率不高时，每一步都要从显存搬运大量数据。

因此，从硬件角度来说，我们往往可以近似地认为：
$$\text{推理速度} \propto \frac{\text{每次生成 token 所需数据量}}{\text{显存带宽}}$$

### KV-cache 对时延的影响

*   当模型在生成第 $n$ 个 token 时，注意力模块需要读取前面所有 token 的 Key/Value。
*   如果上下文很长，KV-cache 会成为一个线性增长的开销：生成第 1 个 token 不需要前面上下文，生成第 $n$ 个 token 需要读 $(n-1)$ 份 KV。
*   对于深层网络，每层都要访问这部分 KV-cache，所以总访问量会乘以层数。

不过，对一个 7B 模型来说，每次都要加载约 14 GB 权重数据（在 FP16 下），对比之下 KV-cache（例如 130 MB）在早期并不是绝对主因。然而在更大模型（如 30B、70B+）或极长上下文中，KV-cache 和权重本身都会显著增加带宽压力。

### 补充：时延估算思路

🫐一个粗略的估算公式
假设我们有：
*   $BW$: GPU 显存带宽（单位: GB/s），例如 900 GB/s。
*   $ModelSize$: 模型大小（单位: GB），例如 14 GB。

则读取模型参数所需的时间为：
$$t_{model} \approx \frac{ModelSize}{BW}$$

假设带宽是 1000 GB/s，模型 14 GB：
$$t \approx 14.2 ms$$

这是非常理想的、纯带宽计算下限。实际推理中，还需要考虑 KV-cache 的读取、Kernel 启动延迟、指令调度等开销，实际速度会更慢。

### 为何「重复读取权重」的说法是近似的？

在实际推理框架中，模型在做推理 Forward 时，并不一定是每产生一个 token 就把所有层权重完整从 0 开始加载（因为权重通常已经在显存中）：

*   当前向计算经过每一层时，需要从显存搬运该层的参数到计算单元（寄存器/L1/L2）进行运算。
*   由于显存容量远大于 SRAM，权重无法全部驻留在 SRAM 中，因此随着层数加深，数据搬运量依然是线性的。
*   所以，「每生成一个 token，需要搬运 14 GB 数据」是一个「总数据搬运量 / 带宽」的思路，方便做理论的上限估算。带宽瓶颈依然存在。

### 实战案例与代码

**实战案例**：在做 LLM 推理加速优化时，发现同样的 A10G (24GB显存) 上，模型量化到 INT4（约 4GB）比 FP16（约 14GB）的 Decode 阶段速度并没有提升 4 倍，仅提升约 1.5-2 倍。**原因**：虽然计算量减少了，但 Decode 阶段主要受限于显存带宽，而非算力，INT4 主要节省了显存容量，使得 Batch Size 可以开得更大，吞吐量上去了，但单 Token Latency 受限于物理带宽上限，提升有限。

**代码示例**：
```python

        # 实际代码中通常通过 cudaGraph 或流水线优化隐式实现
        
        # 模拟带宽受限下的 Token 生成逻辑
        for layer in self.model.layers:
            # 1. 从 HBM 加载权重 -> SRAM
            weights = load_weights(layer) 
            # 2. 从 HBM 加载 KV Cache -> SRAM (大带宽消耗点)
            kv_cache = load_kv_cache(model_input.blocks)

            hidden_states = layer(weights, hidden_states, kv_cache)
            # 4. 写回 KV Cache (HBM, 大带宽消耗点)
            store_kv_cache(kv_cache)
```
