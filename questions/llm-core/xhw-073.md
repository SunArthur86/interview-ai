# DeepSeek-R1上PRM和MCTS是否有用

### DeepSeek-R1 上 PRM 和 MCTS 是否有用？

#### 1. 结论
在 DeepSeek-R1 的实际研究中，**过程奖励模型 (PRM)** 和 **蒙特卡洛树搜索 (MCTS)** 的效果**未达预期**，最终未被采用作为主要方法。取而代之的是更简单高效的 GRPO（Group Relative Policy Optimization）。

#### 2. PRM (Process Reward Model) 的局限性
- **计算开销大**：PRM 需要对推理链的每一步进行评分，在大规模 RL 训练中引入了巨大的计算成本。每个 Token 都需要经过 PRM 网络进行前向传播。
- **奖励黑客**：PRM 容易被模型利用，即模型生成专门让 PRM 打高分但实际逻辑不通的内容，导致评价失真。模型可能学会“讨好” PRM 而非真正解决问题。
- **性价比低**：相比于直接使用 GRPO 这种基于组内比较的方法，PRM 的额外收益不足以覆盖其带来的复杂度和成本。

#### 3. MCTS (Monte Carlo Tree Search) 的局限性
- **搜索空间过大**：不同于围棋或象棋有限的落子点，语言模型的词表极其庞大（通常数万级别），导致搜索树呈爆炸式增长，难以有效扩展。即使限制搜索宽度，搜索深度也非常有限。
- **局部最优**：虽然设置了扩展限制，但在巨大的空间中，MCTS 很容易陷入局部最优解，难以找到全局最优的推理路径。
- **价值模型依赖**：MCTS 的效果严重依赖于价值模型的准确性，而在语言场景下训练一个准确的价值模型本身就很困难。

**MCTS 在 LLM 中的挑战示意图**：
```text
传统游戏 (围棋):
   Root --> [~360个可选节点] --> 下一步 --> 终局评估
   (搜索空间可控)

LLM 推理:
   Root --> [~50,000个可选 Token] --> 下一步 --> [~50,000...] -> ...
   (指数级爆炸，无法穷尽或精准剪枝)
```

#### 4. 实战拓展

**实战案例**：
在尝试使用 MCTS 优化 LeetCode 代码生成任务时，发现树搜索在循环和条件判断节点极易发散，且为了评估一个叶节点的正确性需要执行代码，导致整体吞吐量相比直接 Beam Search 下降了 90% 以上，工程性价比极低。

**代码示例 (MCTS 节点扩展限制)**：
```python
# 简易 MCTS 逻辑 (用于说明局限性)
class MCTSNode:
    def expand(self, model, top_k=5):
        # 真实场景词表 32000，这里被迫只取 top_k=5 进行扩展
        # 这直接导致了搜索范围极其受限，容易错过正确路径
        next_tokens = model.predict_top_k(self.state, k=top_k)
        for token in next_tokens:
            self.children.append(MCTSNode(token))
    
    # 省略 selection, simulation, backpropagation...
```

| 方法 | 优点 | 缺点 (为何 R1 未采用) |
| :--- | :--- | :--- |
| **PRM (过程奖励)** | 提供细粒度步骤反馈 | 需训练额外大模型；推理慢；容易过拟合奖励模型 |
| **MCTS (树搜索)** | 理论上可找到最优路径 | 词表太大导致搜索爆炸；延迟极高；工程复杂 |
| **GRPO (组相对)** | 无需额外模型；计算效率高；鲁棒性强 | 依赖大 Batch Size；主要优化策略而非单步搜索 |

#### 5. 总结
尽管 PRM 和 MCTS 在理论上有助于提升推理能力（如 AlphaProof），但在 DeepSeek-R1 的大规模语言模型训练场景下，**简单的强化学习策略（如 GRPO）配合规则奖励被证明是更高效、更鲁棒的选择**。

## 常见考点
1. **为什么 o1 系列可能用了 MCTS 而 DeepSeek 不用**：这涉及到技术路线的选择，DeepSeek 认为在大规模词表下，纯 RL 的搜索效率优于显式的树搜索。
2. **PRM 与 ORM 的区别**：ORM (Outcome Reward Model) 只看最终结果，PRM 看每一步。虽然 PRM 提供更细粒度的信号，但在工程实现和训练稳定性上代价更高。
3. **GRPO 如何替代了 PRM 的作用**：GRPO 通过组内对比隐式地优化了每一步的生成质量，而不需要显式的每步打分模型。

## 记忆要点

- PRM局限：计算开销大，每步需评分，且容易Reward Hacking。
- MCTS局限：词表过大导致搜索爆炸，推理延迟极高，工程性价比低。
