# PPO在RLHF中的工作流程是什么？

RLHF中使用PPO通常涉及四个模型，以在优化策略的同时保证稳定性和安全性。

**四个模型角色：**

1.  **Actor（Policy Model, π_θ）**：待优化的模型，负责生成Response。参数为θ。
2.  **Reference Model（π_ref）**：冻结参数的SFT模型。用于计算KL散度，防止Actor在优化过程中偏离SFT模型太远（防止Reward Hacking或灾难性遗忘）。
3.  **Reward Model（R_ϕ）**：根据Prompt+Response打分。它独立训练自人类偏好数据，作为人类判断的代理。
4.  **Critic Model（Value Model, V_ω）**：估计状态价值。用于计算Advantage（优势函数），衡量当前Action比平均Action好多少，减少方差。

**训练流程（循环）：**

1.  **生成**：Actor根据Prompt采样生成Response。
2.  **打分**：将Prompt和Response输入Reward Model，获得分数 r。
3.  **评估**：Critic计算当前状态的估计值 v。
4.  **计算**：
    - Advantage = r - v（广义优势估计通常使用GAE）。
    - Ratio = π_θ(a|s) / π_θ_old(a|s)（新旧策略的概率比）。
5.  **优化**：
    - **PPO Clip目标**：min( Ratio × A, clip(Ratio, 1-ε, 1+ε) × A )。这限制了策略更新的幅度。
    - **KL惩罚**：Loss中减去 β × KL(π_θ || π_ref)，进一步约束行为。

**核心问题与挑战：**
- **显存消耗**：4个模型同时加载（Actor, Ref, Reward, Critic），且需要存储梯度和Optimizer状态，对硬件要求极高。
- **训练不稳定**：Reward Model若不准，Actor容易学会“欺骗”Reward Model（例如生成无意义的长字符串）。PPO的超参数（clip ε, KL coeff β）非常敏感。

**替代方案**：
- **DPO**：彻底移除Reward Model和Critic，直接优化偏好数据。
- **GRPO（DeepSeek）**：移除Critic，通过Group采样的方式计算Baseline。

### 实战深化

**实战案例**：
在训练中文对话模型时，曾遇到“Reward Hacking”现象：Actor为了骗取高Reward，开始反复输出“好的，没问题”等固定句式，或者生成非常啰嗦的车轱辘话。通过引入KL Penalty系数的动态调整（当KL超过阈值时增大β），成功遏制了模式崩塌。

**代码示例 (Python/PyTorch)**：
```python
# PPO Loss 核心计算片段
ratio = torch.exp(log_probs - old_log_probs) # 概率比
# GAE (Generalized Advantage Estimation)
advantages = rewards + gamma * next_values * dones - values 

# 1. PPO Clipping
surrogate1 = ratio * advantages
surrogate2 = torch.clamp(ratio, 1.0 - epsilon, 1.0 + epsilon) * advantages
policy_loss = -torch.min(surrogate1, surrogate2).mean()

# 2. KL 散度惩罚 (防止偏离SFT)
kl_penalty = -beta * kl_divergence(actor_logits, ref_logits).mean()

loss = policy_loss + value_loss + entropy_loss + kl_penalty
```

## 记忆要点

- 四个模型：Actor（策略）、Reference（防偏离）、Reward（打分）、Critic（估值）。
- 核心流程：Actor生成 -> Reward打分 -> Critic算优势 -> PPO Clip更新 -> KL惩罚约束。
- PPO Clip：限制新旧策略概率比，防止更新步子过大导致训练崩溃。
- 主要挑战：显存巨大（4模型）、Reward Hacking（模型骗分）、超参数敏感。
