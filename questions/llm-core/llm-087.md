# 为什么要优化KV cache

KV cache(Key-Value 缓存)是在 Transformer 模型推理阶段保存历史输入的键和值的存储机制，目的是在生成序列时避免重复计算，从而大幅提升效率。

### 1. KV cache 的核心作用
*   **避免重复计算**：Transformer 的自注意力机制需要计算当前 Token 与所有历史 Token 的相关性。若无 KV cache，每生成一个新 Token，都需要把之前所有 Token 重新经过神经网络计算一次 $K$ 和 $V$。缓存后，只需计算新 Token 的 $K, V$，并与历史缓存拼接。
*   **计算瓶颈转移**：在 LLM 推理中，计算 $K, V$ 涉及大量的矩阵乘法。虽然 KV cache 减少了计算量，但将瓶颈转移到了**显存带宽**上，因为每个生成都需要读取巨大的 KV Cache 矩阵。

### 2. 为什么要优化 KV cache (显存压力分析)
LLM 的显存主要由三部分组成：模型权重、KV Cache、激活值。随着序列长度增加，KV Cache 的增长远超权重。

**KV Cache 显存占用公式**：
$$ \text{Size} = 2 \times \text{n\_layers} \times \text{d\_model} \times \text{n\_bytes} \times \text{seq\_len} \times \text{batch\_size} $$
*   **2**：分别缓存 Key 和 Value。
*   **n_layers**：层数（如 GPT-3 96 层）。
*   **d_model**：隐藏层维度（Key/Value 的维度通常等于 d_model / num_heads）。
*   **n_bytes**：数据类型字节数（FP16 为 2，FP32 为 4）。
*   **seq_len**：上下文长度（线性增长）。
*   **batch_size**：并发数。

**实战案例**：在做 70B 模型长文本评测时，上下文长度开到 32k，单卡 A100 (80G) 在 Batch Size 仅为 1 时就爆显存了。经过分析发现，模型权重占用 140GB（张量并行），剩下的大部分显存都被 KV Cache 吃掉。引入 8-bit 量化后，同样的显存勉强支持 32k 长度，但吞吐量极低。

**代码示例 (vLLM PagedAttention 概念)**：
```python
# vLLM 伪代码概念：Block 管理而非连续分配
block_manager = BlockManager(block_size=16)
# 动态申请物理 Block
physical_block_idx = block_manager.allocate()
# 将计算出的 KV 写入对应 Block 的槽位
cache_table[physical_block_idx][slot_offset] = kv_tensor
```

**对比表格：优化前后的 KV Cache**

| 特性 | 传统 KV Cache (HuggingFace) | PagedAttention (vLLM) | 量化 (INT8/FP8) |
| :--- | :--- | :--- | :--- |
| **内存分配** | 预分配连续大内存 | 动态分配固定大小 Block | 按位分配 (1-2 bytes) |
| **显存碎片** | 严重 (需按最大长预留) | 极低 (碎片化利用) | 取决于底层分配器 |
| **内存浪费** | 高 (Padding 和 Max Seq) | 低 (按需申请) | 低 (压缩数据本身) |
| **计算精度** | FP16/BF16 | FP16/BF16 | 可能损失 |
| **适用场景** | 短序列/简单部署 | 高并发/长文本推理 | 显存极度受限场景 |

### 3. 优化方向
1.  **量化**：将 KV Cache 从 FP16 (2 bytes) 量化为 FP8 (1 byte) 甚至 INT8，显存减半，但可能带来精度损失。
2.  **PagedAttention (如 vLLM)**：借鉴操作系统的虚拟内存分页思想，将 KV Cache 切分成固定的 Block，动态申请和释放，解决显存碎片问题，提高并发 Batch Size。
3.  **Multi-Query Attention (MQA) / Grouped-Query Attention (GQA)**：让多个注意力头共享同一份 Key 和 Value，大幅减少 Cache 大小（如 LLaMA 3 采用 GQA）。
4.  **Flash Attention**：虽然是计算优化，但其对 HBM 访问的优化也间接降低了 KV Cache 读写延迟。

**## 常见考点**
1.  **Prefill vs Decode 阶段**：KV cache 在这两个阶段的显存和计算特征有何不同？（Prefill 计算密集，Decode 访存密集）。
2.  **长文本瓶颈**：为什么超长文本推理时，即使显存够用，速度也会变慢？（Attention 计算复杂度是 $O(N^2)$，且需要读取过长的 KV 序列）。
3.  **vLLM 原理**：简述 PagedAttention 如何解决传统 KV Cache 预分配导致的显存浪费问题。
