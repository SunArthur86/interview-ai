# 大模型推理中的量化有哪些方案？

量化（Quantization）将FP16/BF16权重压缩到低精度（INT8/INT4），减少显存和加速推理。

**主要方案**：

1. **GPTQ（后训练量化）**：
   - 基于二阶Hessian矩阵信息逐层量化
   - INT4量化效果好，速度快
   - 适合服务端部署

2. **AWQ（Activation-aware Weight Quantization）**：
   - 保护对激活值贡献大的权重通道
   - 引入缩放因子s，使重要权重的量化误差最小
   - INT4量化效果优于GPTQ

3. **BitsAndBytes（NF4/INT8）**：
   - QLoRA使用的NF4量化
   - 适合训练场景（QLoRA微调）

4. **GGUF/GGML（llama.cpp）**：
   - 面向CPU推理的量化方案
   - 支持Q4_K_M、Q5_K_M等多种精度

**效果**：INT4量化使13B模型从26GB降到约7GB，RTX 4090可跑。精度损失通常<1%。

### 实战案例
在本地PC端部署70B大模型时，直接加载FP16模型需要140GB显存，普通多卡服务器难以承担。使用AWQ INT4量化后，显存需求降至约40GB（双张4090即可运行），且在长文本概括任务中，量化后的模型输出质量与原版几乎无肉眼可见的差异，但推理速度提升了2倍以上。

### 代码示例 (AWQ 伪加载流程)
```python
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

# 加载量化模型
model_path = "lmsys/vicuna-7b-v1.5-awq"
model = AutoAWQForCausalLM.from_quantized(model_path, safetensors=True)
tokenizer = AutoTokenizer.from_pretrained(model_path, trust_remote_code=True)

# 推理：内部会自动处理 Scale 的反量化计算
prompt = "You're a helpful assistant."
inputs = tokenizer(prompt, return_tensors="pt").to("cuda")
outputs = model.generate(**inputs)
```

### 对比表格：主流量化方案对比
| 方案 | 数据类型 | 适用场景 | 核心优势 | 潜在劣势 |
| :--- | :--- | :--- | :--- | :--- |
| **GPTQ** | INT4/INT8 | 通用推理，显存受限 | 成熟生态，速度快，支持Per-channel | 极端复杂任务下精度略逊于AWQ |
| **AWQ** | INT4 | 通用推理 | 激活感知，保留重要权重，精度高 | 需要校准数据集，推理时略有显存开销 |
| **BitsAndBytes** | NF4/INT8 | 训练(LoRA)/推理 | 无需数据校准，即插即用 | NF4推理速度通常慢于GPTQ/AWQ |
| **GGUF** | Q2-Q8 | CPU/Mac边缘端 | 极致压缩，支持全平台 | GPU利用率不如CUDA方案，速度较慢 |

## 常见考点
- **Weight-only vs Activation Quantization**：目前推理主流是 Weight-only（只量化权重），因为激活值在推理过程中是动态计算的，量化它们容易引入较大误差且需要动态校准。
- **Per-tensor vs Per-channel**：GPTQ 通常支持 Per-channel（对每个输出通道单独量化），精度通常高于 Per-tensor（整个张量共用一个缩放因子），但计算开销稍大。
- **Quantization-aware Training (QAT) vs Post-training Quantization (PTQ)**：GPTQ/AWQ 属于 PTQ（无需训练数据微调即可量化），而 QAT 需要在微调时模拟量化噪声，通常精度更高但成本也更高。
