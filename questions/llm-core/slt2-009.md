# 什么是PagedAttention？vLLM如何优化推理？

PagedAttention是vLLM的核心创新，灵感来自操作系统的虚拟内存分页机制。

**问题**：传统推理中，每个请求的KV Cache需要连续内存。当请求长度不同、动态变化时，导致严重的显存碎片和浪费（预留最大长度的空间）。

**PagedAttention方案**：
1. 将KV Cache分成固定大小的block（如16个token/block）
2. 用block table（类似页表）映射逻辑block到物理block
3. 物理block不需要连续，减少碎片
4. 支持Copy-on-Write，实现Parallel Sampling（一个prompt生成多个输出）时共享前缀

**Continuous Batching（连续批处理）**：
传统Static Batching要等一个batch中所有请求完成才能处理新请求。Continuous Batching在每个iteration级别动态调度，请求完成后立即加入新请求，GPU利用率大幅提升。

**效果**：vLLM比HuggingFace Transformers推理吞吐量高14-24倍。

### 实战案例
在生产环境部署长文本对话模型时，若使用HuggingFace Transformers的静态缓存策略，一旦某个用户输入超长Prompt（>4k tokens），极易导致OOM且阻塞整个Batch。切换vLLM后，不仅消除了因显存碎片导致的频繁OOM，还能在高并发下维持稳定的TPS（Tokens Per Second）。

### 代码示例 (PagedAttention Block管理逻辑)
```python
# vLLM BlockManager 核心逻辑示意 (Python)
class BlockAllocator:
    def allocate(self, seq_id: int) -> List[PhysicalBlock]:
        # 检查空闲链表，获取非连续物理块
        blocks = self.free_queue.pop(num_blocks_needed)
        # 更新映射表: 逻辑序列 -> 物理块列表
        self.block_table[seq_id].extend(blocks)
        return blocks
    
    def free(self, seq_id: int):
        # 请求结束，物理块归还到空闲链表而非立即释放OS内存
        for block in self.block_table[seq_id]:
            self.free_queue.push(block)
```

### 对比表格：Static Batching vs vLLM Continuous Batching
| 特性 | Static Batching (静态批处理) | vLLM Continuous Batching (连续批处理) |
| :--- | :--- | :--- |
| **显存分配** | 预分配最大连续内存，浪费严重 | PagedAttention 按需分配非连续块，利用率高 |
| **调度粒度** | Batch 级别（必须等所有请求完成） | Iteration 级别（完成一个即可加入新请求） |
| **并发处理能力** | 受限于最慢的请求 | 弱依赖长尾请求，吞吐量大幅提升 |
| **适用场景** | 离线批量处理、请求长度差异小 | 在线实时服务、高并发、变长请求 |

## 常见考点
- **Block Table 的实现位置**：通常存储在 CPU 内存中，但在推理调度时需要快速访问 GPU 内存映射，这也是调度器优化的关键点。
- **vLLM 的迭代级调度**：不同于传统的 Request 级别调度，vLLM 在每次 Forward Pass 后都会检查是否有请求完成或新请求加入，动态调整 Batch 中的序列。
- **显存碎片与 OOM**：解释为什么 PagedAttention 能解决 OOM（Out Of Memory）问题，因为它允许最大限度地利用显存中的微小碎片，而不需要一大块连续内存。

## 记忆要点

- 原理：借鉴OS虚拟内存，将KV Cache切块（Block）非连续存储。
- 效果：消除显存碎片，解决变长序列内存浪费，支持Copy-on-Write。
- 调度：Continuous Batching实现迭代级动态调度，吞吐量提升24倍。
