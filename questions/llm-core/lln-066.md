# Prefix Caching(前缀缓存)的原理是什么?在什么场景下效果最好

### Prefix Caching (前缀缓存)

**Prefix Caching** 是缓存重复前缀的 KV Cache 以加速推理的技术，特别适用于 LLM 推理中大量重复 System Prompt 或上下文的场景。

#### 原理深度解析
1. **Cache 存储粒度**：
   - 并不是缓存整个 Prompt，而是将 Prompt 的 KV Cache 按照 Block（如 vLLM 中的 Page）为单位切分存储。
   - 每个 Block 有唯一的 Hash 指纹。

2. **复用机制**：
   - 当新请求到达时，计算其 Prompt 的 KV Block 指纹。
   - 在全局缓存池中查找，如果命中，直接映射物理内存指针，无需重新计算 Attention。
   - **Copy-on-Write**：如果后续生成导致旧 Cache 需要修改，则复制一份新的 Block，保证缓存不被污染。

3. **增量计算**：
   - 仅计算未命中的那部分 Prompt 的 KV，剩下的部分直接复用。

#### 架构流程图

```text
┌──────────────┐     Compute      ┌──────────────────┐
│   User Req   │ ────────────────> │   KV Generator   │
│ (Prompt+Q)   │                  │  (Attention Op)  │
└──────┬───────┘                  └────────┬─────────┘
       │                                   │
       │ Check Cache Hash                 │
       ↓                                   │
┌──────────────┐                           │
│   KV Cache   │ <─────────────────────────┘
│   Manager    │
│  (Hash Map)  │
└──────┬───────┘
       │
       │ Hit? ─Yes──> Copy Reference (Zero Compute)
       │ No
       ↓
  Schedule GPU
```

#### 典型场景
- **多轮对话**：System Prompt + 历史对话是重复前缀，只有最后一轮用户输入是新的。
- **Few-shot Learning**：大量的 Few-shot 示例完全相同。
- **RAG 系统**：检索到的文档在短时间内可能被不同用户的请求引用。

#### 效果
- **首 Token 延迟 (TTFT)**：降低 50-90%（取决于缓存命中率）。
- **吞吐量**：显存带宽压力减小，整体吞吐量提升。

#### 实战案例
在带有超长 System Prompt（如 2000 Token 的角色设定）的 NPC 对话场景中，未开启 Prefix Caching 时，每次对话都需要重新计算这 2000 Token 的 Attention，导致首字延迟高达 500ms；开启缓存后，复用率达 90% 以上，首字延迟降至 50ms 以内，用户体验从“卡顿”变为“即时”。

#### 关键代码 (Python/Hash逻辑)
```python
class BlockCache:
    def get_or_compute(self, tokens: List[int]):
        # 计算当前 Block 的 Hash 指纹
        block_hash = compute_hash(tokens)
        
        # 尝试在全局缓存中获取物理 Block
        cached_block = global_cache.get(block_hash)
        if cached_block:
            return cached_block # 命中，零拷贝复用
            
        # 未命中，申请显存并计算 KV
        new_block = allocator.allocate()
        new_block.kv = compute_attention(tokens)
        global_cache[block_hash] = new_block
        return new_block
```

#### 常见考点
1. **缓存淘汰策略**：显存有限时，使用什么策略淘汰 KV Cache？（LRU, LFU, 或基于 Token 优先级的策略？）。
2. **内存碎片管理**：既然是 Block 粒度缓存，如何处理跨 Block 的共享和内存碎片问题？（vLLM 的 Radix Tree 或链表结构）。
3. **一致性**：模型权重更新后（如 LoRA 切换），缓存如何处理？（通常需要失效或隔离）。
4. **自动 Prefix Caching**：用户是否需要显式指定共享 Prompt？现代框架通常通过自动 Hash 比对实现透明缓存。

## 记忆要点

- 核心原理：以 Block 为粒度计算 Hash 指纹，命中则直接映射物理内存，跳过 Attention 计算。
- 安全机制：因为采用写时复制，所以能保证多请求复用时缓存不被污染。
- 最佳场景：多轮对话、Few-shot、RAG 等拥有大量重复 System Prompt 或上下文的场景。
- 关键收益：因为免去重复前缀计算，所以大幅降低首字延迟（TTFT），提升并发吞吐。
