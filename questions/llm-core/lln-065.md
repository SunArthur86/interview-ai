# Continuous Batching(连续批处理)的原理是什么?为什么它能大幅提升推理吞吐

### Continuous Batching (连续批处理)

**Continuous Batching** 是 vLLM 等推理框架的核心优化技术，也称为 Iterative Batching 或 In-flight Batching。

#### 传统 Batching 的问题
- **静态 Padding**：在静态 Batching 中，不同请求的生成长度不同。系统必须按最大长度进行 Padding，浪费大量计算资源。
- **队头阻塞**：Batch 中即使短请求已完成，也必须等待长请求生成完毕才能释放 GPU，导致 GPU 空闲（计算气泡），利用率低下。

#### Continuous Batching 的方案
1. **Token 级别的动态调度**：
   - 调度器每隔一个 step（生成一个 token）介入一次。
   - 重新计算当前的 Logical Batch，剔除已完成的序列，插入新来的序列。
   
2. **即时加入/退出**：
   - **加入**：新请求的 Prompt 可以直接进入 Batch 进行 Prefill 阶段。
   - **退出**：一旦序列生成 EOS（End of Sentence），立即释放其占用的计算 slot 和显存。

3. **配合 PagedAttention (核心技术)**：
   - 将 KV Cache 切分为固定大小的 Block（类似操作系统的分页）。
   - 避免了传统连续显存分配产生的碎片，支持动态扩容和回收，是实现 Continuous Batching 的底层基石。

#### 架构图解

```text
时间轴 T1 (Batch 初始状态)
┌─────────────────────────────────────┐
│ Req A (Prompt) │ Req B (Prefill)   │
└─────────────────────────────────────┘
         ↓ 生成1个Token 后
时间轴 T2 (动态调度)
┌─────────────────────────────────────┐
│ Req A (Decode) │ Req B (Decode)    │ ← A转为Decode阶段
└─────────────────────────────────────┘
         ↓ 生成1个Token 后
时间轴 T3 (Req B 完成)
┌─────────────────────────────────────┐
│ Req A (Decode) │ Req C (Prompt)    │ ← B退出，C立即加入填补空缺
└─────────────────────────────────────┘
```

#### 性能提升关键
- **消除气泡**：通过物理上保持 GPU 的满载运行，将 GPU 利用率从静态 Batch 的约 30% 提升到 80% 以上。
- **高吞吐**：在同样显存下，能处理并发更高的请求。
- **指标**：相比静态 Batching，吞吐量通常提升 2-22 倍（取决于请求长度的方差）。

#### 实战案例
在处理高并发在线对话服务时，由于用户输入长度差异极大（有的问“你好”，有的上传长文档），未使用 Continuous Batching 前长请求会阻塞整个 Batch，导致 99% 请求延迟（P99）飙升；引入 vLLM 后，P99 延迟降低了 60% 以上，且无需手动切分 Batch。

#### 关键代码 (Python/vLLM 伪代码)
```python
# vLLM 内部调度逻辑简化示意
scheduler = Scheduler()

while True:
    # 1. 获取当前已完成或需要调度的序列
    seq_group = scheduler.get_seq_group()
    
    # 2. 执行模型推理 (Prefill 或 Decode)
    outputs = model_runner.execute_model(seq_group)
    
    # 3. 核心调度：立即释放完成的请求，插入新请求
    scheduler.schedule(
        finished_seqs=outputs.finished, 
        pending_seqs=waiting_queue.dequeue()
    )
    # 此时 GPU 上的 Batch 在每个 step 后都发生了物理重组
```

#### 常见考点
1. **Prefill 和 Decode 阶段的区别**：Prefill 是并行计算，计算密度高；Decode 是串行计算，受显存带宽限制。Continuous Batching 如何处理这两个阶段的混合调度？（通常利用迭代级调度隔离不同计算特性的批次）。
2. **为什么需要 PagedAttention**：传统 KV Cache 预分配导致显存浪费严重且无法释放，PagedAttention 如何解决 OOM（Out of Memory）问题？
3. **与 Orca 的区别**：Continuous Batching 和 Orca (Continuous Batching 的先驱论文) 的主要优化点是什么？
4. **Chunked Prefill**：当输入 Prompt 极长时，如何避免长请求饿死短请求？（vLLM 采用 Chunked Prefill 将长 Prompt 切片处理）。

## 记忆要点

- 地基支撑：高度依赖PagedAttention，像OS管理虚拟内存一样分页管理KV Cache实现动态扩容。
- 性能结果：彻底消除GPU计算气泡，将吞吐量呈几何级提升（vLLM核心优化点）。
