# KV Cache的原理是什么？为什么能加速推理？

KV Cache是Transformer自回归推理的关键优化。

**原理详解**：
在Decoder-only架构的自回归生成中，生成第t个token时，需要计算它与前t-1个token的Attention分数。由于Attention计算依赖Query(Q)、Key(K)、Value(V)，且前t-1个token的输入未改变，其K和V矩阵是固定的。KV Cache的核心就是将这些静态的K和V矩阵缓存显存中。

**推理流程对比**：
不使用KV Cache时，每一步都要将整个序列输入模型，计算量随序列长度平方增长（O(n²)）。使用KV Cache后，每一步只需输入新生成的token，计算其Q，并与缓存中的K、V进行Attention。

```text
Step 1 (输入 "I"):
┌─────────────────────────────────────┐
│ Model Forward                        │
│ -> Compute Q1, K1, V1                │
│ -> KV Cache: [K1, V1]                │
└─────────────────────────────────────┘

Step 2 (输入 "love"):
┌─────────────────────────────────────┐
│ Input: "love" (only)                │
│ -> Compute Q2, K2, V2                │
│ -> KV Cache Update: [K1, V1, K2, V2] │
│ -> Attention: Q2 dot [K1, K2]       │
└─────────────────────────────────────┘

Step t:
┌─────────────────────────────────────┐
│ Input: New Token (only)             │
│ -> Compute Qt, Kt, Vt                │
│ -> Read Cache: K[1:t-1], V[1:t-1]    │
│ -> Attention: Qt dot K[1:t]          │
└─────────────────────────────────────┘
```

**加速效果**：
- **不使用KV Cache**：每生成一个token，需要重新计算所有前面token的K和V，复杂度O(n²)，且显存读写带宽压力巨大。
- **使用KV Cache**：计算K/V的浮点运算量被复用，每步只需计算新token的K/V（O(1)计算量）及Attention（O(n)计算量）。总复杂度从O(n³)降到O(n²)。

**内存占用**：
KV Cache大小 = 2 × num_layers × seq_len × hidden_size × batch × dtype_size × num_heads（MHA下）
这是LLM推理在Batch Size > 1或长序列下的主要显存瓶颈，往往超过模型权重本身。

**常见问题与优化**：
KV Cache的形状是静态的吗？不是，随着序列生成，序列维度会动态增长。这导致了内存碎片问题。
- **PagedAttention（vLLM）**：借鉴操作系统的虚拟内存和分页机制，将KV Cache切块存储，有效解决显存碎片和浪费。
- **GQA/MQA**：减少KV头数，直接线性降低Cache占用。
- **KV Cache量化**：将FP16/BF16的KV Cache量化为INT8甚至FP4，牺牲微小精度换取显存减半。

**实战案例**：
在多轮对话场景中，如果不实现Prefix Caching，每次新请求都会重新计算System Prompt和历史对话的KV，导致首字延迟（TTFT）极高且算力浪费严重。vLLM等框架通过缓存公共前缀KV，可直接复用，显著加速冷启动。

**代码示例（Python，模拟KV Cache更新）**：
```python
import torch

class KVCache:
    def __init__(self, max_batch, max_len, n_heads, head_dim, dtype):
        # 预分配显存 (batch, seq_len, n_heads, head_dim)
        self.k_cache = torch.zeros(max_batch, max_len, n_heads, head_dim, dtype=dtype).cuda()
        self.v_cache = torch.zeros(max_batch, max_len, n_heads, head_dim, dtype=dtype).cuda()
        self.seq_len = 0

    def update(self, new_k, new_v):
        # new_k shape: [batch, 1, n_heads, head_dim]
        batch = new_k.shape[0]
        self.k_cache[:batch, self.seq_len:self.seq_len+1] = new_k
        self.v_cache[:batch, self.seq_len:self.seq_len+1] = new_v
        self.seq_len += 1
        return self.k_cache[:batch, :self.seq_len], self.v_cache[:batch, :self.seq_len]
```

## 常见考点
1. **Prefix Caching（前缀缓存）**：如何利用KV Cache加速多轮对话或Prompt复用？（vLLM中的CudaGraphs等技术）
2. **Prefill vs Decoding阶段**：KV Cache对这两个阶段计算瓶颈的影响有何不同？（Prefill受限于算力，Decoding受限于显存带宽）
3. **连续批处理**：在有KV Cache的情况下，如何实现不同序列长度和步长的Batching？

## 记忆要点

- 原理：缓存历史Token的K和V矩阵，生成时只计算新Token的Q。
- 效果：避免重复计算，复杂度从O(n²)降至O(n)，大幅加速自回归生成。
- 瓶颈：KV Cache显存占用随序列长线性增长，常超过模型权重本身。
