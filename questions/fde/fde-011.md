# 客户的 AI 应用回答效果不好,你会从哪些角度排查和优化

- **AI 效果优化排查 SOP(5 层模型)**

- **Layer 1: 数据层(最常见)**
- 知识库覆盖够不够?补充文档
- 数据分块合不合理?调整 chunk_size/overlap
- 向量化模型好不好?换更好的 Embedding

- **Layer 2: 检索层**
- 检索结果准不准?人工检查 Top-5
- 是否需要 Rerank?加 Cross-Encoder
- Query 是否需要改写?HyDE/子问题分解

- **Layer 3: Prompt 层**
- System Prompt 够不够清晰?
- 是否给了足够的上下文?
- 有没有约束幻觉的指令?
- Few-shot 示例是否有帮助?

- **Layer 4: 模型层**
- 模型能力够不够?换更大的模型
- Temperature 合不合适?事实问答用 0-0.3
- 上下文窗口够不够?分段处理

- **Layer 5: 评估层**
- 有没有评估体系?建立黄金测试集
- 用户反馈机制?点赞/点踩收集
- 定期回归测试?每次改动跑测试集

- **经验**:80% 的问题出在数据层和 Prompt 层,不要急于换模型.

- **## 常见考点**
1. **HyDE (Hypothetical Document Embeddings)**：原理是什么？答案：让 LLM 先针对问题生成一个“假设性的完美答案”，然后用这个假设答案去向量库里检索，因为“答案对答案”的语义相似度往往高于“问题对答案”，能解决 Query 和 Doc 语义不匹配的问题。
2. **RAGAS 评估框架**：如何量化 RAG 效果？答案：使用 RAGAS 框架计算 Faithfulness（忠实度，答案是否基于上下文）和 Answer Relevance（相关性），无需人工标注即可自动化评估系统质量。
3. **Temperature 调优**：面试官追问参数设置。答案：事实类任务（问答、数据库提取）设为 0（确定性最强）；创意类任务（写邮件、营销文案）设为 0.7-0.9；一般 RAG 任务推荐 0.1-0.3 以平衡准确性和自然度。

---

- **实战案例**：某法律问答系统经常“一本正经胡说八道”。排查发现检索到的法规片段正确，但 LLM 上下文里混杂了多条相似条款导致逻辑混乱。我们将检索策略从“Top 5 拼接”改为“只检索最相关的 1 条并强制引用”，幻觉率直接降低了 80%。

- **代码示例**：
```python
# 混合检索 + Rerank 关键逻辑 (Python)
from rank_bm25 import BM25Okapi
from sentence_transformers import CrossEncoder

def hybrid_search(query, corpus, top_k=5):
    # 1. 关键词检索 (BM25)
    tokenized_corpus = [doc.split() for doc in corpus]
    bm25 = BM25Okapi(tokenized_corpus)
    bm25_results = bm25.get_top_n(query.split(), corpus, n=top_k)

    # 2. 语义检索 (省略向量代码...)
    vector_results = vector_db.search(query, top_k)

    # 3. Rerank (重排序)
    rerank_pairs = [[query, doc] for doc in set(bm25_results + vector_results)]
    scores = cross_encoder.predict(rerank_pairs)
    # 按分数取 Top K
    final_results = [doc for score, doc in sorted(zip(scores, rerank_pairs), reverse=True)][:top_k]
    return final_results
```

- **对比表格**：调优手段效果评估
| 手段 | 实施难度 | 见效速度 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **Prompt 优化** | 低 (立竿见影) | 快 | 指令不清、格式要求严格 |
| **Rerank 模型** | 中 (需推理时间) | 中 | 检索结果召回多但排序乱 |
| **增加数据** | 高 (需ETL清洗) | 慢 | 知识库未覆盖、明显缺失 |
| **微调模型** | 极高 (需算力) | 慢 | 特定领域风格、格式极其特殊 |
| **更换 Embedding** | 中 (需重建库) | 慢 | 语义理解偏差大（跨语言等） |
