# 客户的 AI 应用 P95 延迟 8 秒,用户抱怨太慢,你会怎么优化到 3 秒以内

- **AI 应用延迟优化方案(8s -> 3s)**

- **诊断:延迟分解**
用 OpenTelemetry 追踪每个环节耗时:
- Embedding 查询向量化:100ms
- 向量检索:200ms
- Rerank:500ms
- LLM 生成(最大头):6000ms
- 网络传输:200ms

- **优化措施(按收益排序)**

1. **流式输出**(体感延迟降低 50%)
- 首 token 延迟 < 1s 即可
- 用户看到流式输出不会觉得慢

2. **减小上下文**(LLM 延迟降低 30-40%)
- 检索结果从 Top-10 减到 Top-5
- 去掉冗余的 system prompt
- 上下文 token 数直接线性影响生成时间

3. **模型降级**(LLM 延迟降低 50%)
- 简单问题用 GLM-4-Flash(2s 内完成)
- 复杂问题才用 GLM-4-Plus
- 模型路由实现自动选择

4. **语义缓存**(缓存命中直接返回)
- 高频问题直接返回缓存
- 命中率 20-30% 时整体延迟降低 20%

5. **Rerank 异步化**
- 先返回 Top-3 初步结果
- Rerank 在后台异步执行
- 用户不需要等待 Rerank 完成

- **预期效果**
- 流式首 token < 1.5s
- 完整生成 P95 < 3s
- 缓存命中 < 200ms

- **## 常见考点**
1. **TTFT vs TPOT**：解释 TTFT (Time To First Token) 和 TPOT (Time Per Output Token) 的区别。用户体感主要取决于 TTFT，而生成速度取决于 TPOT（受限于模型大小和显存带宽）。
2. **KV Cache 作用**：推理服务端（如 vLLM/TGI）开启 KV Cache 对延迟有何影响？极大降低 TPOT，因为无需重复计算已生成 Token 的 Attention 矩阵，但在高并发下会消耗大量显存。
3. **Rerank 异步化实现**：如何实现先返回结果后 Rerank？通常利用 WebSocket 或 SSE 发送 "update" 事件，前端收到后动态替换页面上的引用来源或调整答案排序。

- **实战案例**：某物流问答系统 P95 延迟高达 12 秒。通过 Profiler 发现，System Prompt 中包含长达 2000 token 的“公司历史介绍”和“冗长的格式要求”。我们将这部分硬知识移入向量库，并精简 Prompt 指令为“简洁回答”，仅此一项修改，LLM 推理耗时下降了 45%。

- **优化前后对比**

| 优化项 | 优化前耗时 | 优化后耗时 | 技术手段 |
| :--- | :--- | :--- | :--- |
| **检索阶段** | 800ms | 200ms | 索引优化(HNSW) + 减少召回数(20->5) |
| **Rerank阶段** | 600ms | 0ms (异步) | 移除同步 Rerank 或后台重排引用 |
| **Prompt构建** | N/A | -400ms | 精简 System Prompt (压缩50%) |
| **LLM推理** | 6500ms | 2200ms | 模型降级(GPT-4->GPT-3.5-Turbo) + 流式 |
| **整体首字(TTFT)** | 7900ms | **1400ms** | 综合手段 |

- **关键代码 (Python - 简单路由策略)**
```python
# 意图识别路由：简单问题用小模型，复杂问题用大模型
def route_query(question: str):
    # 简单规则或小模型判断意图
    simple_keywords = ["查询", "多少", "什么时候"]
    if any(kw in question for kw in simple_keywords):
        return "gpt-3.5-turbo" # 快速，便宜
    else:
        return "gpt-4" # 复杂推理，准确

# 调用时选择模型
model_name = route_query(user_question)
response = client.chat.completions.create(
    model=model_name,
    messages=[...],
    max_tokens=256 # 限制输出长度也能提速
)
```

## 记忆要点

- 流式优先：首字延迟(TTFT)小于 1.5 秒，利用流式输出大幅改善用户体感。
- 模型路由：简单问题用小模型(Flash)，复杂用大模型，平衡速度与质量。
- 精简上下文：减少检索数量和 System Prompt 长度，线性降低推理耗时。
- 语义缓存：高频问题命中直接返回，20-30% 命中率可显著降低整体延迟。
- 异步 Rerank：先返回初步结果，重排在后台异步执行，不阻塞用户等待。
