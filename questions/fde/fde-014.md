# 客户的数据不能出公司网络,你怎么帮他部署 AI 方案

- **数据不出网的 AI 部署方案**

**架构对比图：**

- **方案详解**

**方案 1: 全本地化部署**
*   **栈**: 开源模型 (Llama 3 8B/70B 或 Qwen-72B) + vLLM/TGI 推理加速 + 本地向量库 (Milvus/PGVector) + 应用层 (FastAPI/LangChain)。
*   **优势**: 数据完全不出网，合规性最高。
*   **劣势**: 需要 GPU 服务器集群，运维成本高，模型效果通常弱于 SOTA 闭源模型。

**方案 2: 混合模式**
*   **机制**:
    *   **敏感数据**: 存储和处理在本地向量库。
    *   **非敏感推理**: LLM 推理走 API。
    *   *关键细节*: 构造 Prompt 时，不将原文传给云端，而是只传检索到的元数据或经过高度脱敏的特征；或者在本地运行小模型进行脱敏重写后再发给云端。
*   **优势**: 利用云端模型强能力，减少数据暴露。

**方案 3: 专用云 (VPC / 私有化云)**
*   **机制**: 云厂商提供专用实例 (如火山引擎 VPC、AWS Outposts)。
*   **数据隔离**: 数据在客户专属的云空间，物理或逻辑隔离，符合部分监管要求。

- **实战案例**
  某银行项目要求核心账务数据绝对内网处理，但希望利用 GPT-4 的理解能力。我们采用“本地 LLaMA 做指令路由 + 本地 NER 做脱敏”的方式，将敏感字段替换为占位符后请求云端，云端返回结果后再由本地模型将占位符还原回真实数据，实现了性能与合规的平衡。

- **代码示例**
```python
# Python: 使用 Microsoft Presidio 进行本地脱敏
from presidio_analyzer import AnalyzerEngine
from presidio_anonymizer import AnonymizerEngine

analyzer = AnalyzerEngine()
anonymizer = AnonymizerEngine()

text = "用户张三的身份证号是 110101199001011234"

# 1. 分析敏感实体
results = analyzer.analyze(text=text, language='zh')
# 2. 执行脱敏
anonymized_text = anonymizer.anonymize(text=text, analyzer_results=results)

print(anonymized_text.text) 
# 输出: "用户<PERSON>的身份证号是 <ID_NUMBER>"
```

- **方案对比表格**

| 特性 | 全本地化部署 (私有化) | 混合模式 (脱敏) | 专用云 (VPC) |
| :--- | :--- | :--- | :--- |
| **数据安全性** | 最高 (物理隔离) | 中 (依赖脱敏算法强度) | 高 (逻辑/物理隔离) |
| **模型效果** | 中等 (依赖开源模型) | 高 (可调用 SOTA 模型) | 高 (可调用 SOTA 模型) |
| **硬件成本** | 极高 (需自建 GPU 集群) | 低 (仅需 CPU/少量 GPU) | 高 (专用资源租赁) |
| **运维难度** | 高 (需维护 MLOps) | 中 (需维护脱敏网关) | 低 (云厂商托管大部分) |
| **适用场景** | 军工、核心金融、涉密 | 通用企业、敏感度中等 | 政企、大中型金融 |

- **合规要点与技术实现**
1.  **数据脱敏**:
    *   API 调用前自动替换 PII (姓名/手机号/身份证) 为 `<PERSON>`, `<PHONE>` 等占位符。
    *   可使用微软 Presidio 或专门的 NER 模型进行预处理。
2.  **审计日志**:
    *   记录所有 AI 调用的输入/输出。注意：审计日志本身也是敏感数据，需加密存储并严格控制访问权限。
3.  **权限控制 (RBAC)**:
    *   不同角色访问不同知识库。例如：HR 只能访问 HR 文档的向量库空间，技术部访问技术文档。
4.  **国密算法**:
    *   政府/金融客户可能要求传输和存储使用 SM2/SM4 加密。需确保推理框架和通信库支持国密。
5.  **等保三级**: 需满足网络安全等级保护制度要求，包括物理环境、通信网络、区域边界等的安全防护。
