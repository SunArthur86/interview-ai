# 设计一个企业 RAG 系统的完整技术方案,从数据接入到用户查询的全流程是什么

- **企业 RAG 系统完整架构**

**数据流：** 数据源(PDF/Word/Web/DB) -> 数据处理管道(解析->分块->Embedding) -> 向量数据库(Milvus/Chroma)

**查询流：** 用户提问 -> Query改写 -> 多路检索(向量+BM25) -> RRF融合 -> Rerank -> LLM生成 -> 引用标注 -> 流式输出

- **Phase 1: 数据接入(离线)**
1. 文档解析:Unstructured/DeepDoc 处理 PDF(含表格/图片)
2. 清洗:去除页眉页脚、修复换行、合并段落
3. 分块:RecursiveCharacterSplitter(chunk=512, overlap=64)
4. 元数据标注:来源、页码、时间、权限标签

- **Phase 2: 检索(在线)**
1. Query 改写:HyDE / 子问题分解
2. 多路检索:向量检索 Top-20 + BM25 Top-20
3. RRF 融合:reciprocal_rank_fusion 合并
4. Rerank:Cross-Encoder 精排 Top-5
5. 上下文组装:父子块 + 窗口扩展

- **Phase 3: 生成(在线)**
1. Prompt 模板:System + Context + Question
2. LLM 生成:带引用标注的回答
3. 安全检查:幻觉检测 + 敏感词过滤
4. 流式输出:SSE 推送

- **实战案例**：在处理一份 200 页的财报 PDF 时，常规文本解析器将表格拆解成了乱码。我们引入了 Unstructured 库的高级表格提取模式，将表格转换为 HTML 格式存入向量库，LLM 检索时能完美还原表格逻辑进行计算。

- **代码示例**：
```python
# RRF (Reciprocal Rank Fusion) 融合算法 (Python)
def reciprocal_rank_fusion(results_dict, k=60):
    fused_scores = {}
    for system, doc_list in results_dict.items():
        for rank, doc in enumerate(doc_list):
            doc_id = doc['id']
            if doc_id not in fused_scores:
                fused_scores[doc_id] = 0
            fused_scores[doc_id] += 1 / (k + rank + 1)
    # 按分数降序排列
    reranked = sorted(fused_scores.items(), key=lambda x: x[1], reverse=True)
    return reranked
```

- **对比表格**：向量检索 vs 关键词检索 (BM25)
| 特性 | 向量检索 | BM25 关键词检索 |
| :--- | :--- | :--- |
| **匹配逻辑** | 语义相似度 | 词频/逆文档频率 (TF-IDF) |
| **强项** | 处理同义词、长尾语义 | 精确匹配人名、型号、特定术语 |
| **弱项** | 难以处理“生僻词” | 无法理解“智能手机”≈“手机” |
| **计算成本** | 高 (需做 Embedding) | 低 (倒排索引查表) |
| **RAG 建议** | 必选 (语义理解) | 必选 (精确召回补充) |
