# FDE 和咨询顾问、解决方案架构师(SA)有什么区别

- **三者对比**

| 维度 | 咨询顾问 | 解决方案架构师 (SA) | FDE (前沿部署工程师) |
|------|---------|-------------------|-----|
| **核心产出** | PPT + 战略报告 | 架构图 + 技术方案文档 | **可运行的代码/系统** |
| **是否写代码** | ❌ 完全不写 | ❌ 极少写 (只写 Demo) | ✅ **亲手写核心逻辑** |
| **关注深度** | 战略层面 | 架构层面 | **实现与集成层面** |
| **交付周期** | 数周到数月 | 数天到数周 | **数天到数周 (高频迭代)** |
| **客户关系** | 高层对接 | 技术总监对接 | **一线执行团队深度绑定** |

- **实战案例**：SA 设计了完美的微服务架构，但客户现场仅提供了两台 8 核 16G 的虚拟机且禁止 K8s。FDE 需现场将微服务架构改为单机 Docker Compose 部署，并引入 Nginx 做轻量级路由，在资源受限下强行跑通业务流程，避免了项目延期。

- **FDE 的核心差异化**
  - **动手能力**: 顾问告诉你“应该怎么做”，SA 画图纸告诉你“长什么样”，FDE **直接帮你做到**。
  - **解决细节**: SA 解决宏观架构，FDE 还要**上手敲代码、调接口、Debug 异常数据、处理网络不通**。
  - **交付物**: FDE 不只交付方案，还要**交付一个真实能跑起来的系统**，哪怕充满了临时的补丁。

- **代码示例（适配受限环境的 Docker Compose）**：
```yaml
# docker-compose.yml - 简化版生产部署
version: '3.8'
services:
  app:
    image: my-ai-app:v1
    ports:
      - "5000:5000"
    environment:
      - ENV=production
      - DB_HOST=db
    # FDE现场调整：限制内存以防OOM
    deploy:
      resources:
        limits:
          memory: 2G
```

- **协作关系**
```text
[客户] <--(战略规划)-- [咨询顾问]
   │
   +--(技术蓝图)-- [解决方案架构师 SA]
                      │
                      +--(落地实现/现场部署)-- [FDE]
                                                │
                                                +--(反馈)-- [产品团队]
```

- **形象比喻**
  - **顾问** = 军师 (出谋划策，指明方向)
  - **SA** = 建筑师 (画图纸，设计结构)
  - **FDE** = 施工队队长 + 工程师 (既画图，又亲自盖楼，还要修水管)

## 常见考点
1. **SA 和 FDE 在项目中如何协作避免冲突？**：SA 设计的架构如果不切实际（忽略了客户环境限制），FDE 需要现场反馈并调整架构，考察协作沟通机制。
2. **FDE 的产出如何转化为公司的通用产品资产？**：考察 FDE 是否具备“反向工程”能力，将客户现场的定制化修补转化为产品的 Feature。
3. **为什么现在 AI 公司特别强调 FDE 而不是 SA？**：AI 技术的非确定性导致仅靠设计图纸无法保证效果，必须在现场通过不断调试才能达到可用标准。
