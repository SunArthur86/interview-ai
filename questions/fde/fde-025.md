# 请设计一个企业级智能客服 AI 平台,日处理 10 万工单,支持 50 个租户.

- **日 10 万工单、50 租户智能客服平台设计**

- **容量估算**
- 日 10 万工单 -> 日均 1.2 QPS,峰值 10 QPS (假设 8 小时工作制)
- **增强细节**：考虑早高峰效应，峰值可能达到平均值的 5-10 倍，设计需预留 50 QPS 缓冲。
- 每工单平均 3 轮对话 -> 日 30 万次 LLM 调用
- 50 租户,每租户平均 1 万篇知识库文档 -> 总计 50 万篇

- **架构设计**

1. **接入层**
- API Gateway: Nginx/Kong + 认证 + 限流(每租户 QPS 上限)
- WebSocket: 支持流式输出,保持长连接

2. **核心服务层**(无状态,水平扩展)
- 意图分类服务: BERT 分类器(8类意图)
- RAG 引擎: 向量检索 + BM25 混合,重排序
- Agent 编排器: ReAct / Plan-and-Execute
- 质量检查服务: 幻觉检测 + 敏感词

3. **模型层**
- 主力模型: GLM-4-Plus (API) 或 Qwen-Max
- 备用模型: GPT-4 (故障 fallback, 成本高需限流)
- 分类模型: 本地部署 BERT (低延迟意图识别)
- Embedding: BGE-M3 (本地部署, 支持 GPU 加速)

4. **存储层**
- 向量库: Milvus/Elasticsearch 集群(50 个 Collection,每租户1个,物理隔离或多租户路由)
- 关系DB: PostgreSQL (工单 + 用户 + 配置)
- 缓存: Redis (会话上下文 + 语义缓存, 缓存命中率目标 40%)
- 对象存储: MinIO (原始文档 + 图片)

5. **运维层**
- 监控: Prometheus + Grafana + Langfuse (LLM 专用观测)
- 日志: ELK Stack
- 部署: K8s + ArgoCD (金丝雀发布)

- **关键指标**
- P95 延迟 < 3s (首字生成 < 1s)
- 自动解决率 > 60%
- 幻觉率 < 5% (基于人工抽检或模型打分)
- 可用性 99.9%
- 月度 API 成本 < 5 万元

- **## 常见考点**
1. **多租户隔离**：在向量库层面如何实现数据隔离？（Collection 隔离 vs TenantID 字段隔离，及其优缺点）
2. **RAG 准确率优化**：当检索结果不相关时，系统如何处理？（引出 Re-ranking 机制和“不知道”的回复策略）
3. **上下文管理**：如何处理多轮对话中的长上下文摘要，防止超过 Token 限制？（引出 Rolling Summary 或 历史摘要策略）
