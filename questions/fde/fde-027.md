# 客户的知识库有 10 万篇文档,每周新增 500 篇,如何设计高效的数据更新管道

- **大规模知识库增量更新管道**

- **全量初始化(一次性)**
1. 批量处理 10 万篇文档
2. 分布式处理(10 个 worker 并行, 利用 Celery 或 Kafka)
3. 分块 + Embedding + 入向量库
4. 预计时间: 10 万篇 x 2s/篇 / 10 worker = ~5.5 小时

- **增量更新管道**

1. **变更检测**
- 监控文档目录/SharePoint/Confluence Webhook
- 文件 hash (MD5/SHA256) 比对(新增/修改/删除)
- Cron job 每 1 小时扫描一次 (作为兜底)

2. **处理流水线架构**
```text
  数据源 (SharePoint/S3/OSS)
      │
      ▼ (Webhook / Polling)
┌─────────────┐
│   消息队列   │ (Kafka/RabbitMQ 缓冲变更)
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  ETL Worker │ (分布式消费者)
└──────┬──────┘
       │
  ┌────┴────┬──────────┐
  ▼         ▼          ▼
[新增]    [修改]     [删除]
  │         │          │
  ▼         ▼          ▼
解析→   删除旧块→   标记删除→
分块→   重新解析→   (软删除)
向量化  分块/向量化
  │         │
  └────┬────┘
       ▼
┌─────────────┐
│   向量数据库  │ (Upsert 操作)
└─────────────┘
```

3. **去重与合并**
- 同一文档的不同版本只保留最新 (基于 Doc ID 覆盖)
- 跨文档去重 (MinHash LSH 近似检测相似度 > 95% 的文档)

4. **质量检查**
- Embedding 成功率 > 99%
- 分块大小分布检查 (异常值告警, 如块大小 < 50 字符)
- 随机抽检 10 篇,人工验证检索相关性

5. **版本管理**
- 每次更新记录版本号 (如 Snowflake ID 或时间戳)
- 支持回滚到上一个版本 (向量库支持时间旅行或快照)
- 向量库快照备份 (每日冷备)

- **性能优化**
- Embedding 用批量 API (一次 100 条, 利用 GPU 并行)
- 向量库 upsert 用批量操作 (减少网络 RTT)
- 增量更新在低峰期执行 (凌晨 2 点)
- **增强细节**：对于“修改”操作，采用“读-改-写”模式，必须确保原子性，避免出现查询到新旧块混合的情况。

- **## 常见考点**
1. **向量删除机制**：向量数据库通常不支持高效的精确删除，如何处理？（引出软删除标记 Bitset 或 Compaction 机制）
2. **一致性保证**：在更新过程中，用户查询到“旧文档已删，新文档未入”的空窗期怎么解决？（双写策略或读写分离）
3. **文档解析难点**：PDF 中的表格、图片、双栏排版如何解析准确？（引出 OCR 版面分析或多模态解析技术）

## 记忆要点

- 增量管道：Webhook 监听 + 消息队列缓冲 + 分布式 Worker 处理。
- 变更检测：文件 Hash 比对识别增删改，Cron 扫描兜底。
- 处理策略：新增直接入库，修改先删后写，删除标记软删除。
- 去重合并：同文档版本覆盖，跨文档用 MinHash LSH 检测相似。
- 原子性：修改操作采用“读-改-写”模式，确保查询不出现新旧混合。
