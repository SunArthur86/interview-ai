# RAG 两步流水线(离线与在线).

**RAG 系统架构分为离线处理与在线服务两部分，形成闭环。**

**1. 离线流水线：负责将非结构化数据转化为可检索的向量索引**
- **数据解析**：处理 PDF、Markdown、HTML 等格式，提取纯文本。
- **清洗**：去除特殊字符、无意义的页眉页脚、广告噪音。
- **分块**：将长文本切分为固定大小（如 512 token）或语义连贯的片段。关键点：块之间保持一定的重叠（如 10-20%），以保留语义边界。
- **嵌入**：调用 Embedding 模型（如 BGE, OpenAI text-embedding-3）将文本块转化为高维向量。
- **建索引**：将向量存入向量数据库（如 FAISS, Milvus, Pinecone），并构建索引结构（IVF, HNSW）以加速检索。

**2. 在线流水线：处理用户请求并生成回答**
- **Query 处理**：
  - *改写*：将用户简短的 Query 扩写为更丰富的描述。
  - *路由*：判断 Query 该走向量检索、关键词检索（BM25）还是直接走知识库问答。
- **检索**：
  - *向量检索*：计算 Query 向量与库中向量的相似度。
  - *混合检索*：结合向量检索和关键词检索（BM25/RRF），提升准确率。
- **重排**：对召回的 Top-K 文档（如 Top 50）进行精细排序，筛选出最相关的 Top 5。
- **提示工程**：将 Query 和检索到的文档片段拼接成 Prompt，模版通常为：`"上下文信息：\n{context}\n\n问题：{question}\n\n回答："`。
- **生成**：调用 LLM 生成最终答案。

## 边界情况
1. **空结果处理**：当检索阶段没有匹配到任何文档时，系统应具备降级策略（如回答“不知道”或利用通用模型能力回答），而不是强行让 LLM 基于空上下文编造。
2. **多模态数据**：如果数据包含表格或图片，纯文本分块会丢失结构信息，需要使用专门的 Table Parsing 或多模态 Embedding 策略。
3. **索引一致性**：离线流水线更新数据后，在线向量索引可能存在延迟，导致用户无法搜到最新发布的内容，需考虑近实时索引构建方案。

## 面试追问
1. 在分块时，如何平衡“语义完整性”和“检索召回率”？如果切分太碎会有什么后果？
2. 当向量库的数据量达到亿级时，检索延迟成为瓶颈，除了优化索引算法，还有哪些架构层面的优化手段？
3. 如何评估 RAG 系统检索环节的质量？有哪些常用的指标（如 Recall@K, MRR）？

## 易错点
1. **忽视元数据**：只存文本块而不存对应的文件名、页码、时间戳等元数据，导致用户问到“最新的数据”时无法进行过滤。
2. **过度依赖向量检索**：对于专有名词、缩写或特定 ID（如订单号），向量检索往往效果很差，必须引入关键词检索（BM25）作为补充。
