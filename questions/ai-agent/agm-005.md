# 长期记忆为什么常用向量数据库?有什么局限

向量数据库是 Agent 长期记忆的主流选择，核心原因在于其支持**语义检索**，能解决传统关键词搜索无法处理的“同义词”或“表述差异”问题。但其在精确性和一致性上存在局限。

**为什么常用向量数据库？**
- **语义匹配能力**：将文本转化为高维向量，通过计算余弦相似度，使得“苹果”和“水果”、“手机”和“移动电话”在向量空间距离相近。这允许用户用不同的说法问同一个问题，Agent 仍能检索到相关信息。
- **处理非结构化数据**：适合存储聊天记录、文档片段、用户偏好描述等非结构化长文本。

**局限性**：
1.  **相似 $\neq$ 正确**：
    - 向量检索可能召回表面相似但语义无关的噪声（例如：用户问“如何退款”，可能召回一篇关于“如何付款”的文章，因为词汇重叠度高）。
    - 对反义、否定关系的区分能力较弱。
2.  **难以处理精确匹配**：
    - 对于账号、订单号、API Key、具体数值等需要字符级精确匹配的场景，向量检索效果很差（“12345”和“12346”向量距离极近，但意义完全不同）。这类数据适合用关键词索引（如 Elasticsearch）或关系型数据库。
3.  **更新与一致性**：
    - 向量库通常不支持复杂的 ACID 事务。当现实世界的数据发生变化（如用户改名），向量库可能存在脏数据，需要业务层额外保障同步更新（Upsert 机制）。
4.  **边界情况**：
    - **多语言混淆**：相同含义不同语言（如中文“苹果”和英文“Apple”）的距离取决于 Embedding 模型的训练数据，未经过多语言对齐训练的模型可能导致跨语言检索失效。
    - **低频词/生僻实体**：在训练语料中出现次数极少的专有名词，生成的向量可能不稳定，导致检索时无法准确召回。
    - **空查询/极短查询**：仅输入标点或语气词（如“？”、“好的”），生成的向量可能具有随机性，导致召回结果不可控。

### 实战案例
在做金融财报问答 Agent 时，我们发现单纯用向量检索经常将“2023年净利润”和“2022年净利润”混淆，因为两段文本描述结构高度相似，只有数字不同，导致模型回答数值时产生严重的“指代不清”错误。

### 对比表格
| 维度 | 向量检索 | 关键词检索 (BM25/ES) | 
| :--- | :--- | :--- |
| **核心优势** | 理解意图，处理同义词、模糊查询 | 精确匹配，专有名词强召回，结果可解释性高 |
| **典型场景** | “我想把货退了”、“这个东西怎么修” | 订单号查询、API 匹配、人名精确查找 |
| **主要缺陷** | 算力成本高，对生僻词/数字敏感度低 | 无法处理变体词（如“手机”vs“移动电话”） |

## 常见考点
1.  **如何缓解向量检索的精度问题？**：使用 **Re-rank（重排序）** 模型。先用向量库粗略召回 Top-K，再用一个精准的 Cross-Encoder 模型对这 K 个结果进行精细打分排序。
2.  **什么是混合检索？**：结合向量检索（语义）和关键词检索（字面），通过加权融合（如 RRF算法）互补，既解决同义问题又解决精确匹配问题。

## 易错点
1.  **忽视“否定”陷阱**：很多人误以为向量检索能完美理解逻辑。实际上，“我想要**不**辣的食物”和“我想要辣的食物”在向量空间中可能非常接近，导致召回错误内容。
2.  **混淆索引与存储**：误以为向量数据库可以直接替代关系型数据库。向量数据库擅长检索，不擅长处理复杂的关联查询和事务，必须搭配 KV 存储或 SQL 数据库使用。

## 面试追问
1.  **关于嵌入模型的选择**：如果你的业务场景包含大量行业专有名词（如医疗、法律），通用 Embedding 模型效果不佳，你会如何优化？（考察微调或领域适配知识库）
2.  **关于性能瓶颈**：当向量规模达到亿级时，检索延迟明显增加，除了增加索引参数（如 ef_construction），你会从架构上做什么优化？（考察分片、量化或部署层级缓存）
3.  **关于相关性评估**：在离线评估中，如何自动化判断向量检索召回的内容是否真的回答了用户问题？（考察 LLM-as-a-judge 的评估体系）
