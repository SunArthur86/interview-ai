# RAG 生产优化

### RAG 生产优化

**ANN 参数优化**
调整 HNSW 的 `M`（连接数）、`efConstruction`（构建时搜索范围）、`efSearch`（查询时范围）以平衡召回率与速度。

**分段分区**
按租户、时间、产品线分区，减少搜索空间，提升检索效率。

**索引维护**
权衡定期重建与增量插入：大批量用离线任务重建分区索引，小批量实时 upsert；文档变更需版本化，删除旧向量按 `doc_id`。

**缓存策略**
1. **Query 级缓存**：相同问题直接返回（注意权限与 TTL）。
2. **Embedding 缓存**：缓存热门 Query 的向量。
3. **LLM 响应缓存**：对低敏场景缓存生成结果，缩短延迟。

**多租户设计**
1. **逻辑隔离**：`tenant_id` 元数据强制过滤。
2. **物理隔离**：大客户独立集群或命名空间。
3. **配额限流**：按租户限制 QPS 与存储。

**Token 成本控制**
1. 控制检索条数与块长度。
2. 选用小上下文模型做摘要与路由。
3. 压缩检索结果（抽取句子级证据再喂给主模型）。

**实战案例**：
在某金融 RAG 项目中，初期直接拼接 Top 5 块内容导致上下文超出 128k 限制且模型注意力被无关财报噪声干扰。通过引入 `LongContextReorder`（将最相关的文档放在首尾）及仅抽取包含 Query 关键词的句子喂给 LLM，Token 成本降低 60%，准确率提升 15%。

**代码示例（LLM 友好的检索上下文压缩）**：
```python
from langchain_core.prompts import PromptTemplate
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor

# 仅提取与 Question 相关的句子，去除废话
compressor = LLMChainExtractor.from_llm(llm)
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor, base_retriever=vector_retriever
)

compressed_docs = compression_retriever.get_relevant_documents("公司净利润是多少？")
```

**常见追问**
**Q：多租户下最常见的安全事故是什么？**
**A：** 检索未带租户过滤导致跨租户数据泄露；必须在数据库层与查询层双重校验。

**核心面试题精选**
1. **RAG 两步流水线**：离线（解析→清洗→分块→嵌入→建索引）；在线（Query→检索→重排→Prompt→生成）。
2. **RAG 与微调**：微调改善格式与领域表达，RAG 提供可更新事实；事实类优先 RAG。
3. **Chunk Overlap**：防止关键句被切断在两块边界，检索时丢失上下文。一般设 10-20%。
4. **RecursiveCharacterTextSplitter**：优先在段落断开，再退到句子、空格，减少语义碎片化。
5. **父子文档**：子块带 `parent_id`，检索子块（高精度，指向性强）→映射父块文本（提供丰富上下文）再生成。
6. **Embedding 归一化**：若用内积/余弦且框架假设归一化，应归一化以稳定相似度。
7. **FAISS IndexFlatIP vs IndexHNSW**：Flat 精确但慢；HNSW 近似快，适合大规模。
8. **混合检索**：验证集网格搜索定权重，或用 RRF（Reciprocal Rank Fusion）融合。
9. **HyDE 风险缓解**：重排序、引用约束、拒答、对比多条检索结果。
10. **Cross-Encoder**：无法替代向量索引，因需对每个 doc 与 query 运算，复杂度过高。
11. **GraphRAG**：解决多跳关系与全局聚合类问题。
12. **Prompt 注入防护**：文档清洗、权限隔离、输出引用限制、检测异常指令。
13. **长上下文模型**：RAG 不会消失，因私域数据规模、成本、检索聚焦、合规审计仍需 RAG。

**对比表格：向量检索 vs 关键词检索**

| 特性 | 向量检索 | 关键词检索 (BM25) |
| :--- | :--- | :--- |
| **原理** | 语义相似度 | 词汇精确匹配 (TF-IDF) |
| **优势** | 处理同义词、概念模糊匹配 | 处理专有名词、精确 ID 匹配成本低 |
| **劣势** | 无法处理精确 ID (如 SKU12345)，计算成本高 | 无法理解语义变体 (如 "省钱" vs "便宜") |
| **适用场景** | 问答、摘要、语义理解 | 搜索日志、代码搜索、实体查找 |
| **融合策略** | RRF 或 加权打分 | RRF 或 加权打分 |
