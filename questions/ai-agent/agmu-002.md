# 什么是「注意力漂移」?多 Agent 如何缓解

### 什么是「注意力漂移」? 多 Agent 如何缓解

**定义**：
注意力漂移指模型在长上下文或多目标提示下，对关键约束的关注度下降，导致输出偏离要求。这通常是由 LLM 的「遗忘窗口」或「中间迷失」现象引起的。

**原理细节**：
- **长上下文稀释**：当 Prompt 变长，关键信息淹没在大量 Token 中，模型注意力权重分散。
- **多目标干扰**：单 Prompt 包含过多指令（如：既要写代码、又要讲笑话、还要符合格式），导致模型顾此失彼。

**多 Agent 缓解方式**：
1. **子目标拆解**：将长任务拆分为多个短上下文子任务，每个 Agent 专注局部窗口。
2. **专职角色**：单 Agent 单责，减少单次 Prompt 中的指令维度（如：Coder 只写代码，Reviewer 只检查规范）。
3. **结构化中间态**：使用 JSON 或状态机存储中间结果，而非让模型在自然语言对话中「传递」所有上下文。

**单 Agent vs 多 Agent 缓解效果对比**：
| 方案 | 上下文长度 | 指令复杂度 | 耦合度 | 缓解效果 |
| :--- | :--- | :--- | :--- | :--- |
| **单 Agent (长 Prompt)** | 极长 (>10k) | 高 (混合指令) | 高 | 差 (易迷失首尾) |
| **多 Agent (分而治之)** | 短 (<2k/节点) | 低 (单一指令) | 低 | 优 (注意力聚焦) |

**追问应对**：
若问「不用多 Agent 怎么缓解？」——答：使用 Sliding Window + 关键信息摘要、RAG 注入关键约束句、约束指令前置、CoT 链式调用配合输出校验器等。

### 深化实战
- **实战案例**：在做复杂 SQL 生成任务时，单 Agent 经常在生成 100 行代码后忘记“只查询只读库”的安全约束。引入 Safety Agent 专门审核生成语句的权限关键字后，误删率降至 0。
- **代码示例（Python - 结构化通信）**：
```python
# 强制结构化通信，避免信息过载导致的漂移
message = {
    "role": "coder", 
    "payload": {"code": "..."}, # 只传必要数据，不传全部历史
    "constraints": ["use_readonly_repo"] # 显式携带约束
}
```

## 边界情况
- **循环漂移**：在多轮迭代（如 Agent 之间互相 Review）中，原始约束在多次重写后逐渐被“冲淡”或曲解，最终导致南辕北辙。
- **新生冲突**：长上下文中可能包含自相矛盾的指令（如 Prompt 前段要求“简短”，后段要求“详细”），模型可能优先关注最近出现的指令，导致前半部分约束失效。
- **静默失败**：Agent 没有明确报错，而是生成了符合格式但内容完全错误的幻觉结果，这是最难检测的漂移形式。

## 常见考点
1. **如何量化注意力漂移？**
   答：通过人工评估关键约束满足率，或使用约束检查模型自动测试在不同 Context Length 下的通过率。
2. **Agent 间通信如何避免漂移？**
   答：使用结构化消息（如 JSON Schema）而非纯文本聊天，确保接收方只解析关键字段。

## 易错点
- **误区**：认为增加 Token 数量总是能提高效果。
  **正解**：超过一定阈值后，关键信息的召回率反而下降（“大海捞针”失效），必须配合 RAG 或分层摘要。
- **误区**：认为多个 Agent 一定能解决所有漂移问题。
  **正解**：如果 Agent 之间的信息传递依然依赖长文本拼接，漂移依然会在通信链路中发生，必须坚持“结构化中间态”传递。

## 面试追问
1. 如果任务必须在一个 Agent 内完成（无法拆解），如何从 Prompt 工程角度最大程度缓解漂移？
2. 在多 Agent 协作中，如果下游 Agent 依据上游 Agent 的错误中间态继续工作，会产生“错误级联”，如何设计机制切断这种级联？
3. 如何通过自动化测试（如基于 LLM 的 Evaluator）来监控线上 Agent 系统的注意力漂移情况？
