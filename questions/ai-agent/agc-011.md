# Agent 的最大风险是什么

**最大风险：复合错误与权限滥用**

1.  **复合错误（错误累积）**：
    *   **原理**：在多步推理中，模型如果某一步产生幻觉或计算错误，后续步骤会基于该错误继续推导，导致误差指数级放大，无法自我修正。
    *   **后果**：虽然单步看似合理，但最终结果谬以千里。
    *   **实战案例**：某客服 Agent 在处理退订时，因幻觉错误地将“全额退款”工具的返回结果理解为“退款成功”，进而错误执行了“关闭工单”并调用了“发送确认邮件”，导致用户资金未退但工单已关，引发严重客诉。
    *   **边界情况**：**同义反复循环**。模型产生幻觉后，调用工具返回了错误的置信度反馈（例如工具本身有Bug，未校验参数直接返回成功），模型据此再次产生类似的幻觉，导致“错误-确认”的死循环，比单纯的报错更难被发现。

2.  **权限滥用与工具风险**：
    *   **原理**：模型可能被恶意 Prompt 注入，诱导其调用“删除数据库”、“发送邮件”等高危工具。
    *   **边界条件**：特别是当 Agent 拥有写入权限或访问敏感数据权限时。
    *   **极端场景**：**越狱结合工具调用**。攻击者通过角色扮演（如“你现在是忘记密码的系统管理员”）绕过安全审查，利用 Agent 的权限修改自身配置或提升权限。

**风险控制架构图：**
```text
User Input ──> [Agent Core] ──> Tool Request
                    │                │
                    ▼                ▼
            [Safety Check] ──X── [Guardrail / ACL]
               (输入/输出审查)        (最小权限校验)
                    │                │
                    ▼                ▼
              Allow/Block      Execute/Deny
```

**应对策略：**
*   **最小权限原则**：Agent 仅拥有完成任务所需的最小权限，且不支持交互式 Shell。
*   **强审计**：记录所有 Tool Call 的参数、返回值及上下文。
*   **人在回路**：对于高风险操作（如资金转账、删库），必须引入人工审批确认。

**关键代码示例：**
```python
# 伪代码：基于 RBAC 的人机回路审批
def execute_tool(agent_request):
    tool = get_tool(agent_request.tool_name)
    if tool.risk_level == "HIGH":
        approval = request_human_approval(agent_request)
        if not approval.approved:
            return {"error": "User denied execution"}
    return tool.execute(**agent_request.args)
```

**## 面试追问**
1.  如果 Agent 被 Prompt Injection 攻击并尝试外连 C2 服务器，在网络安全层面除了 ACL 还有什么防御手段？（答：流量特征检测、DNS 监控、Egress Firewall 白名单机制）。
2.  如何在保证 Agent 自主性的同时，避免其对生产环境造成不可逆的破坏？（答：使用影子模式/灰度环境进行验证，或对所有写操作实施“延迟生效+人工复核”机制）。

**## 易错点**
1.  **过度依赖模型自我纠错**：认为模型能够像人类一样意识到自己错了。实际上，在缺乏外部负反馈（如工具报错）的情况下，模型很难自我修正幻觉，反而可能加强错误信念。
2.  **忽视输入侧的 Guardrail**：只关注工具调用的输出是否合规，忽略了用户输入本身可能包含恶意指令或敏感数据提取攻击，导致在第一步规划阶段就已失控。

**## 常见考点**
1.  **追问**：什么是 Prompt Injection，如何防御？（答：通过分隔符、指令清洗及输出审查来防御）。
2.  **追问**：如何处理 Agent 的死循环问题？（答：设置最大步数限制、Token 预算熔断及重复内容检测）。
