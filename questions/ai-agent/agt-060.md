# 前沿模型与选型

### 概念解释
选型需在闭源（API）与开源（可自托管）之间权衡能力、成本、合规与迭代速度。

### 选型要点
#### 1. 闭源 vs 开源
- **开源**：数据自主、可微调、离线部署；需自备算力运维。适用：高合规、深度定制。
- **闭源**：上手快、持续升级；依赖供应商。适用：快速验证、通用任务。

#### 2. 主流模型对比
| 维度 | GPT-4o | Claude | Gemini | DeepSeek (开源代表) |
| :--- | :--- | :--- | :--- | :--- |
| **特点** | 语音图像端到端 | 长文本与代码 | 原生多模态 | 工程效率、推理强 |
| **优势** | 综合生态强 | 上下文超长 | 谷歌生态集成 | 性价比高、可微调 |

#### 3. 选型策略
- **混合模式**：敏感数据本地开源模型，通用能力调用 API。
- **评估维度**：Benchmark 需结合具体业务任务，避免仅看公开榜单。

#### 4. 蒸馏与量化优化
- **模型蒸馏**：在闭源大模型（如GPT-4）指导下训练小模型，保留90%能力但降低推理成本。
- **量化技术**：FP16/INT8/INT4量化，显存占用减半，推理速度提升（需关注精度损失）。

#### 5. 评估与边界
- **评估指标**：准确率、召回率、响应延迟、Tokens/成本。
- **边界条件**：长文本截断、Rate Limit限制、输出格式稳定性。

### 实战案例
某企业内部知识库项目初期直接调用 GPT-4 API，但因涉及薪资等核心数据无法上传云端，且 API 延迟较高导致用户等待。最终切换为 DeepSeek-Coder 并利用公司私有代码库进行 LoRA 微调，不仅解决了数据合规问题，还将单次查询成本降低了 95%。

### 关键代码示例 (路由策略模拟)
```python
def route_request(prompt):
    # 简单的关键词路由逻辑
    sensitive_keywords = ["薪资", "保密", "内部数据"]
    if any(keyword in prompt for keyword in sensitive_keywords):
        # 路由到本地开源模型 (如 vLLM 实例)
        return query_local_model(prompt)
    else:
        # 路由到云端高性能 API (如 OpenAI)
        return query_cloud_api(prompt)
```

### 边界情况补充
1. **幻觉处理**：开源模型在垂直领域微调后，可能产生“固执幻觉”（对错误答案置信度极高），需在 Prompt 中加入“不确定时回复不知道”的指令或引入检索增强（RAG）校验。
2. **多语言与长尾语言**：部分模型（如某些英文主导的开源模型）在中文或小语种上表现断崖式下跌，选型时需针对目标语言做 AB 测试，而非仅参考英文 Benchmark。
3. **故障转移**：混合架构中，当云端 API 超时或宕机时，本地模型需具备承接流量的兜底能力，即使精度下降也要保证服务不中断（降级策略）。
4. **Token 泄露**：使用闭源 API 时，需注意 System Prompt 或上下文中的 PII（个人敏感信息）被用于模型训练的风险，需查阅供应商最新隐私政策并配置数据不留存选项。

### ## 面试追问
1. **模型漂移**：如果云端闭源模型突然更新版本导致输出格式变化或逻辑变差，你们的架构中如何做版本兼容和快速回滚？
2. **评估集构建**：在业务初期没有标准答案的情况下，如何构建高质量的 Golden Set（黄金数据集）来客观评估不同模型的表现？
3. **冷启动问题**：本地私有化部署时，如果显存资源有限，如何利用如 vLLM 的 PagedAttention 或 Continuous Batching 技术来提升并发吞吐量？

### ## 易错点
1. **迷信榜单**：错误地认为 Open LLM Leaderboard 排名高的模型一定适合业务。实际上，榜单多为 MMLU、GSM8K 等通用测试，与特定业务场景（如特定格式的 JSON 提取、医疗术语理解）的相关性极弱。
2. **忽视推理成本结构**：只关注模型的 Token 定价，忽略了 Context 的计费差异。例如，某些模型 Prompt Token 极贵但 Completion 便宜，适合 Few-shot 场景；而某些模型 Input/Output 同价，适合对话场景。
3. **过度微调**：认为开源模型必须经过全量微调才能用。实际上，很多 70B+ 级别的模型通过 Prompt Engineering 或 Few-shot 即可满足需求，微调反而可能导致灾难性遗忘。

### ## 常见考点
1. **参数效率微调（PEFT）**：LoRA与全量微调的区别及显存节省原理。
2. **上下文窗口限制**：处理超长文档时的RAG（检索增强生成）与长文本模型选择。
3. **MoE（混合专家模型）**：如 DeepSeek-V2 或 Mixtral 的推理特性（激活参数少但显存占用大）及对硬件的要求。
