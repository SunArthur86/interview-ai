# Agent记忆的Token预算如何分配才不容易翻车?不同记忆类型的Token配比建议

### Token 预算分配框架（假设总上下文 128K tokens）
- **System Prompt**: 2K (1.5%)
- **工具定义**: 3K (2.5%)
- **长期记忆检索**: 10K (8%)
- **对话历史(近期)**: 15K (12%)
- **RAG 文档检索**: 30K (24%)
- **当前任务输入**: 5K (4%)
- **输出预留**: 4K (3%)
- **剩余空间（安全余量）**: 59K (46%)

### 关键原则
1.  **永远留余量**：不要把上下文填满，留 20-30% 余量。
2.  **动态调整**：根据任务类型调整配比。
3.  **优先级排序**：当前任务 > 对话历史 > RAG 文档 > 长期记忆。

### 记忆 Token 控制策略
| 记忆类型 | 压缩方式 | 目标 Token |
|---------|---------|-----------|
| 对话历史 | 滑动窗口(旧) + 摘要(更旧) + 原文(最新3轮) | <15K |
| 工具输出 | 截断(取前/后部) 或 结构化提取 | <5K/次 |
| RAG 文档 | 精排(Cross-Encoder取 Top-3) | <30K |
| 长期记忆 | 检索(取相关 Top-5) + 重要性过滤 | <10K |

### 翻车场景与应对
- **场景1**：工具返回 10MB 日志 → **撑爆上下文**
  - **对策**：LLM 无法直接阅读。必须在 Tool 层做预处理（提取 Error 级别日志、统计摘要），只传结果给 LLM。
- **场景2**：对话 100 轮未压缩 → **超限**
  - **对策**：当历史超过阈值（如 8K），触发异步摘要任务，将旧对话替换为 Summary。
- **场景3**：RAG 返回 50 个文档 → **太多噪音**
  - **对策**：必须使用两阶段检索（向量粗排 + 重排 Rerank），严格控制进入 Context 的文档数量。

### ## 常见考点
- 如果留的余量太大，会不会浪费 Token 成本？（答案：可以使用流式处理或动态 Context Window 扩展技术，但在推理前预留 buffer 是为了防止生成长文本被截断导致报错）
- RAG 检索结果和记忆冲突了怎么办？（答案：通常 Prompt 中规定 RAG 文档（实时知识）优先级高于长期记忆（陈旧知识），或者让 LLM 进行显式对比判断）
