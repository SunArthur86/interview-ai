# 生产环境记忆系统最容易出的事故是什么?怎么防

最常见事故是租户隔离失败与把敏感数据写入长期记忆未加密。

### 关键原理与细节
1.  **租户隔离失效**：通常发生在向量检索或中间件层。若仅在应用层过滤，当底层查询逻辑（如 `filter` 参数）构造错误时，会导致 A 用户检索到 B 用户的记忆。
2.  **敏感数据泄露**：PII（个人身份信息）如手机号、身份证号直接嵌入向量，可能通过近似搜索被还原。
3.  **一致性事故**：写入向量库成功但元数据（如 SQL）写入失败，导致“僵尸数据”堆积，影响检索精度。

### 防护措施（架构级）
```text
写入流程防护：
┌─────────────┐    PII扫描     ┌─────────────┐   加密/脱敏    ┌─────────────┐
│ 原始输入数据 │ ─────────────> │ 敏感识别层   │ ────────────> │ 向量化/存储  │
└─────────────┘                └─────────────┘                └─────────────┘
                                    │                              │
                                    v                              v
                           [阻断/打码] PII字段            [强校验] Tenant_ID

查询流程防护：
┌─────────────┐  强制注入     ┌─────────────┐  权限校验     ┌─────────────┐
│ User Query  │ ────────────> │ 查询构造器   │ ────────────> │ 向量数据库   │
└─────────────┘  Tenant_ID   └─────────────┘  Row-Level    └─────────────┘
```
1.  **强制租户过滤**：在数据库中间件或 ORM 层实现 Row-Level Security (RLS)，确保所有查询自动拼接 `WHERE tenant_id = ?`，禁止全表扫描。
2.  **PII 防护**：在 Embedding 前部署正则或 NER 模型识别敏感字段，进行掩码（如 `138****0000`）或哈希处理。
3.  **审计与最小权限**：数据库账号仅授予 `读写` 权限，禁止 `DROP`；开启审计日志记录所有查询的 `user_id` 和 `filter` 条件。
4.  **测试覆盖**：编写混沌测试或集成测试，模拟错误 ID 注入，验证是否返回空集而非他人数据。

### ## 常见考点
- 如果向量数据库不支持行级权限，应用层该如何保证隔离？（答案：使用多租户集群隔离或逻辑命名空间）
- 敏感数据在 Embedding 阶段被脱敏了，生成时需要还原怎么办？（答案：单独维护映射表，仅在有权限时回填）
- 如何检测记忆污染？（答案：定期人工抽检 Top 检索结果，或计算检索结果与当前 Query 的语义突变率）
