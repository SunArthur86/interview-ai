# 生产环境记忆系统最容易出的事故是什么?怎么防

最常见事故是租户隔离失败与把敏感数据写入长期记忆未加密。

### 关键原理与细节
1.  **租户隔离失效**：通常发生在向量检索或中间件层。若仅在应用层过滤，当底层查询逻辑（如 `filter` 参数）构造错误时，会导致 A 用户检索到 B 用户的记忆。
2.  **敏感数据泄露**：PII（个人身份信息）如手机号、身份证号直接嵌入向量，可能通过近似搜索被还原。
3.  **一致性事故**：写入向量库成功但元数据（如 SQL）写入失败，导致“僵尸数据”堆积，影响检索精度。

**边界情况**：
- **Embedding 模型自身泄露**：即使做了元数据隔离，如果攻击者通过特定的 Prompt（如“请输出与薪资最相似的内容”）诱导模型输出，且敏感信息已被向量化，模型仍可能通过语义关联“复述”出敏感信息。
- **批量操作的原子性**：在批量 Upsert 时，如果网络中断，可能导致部分数据写入、部分失败。没有事务机制的向量库会产生数据不一致。
- **孤儿数据**：用户注销账号后，其对应的情景/语义记忆可能因清理脚本遗漏而残留在库中，成为隐私隐患。

### 防护措施（架构级）
```text
写入流程防护：
┌─────────────┐    PII扫描     ┌─────────────┐   加密/脱敏    ┌─────────────┐
│ 原始输入数据 │ ─────────────> │ 敏感识别层   │ ────────────> │ 向量化/存储  │
└─────────────┘                └─────────────┘                └─────────────┘
                                    │                              │
                                    v                              v
                           [阻断/打码] PII字段            [强校验] Tenant_ID

查询流程防护：
┌─────────────┐  强制注入     ┌─────────────┐  权限校验     ┌─────────────┐
│ User Query  │ ────────────> │ 查询构造器   │ ────────────> │ 向量数据库   │
└─────────────┘  Tenant_ID   └─────────────┘  Row-Level    └─────────────┘
```
1.  **强制租户过滤**：在数据库中间件或 ORM 层实现 Row-Level Security (RLS)，确保所有查询自动拼接 `WHERE tenant_id = ?`，禁止全表扫描。
2.  **PII 防护**：在 Embedding 前部署正则或 NER 模型识别敏感字段，进行掩码（如 `138****0000`）或哈希处理。
3.  **审计与最小权限**：数据库账号仅授予 `读写` 权限，禁止 `DROP`；开启审计日志记录所有查询的 `user_id` 和 `filter` 条件。
4.  **测试覆盖**：编写混沌测试或集成测试，模拟错误 ID 注入，验证是否返回空集而非他人数据。

**实战案例**：
某次代码重构中，开发者误将向量检索的 `filter` 参数从字典写成了字符串，导致 `must_contain` 过滤失效，造成了跨租户数据泄露。事后我们在数据入库层增加了“元数据强制校验中间件”，若记录中缺失 `tenant_id` 字段则直接抛出异常拒绝入库，从源头堵住了漏洞。

**代码示例**：
```python
def safe_upsert(vectors, metadata):
    # 强制元数据校验，防止隔离失效
    if "tenant_id" not in metadata or not metadata["tenant_id"]:
        raise ValueError("SecurityError: tenant_id is mandatory for write operations.")
    
    # PII 脱敏检查
    if PII_DETECTOR.contains_secret(vectors):
        logging.warning("Blocked potential PII data injection.")
        return False
    
    return vector_db.upsert(vectors, metadata=metadata)
```

**## 面试追问**
1. 如果向量数据库（如某些开源版本）原生不支持 Row-Level Security，你如何在应用架构层面实现高性能的租户隔离？（考虑分库分表或命名空间）
2. 既然 PII 在入库前被脱敏了（如变成 `***`），当用户询问“我的手机号是多少”时，系统如何既保护隐私又能正确回答？
3. 面对海量历史数据的“被遗忘权”请求（GDPR），如何保证向量库中的数据被彻底物理删除，而非仅仅标记为删除？（特别是涉及到全量索引时）

**## 易错点**
- **过度信任模型**：认为在 Prompt 里写了“不要透露隐私”就能防止泄露。实际上，模型生成的上下文是基于检索结果的，如果检索结果本身包含了隐私，模型很难违背事实进行完全的掩盖。
- **忽视训练数据泄露**：在使用定制化微调 Embedding 模型时，如果训练集中包含了特定租户的敏感信息，模型可能将这些信息编码进权重中，导致即使数据库清空了，模型仍能“记住”并泄露这些信息。需警惕模型权重层面的隐私风险。

**## 常见考点**
- 如果向量数据库不支持行级权限，应用层该如何保证隔离？（答案：使用多租户集群隔离或逻辑命名空间）
- 敏感数据在 Embedding 阶段被脱敏了，生成时需要还原怎么办？（答案：单独维护映射表，仅在有权限时回填）
- 如何检测记忆污染？（答案：定期人工抽检 Top 检索结果，或计算检索结果与当前 Query 的意外相关性）

## 记忆要点

- 核心事故：租户隔离失效（A用户查到B数据）与敏感数据（PII）未加密入库。
- 防护措施：中间件强制注入Tenant_ID过滤；入库前PII扫描脱敏或哈希。
- 数据一致性：防止向量库写入成功但元数据失败的“僵尸数据”，需事务保障。
- 审计测试：编写混沌测试模拟错误ID注入，验证返回空集而非他人数据。
