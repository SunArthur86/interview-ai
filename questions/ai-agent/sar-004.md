# Agentic RAG是什么？它和传统RAG有什么本质区别？

🎯 本质：Agentic RAG将RAG从"单次检索-生成"升级为"智能体多轮迭代检索"，让LLM作为Agent主动规划检索策略、评估结果、决定是否需要进一步检索。

📊 Agentic RAG vs 传统RAG：
| 维度 | 传统RAG | Agentic RAG |
|------|---------|-------------|
| 检索次数 | 1次 | 多次（按需迭代） |
| 检索策略 | 固定(向量搜索) | 动态(改写、分解、多源) |
| 决策能力 | 无 | 自主规划和评估 |
| 查询方式 | 原始查询 | 改写+分解+子查询 |
| 工具使用 | 仅向量搜索 | 向量+SQL+API+Web |

Agentic RAG核心流程：
1. Query理解与分解
  Agent分析复杂问题，分解为子问题
  例："比较2023和2024年的GDP增长" -> 2个子查询
  
2. 多轮检索
  Round 1: 检索 -> 评估 -> 信息不足？
  Round 2: 改写查询 -> 重新检索 -> 评估
  直到获取足够信息或达到最大轮次

3. 多源检索
  向量数据库（语义匹配）
  SQL数据库（精确查询）
  Web搜索（实时信息）
  知识图谱（关系查询）

4. 答案合成与验证
  综合多轮检索结果
  交叉验证不同来源的信息
  标注引用来源

实现框架：
- LangGraph: 基于状态图的Agentic RAG
- LlamaIndex: Workflows + Agent
- CrewAI: 多Agent协作RAG

Agentic RAG是Self-RAG + CRAG + 工具使用 + 规划能力的集大成者。适用于复杂研究、多跳推理、实时信息综合等场景。

💡 **实战案例**：在"竞品分析"自动生成系统中，Agentic RAG会先调用API获取竞品财务数据，发现缺少"市场份额"信息后，自主决策发起Google搜索抓取研报，最后查询内部数据库对比自家产品线。这种多工具协同打破了传统RAG只能查静态文档的限制。

```python
# 伪代码：基于 LangGraph 的 Agentic RAG 节点定义
# 语言：Python

from langgraph.graph import StateGraph

def agent_node(state):
    """决策节点：决定下一步行动"""
    query = state["query"]
    # Agent 思考：还需要什么信息？
    thought = llm.predict(f"Current info: {state['context']}. Need: {query}")
    if "financial_data" in thought and not state["has_sql"]:
        return "tool_sql"
    elif "market_news" in thought and not state["has_web"]:
        return "tool_web"
    else:
        return "generate_answer"

def sql_tool_node(state):
    """工具节点：查询数据库"""
    data = db.query(state["sql_query"])
    state["context"].append(data)
    state["has_sql"] = True
    return state

# 构建状态图，实现动态路由
workflow = StateGraph(AgentState)
workflow.add_node("agent", agent_node)
workflow.add_node("sql_tool", sql_tool_node)
workflow.add_conditional_edges("agent", should_continue) # 动态判断流向
```

## 记忆要点

- 本质区别：从单次检索升级为Agent多轮迭代，具备自主规划和工具调用能力
- 核心流程：理解分解 -> 多轮检索（动态改写） -> 多源融合 -> 答案合成
- 工具扩展：除向量库外，还能调用SQL、API、Web搜索等多源数据
- 集大成者：Self-RAG反思 + CRAG纠正 + 工具使用 + 规划能力的综合
