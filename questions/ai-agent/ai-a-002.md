# ReAct框架的核心思想是什么?Thought-Action-Observation循环如何工作

- **ReAct = Reasoning + Acting**

让LLM交替进行推理和行动,使用工具后观察结果.

- **工作流程:**
```
**用户：** 北京明天天气如何?

**Thought：** 我需要查询北京的天气预报
**Action：** weather_api(city="北京", date="tomorrow")
**Observation：** 晴, 25°C, 东南风3级

**Thought：** 我拿到了天气信息,可以回答了
**Action：** finish(answer="北京明天晴,25°C,东南风3级")
```

- **优势:**
1. **减少幻觉** - 基于工具返回的真实数据
2. **可追溯** - 推理链可审查
3. **可纠错** - 工具失败后可重试或换方案

- **局限:**
- Token消耗大(每次都带完整推理历史)
- 延迟高(多轮工具调用)
- 需要精心设计工具描述

- **实战案例**：在运维告警处理中，当服务器API报错时，LLM不会凭空捏造原因，而是通过ReAct调用日志查询工具，发现是磁盘满，再自动调用清理脚本，避免了人工排查的滞后。

- **代码示例**：
```python
# 模拟 ReAct 循环解析伪代码
prompt = """
Question: {input}
Thought: {agent_scratchpad}
"""

while True:
    response = llm.predict(prompt) # LLM生成 Thought 和 Action
    thought, action, action_input = parse_react(response)
    if action == "Finish": break
    
    observation = tools[action].run(action_input) # 执行工具
    prompt += f"\nObservation: {observation}" # 将结果回填
```

- **提示词工程细节：**
  - 使用 `Thought:` 前缀强制模型显式推理。
  - 限制 `Action Input` 为 JSON 格式以便解析。
  - 设置 `max_iterations` 防止无限循环。

- **边界情况**：
  - **工具解析失败**：LLM输出格式错误（如JSON不合法），导致代码无法解析Action，需要增加Try-Catch重试或CoT提示纠正。
  - **Observation过长**：工具返回内容（如长文章）超出模型处理能力，必须进行截断或摘要后再填入Prompt。
  - **循环依赖**：Thought误判导致反复调用同一个工具（如无限翻页），需通过检测重复Action打断循环。

- ## 面试追问
  1. **当工具返回的结果是空或异常时（如404），如何在Prompt中引导LLM进行错误恢复而不是直接回答“我不知道”？**
  2. **ReAct模式推理链很长，如何优化Prompt以减少Token消耗？例如使用Few-shot还是结构化输出？**
  3. **如何处理并发的Action请求？ReAct是串行的，如果任务允许并行，架构上该如何改造？**

- ## 易错点
  - **过度依赖Prompt**：以为只要Prompt写得好，ReAct就能稳定运行。实际上，工具的Schema定义和错误处理逻辑同样关键。
  - **混淆Observation与Answer**：Observation是工具的原始输出（如JSON），Answer是给用户的最终结论。LLM容易直接把原始结果丢给用户，需要进行格式化指令。
