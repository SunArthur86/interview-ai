# Self-Refine和Reflection机制如何提升Agent质量?具体怎么实现

- **Self-Refine流程:**

```
生成初稿 → 自我批评 → 修改 → 自我批评 → 修改 → ... → 最终版
```

- **实现:**
```python
# 1. 生成
output = llm(task)

# 2. 批评
feedback = llm(f"审查以下输出的问题:\n{output}")

# 3. 修改
output = llm(f"根据反馈改进:\n反馈:{feedback}\n原文:{output}")

# 重复2-3直到满意或达到最大次数
```

- **Reflection vs Self-Refine:**
- **Self-Refine:** 同一模型自我改进
- **Reflection:** 可以用不同模型/角色来批评

- **效果:**
- 代码生成:通过率提升10-20%
- 推理任务:准确率提升5-15%
- 写作任务:质量提升但增加2-3倍Token

- **关键:** 批评prompt要具体(不要说「改进」,要说「检查边界条件、错误处理、代码风格」)

- **迭代优化循环图:**
```text
┌────────────┐
│  Initial   │
│  Draft     │
└─────┬──────┘
      │
      ▼
┌────────────┐
│  Critique  │ <─── [Prompt: List specific flaws]
│  /Feedback │
└─────┬──────┘
      │
      ▼
┌────────────┐
│  Refine    │ <─── [Prompt: Fix based on feedback]
└─────┬──────┘
      │
      ▼
  (Check Quality)
      │
   Pass │ ───────────> [Final Output]
      │
      Fail ┘ (Loop back to Critique)
```

- **实战案例**: 在生成SQL查询时，LLM初次生成常忘记JOIN条件或数据类型转换。通过引入Reflection，让Critic模型先检查“是否有隐式笛卡尔积风险”，Generator根据此反馈添加明确的JOIN ON子句，将生产环境SQL报错率降低了40%。

- **代码示例 (Python - 进阶版):**
```python
class SelfRefineAgent:
    def run(self, task):
        response = self.generator.generate(task)
        for _ in range(self.max_loops):
            feedback = self.critic.generate(
                f"Analyze this output for logic errors: {response}\n"
                f"If perfect, say 'PASS'. Else list issues."
            )
            if "PASS" in feedback: break
            
            response = self.generator.generate(
                f"Revise the output based on issues:\n{feedback}\n"
                f"Previous output:\n{response}"
            )
        return response
```

## 常见考点
1. **何时停止迭代？**
   - 问点：一直循环怎么办？
   - 答案：设定最大迭代次数（如3-5次），或者引入"自我评估"步骤，当模型认为"无需改进"时停止。
2. **Critic 模型是否必须比 Generator 模型强？**
   - 问点：自己批评自己有效吗？
   - 答案：Self-Refine 证明同模型有效，但如果有能力更强的模型（如 GPT-4 批评 GPT-3.5 生成的代码）作为 Critic，效果会更好。
3. **Reflection 的代价？**
   - 问点：Token 消耗增加多少？
   - 答案：每次迭代大约增加 1.5倍~2倍的 Token 消耗（原文+反馈+新文）。适用于高质量要求但对延迟不敏感的场景。

## 易错点
1. **过度修正**：如果 Critic 的 Prompt 过于严格或指出了不存在的问题（幻觉），Generator 可能会把正确的答案改错。需要在 Critic 阶段加入置信度判断。
2. **循环依赖**：在某些复杂的逻辑任务中，Refine 可能会在 A 和 B 两种解决方案之间反复横跳，导致无法收敛。需要记录历史版本并在 Prompt 中明确禁止回退到之前的状态。

## 面试追问
1. 如果 Critic 模型给出的 Feedback 是模糊的建议（如“优化代码逻辑”）而非具体的错误，你会如何设计系统来强制其输出可执行的指令？
2. 在多轮对话的 Agent 场景中，Self-Refine 应该作用于单轮回复还是整个对话历史？如何处理“修正单轮回复导致上下文不一致”的问题？
3. 除了文本层面的反思，如何结合外部环境（如代码执行结果、用户点击反馈）作为 Reflection 的信号？
