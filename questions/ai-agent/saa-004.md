# Function Calling和工具调用有什么区别？

Function Calling：LLM厂商提供的原生API能力，LLM根据用户意图输出结构化函数调用。

**流程**：
1. 定义函数schema（名称、参数、描述）
2. 用户输入 + schema发给LLM
3. LLM返回要调用的函数名和参数（JSON）
4. 应用执行函数，将结果返回给LLM
5. LLM基于结果生成最终回答

**核心定义**：
工具调用 = Function Calling（意图识别与参数提取） + 执行环境（实际运行逻辑）

**边界情况**：
1. **循环调用**：当工具返回的结果仍需进一步处理时，如何控制递归深度，防止无限循环消耗Token。
2. **函数不存在/冲突**：LLM可能会幻觉出未定义的函数，或者同时调用多个互斥的函数（如“订票”和“退票”），需要在代码层做校验。
3. **超长上下文**：当工具返回结果极大（如长文档检索）超过Context Window时，如何进行截断或摘要处理。

**实战案例**：
在做电商客服机器人时，LLM经常会把非商品的数字（如订单号中的随机串）误识别为商品ID调用库存查询。解决方法是在函数描述中显式加入“参数必须为特定格式的数字ID”约束，或者在代码层做正则校验拦截。

**代码示例（Python - OpenAI）**：
```python
import openai

def get_weather(location: str, unit: str = "celsius"):
    return f"The weather in {location} is 22{unit}"

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "北京天气怎么样？"}],
    tools=[{
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取指定城市的天气",
            "parameters": {
                "type": "object",
                "properties": {"location": {"type": "string"}},
                "required": ["location"]
            }
        }
    }]
)
```

**关键区别对比**：
| 维度 | Function Calling | 工具调用 |
|------|------------------|----------|
| 核心职责 | 输出意图和结构化参数 | 执行逻辑并返回结果 |
| 执行方 | 仅LLM | 外部运行环境 |
| 依赖性 | 无需外部环境 | 依赖特定库/API |

**最佳实践**：
1. 函数描述要清晰（让LLM理解何时用）
2. 参数schema要精确（类型、必填、枚举）
3. 并行调用：现代LLM支持一次返回多个函数调用
4. 错误处理：函数执行失败时返回错误信息给LLM重试

## 面试追问
1. 如果LLM生成的JSON参数格式不符合定义（如缺少必填项或类型错误），你的容错机制是怎样的？
2. 在处理工具返回的过长文本时，你会如何设计截断策略以保证LLM能获取关键信息而不丢失上下文？
3. 如何实现并发的Function Calling，以及如何聚合多个并发调用的结果？

## 易错点
1. **混淆概念**：误以为Function Calling包含了代码执行步骤，实际上它只负责生成调用指令，执行必须由外部代码完成。
2. **忽略Prompt注入**：直接将用户输入拼接到工具参数中，可能导致注入攻击，应对用户输入进行清洗或使用LLM进行参数重写。

## 记忆要点

- 核心区别：Function Calling只负责输出结构化参数，工具调用负责实际执行逻辑。
- 流程：定义Schema -> LLM生成参数 -> 代码执行函数 -> 结果回传LLM生成回答。
- 边界控制：需校验幻觉函数、互斥调用，防止递归死循环消耗Token。
- 最佳实践：描述清晰、Schema精确，并行调用需聚合结果。
