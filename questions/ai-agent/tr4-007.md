# Semantic Kernel 的 Planner（规划器）是如何实现复杂任务分解的？它和 LangChain 的 Agent 有何不同？

Semantic Kernel (SK) 的 Planner 利用 LLM 来理解用户的意图，并将其分解为一系列可执行的函数调用链。SK 强调**原生代码集成**，Skills/Plugins 是通过 C# 或 Python 代码定义的强类型函数。Planner 会根据函数的语义描述和参数 schema 来构建一个执行计划。与 LangChain 的 Agent 相比，SK 的优势在于其**类型安全**和**企业级集成能力**：LangChain 更侧重于链式调用和 Prompt 模板的编排，灵活性高但调试复杂；SK 则更像是一个编排层，能够更自然地利用现有的业务逻辑代码库。此外，SK 的 Handlebars 模板允许更复杂的变量传递和计划结构化生成，适合需要精确控制执行流程的企业级应用。

## 边界情况
1. **循环依赖检测**：当 SK 的 Planner 分解任务时，如果 Function A 依赖 Function B 的输出，而 Function B 反过来又依赖 Function A，Planner 必须具备检测这种死锁的能力，否则会陷入无限规划。SK 需要在构建 Dependency Graph 时进行环检测。
2. **非确定性输入**：如果传入 Planner 的目标过于模糊（例如“帮我优化一下系统”），LLM 可能生成的计划跨度极大且不可执行，或者陷入反复重新规划的泥潭。需要通过 Few-Shot Prompting 限制规划粒度或引入人工确认节点。
3. **Schema 不匹配**：当原生代码函数的参数非常复杂（如嵌套的类对象），LLM 可能难以准确生成符合 JSON Schema 的参数填充，导致运行时反序列化失败。

## 面试追问
1. Semantic Kernel 的 Planner 在处理函数参数缺失时，是如何通过上下文自动填充的？如果必须由用户提供，流程是怎样的？
2. 相比 LangChain 的 ReAct 模式（推理-行动循环），SK 的“一次性生成完整计划再执行”模式在面对需要动态调整的任务时有什么劣势？
3. SK 强调类型安全，但在 Python 这种动态语言中，它是如何实现类似 C# 的强类型约束和 Schema 验证的？

## 易错点
1. **混淆 SK 与 LangChain 的定位**：LangChain 是“框架优先”，强调用代码定义 Prompt 和 Chain；SK 是“代码优先”，强调把现有的代码函数变成 Skill。不要试图用 SK 去做纯文本生成的 Chain 编排，那不是它的强项。
2. **忽视 Planner 的顺序执行限制**：标准的 Stepwise Planner 通常是串行执行的。如果需要并行执行独立的函数（如同时查天气和查股价），SK 需要特定的 Planner 实现（如 ActionPlanner 的手动编排或树状规划），默认配置下不会自动并行化。
