# 工具描述(tool description)为什么非常重要

**重要性核心原因：它是模型理解世界的唯一接口**
LLM 无法执行代码或直接联网，它完全依赖 JSON Schema 或自然语言描述来决定“调用什么”以及“怎么调用”。工具描述的质量直接决定了 Agent 的智能上限。

**描述不当带来的问题：**
*   **工具选错**：模型混淆了相似功能的工具（如 `create_user` 和 `update_user`）。
*   **参数幻觉**：模型发明了不存在的参数，或参数格式错误。

**优秀的 Tool Description 包含要素：**
1.  **意图说明**：清晰描述“何时使用该工具”（e.g., “仅当需要查询实时天气时使用”）。
2.  **否定约束**：明确“何时不用”（e.g., “不要用于查询历史天气”）。
3.  **参数细节**：每个参数的类型、取值范围、示例值。
4.  **副作用说明**：是否修改数据、是否有延迟。
5.  **返回样例**：提供标准的返回 JSON 样例，帮助模型理解后续如何处理数据。

**对比示例：**
*   *差的描述*：`search_web(query): Search the web.`
*   *好的描述*：`search_web(query): A search engine for recent web content. Use this when user asks about current events or data not in training set. Input should be a concise search keyword string. Returns a list of URLs and snippets.`

**实战案例**：
在一个电商 Agent 中，`get_product_price` 和 `get_product_stock` 两个工具描述模糊，导致 Agent 在用户询问“有货吗”时，错误地调用了查询价格接口而非库存接口，返回了无关信息。优化描述后，在 `get_product_stock` 中增加“Use only when checking availability or quantity”，Agent 识别准确率提升至 98%。

**关键代码示例**：
```python
# 定义高质量的工具描述 (Python LangChain风格)
tools = [
    {
        "name": "multiply",
        "description": "Useful for when you need to multiply two numbers together. "
                        "Do NOT use this for addition or subtraction.",
        "parameters": {
            "type": "object",
            "properties": {
                "a": {"type": "number", "description": "The first number to multiply"},
                "b": {"type": "number", "description": "The second number to multiply"}
            },
            "required": ["a", "b"]
        }
    }
]
```

**## 常见考点**
1.  **追问**：工具描述太长会导致什么问题？（答：消耗 Context Window，可能导致模型“遗忘”或产生注意力分散，需精简描述）。
2.  **追问**：如果模型总是选错工具，除了改描述还能怎么做？（答：利用 Few-Shot 在 System Prompt 中给几个选对工具的示例）。
