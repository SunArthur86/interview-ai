# Plan-and-Execute Agent和ReAct有什么区别？

ReAct：边想边做，每步 Thought→Action→Observation 循环。适合简单、动态的任务。

**Plan-and-Execute**：
1. **Planner**：LLM 先基于目标制定完整的步骤列表（多步骤规划）
2. **Executor**：逐步执行每个子任务，通常不进行深度推理，只负责调用
3. **Re-Plan**（可选）：根据执行结果和中间反馈调整后续计划

**架构对比图**：
```text
ReAct 模式 (串行交互):
User -> LLM -> Tool -> LLM -> Tool -> LLM -> Answer
(决策紧耦合，每一步都消耗大模型推理成本)

Plan-and-Execute 模式 (规划与执行分离):
     ┌─────────┐
     │ Planner │ (一次性生成计划: 1->2->3)
     └────┬────┘
          ▼
   ┌──────────────┐
   │ Step List:   │
   │ 1. Search...  │
   │ 2. Calc...   │
   │ 3. Summarize │
   └──────┬───────┘
          ▼
     ┌─────────┐
     │Executor │ ──▶ Tool 1
     │(轻量LLM)│ ──▶ Tool 2
     │ or Rule │ ──▶ Tool 3
     └────┬────┘
          ▼
       Answer
```

**优势**：
- **推理成本优化**：Planner 只需调用一次（通常用强模型），Executor 步骤可用更小、更快的模型甚至规则引擎
- **全局视野**：在执行前就有完整路径，减少陷入局部死胡同的概率
- **并行化**：若计划中的步骤无依赖关系，Executor 可并行执行

**劣势**：
- **脆弱性**：如果初始计划有误，执行过程很难补救（除非引入 Re-Plan 机制）
- **延迟问题**：必须等所有计划生成后才能开始第一步执行
- **缺乏反馈**：Executor 通常不修正计划，遇到错误容易直接失败

**实际应用**：
- **LangChain**：`PlanAndExecute` Agent
- **BabyAGI**：基于任务列表循环生成、执行和优先级排序
- **AutoGPT**：自主设定目标、拆解任务并执行

## 常见考点
1. **什么情况下 Plan-and-Execute 效果不如 ReAct？**
   - 任务环境高度动态，无法预先规划（如与人类对话或探索未知API）
   - 简单任务，ReAct 的响应更快（省去了生成计划的步骤）
2. **如何解决计划错误的问题？**
   - 引入 **Re-Plan** 机制：当 Executor 失败时，将错误返回给 Planner 重新生成剩余步骤
   - 使用 **Self-Reflection**：让 Executor 评估计划可行性并给出反馈
3. **Planner 和 Executor 是否可以使用不同的模型？**
   - 是的，这是典型最佳实践。Planner 用 GPT-4 确保逻辑严密，Executor 用 GPT-3.5 或 Llama 3 降低成本。
