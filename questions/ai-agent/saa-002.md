# Plan-and-Execute Agent和ReAct有什么区别？

ReAct：边想边做，每步 Thought→Action→Observation 循环。适合简单、动态的任务。

**Plan-and-Execute**：
1. **Planner**：LLM 先基于目标制定完整的步骤列表（多步骤规划）
2. **Executor**：逐步执行每个子任务，通常不进行深度推理，只负责调用
3. **Re-Plan**（可选）：根据执行结果和中间反馈调整后续计划

**架构对比图**：
```text
ReAct 模式 (串行交互):
User -> LLM -> Tool -> LLM -> Tool -> LLM -> Answer
(决策紧耦合，每一步都消耗大模型推理成本)

Plan-and-Execute 模式 (规划与执行分离):
     ┌─────────┐
     │ Planner │ (一次性生成计划: 1->2->3)
     └────┬────┘
          ▼
   ┌──────────────┐
   │ Step List:   │
   │ 1. Search...  │
   │ 2. Calc...   │
   │ 3. Summarize │
   └──────┬───────┘
          ▼
     ┌─────────┐
     │Executor │ ──▶ Tool 1
     │(轻量LLM)│ ──▶ Tool 2
     │ or Rule │ ──▶ Tool 3
     └────┬────┘
          ▼
       Answer
```

**优势**：
- **推理成本优化**：Planner 只需调用一次（通常用强模型），Executor 步骤可用更小、更快的模型甚至规则引擎
- **全局视野**：在执行前就有完整路径，减少陷入局部死胡同的概率
- **并行化**：若计划中的步骤无依赖关系，Executor 可并行执行

**劣势**：
- **脆弱性**：如果初始计划有误，执行过程很难补救（除非引入 Re-Plan 机制）
- **延迟问题**：必须等所有计划生成后才能开始第一步执行
- **缺乏反馈**：Executor 通常不修正计划，遇到错误容易直接失败

**对比表格**：

| 维度 | ReAct | Plan-and-Execute |
| :--- | :--- | :--- |
| **执行方式** | 串行推理 + 执行 | 先全局规划，后执行 |
| **模型调用** | 高频调用强模型 | Planner(强) + Executor(弱/规则) |
| **纠错能力** | 强（每步可调整） | 弱（依赖 Re-Plan 机制） |
| **并行能力** | 差（天然串行） | 好（执行阶段可并行） |
| **适用场景** | 探索性任务、复杂逻辑 | 流程化任务、成本敏感场景 |

- **边界情况与极端场景**
- **Plan 长度限制**: Planner 生成的步骤数受限于上下文窗口。对于极度复杂的任务（如编写整个项目），需引入“分层规划”，即先生成主阶段计划，再动态展开子计划。
- **中间步骤变更**: 如果执行过程中外部环境发生了变化（如 API 接口变动），Pre-defined Plan 会失效。必须在 Executor 中加入异常回调机制，强制触发 Re-Plan。
- **幻觉计划**: Planner 可能会生成不存在工具的步骤。Executor 需具备工具可用性检测能力，当找不到对应工具时应抛出明确错误而非盲目尝试。

- **面试追问**
1. **追问 1**: Plan-and-Execute 在 Executor 失败时，如何设计“回滚”或“Re-Plan”策略，避免陷入无限重试？
2. **追问 2**: 如果 Executor 使用规则引擎（非 LLM），如何解析 Planner 输出的自然语言指令并映射到具体的函数调用？
3. **追问 3**: 在多 Agent 协作场景中，Plan-and-Execute 如何分配子任务给不同的 Agent，如何处理 Agent 间的依赖关系？

- **易错点**
1. **易错点**: 认为 Plan-and-Execute 总是比 ReAct 快。如果 Planner 花费大量时间思考，或者计划步骤极其繁琐，总耗时可能超过 ReAct 的“边做边改”。
2. **易错点**: 忽视 Executor 的错误处理。如果 Executor 只是简单的函数调用，一旦参数缺失就会报错，应在 Prompt 中要求 Planner 输出完整的参数 Schema。
