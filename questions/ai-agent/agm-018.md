# Embedding 模型升级后旧向量怎么办

旧向量与新模型不在同一向量空间，不可直接混合比较，否则余弦相似度计算失效（即语义不匹配）。

### 对比表格：升级策略权衡
| 策略 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **全量重算** | 效果无损，架构最清晰 | 耗时长，成本高，需停机或双写 | 原始文本保留完整，数据量中等 |
| **向量空间对齐** (如 Matryoshka) | 无需重算旧数据，速度快 | 存在精度损失，需训练对齐矩阵 | 原始文本丢失，或计算资源极度受限 |
| **双塔并行 (灰度)** | 风险低，可随时回滚 | 维护两套库，逻辑复杂，成本翻倍 | 高可用要求，且 V1 向量数据量巨大 |

### 处理方案架构
```text
阶段一: 双写期
┌─────────┐         ┌──────────────┐
│ 新数据   │ ──┬───> │ Embedding V1 │ (旧逻辑)
└─────────┘   │    └──────────────┘
              │           │
              └───┬───────┘
                  │           ▼
                  │      ┌──────────────┐
                  └────> │ Embedding V2 │ (新逻辑) ──> 写入新库/新字段
                         └──────────────┘

阶段二: 查询与回填
┌─────────┐    查询 V1      ┌──────────────┐
│ 用户Query│ ────────────> │ 向量库 (V1)   │ ──> 若未命中
└─────────┘                 └──────────────┘      │
                                                │ 离线重算 V2
                                                ▼
                                         ┌──────────────┐
                                         │ 向量库 (V2)   │ (全量重写完成后切换)
                                         └──────────────┘
```

### 关键细节
1.  **模型版本标识**：在向量记录的 Metadata 中强制存储 `embedding_model_version` 字段。
2.  **灰度发布**：先对新数据使用新模型，旧数据保留旧向量。查询时，分别检索 Top-K V1 和 Top-K V2 结果，在应用层进行 Rerank 混合，逐步验证 V2 效果。
3.  **全量重嵌入**：后台异步任务读取所有原始文本，用新模型重新向量化并覆盖。注意对在线服务的影响，需限流。
4.  **兼容性处理**：切换期间，若用户 Query 使用 V2 向量，必须确保检索参数中指定 `filter: model_version='v2'`，避免跨空间比对。

### 实战案例
在从 bge-small 升级到 bge-large 时，未做隔离直接混合查询，导致检索到的全是语义接近但无关的旧数据（因为不同维度的向量空间度量标准不同）。最终采用“蓝绿部署”：新建 Collection 存 V2 数据，通过应用层 Routing 流量，待全量迁移完成后切换 DNS。

### 代码示例 (Python - 混合检索与元数据过滤)
```python
# 查询时强制通过元数据过滤，防止跨模型比对
def search_hybrid(query_vector, top_k=5):
    # 查询新模型数据
    results_v2 = v2_client.search(
        collection_name="memories_v2",
        query_vector=query_vector,
        limit=top_k,
        expr="model_version == 'v2'" # 关键：确保空间一致性
    )
    
    if len(results_v2) < top_k:
        # 回退：如果 V2 数据不足，可查询 V1
        results_v1 = v1_client.search(...)
        return rerank_mix(results_v1, results_v2)
    return results_v2
```

### ## 边界情况
1.  **原始文本丢失**：很多向量库为了节省存储只存了 Vector 和 ID，丢了原始 Text。此时无法全量重算，只能通过“回放”业务日志来补救，或者采用“向量对齐”技术训练一个 V1->V2 的映射矩阵（有损）。
2.  **维度不一致**：旧模型 1024 维，新模型 768 维。某些向量数据库（如 Milvus）不支持同一 Collection 存不同维度向量，必须新建 Collection 或使用 Padding（不推荐），必须进行物理隔离。
3.  **量化参数变化**：如果新模型引入了不同的量化方式（如从 Float32 变为 Int8），余弦相似度的归一化方式会变，导致分数不可比，需要重新校准阈值。

### ## 面试追问
1.  如果原始文本丢失了，除了向量对齐，还有什么办法利用旧数据？（答案：可以用 V2 模型检索旧向量，虽然空间不通，但在某些场景下可通过 Adapter 进行特征转换，或仅作为最后兜底参考）
2.  在灰度发布阶段，如何保证用户体验的一致性？（答案：在 Rerank 阶段加入“版本平滑因子”，或通过 A/B Test 对比两组效果，确保 V2 效果显著优于 V1 才全量切换）
3.  “向量空间对齐”的具体技术原理是什么？（答案：通常训练一个线性映射矩阵 W，使得 ||W * V1_old - V2_target|| 最小化，常使用 DSTL 或 Skip-Thought 等对齐技术）

### ## 易错点
1.  **Metadata 过滤性能陷阱**：在大规模数据下，查询时增加 `model_version` 过滤器可能会导致索引失效或性能大幅下降（取决于 VectorDB 的实现，如部分索引不支持高效标量过滤），需测试混合查询性能。
2.  **归一化问题**：某些模型输出是 L2 归一化的，某些不是。直接比对余弦相似度时，必须确保 Query 和 Document 都在相同的度量空间（是否归一化）。

## 记忆要点

- 核心问题：新旧向量不在同一空间，不可直接混合比较，否则相似度失效。
- 升级策略：全量重算（效果最好）、向量空间对齐（无原始文本时）、双塔并行（灰度）。
- 灰度方案：Metadata标记版本，查询时强制过滤 `model_version`，避免跨空间比对。
- 实战陷阱：若原始文本丢失，只能通过日志回放或训练映射矩阵（有损）补救。
