# Planner-Executor 架构及优缺点.

**架构说明**：
- **Planner（规划器）**：负责理解用户目标，将其拆解为步骤列表或 DAG（有向无环图），处理任务分解。
- **Executor（执行器）**：逐步执行 Planner 产出的步骤，调用工具或子 Agent。
- **重规划**：Executor 将执行结果反馈给 Planner，若遇到错误或环境变化，Planner 修正计划。

**优缺点**：
- **优点**：将“思考”与“行动”分离，结构清晰，易于对 Plan 进行校验、人工审核或并行化执行（DAG）。
- **缺点**：两阶段调用增加 Token 成本和延迟；规划一次可能不准确（静态规划无法预知执行中的动态障碍），需要多次迭代重规划。

**流程与状态流转图**：

```text
用户目标
   │
   ▼
┌──────────────┐
│  Planner     │◄──────────────────────┐
│  (生成 Plan) │                       │
└──────┬───────┘                       │
       │ Plan (Step 1, 2, 3)          │ Obs/Error
       ▼                               │
┌──────────────┐                       │
│  Executor    │                       │
│  (执行 Step) │                       │
└──────┬───────┘                       │
       │ Result                       │
       ├──────────► (所有步骤完成?) ────┘
       │          │ No
       │          ▼
       │    [判断: 需重规划吗?]
       │          │ Yes
       └──────────┘
```

**补充细节**：
- **Plan 的格式**：常用 JSON 格式描述，包含 `step_id`, `description`, `tool`, `dependencies` (用于 DAG 并行)。
- **静态 vs 动态规划**：Planner-Executor 通常指静态规划；而 ReAct 模式更像是交替进行的动态规划，Planner-Executor 更适合需要复杂编排的场景。

### 实战案例
在某数据分析 Agent 中，Planner 生成 SQL 语句，但 Executor 执行时发现目标表结构已变更（字段不存在）。若不设计重规划机制，Executor 会反复报错死循环；引入重规划后，Planner 能根据新的 Schema 修正 SQL 并重新执行。

### 代码示例 (Python)
```python
def execute_plan(plan, max_steps=10):
    for step in plan.steps:
        for attempt in range(3): # 最多重试3次
            result = tool_executor.run(step.tool, step.args)
            if result.status == "SUCCESS":
                break
            if result.status == "FATAL_ERROR":
                return replan(current_goal, error=result.error_msg) # 触发重规划
    return "All steps completed"
```

### 对比表格

| 特性 | Planner-Executor 模式 | ReAct 模式 |
| :--- | :--- | :--- |
| **思考方式** | 全局规划，生成完整 DAG | 逐步推理，走一步看一步 |
| **适用场景** | 复杂任务编排、多步骤并行 | 简单任务、探索性任务 |
| **Token消耗** | 较高（需生成完整计划） | 较低（单步思考） |
| **纠错能力** | 弱（计划错误需整体推翻） | 强（立即修正下一步动作） |
| **并行能力** | 支持 DAG 并行执行 | 本质串行 |

## 常见考点
1. **Executor 执行失败时，如何决定是重试当前步骤还是让 Planner 重规划？**（答：看错误类型，网络超时可重试；参数错误或逻辑不通通常需重规划）
2. **如何减少两阶段调用带来的延迟？**（答：使用流式输出，或者当计划很置信时并行执行非依赖步骤）
3. **如果 Planner 生成了一个无法完成的 Plan，Executor 会陷入死循环吗？**（答：必须设定最大执行步数或时间限制，超过阈值强制终止并报错）
