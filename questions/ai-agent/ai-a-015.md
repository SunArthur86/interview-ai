# Agent工具调用失败时的错误恢复策略有哪些?如何设计鲁棒的Agent

- **错误类型与恢复策略:**

| 错误类型 | 示例 | 恢复策略 |
|---------|------|---------|
| 网络超时 | API不响应 | 重试+指数退避 |
| 参数错误 | 传错参数 | LLM修正参数后重试 |
| 权限不足 | 403 | 换工具/降级方案 |
| 结果异常 | 返回空/格式错误 | 验证+重新调用 |
| 工具不存在 | 404 | 换替代工具 |
| 逻辑冲突 | 并发写冲突 | 重试/串行化 |

- **鲁棒Agent设计原则:**

1. **结构化错误处理**
```python
try: 
    result = tool.execute(params)
    if not validate(result):
        raise ToolError("Invalid result")
except RetryableError:
    retry_with_backoff()
except FatalError:
    fallback_strategy()
```

2. **让LLM自己处理错误**
- 将错误信息返回给LLM
- LLM决定:修正参数/换工具/放弃
- 这是Agent的核心优势

3. **断路器模式**
- 连续N次失败后停止调用该工具
- 避免浪费Token

4. **检查点与回滚**
- 每N步保存状态
- 失败后可回滚到检查点

- **## 常见考点**
1. **错误 Prompt 设计**：如何将错误信息传递给 LLM 才能最大化修复成功率？（例如：明确指出是参数类型错误还是值越界，并提供 Schema 片段）。
2. **指数退避参数**：在重试策略中，初始等待时间、最大重试次数和退避倍数（如 base=2, cap=60s）应该如何根据业务的 SLA 设定？
3. **非致命错误判断**：如何定义哪些错误是 `Retryable`（如 5xx 网络错误）而哪些是 `Fatal`（如 401 认证失败）？错误的分类边界在哪里？
