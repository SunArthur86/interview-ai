# Plan-and-Execute 与 ReAct 两种 Agent 模式有什么核心区别？在什么场景下应该选择 Plan-and-Execute？

ReAct 和 Plan-and-Execute 是两种主要的 Agent 任务规划范式。

**ReAct** 是“边想边做”， interleaving 地进行推理和行动。每一步都基于上一步的观测结果决定下一步行动。这种方式灵活、反馈快，适合路径探索或未知环境，但容易陷入局部循环或因早期错误导致后续动作全部失败，Token 消耗不可控。

**Plan-and-Execute** 是“先想后做”，分为两个阶段：
1.  **Plan**：模型首先一次性规划出完成任务的全局步骤列表（如：1. 搜索；2. 读取；3. 总结）。这通常利用 LLM 强大的逻辑推理能力生成结构化的任务链。
2.  **Execute**：模型按顺序执行步骤，通常不再修改计划（或者引入一个“Re-planner”仅在遇到无法解决的错误时介入）。

**选择场景**：
-   **Plan-and-Execute** 适合步骤明确、逻辑链条较长的复杂任务（如编写长篇代码、多步骤的数据分析流程或旅游规划）。它能减少执行过程中的幻觉，避免在中间步骤频繁试错，Token 总成本往往更低且更可控。
-   **ReAct** 更适合需要动态调整策略、环境高度不确定的探索性任务（如网页导航、调试未知错误），因为初始计划可能会因为环境变化而完全失效。

```text
      ReAct Loop                  Plan-and-Execute Flow

   ┌─────────────┐             ┌───────────────┐
   │   Thought   │             │   Planner     │
   └──────┬──────┘             └───────┬───────┘
          │                           │
          ▼                           ▼ (Plan Steps: [1,2,3])
   ┌─────────────┐             ┌───────────────┐
   │   Action    │             │   Executor    │◄─────┐
   └──────┬──────┘             └───────┬───────┘      │
          │                           │             │
          ▼                           ▼             │
   ┌─────────────┐             ┌───────────────┐      │
   │ Observation │             │   Step 1 Act  │      │
   └──────┬──────┘             └───────┬───────┘      │

                                     └───────────────┘
```

### 实战深化

**1. 实战案例**：在自动化撰写竞品分析报告时，使用 **Plan-and-Execute** 表现更佳。Planner 首先规划出：[1. 搜索公司A营收, 2. 搜索公司B营收, 3. 搜索公司C营收, 4. 对比数据, 5. 生成报告]。Executor 按序执行。若中途第 2 步搜不到数据，只需针对该步骤重试，不会导致整个流程重跑，效率远高于 ReAct 的“走一步看一步”。

**2. 代码示例 (伪代码)**：
```python
# Plan Stage
plan = planner_llm.invoke(
    f"Task: {user_task}. Output a list of steps JSON."
)
# plan = ["Search A", "Search B", "Summarize"]

# Execute Stage
for step in plan:
    try:
        result = executor_tool.run(step)
        context.append(f"Step {step} result: {result}")
    except Exception as e:
        # 简单的错误重试或标记
        context.append(f"Step {step} failed: {e}")

final_answer = summarizer_llm.invoke(context)
```

**3. 模式对比表**：

| 特性 | ReAct | Plan-and-Execute |
| :--- | :--- | :--- |
| **决策方式** | 分布式决策（每一步都决策） | 集中式决策（先规划后执行） |
| **纠错能力** | 弱（容易陷入死循环，需 Max Turns 限制） | 强（可针对单步失败进行重试或局部重规划） |
| **Token 成本** | 较高（每一步都要带全量 Context） | 较低（Executor 可轻量化，只需关注当前步骤） |
| **适用任务** | 动态环境、网页浏览、即时交互 | 复杂逻辑、多步骤批处理、数据分析 |

## 常见考点
1.  **Plan-and-Execute 如果中间某一步失败了怎么办？**
    -   通常做法是让 Executor 将错误反馈给 Planner，触发“重规划”过程，或者在 Plan 阶段就预设条件分支和异常处理逻辑。
2.  **为什么 ReAct 容易产生死循环？**
    -   因为模型在 Observation 状态下可能会做出相同的错误推断，陷入 Thought-Action-Observation 的无限循环，通常需要设置“最大步数”限制来兜底。
3.  **如何结合两者的优点？**
    -   可以采用“规划-执行-再规划”的混合模式（如 BabyAGI 或 DoGPT），先做整体 Plan，每执行完一步后，根据 Observation 决定是继续下一步还是微调后续计划。
