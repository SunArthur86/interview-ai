# Plan-and-Execute 与 ReAct 两种 Agent 模式有什么核心区别？在什么场景下应该选择 Plan-and-Execute？

ReAct 和 Plan-and-Execute 是两种主要的 Agent 任务规划范式。

**ReAct** 是“边想边做”， interleaving 地进行推理和行动。每一步都基于上一步的观测结果决定下一步行动。这种方式灵活、反馈快，适合路径探索或未知环境，但容易陷入局部循环或因早期错误导致后续动作全部失败，Token 消耗不可控。

**Plan-and-Execute** 是“先想后做”，分为两个阶段：
1.  **Plan**：模型首先一次性规划出完成任务的全局步骤列表（如：1. 搜索；2. 读取；3. 总结）。这通常利用 LLM 强大的逻辑推理能力生成结构化的任务链。
2.  **Execute**：模型按顺序执行步骤，通常不再修改计划（或者引入一个“Re-planner”仅在遇到无法解决的错误时介入）。

**选择场景**：
-   **Plan-and-Execute** 适合步骤明确、逻辑链条较长的复杂任务（如编写长篇代码、多步骤的数据分析流程或旅游规划）。它能减少执行过程中的幻觉，避免在中间步骤频繁试错，Token 总成本往往更低且更可控。
-   **ReAct** 更适合需要动态调整策略、环境高度不确定的探索性任务（如网页导航、调试未知错误），因为初始计划可能会因为环境变化而完全失效。

```text

   ┌─────────────┐             ┌───────────────┐
   │   Thought   │             │   Planner     │
```

### 实战深化

**1. 实战案例**：在自动化撰写竞品分析报告时，使用 **Plan-and-Execute** 表现更佳。Planner 首先规划出：[1. 搜索公司A营收, 2. 搜索公司B营收, 3. 搜索公司C营收, 4. 对比数据, 5. 生成报告]。Executor 按序执行。若中途第 2 步搜不到数据，只需针对该步骤重试，不会导致整个流程重跑，效率远高于 ReAct 的“走一步看一步”。

**2. 代码示例 (伪代码)**：
```python
# Plan Stage
plan = planner_llm.invoke("Create a plan for competitor analysis.")
steps = parse_steps(plan) # e.g. ["Search A", "Search B", "Write Report"]

# Execute Stage
results = []
for step in steps:
    # Executor 仅负责执行单步，不负责全局规划
    result = executor_llm.invoke(f"Execute this step: {step}") 
    results.append(result)
    # 如果某步失败，可引入 Re-planner 修正后续步骤
    if not result.success:
        steps = re_planner.adjust_plan(steps, current_step_index)
```

**边界情况**：Plan-and-Execute 面临最大的挑战是**“计划初期的一步错，步步错”**（Error Propagation）。如果在 Plan 阶段生成了一个包含逻辑错误的步骤（如需要不存在的工具或权限），Execute 阶段往往会机械地执行直到崩溃。因此，工业级实现必须在 Execute 循环中引入**“检查点”**（Checkpoint），每执行完一步由 Verifier 模型验证结果是否偏离预期，若偏离则触发 Re-planning。另一个极端场景是**“过度规划”**，即任务在第二步已经完成，但计划仍有十步，需要设计“早停机制”来判断任务是否已提前达成。

**易错点**：
1. **执行僵化**：Executor 严格按照 Plan 执行，即使中间结果已经暗示 Plan 失效也继续执行，导致资源浪费。
2. **步骤粒度**：Plan 的步骤过粗会导致 Executor 无法执行（如“解决bug”），过细则失去规划意义（如“敲下回车键”）。
