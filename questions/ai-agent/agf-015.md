# LATS 与 ReAct 在「探索能力」上如何对比

**核心对比**：ReAct 通常基于当前观察进行贪心决策，主路径线性推进，难以回溯，容易陷入局部最优；LATS（基于树的语言模型搜索）引入 MCTS 思想，显式维护一颗搜索树，通过多分支探索、价值评估与回传，能更系统地权衡不同路径的潜在收益。

**细节增强**：
1. **探索机制**：ReAct 的 Thought 动作虽然包含推理，但本质上是在单一思维链上的局部优化。LATS 利用 MCTS 的 Selection（选择）、Expansion（扩展）、Evaluation（评估）、Backpropagation（回传）循环，能够动态评估“未选路径”的价值。
2. **成本与权衡**：ReAct 调用成本低（只需生成一条链）；LATS 需要多次模拟和评估，计算成本高（Token 消耗大），但在复杂任务（如编程、数学推理）中成功率显著提升。

**架构流程图**：
```text
ReAct (线性链):
Observation -> Thought -> Action -> Observation -> ... (不可回溯)

LATS (树搜索):
          Root (Q)
        /   |   \
    Action1 Action2 Action3  <-- 扩展与评估
      |      |      |
    Child  Child  Child      <-- 价值回传
      ^      |      |
      |______|______|        <-- 选择最优路径继续
```

**实战案例**：
在一个类 LeetCode 的算法解题 Agent 中，ReAct 模型一旦选择了错误的算法思路（如本该用 BFS 却用了 DFS），后续代码编写会一直基于错误前提“死磕”直到 Token 耗尽；而引入 LATS 后，Agent 会并行尝试“贪心算法”和“动态规划”两个分支，通过评估单元测试用例的通过率来回传价值，最终自动放弃低分路径，回溯并选择正确的 DP 分支。

**代码示例 (Python)**：
```python
# LATS 节点价值回传与选择伪代码
def mcts_search(root, iterations):
    for _ in range(iterations):
        node = select_best_child(root)  # UCB公式选择
        if not node.is_terminal:
            node = expand(node)          # LLM生成新Action/Thought
        value = evaluate(node)          # 环境反馈或LLM打分
        backpropagate(node, value)      # 更新路径上所有节点Q值
    return root.max_child_value_path()  # 返回最优路径
```

**对比表格**：
| 特性 | ReAct | LATS |
| :--- | :--- | :--- |
| **搜索策略** | 单路径线性探索 (贪心) | 多分支树状探索 (MCTS) |
| **回溯能力** | 弱 (难以撤销已执行步骤) | 强 (显式回溯并选择兄弟节点) |
| **计算成本** | 低 (单次推理) | 高 (需多次模拟与评估) |
| **适用场景** | 简单问答、路径明确的任务 | 复杂推理、多步规划、高容错场景 |

## 常见考点
1. **剪枝策略**：LATS 在资源有限时如何剪枝（如基于 UCB 公式或价值阈值）？
2. **价值函数设计**：LATS 如何评估节点价值（是直接用 LLM 打分，还是基于外部反馈/Reward）？
3. **ReAct 优化**：如何在不引入树搜索的情况下改进 ReAct 的探索能力（如 CoT-Self Consistency）？
