# MCP(Model Context Protocol)的完整架构是什么?它如何解决 N×M 工具集成问题

- **MCP (Model Context Protocol)** 是 Anthropic 提出的开放协议,标准化 LLM 与外部工具/数据的连接方式.

- **N×M 问题**
- N 个 LLM 应用 × M 个工具 = N×M 个集成点
- 每个 LLM 框架都有自己的工具定义格式
- 工具开发者要为每个框架写适配

- **MCP 架构全貌**
```text
┌───────────────────────────────────────────────────────────────────────┐
│                           LLM Application (Host)                       │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │                     MCP Client (e.g. Claude Desktop)             │  │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐   │  │
│  │  │   Local Tool │  │   Resource  │  │      Prompt         │   │  │
│  │  │   Executor   │  │    Reader   │  │    Template         │   │  │
│  │  └──────┬───────┘  └──────┬───────┘  └──────────┬───────────┘   │  │
│  └─────────┼────────────────┼─────────────────────┼───────────────┘  │
└────────────┼────────────────┼─────────────────────┼──────────────────┘
             │                │                     │
             │      JSON-RPC 2.0 (Standardized Protocol)               │
             ▼                ▼                     ▼
      ┌────────────┐   ┌─────────────┐       ┌──────────────┐
      │  stdio     │   │    SSE      │       │   (Other)    │  <-- Transport Layer
      │  (Local)   │   │  (Remote)   │       │              │
      └─────┬──────┘   └──────┬──────┘       └──────────────┘
            │                 │
            ▼                 ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                        MCP Server Ecosystem                             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────────────┐  │
│  │  File System │  │   Database   │  │   Git / GitHub Integration   │  │
│  │   Server     │  │   Server     │  │         Server               │  │
│  └──────────────┘  └──────────────┘  └──────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
```

- **核心组件**
1. **MCP Client**: 集成在宿主应用（如 IDE、浏览器）中，负责启动 Server、管理生命周期、将 LLM 的工具调用请求转化为 JSON-RPC。
2. **MCP Server**: 独立进程，封装具体的工具逻辑（如 `fetch_gmail`、`execute_sql`），通过标准接口暴露资源、提示词和工具。
3. **Transport Layer**: 定义通信信道，支持 `stdio`（本地进程间通信）和 `SSE`（远程服务器通信），确保协议在不同网络环境下通用。
4. **JSON-RPC 2.0**: 统一的消息格式，包含 `jsonrpc`、`method`、`params` 和 `id`，支持双向通信与请求/响应匹配。

- **解决 N×M 问题的机制**
- **一次编写，到处运行**: 工具开发者只需实现一个 MCP Server，任何支持 MCP 协议的 Client（LLM 应用）均可直接连接，无需为每个框架单独开发 Adapter。
- **标准化的能力描述**: 通过 `tools/list`、`resources/list` 等 JSON-RPC 方法，统一了工具元数据的描述格式，消除了框架间的定义差异。

- **边界情况与极端场景**
- **Server 崩溃检测**: Client 需通过 `stdio` 进句柄或 SSE 心跳机制检测 Server 异常退出，并支持自动重启策略，防止 LLM 调用工具时永久挂起。
- **大数据传输**: MCP 虽基于 JSON-RPC，但直接传输大文件（如 100MB PDF）会阻塞通信。最佳实践是 MCP Server 提供下载链接或使用 `resources` 机制流式读取，而非直接在 Response 中二进制传输。
- **并发控制**: LLM 可能并行调用多个工具。Server 需支持并发请求处理，Client 需正确处理 `id` 匹配，防止回调混乱。

- **面试追问**
1. **追问 1**: 既然 MCP 支持 SSE 远程连接，如何保证跨域通信的安全性？
2. **追问 2**: 如果 LLM 调用 MCP 工具超时（例如长 SQL 查询），JSON-RPC 层面如何处理通知与中断？
3. **追问 3**: 相比于直接调用 OpenAPI，MCP 的核心劣势是什么（如额外的进程管理开销）？

- **易错点**
1. **易错点**: 认为 MCP 只能用于本地工具集成。实际上它通过 SSE Transport 广泛支持远程 Server 部署（企业内网私有云工具）。
2. **易错点**: 混淆 MCP 和 OpenAI Function Calling。MCP 是通信协议，Function Calling 是 LLM 的输出格式；MCP Server 可以兼容 Function Calling 的 JSON Schema，但两者的抽象层级不同。
