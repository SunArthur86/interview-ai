# Agent的短期记忆(工作记忆)如何设计?上下文窗口满了怎么办

- **Agent记忆层级:**

```
即时记忆 = 当前prompt上下文窗口(如128K tokens)
↓ 溢出
短期记忆 = 最近N轮对话摘要
↓ 归档
长期记忆 = 向量数据库中的历史交互
```

- **上下文满了的处理策略:**

1. **滑动窗口** - 只保留最近K轮对话
- 简单但丢失早期上下文

2. **摘要压缩** - 定期总结旧对话
- 每N轮将前N-1轮总结为一段摘要
- LangChain的ConversationSummaryMemory

3. **向量检索** - 将历史存入向量库
- 当前问题embedding检索相关历史
- 只注入相关的历史片段

4. **实体记忆** - 提取并记住关键实体
- 记住用户提到的人名/项目名/偏好
- LangChain的EntityMemory

5. **分层压缩**
- 最近3轮:完整保留
- 3-10轮前:摘要
- 10轮前:只存向量索引

- **Token管理流程图:**
```text
┌─────────────────────┐
│   New User Input    │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ Check Context Limit │────── No ────>  Continue (Inject Full)
└──────────┬──────────┘
           │ Yes (Overflow)
           ▼
┌─────────────────────┐
│  Compression Engine │
│  1. Summarize Old   │ <───┐

```

- **实战案例:**：某Code Assistant项目在处理长文件重构任务时，简单的滑动窗口策略导致上下文中丢失了文件头部的“全局变量定义”，致使LLM生成了错误的代码。改进方案采用了`Token Streaming + Semantic Window`，即保留最新代码的同时，强制通过语义检索回填头部定义部分，确保上下文完整性。

- **代码示例 (Python - LangChain风格):**
```python
from langchain_community.chat_message_histories import RedisChatMessageHistory
from langchain_core.prompts import ChatPromptTemplate

def build_prompt(session_id: str, new_input: str):
    history = RedisChatMessageHistory(session_id)
    
    # 1. 获取完整历史
    full_messages = history.messages
    
    # 2. 简单滑动窗口策略：取最后 6 轮 + 预设 System Prompt
    recent_context = full_messages[-12:] # 假设一轮包含user/assistant两条
    
    prompt = ChatPromptTemplate.from_messages([
        ("system", "你是一个编程助手..."),
        *recent_context,
        ("user", "{input}")
    ])
    return prompt.format_prompt(input=new_input)
```

- **策略对比:**

| 策略 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- |
| 滑动窗口 | 实现简单，零推理成本 | 丢失早期关键信息，上下文断裂 | 短对话、无状态任务 |
| 摘要压缩 | 大幅减少Token，保留大意 | 细节丢失，可能产生幻觉 | 长篇叙事类对话 |
| 向量检索 | 精准召回相关历史，节省空间 | 需维护向量库，存在召回延迟 | 需引用很久之前的事实/知识 |
| 分层压缩 | 平衡成本与信息完整性 | 逻辑复杂，需维护多级索引 | 通用长对话Agent |

- **最佳实践:** 滑动窗口 + 摘要压缩 + 向量检索的组合

## 常见考点
1. **系统提示词 占用的空间怎么处理？**
   - 问点：系统指令很长，会被压缩吗？
   - 答案：System Prompt通常作为"永久保留区"，不参与滑动窗口的淘汰，除非模型支持动态注入。
2. **如何判断哪些历史是重要的？**
   - 问点：压缩时怎么防止把关键信息丢了？
   - 答案：可以使用LLM对历史对话进行"重要性评分"，或者基于用户显式反馈（如点赞/修改）来标记高权重片段。
3. **摘要产生的幻觉问题？**
   - 问点：摘要如果是错的怎么办？
   - 答案：保持原始对话的哈希索引，当检测到逻辑冲突时，可以回溯原文进行核实。
