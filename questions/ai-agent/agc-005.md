# Agent 的记忆一般怎么设计

Agent 的记忆设计核心目标是解决 **LLM 上下文窗口有限** 和 **长期知识持久化** 的问题。通常采用分层架构设计。

**1. 记忆分层架构**

*   **Sensory Memory (感官记忆)**
    *   **作用**：最原始的输入缓冲，如当前 Prompt、最新的几轮对话。
    *   **实现**：直接作为 LLM 的 Input Context。

*   **Short-term Memory (短期记忆 / 工作记忆)**
    *   **作用**：存储当前任务链中的关键信息、中间步骤的推理结果。
    *   **实现**：通常使用滑动窗口或 Token 限制内的摘要机制。

*   **Long-term Memory (长期记忆)**
    *   **作用**：存储跨会话的用户偏好、历史事实、知识库。
    *   **实现**：向量数据库 存储。通过 Embedding 相似度检索 (RAG) 将相关记忆注入当前上下文。

**2. 关键设计细节**
*   **Reflect / 重要性判断**：不是所有信息都值得存入长期记忆。需要有一个机制（让 LLM 评分或筛选）判断哪些是「重要事实」，哪些是「临时寒暄」。
*   **带时间戳与来源**：每条记忆应记录 `timestamp` 和 `source`，支持时间范围检索和溯源，方便更新或撤销过时信息。
*   **结构化与半结构化**：对于关键数据（如用户姓名、喜好），尽量提取为结构化 JSON 存储，而非纯文本，以便精准查询。

**记忆读写流程图：**
```text
  ┌──────────────┐       Read (Search)       ┌───────────────────┐
  │ User Query   │ ────────────────────────▶ │ Vector Database   │
  └──────┬───────┘                          │ (Long-term Memory)│
         │                                  └─────────┬─────────┘
         │                                              │
         ▼                                              ▼
  ┌──────────────┐   Embed & Store        ┌───────────────────┐
  │  LLM Agent   │ ◀────────────────────── │  Reflection /     │

```

**实战案例：**
在构建企业知识库 Agent 时，我们遇到一个严重问题：当公司政策（如报销额度）更新后，Agent 依然检索到旧的向量文档并给出错误建议。解决方案是引入“记忆衰减与版本控制”：将旧记忆标记为 `is_obsolete` 而非物理删除，并在 Prompt 中强调“优先检索时间戳在最近一个月内的记忆”，确保信息的时效性。

**代码示例（Memory 检索与写入）：**
```python
import chromadb
from datetime import datetime

class AgentMemory:
    def __init__(self):
        self.client = chromadb.Client()
        self.collection = self.client.create_collection("user_memories")

    def write(self, content, metadata={}):
        # 写入时带时间戳，便于后续清理或排序
        metadata['timestamp'] = datetime.now().isoformat()
        self.collection.add(
            documents=[content],
            metadatas=[metadata],
            ids=[f"mem_{datetime.now().timestamp()}"]
        )

    def read(self, query, n_results=3):
        # 简单的语义检索
        results = self.collection.query(
            query_texts=[query],
            n_results=n_results
        )
        return results['documents'][0]
```

## 面试追问
1. **记忆冲突**：如果检索到的两条记忆互相矛盾（例如用户在不同时间表达了相反的观点），Agent 应该如何处理？
2. **检索策略优化**：简单的向量检索 (KNN) 往往无法捕捉复杂的查询意图（如“上周所有关于价格大于100的记录”）。如何结合混合检索或关键词过滤来优化？
3. **容量与遗忘**：长期记忆会无限膨胀，如何设计“遗忘机制”或“记忆重要性评分”来自动清理冗余记忆？

## 易错点
1. **混淆上下文窗口与记忆**：认为扩大 LLM 的 Context Window（如 128k/1M）就不需要复杂的记忆设计。实际上，将海量历史数据直接塞入 Context 会导致“迷失中间”现象和极高的推理成本，分层记忆仍然是必要的。
2. **忽视记忆的隐私与隔离**：多租户场景下，简单的向量库检索可能导致 A 用户的记忆泄露给 B 用户。必须在 Metadata 中强制绑定 `user_id` 并在查询时作为强过滤条件。
