# GraphRAG 的原理是什么?它与传统向量 RAG 有什么区别

- **GraphRAG** 是微软提出的知识图谱增强 RAG 方案,通过构建结构化知识图谱来增强检索.

- **ASCII 架构图 (GraphRAG 流程):**
```
输入文档集合
  │
  ▼ [阶段 1: 索引构建 (离线)]
┌─────────────────────────────────────┐
│ 1. 文本分块 → 实体/关系抽取         │
│    (LLM 提取 Nodes & Edges)         │
├─────────────────────────────────────┤
│ 2. 构建同质图 → Leiden 算法社区发现 │
│    (将节点聚类为 Community)         │
├─────────────────────────────────────┤
│ 3. 生成社区摘要                     │
│    (LLM 总结每个社区的 "主题")      │
└─────────────────────────────────────┘
  │
  ▼ [阶段 2: 检索生成 (在线)]
用户查询
  │
  ├─→ 检索相关实体 (社区内)
  │
  ├─→ 检索相关社区摘要 (全局)
  │
  ▼ → 组装 Context → LLM 生成答案
```

- **核心原理:**
1. 文档预处理阶段:用 LLM 从文档中抽取实体和关系
2. 构建知识图谱:实体为节点,关系为边
3. 社区发现:将相关实体聚类
4. 生成社区摘要:为每个社区生成高层摘要
5. 查询时:先匹配社区,再在社区内精确检索

- **与传统向量 RAG 的区别:**

| 维度 | 传统向量 RAG | GraphRAG |
| :--- | :--- | :--- |
| **索引结构** | 向量数据库 | 知识图谱 + 社区摘要 |
| **检索逻辑** | 语义相似度匹配 | 实体遍历 + 社区层级检索 |
| **全局理解** | 差 (难以跨文档汇总) | 强 (基于社区摘要) |
| **多跳推理** | 依赖向量偶然碰撞 | 基于显式路径推理 |
| **构建成本** | 低 (仅 Embedding) | 高 (大量 LLM 抽取) |
| **更新维护** | 容易 (增量 Vector) | 困难 (需局部重构图) |

- **适用场景:**
- 需要多跳推理的复杂问答
- 实体关系密集的领域(金融、法律、医学)
- 全局摘要类查询

- **代价:** 构图成本高(需要大量 LLM 调用),更新维护复杂.

- **实战案例:** 在某金融研报分析项目中，使用传统 RAG 无法回答“某行业全年的整体风险趋势”，因为答案散落在数百份报告中；切换到 GraphRAG 后，利用社区摘要直接生成了各风险主题的宏观分析，召回率提升了 40%，但索引构建耗时从 30 分钟增加到了 4 小时。

- **代码示例:**
```python
# 使用 LlamaIndex 模拟 GraphRAG 的核心：基于实体关系的检索
from llama_index.core import KnowledgeGraphIndex
from llama_index.graph_stores import NebulaGraphStore

# 1. 构建图索引 (离线)
kg_index = KnowledgeGraphIndex.from_documents(
    documents,
    storage_context=storage_context,
    max_triplets_per_chunk=10,
    include_embeddings=True
)

# 2. 查询时进行多跳推理 (在线)
# 引用同质图中发现的社区路径
response = kg_index.as_query_engine(
    include_text=True, 
    retrieval_mode="keyword" # 显式基于实体/关键词查找
).query("A公司收购B公司后对供应链有什么影响?")
```

## 常见考点
1. **GraphRAG 中的 Community Detection 有什么作用**？（将密集的图结构压缩成高层语义，解决图谱遍历深度过大导致的检索爆炸问题）
2. **GraphRAG 如何处理图谱更新**？（通常需要增量抽取或周期性重建，目前的原生实现更新成本较高）
3. **与传统知识图谱(如 Neo4j) 结合 RAG 的区别**？（GraphRAG 强调从非结构化文本自动构建社区摘要，而传统 KG-RAG 通常依赖预定义的 Schema）

## 记忆要点

- 核心流程：文档抽取实体关系→构建图谱→社区发现(Leiden算法)→生成社区摘要。
- 区别于向量RAG：利用显式结构进行多跳推理，具备全局理解能力，能回答跨文档总结。
- 优势：解决向量检索无法处理复杂关系的问题，召回更精准。
- 代价：索引构建成本极高(需大量LLM调用)，更新维护困难，适合高价值静态知识库。
