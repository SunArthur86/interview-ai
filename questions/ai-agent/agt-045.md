# 通信机制

通信机制决定 Agent 之间如何交换信息，常见四类：直接消息、共享黑板、发布-订阅、消息队列。

### 核心机制对比
1.  **直接消息**：A 显式发给 B（点对点）。简单易追踪，但 N² 连接复杂度高。
2.  **共享黑板**：公共读写区。解耦了“谁知道谁”，适合协作求解，但并发写需处理锁冲突。
3.  **发布-订阅**：主题广播，订阅者自选。扩展性好，适合事件通知。
4.  **消息队列**：先入先出，异步解耦。提供削峰填谷、重试、持久化能力，适合任务流转。

### 选型建议
*   强审计、强顺序、高吞吐 -> **消息队列**。
*   探索式协作、快速原型 -> **共享黑板**。
*   简单多角色交互 -> **直接消息**。

### 通信拓扑示意图

```text
直接消息 (P2P):             共享黑板:

   (A) ──────> (B)               ┌───────────────────┐
    |                           │                   │
    v                           │    Blackboard     │
   (C)              (D) ───────>│ (Read/Write Data) │<────── (E)
                              │                   │
                              └───────────────────┘

发布-订阅 (Pub/Sub):          消息队列:

   Publisher                     Producer
      │                             │
      v                             v
  ┌─────────┐                   ┌─────┐   ┌─────┐   ┌─────┐
  │  Topic  │                   │Queue│──>│Cons1│   │Cons2│
  └────┬────┘                   └─────┘   └─────┘   └─────┘
       │    Subscribe           (持久化/顺序)
   ┌────┴────┐
   │         │
Sub1      Sub2
```

### 代码逻辑（示意）
```python
class Blackboard:
    def write(self, key, value):
        with self._lock: self._data[key] = value
    def read(self, key): return self._data.get(key)

class PubSub:
    def publish(self, topic, payload):
        for handler in self._subs.get(topic, []):
            handler(topic, payload)
```

### 实战案例
在多模态内容处理流水线中，使用 RabbitMQ 实现队列通信。当图片处理 Agent 崩溃时，消息自动回到 Unacked 状态，重新入队供其他 Worker 消费，成功避免了因单节点故障导致整个长视频处理任务丢失的问题。

### 代码示例（基于 Redis 的简单队列）
```python
import redis
import json

class AgentQueue:
    def __init__(self, queue_name):
        self.r = redis.Redis()
        self.q_name = queue_name

    def send_task(self, agent_name, task_data):
        # 模拟消息投递
        payload = {"target": agent_name, "data": task_data}
        self.r.lpush(self.q_name, json.dumps(payload))

    def receive_task(self):
        # 阻塞式获取（BRPOP）
        _, msg = self.r.brpop(self.q_name)
        return json.loads(msg)
```

### 对比表格
| 机制 | 耦合度 | 可靠性 | 典型应用场景 |
| :--- | :--- | :--- | :--- |
| **直接消息** | 高 | 依赖网络，无持久化 | 少量 Agent，简单请求/响应 |
| **共享黑板** | 低（空间耦合） | 中（需自行处理并发） | 软件工程 IDE 插件、协同编辑 |
| **发布-订阅** | 低 | 低（通常发后即焚） | 实时通知、日志广播 |
| **消息队列** | 低 | 高（持久化、ACK） | 异步任务流、高并发处理 |

## 常见考点
1. **黑板锁机制**：高并发读写黑板时，如何避免脏读？（答：使用乐观锁（版本号）或读写锁；对于 Agent 场景，通常写入频率低于读取，可接受简单的互斥锁）。
2. **消息丢失**：使用 RabbitMQ/Kafka 等队列时，如何保证 Agent 消息不丢失？（答：开启 ACK 确认机制，消费者处理成功后才回执，结合持久化磁盘）。
3. **上下文窗口**：在共享黑板模式中，历史记录过多怎么办？（答：实施「滚动摘要」机制，保留最近 N 轮关键信息，旧信息压缩后归档）。
