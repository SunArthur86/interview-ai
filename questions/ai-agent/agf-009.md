# 什么时候选 LangGraph 而不是 AgentExecutor

选择 LangGraph 而不是 AgentExecutor，本质上是选择**状态机编排**还是**固定循环模式**。

**应选 LangGraph 的场景**：
1.  **复杂流程控制**：业务流程不是简单的“思考-行动”线性循环，而是包含多阶段流水线（如：先写大纲，再写正文，最后审核）。
2.  **自定义条件路由**：需要根据中间结果进行复杂的分支判断（如：用户满意度 > 80% 则结束，否则转入人工服务）。
3.  **状态持久化与人工介入**：需要在执行过程中暂停，等待人工审核或输入后再恢复执行（Human-in-the-loop）。
4.  **多 Agent 协作**：需要定义多个 Agent 节点及其特定的交互拓扑（如 Supervisor 模式或层级结构）。
5.  **循环与回溯**：需要特定的回环逻辑，而不只是通用的 ReAct 循环。

**架构对比图**：
```text
AgentExecutor (Standard Loop)      LangGraph (Cyclic Graph)
┌─────────────────────┐           ┌──────────────────────────────┐
│   ┌───────────────┐ │           │  ┌──────┐      ┌──────────┐  │
│   │  Think & Act  │ │           │  │Node A│ ──> │ Node B  │  │
│   └───────┬───────┘ │           │  └──┬───┘      └────┬─────┘  │
│           │         │           │     │(cond)        │       │
│   ┌───────▼───────┐ │           │     ▼              │       │
│   │  Finish? (No) │ │           │  ┌──────┐          │       │
│   └───────┬───────┘ │           │  │Node C│ <─────────┘       │
│           │(Yes)    │           │  └──┬───┘                  │
│           ▼         │           │     │(end)                 │
│   ┌───────────────┐ │           │     ▼                      │
│   │    Return     │ │           │  ┌──────────┐              │
│   └───────────────┘ │           │  │  End     │              │
└─────────────────────┘           │  └──────────┘              │
                                  └──────────────────────────────┘
```

**实战案例**：
在构建内容风控系统时，使用 AgentExecutor 很难实现“机器初审 -> 人工复核 -> 机器发布”的混合流程。切换到 LangGraph 后，利用 `interrupt` 机制在“人工复核”节点暂停，保留状态供审核员在前端点击通过/驳回，流程无缝继续，这是 AgentExecutor 无法做到的。

**代码示例**：
```python
from langgraph.graph import StateGraph, END

def should_continue(state):
    # 复杂条件路由：根据状态中的情绪分数决定是否继续
    return "human_agent" if state["sentiment"] < 0.5 else "auto_agent"

workflow = StateGraph(AgentState)
workflow.add_node("draft", draft_node)
workflow.add_node("human_agent", human_node)  # 人工介入节点
workflow.add_node("auto_agent", auto_node)

# 添加条件边
workflow.add_conditional_edges(
    "draft",
    should_continue,
    {
        "human_agent": "human_agent",
        "auto_agent": "auto_agent"
    }
)
workflow.add_edge("human_agent", END)
workflow.add_edge("auto_agent", END)

app = workflow.compile()
```

**边界情况**：
1. **状态版本冲突**：在长时间运行的人工介入流程中，如果底层数据或工具 Schema 发生了变化，而 Graph 的 State 还是旧版本，恢复执行时可能会导致逻辑错误。需要 State 的版本管理或 Migration 策略。
2. **循环引用死锁**：LangGraph 允许自定义循环，但若条件边设置不当（如 A -> B -> A 的条件永远满足），可能导致无限运行，需要超时或步数限制兜底。
3. **并行竞态**：如果使用了 `Send` 机制进行并行分支执行，多个分支同时修改 State 中的同一个 Key，Reducer 的逻辑必须非常严谨，否则容易产生数据覆盖或丢失。

## 面试追问
1. LangGraph 的 State 是全局共享的，如果多个节点需要修改 State 的不同部分，如何保证数据隔离性和并发安全性？
2. 在 AgentExecutor 中我们很容易实现 `max_iterations`，在 LangGraph 的循环图中，你应该如何优雅地实现全局最大步数限制？
3. LangGraph 的 Checkpointer 机制对于实现「对话历史记忆」和「流程断点续传」分别有什么作用？

## 易错点
1. **过度设计**：对于简单的“单工具调用”或“线性问答”任务，引入 LangGraph 会增加不必要的开发复杂度和状态管理开销，不如直接使用 AgentExecutor 或简单的 Chain。
2. **滥用 `interrupt`**：误认为 `interrupt` 只是用来暂停，实际上它本质上是一种将执行权交还给外部的信号，必须配套完善的 API 来处理 `None` 状态的恢复，否则状态会永久挂起。
