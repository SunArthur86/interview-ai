# 什么时候选 LangGraph 而不是 AgentExecutor

选择 LangGraph 而不是 AgentExecutor，本质上是选择**状态机编排**还是**固定循环模式**。

**应选 LangGraph 的场景**：
1.  **复杂流程控制**：业务流程不是简单的“思考-行动”线性循环，而是包含多阶段流水线（如：先写大纲，再写正文，最后审核）。
2.  **自定义条件路由**：需要根据中间结果进行复杂的分支判断（如：用户满意度 > 80% 则结束，否则转入人工服务）。
3.  **状态持久化与人工介入**：需要在执行过程中暂停，等待人工审核或输入后再恢复执行。
4.  **多 Agent 协作**：需要定义多个 Agent 节点及其特定的交互拓扑（如 Supervisor 模式或层级结构）。
5.  **循环与回溯**：需要特定的回环逻辑，而不只是通用的 ReAct 循环。

**架构对比图**：
```text
AgentExecutor (Standard Loop)      LangGraph (Cyclic Graph)
┌─────────────────────┐           ┌──────────────────────────────┐
│   ┌───────────────┐ │           │  ┌──────┐      ┌──────────┐  │
│   │  Think & Act  │ │           │  │Node A│ ──> │ Node B  │  │
│   └───────┬───────┘ │           │  └──┬───┘      └────┬─────┘  │
│           │         │           │     │(cond)        │       │
│   ┌───────▼───────┐ │           │     ▼              │       │
│   │  Finish? (No) │ │           │  ┌──────┐          │       │
│   └───────┬───────┘ │           │  │Node C│ <─────────┘       │
│           │(Yes)    │           │  └──┬───┘                  │
│           ▼         │           │     │(end)                 │
│   ┌───────────────┐ │           │     ▼                      │
│   │    Return     │ │           │  ┌──────────┐              │
│   └───────────────┘ │           │  │  End     │              │
└─────────────────────┘           │  └──────────┘              │
                                  └──────────────────────────────┘
```

**实战案例**：
在构建内容风控系统时，使用 AgentExecutor 很难实现“机器初审 -> 人工复核 -> 机器发布”的混合流程。切换到 LangGraph 后，利用 `interrupt` 机制在“人工复核”节点暂停，保留状态供审核员在前端点击通过/驳回，流程无缝继续，这是 AgentExecutor 无法做到的。

**代码示例**：
```python
from langgraph.graph import StateGraph, END

def should_continue(state):
    # 复杂条件路由：根据状态中的情绪分数决定是否继续
    return "human_agent" if state["sentiment"] < 0.5 else "auto_agent"

workflow = StateGraph(AgentState)
workflow.add_node("draft", draft_node)
workflow.add_node("human_agent", human_node)  # 人工介入节点
workflow.add_node("auto_agent", auto_node)

# 添加条件边
workflow.add_conditional_edges(
    "draft",
    should_continue,
    {
        "human_agent": "human_agent",
        "auto_agent": "auto_agent"
    }
)
workflow.set_entry_point("draft")
app = workflow.compile()
```

**边界情况**：
1.  **状态版本冲突**：当 Graph 定义发生变化（如节点增删）但旧的 Checkpoint 数据结构未迁移时，恢复执行会导致反序列化错误。
2.  **循环依赖检测**：LangGraph 允许构建循环，但若配置不当可能导致图结构中有死节点或永远无法到达的终态，需要预编译检测。
3.  **事件流中断**：在长时间运行的流式执行中，如果客户端断开连接，服务端如何正确处理已部分更新的状态和正在进行的异步操作。

## 面试追问
1. LangGraph 的 State 是全局共享的，当同一个 Graph 处理海量并发请求时，如何保证不同会话间状态不串扰？（考察对 Checkpointer 和 Thread ID 机制的理解）
2. 如果要在 LangGraph 中实现一个“带回退”的 DAG（例如如果 B 节点失败，尝试 C 节点，若 C 也失败则回滚到 A 节点重试），你会如何设计边和条件逻辑？（考察对图拓扑设计的深度）
3. 相比于 AgentExecutor 的“黑盒”执行，LangGraph 的白盒化带来了复杂性，在团队协作开发（多人维护不同 Node）时，你如何制定规范以防止状态污染？（考察工程治理能力）

## 易错点
1.  **状态过度膨胀**：在 `State` 中存储了不必要的历史记录或大文件对象，导致 Checkpoint 存储成本激增且序列化变慢。
2.  **混淆 Node 与 Chain**：将复杂的 LangChain Expression Language (LCEL) 逻辑直接塞进一个 Node 函数中，导致代码可读性差，且难以复用。

## 记忆要点

- 本质区别：LangGraph 是状态机编排，AgentExecutor 是固定循环。
- 选 LangGraph：需复杂流程控制、条件路由、人工介入或多 Agent 协作。
- AgentExecutor 适合简单线性 ReAct 任务，LangGraph 支持自定义回环。
- LangGraph 支持 interrupt 机制暂停流程，实现人机协同。
