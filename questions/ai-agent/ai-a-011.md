# Function Calling的工作原理是什么?如何设计好的工具描述

- **Function Calling流程:**

1. 定义工具schema(name/description/parameters)
2. LLM根据用户意图选择工具并生成参数
3. 应用执行工具调用
4. 将结果返回给LLM
5. LLM基于结果生成最终回复

- **设计好的工具描述:**

```json
{
  "name": "search_docs",
  "description": "搜索内部文档库.当用户询问产品文档、API文档或技术规范时使用.不要用于搜索网页.",
  "parameters": {
    "query": {"type": "string", "description": "搜索关键词,英文优先"},
    "limit": {"type": "integer", "description": "返回结果数,默认5", "default": 5}
  }
}
```

- **最佳实践:**
1. **描述明确** - 说明何时用、何时不用
2. **参数语义** - 每个参数描述清楚期望值
3. **工具粒度** - 不太粗(一个工具做太多)也不太细(太多小工具)
4. **错误提示** - 工具返回错误时给LLM可操作的信息
5. **少即是多** - 工具数量控制在5-15个为佳

- **实战案例:**
在早期的股票查询Agent中，我们将工具描述写为"获取股票数据"，导致LLM频繁尝试查询不支持的实时分时数据。将描述改为"**仅支持获取A股日K线及财务指标**"后，无效请求减少了90%，不仅节省了Token，还避免了下游接口因不支持的高频查询而熔断。

- **代码示例 (Python):**
```python
import openai

tools = [{
    "type": "function",
    "function": {
        "name": "get_stock_price",
        "description": "获取A股日K线收盘价，不支持实时分时或美股数据",
        "parameters": {
            "type": "object",
            "properties": {
                "symbol": {"type": "string", "description": "股票代码，如 600519.SH"},
                "date": {"type": "string", "description": "日期，格式 YYYY-MM-DD"}
            },
            "required": ["symbol"]
        }
    }
}]
response = openai.chat.completions.create(model="gpt-4o", messages=[...], tools=tools)
```

- **Function Calling 内部流程图:**
```
User Query
    │
    ▼
┌─────────────────────────────────────┐
│  LLM Inference (System Prompt +     │
│  Tools Definitions in JSON Schema) │
└──────────────┬──────────────────────┘
               │
               │ 1. Decide to call tool
               ▼
┌─────────────────────────────────────┐
│  LLM Output (Special Token Mode)    │
│  { "name": "search", "arguments": {...}}│
└──────────────┬──────────────────────┘
               │
               │ 2. App parses JSON
               ▼
┌─────────────────────────────────────┐
│  Local Code Execution (Python/JS)   │
│  result = tool.search(args)         │
└──────────────┬──────────────────────┘
               │
               │ 3. Append result to history
               ▼
┌─────────────────────────────────────┐
│  LLM 2nd Pass (Synthesize Answer)   │
│  Context: Query + Tool Call + Result│
└──────────────┬──────────────────────┘
               │
               ▼
        Final Response
```

- **## 常见考点**
1. **参数定义规范**：为什么 `parameters` 必须定义为 JSON Schema 格式？`enum` 类型在定义中如何帮助 LLM 减少幻觉？
2. **Parallel Calling**：如何在一个 Response 中同时返回多个 `tool_calls`（例如 OpenAI 的格式），并处理并发执行？
3. **Null/None 处理**：当工具不需要参数时，Schema 应该定义为空对象 `{}` 还是省略该字段？错误的定义会导致什么问题？
