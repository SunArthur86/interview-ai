# Plan-and-Execute（规划与执行）与 ReAct（推理与行动）范式在解决复杂任务时有哪些核心区别？在什么场景下应优先选择 Plan-and-Execute？

ReAct 采用推理与行动的交错迭代，适合探索性任务，但 Token 消耗高且易陷入循环；Plan-and-Execute 采用先规划后执行的分阶段模式，Token 成本低且具备全局视角，适合路径清晰、步骤固定的复杂长链条任务。

### 边界情况与极端场景
1.  **规划阶段的目标坍塌**：若 Plan 阶段生成的计划本身有误（如步骤缺失或逻辑错误），Execute 阶段会机械地执行错误指令，且 ReAct 机制很难中途自我修正。需引入“Plan Review”步骤，在执行前让模型二次检查。
2.  **执行过程中的环境突变**：在 Execute 阶段，若外部环境状态改变（如 API 接口返回值与 Plan 预期不符），Plan-and-Execute 可能因缺乏动态推理能力而卡死。需设计“异常回退机制”，允许在执行失败时触发局部 ReAct。
3.  **死循环与无限探索**：ReAct 在处理无解问题时可能无限尝试新的 Action。必须设置最大步数限制或“放弃令牌”，防止成本失控。

## 易错点
1.  **范式错配**：将 Plan-and-Execute 用于完全未知的探索性任务（如“帮我找一下这个冷门漏洞的资料”），会导致生成的计划全是“百度搜索”、“谷歌搜索”等无效通用步骤，效率极低。
2.  **过度依赖静态规划**：认为 Plan 生成后就一劳永逸，忽视了工具执行过程中的依赖变化（如 Token 失效），导致批量失败。

## 面试追问
1.  如何量化评估 Plan-and-Execute 中的 Plan 质量？是否有办法通过 Reward Model 来优化 LLM 的规划能力？
2.  如果我要实现一个既能 ReAct 动态调整又能全局规划的 Agent，你会设计什么样的架构或流程？（提示：ReWoo、TaskWeaver）
3.  在多 Agent 协作场景中，Plan-and-Execute 的“Plan”是由一个 Manager Agent 统一下发，还是各 Agent 自行规划？各有什么优劣？

## 记忆要点

- ReAct 是推理行动交错迭代，适合探索但 Token 高且易循环。
- Plan-and-Execute 先规划后执行，全局视角强，适合步骤固定的长链任务。
- Plan 阶段错误会导致执行机械失败，需引入 Plan Review 二次检查。
- 环境突变时需设计异常回退，允许局部触发 ReAct 动态调整。
- 范式错配：未知探索任务用 ReAct，路径清晰任务用 Plan-and-Execute。
