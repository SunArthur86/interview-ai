# MMR(Maximal Marginal Relevance)的lambda参数是什么含义?如何调节检索结果的多样性

**MMR (Maximal Marginal Relevance) 是解决 RAG 检索结果冗余、提升多样性的经典算法。**

- **核心原理**：
  MMR 在选择下一个文档时，不仅考虑其与 Query 的**相关性**，还要减去其与**已选文档**的相似度。
  公式：$MMR = \lambda \cdot Sim(q, d_i) - (1-\lambda) \cdot \max_{d_j \in R} Sim(d_i, d_j)$

- **Lambda ($\lambda$) 参数含义**：
  - **$\lambda \approx 1$**：侧重相关性。此时算法几乎退化为按相似度排序，结果可能高度冗余（例如连续检索到同一文档的不同片段）。
  - **$\lambda \approx 0$**：侧重多样性。算法会刻意挑选与已选内容差异大的文档，哪怕相关性稍低。
  - **通常取值 0.5 ~ 0.7**，在相关性和多样性间寻找平衡。

**实战案例**：在做医疗问答时，用户问“糖尿病饮食禁忌”。如果不使用 MMR，检索到的 Top-5 结果可能全是关于“糖分摄入”的重复段落。引入 MMR ($\lambda=0.5$) 后，结果覆盖了“糖分”、“水果选择”、“酒精影响”等多个维度，大大丰富了生成答案的全面性。

**代码示例（手动实现 MMR 逻辑）**：
```python
import numpy as np

def mmr(query_embedding, document_embeddings, lambda_param=0.5, k=5):
    # query_embedding: (1, d)
    # document_embeddings: (n, d)
    
    # 1. 计算所有文档与 Query 的相似度
    query_sim = np.dot(query_embedding, document_embeddings.T).flatten()
    
    # 2. 初始化
    selected_indices = []
    remaining_indices = list(range(len(document_embeddings)))
    
    while len(selected_indices) < k:
        # 获取余下文档与 Query 的相关性
        relevance = query_sim[remaining_indices]
        
        if not selected_indices:
            # 第一步直接选最相关的
            best_idx = remaining_indices[np.argmax(relevance)]
        else:
            # 计算余下文档与已选文档的最大相似度 (冗余度)
            selected_docs_emb = document_embeddings[selected_indices]
            redundancy = np.max(
                np.dot(document_embeddings[remaining_indices], selected_docs_emb.T), 
                axis=1
            )
            # MMR 分数计算
            mmr_score = lambda_param * relevance - (1 - lambda_param) * redundancy
            best_idx = remaining_indices[np.argmax(mmr_score)]
            
        selected_indices.append(best_idx)
        remaining_indices.remove(best_idx)
        
    return selected_indices
```

**对比表格**：

| 策略 | 侧重点 | 典型场景 | 缺点 |
| :--- | :--- | :--- | :--- |
| **Top-K (相似度排序)** | 准确性 | 事实核查、精确匹配 | 结果高度重复，信息覆盖面窄 |
| **MMR ($\lambda$)** | 平衡 | 综合问答、多视角总结 | $\lambda$ 难以调优，可能引入噪声 |

**## 边界情况**
1. **候选池过小**：当 `k` 接近候选文档总数时，MMR 会被迫选择相关性极低的文档，导致生成质量下降。需设置 `k` 小于候选池总数的一定比例（如 50%）。
2. **Query 本身模糊**：如果用户 Query 语义不明，相关性分数普遍较低，MMR 可能会退化成随机挑选，导致结果完全跑偏。

**## 易错点**
1. **只看多样性不顾相关性**：盲目调低 `lambda` 导致检索到的内容与 Query 毫无关系，LLM 只能基于噪声生成幻觉答案。
2. **未归一化向量计算相似度**：在计算 `Sim(d_i, d_j)` 时，如果向量未归一化，点积值不能直接代表余弦相似度，会导致冗余惩罚计算失真。
