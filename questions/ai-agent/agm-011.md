# 混合检索怎么去重与限长

**去重**：
- **语义去重**：计算片段间的余弦相似度，超过阈值（如 0.85）则视为重复，保留评分更高的片段。
- **Canonical Key（实体对齐）**：针对实体（如人名、项目名）建立标准化 ID，合并指向同一实体的所有描述，避免“散点”信息。

**限长**：
- **Token 装箱**：按最终的 `rerank_score` 排序，执行 0/1 背包问题算法或贪心策略，使总 Token 数不超过上下文窗口上限。
- **分层注入**：优先注入高层级摘要（主线），若还有空间再按需加载细粒度的向量块（证据）。

**边界情况**：
- **去重导致的逻辑断裂**：两段文本相似度极高（0.9），但一个是“前提条件”，一个是“执行结果”。简单去重可能导致逻辑链条缺失，需保留结构化引用关系而非仅保留文本。
- **动态截断与死循环**：当 Context Window 极度紧张时，反复截断可能只剩下系统提示词，导致 Agent 进入无限自言自语或死循环，需设定“最小有效信息量”阈值。
- **表格/代码片段处理**：对代码或表格进行 Token 装箱时，截断可能破坏语法结构（如少了半个括号），导致 LLM 解析错误。需基于 AST 或逻辑块进行原子性截断。

**实战案例**：
在处理长 PDF 技术文档时，目录和正文分别切片会导致“重复引用”，消耗大量 Token。通过引入层级去重，优先保留目录级摘要，仅在 Rerank 确认高度相关时才展开正文，成功将上下文占用降低了 40%。

**代码示例**：
```python
def dedupe_and_truncate(docs, max_tokens):
    seen = set()
    unique_docs = []
    for doc in sorted(docs, key=lambda x: x.score, reverse=True):
        # MinHash 或 SimHash 用于快速去重指纹
        fingerprint = simhash(doc.content)
        if fingerprint not in seen and fingerprint.is_distinct(seen):
            unique_docs.append(doc)
            seen.add(fingerprint)
    
    # 0/1 背包策略限长
    return knapsack_pack(unique_docs, max_tokens, weight_fn=len_tokens)
```

**## 面试追问**
1. 当检索到的多个文档片段来自同一长文档的不同页码，且内容高度重叠（如页眉页脚），如何在去重时保留最佳的上下文完整性？
2. 0/1 背包算法计算时间复杂度较高，在毫秒级响应要求的在线推理场景中，你会如何做工程优化？（如贪心策略+启发式修正）
3. 如何处理去重后的“引用归因”问题？即合并了多个相似片段后，生成的内容引用哪个原始来源？

**## 易错点**
- **唯分数论**：仅按分数高低去重，可能会保留内容全面但分数略低的片段，而删掉了虽然短但精度极高的片段。应结合“信息熵”或“覆盖率”综合考量。
- **忽视结构化数据**：尝试对 Markdown 表格或 JSON 进行普通文本去重，一旦被截断（如少了一个 `]`），会严重影响 LLM 的理解能力。结构化数据应被视为不可分割的整体。

**## 常见考点**
- 去重会不会误删“表述相似但约束不同”的信息（如“把A改成B”和“把A改成C”）？
- 在 Token 极度受限时（如长对话后期），如何保证核心指令不被截断？
- 多路召回的结果如何进行归一化打分？

## 记忆要点

- 去重策略：语义去重（余弦相似度阈值）+ 实体对齐，避免重复内容消耗Token。
- 限长策略：按Rerank分数排序，执行0/1背包算法或贪心策略，优先注入高层级摘要。
- 结构化保护：代码或表格需基于逻辑块原子性截断，防止破坏语法结构。
- 引用归因：去重后需保留原始来源引用，解决合并片段后的出处追溯问题。
