# LangChain的RecursiveCharacterTextSplitter的分隔符顺序为什么重要

- **RecursiveCharacterTextSplitter 核心思想:**

递归尝试分隔符列表,优先使用「更自然」的边界.

- **默认分隔符顺序:**
```
['\n\n', '\n', ' ', '']
```

- **为什么顺序重要:**

1. **语义完整性** - 先按段落(\n\n)切分,保持语义完整
2. **递归回退** - 段落太长 → 按句子(\n)切 → 句子太长 → 按词( )切
3. **避免硬截断** - 不会在一个句子/词中间切断

- **ASCII 流程图 (递归切分逻辑):**
```
输入: "一段很长的文本...\n\n 另一段更长的文本...超长..."
  │
  ▼
[尝试分隔符 1: \n\n (段落)]
  ├─ 成功且 ChunkSize 合适? ──> 返回 Chunk
  │
  ▼ (失败或 Chunk 过大)
[尝试分隔符 2: \n (句子)]
  ├─ 成功且 ChunkSize 合适? ──> 返回 Chunk
  │
  ▼ (失败或 Chunk 过大)
[尝试分隔符 3: ' ' (单词)]
  ├─ 成功且 ChunkSize 合适? ──> 返回 Chunk
  │
  ▼ (失败或 Chunk 过大)
[尝试分隔符 4: '' (字符)]
  └─ 强制按长度截断 (最后手段)
```

- **实战案例**：在处理PDF合同解析时，若不调整分隔符顺序，会导致"金额条款"中的数字被截断（如"10,000"分成"10"和"000"），通过将"\n"置于" "之前，确保了按行优先，保证了关键数字的完整性，避免了后续LLM回答时的金额幻觉。

- **自定义分隔符示例:**
```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

# 针对Markdown文档的实战配置
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    length_function=len,
    separators=[
        "\n## ",      # Markdown H2 (最高优先级)
        "\n### ",     # Markdown H3
        "\n\n",       # 段落
        "\n",         # 行
        ". ",         # 句子(英文句号+空格)
        "。",         # 句子(中文句号)
        " ",          # 词
        ""            # 字符(兜底)
    ]
)
```

- **对比表格: 递归切分 vs 固定长度切分**

| 特性 | RecursiveCharacterTextSplitter | Fixed-size TextSplitter |
| :--- | :--- | :--- |
| **切分策略** | 按语义层级递归回退 | 简单粗暴按字符数截断 |
| **语义完整性** | 高 (保留句子/段落边界) | 低 (可能在词语中间断裂) |
| **Context利用** | 优 (Overlap信息密度高) | 差 (Overlap包含断裂内容) |
| **计算开销** | 稍高 (多次正则匹配) | 极低 (直接切片) |
| **适用场景** | 生成式RAG、问答系统 | 关键词检索、大范围扫描 |

- **关键参数边界说明:**
  - `chunk_overlap`: 必须小于 `chunk_size`，通常为 10%-20%。若 overlap 过大，上下文窗口消耗过快；若过小，边界语义易丢失。
  - `length_function`: 默认为 `len` (字符数)。对于中文场景，建议根据 Tokenizer 计算 token 数，避免超出模型 Context Window。

- **为什么比Fixed-size好:** 固定长度切分会在句子中间断裂,Recursive优先在自然边界切分。

- **边界情况处理:**
  - **无分隔符文本**: 对于纯代码或连续字符串，若所有分隔符都不存在，递归逻辑最终会使用空字符串 `''` 强制按 `chunk_size` 进行字符级截断，此时等同于 Fixed-size 切分。
  - **超长Token**: 即使指定了 `length_function=len`，若某行或某段落的 Token 数远超模型限制（如模型支持4k，但某行代码有10k字符），会导致 Chunk 依然无法被模型处理。需配合「硬截断」逻辑或预检测。
  - **表格与列表**: 对于Markdown表格，简单的 `\n` 切分会破坏表格结构。实战中需先提取表格块单独处理，或优先使用 `|` 等表格特定分隔符。

## 易错点
1. **误认为递归切分一定能保证语义完整**: 递归切分依赖于预设的「分隔符」。如果输入是一串无换行符的超长JSON字符串，默认的 `\n\n`, `\n`, ` ` 都无法命中，最终会退化为按字符硬截断，导致JSON结构被破坏。
2. **混淆字符长度与Token长度**: 在设置 `chunk_size` 时直接使用字符数（如 `len(text)`），导致实际输入LLM的Token数超出上下文窗口限制。特别是中英文混排场景，1个中文字符可能对应多个Token，必须使用 `tiktoken.count_tokens` 等函数作为 `length_function`。

## 记忆要点

- 核心是递归回退：优先按段落(\n\n)切，太长则按句子(\n)，最后按字符强制切。
- 顺序决定语义完整性：先尝试自然边界，避免在句子或单词中间硬截断。
- 实战需自定义：针对Markdown加`##`，针对代码加`;`或函数名，适配文档结构。
- 对比Fixed-size：递归切分保留语义上下文，Overlap信息密度更高，适合RAG生成。
