# 只用向量检索、不做摘要可以吗

可以，但你会失去低成本的全局叙事（例如任务阶段、未决事项）。

**本质区别**：
- **摘要**：擅长提供压缩后的“主线”和“上下文预览”，Token 效率高，适合构建全局视角。
- **向量检索**：擅长提供精准的“细节证据”和“语义匹配”，适合回溯具体事实。

**最佳实践**：采用 **摘要 + 检索并存**的分层架构。先检索摘要快速定位上下文区域，再从该区域下沉检索原始向量块，既保证效率又避免幻觉。

**边界情况**：
- **信息密度极低的闲聊**：如果是大量无意义的寒暄（如“哈哈”、“真的吗”），生成摘要不仅浪费成本，还可能产出无意义的噪声摘要。此时应设置过滤阈值，仅对包含关键信息的对话进行摘要。
- **多线程任务**：当对话中同时存在多个话题（如一边修代码一边讨论午饭），单一线性摘要会将不同话题混淆。需要按“话题树”结构生成多分支摘要，而非单一摘要。
- **摘要索引失效**：摘要过于抽象（如“讨论了项目进度”），导致无法匹配用户的具象查询词（如“Button颜色”）。需确保摘要中包含实体的原始关键词或引用。
- **长尾信息覆盖**：向量检索能召回“深埋”的细节，而摘要往往倾向于保留最近的高频信息。仅依赖摘要会导致长尾细节永久丢失，必须有向量检索作为兜底。

### 实战案例
开发过一个代码重构 Agent，初期仅使用向量检索。用户提问“我之前打算怎么优化这个循环？”，向量检索虽然召回了几段代码片段，但因为没有摘要记录“当前处于分析阶段、计划用并行算法”这一上下文，导致 Agent 无法给出连贯的建议，只能瞎猜片段意图。

### 代码示例
```python
# 伪代码：先摘要后细节的分层检索策略
def retrieve_context(query, session_id):
    # 1. 先在摘要层（摘要Token少，匹配快）查找相关话题摘要
    summary = vector_store_summary.search(query, k=1)
    
    if summary and summary.score > threshold:
        # 2. 如果摘要相关，限定在该摘要对应的消息ID范围内检索原始细节
        related_msg_ids = summary.linked_message_ids
        details = vector_store_details.search(query, filter={"id": related_msg_ids}, k=5)
        return summary.content + "\n" + details.content
    else:
        # 3. 降级为全局检索
        return vector_store_details.search(query, k=5)
```

### 对比表格
| 维度 | 纯向量检索 | 摘要+检索 |
| :--- | :--- | :--- |
| **上下文连贯性** | 差，片段化严重 | 好，拥有全局主线 |
| **Token 消耗** | 高（需召回大量片段拼凑） | 低（摘要高度浓缩） |
| **实现复杂度** | 低，只需一套向量库 | 中，需维护摘要与原文的映射关系 |

## 常见考点
- 如果纯向量检索效果已经很好，引入摘要的 ROI（投入产出比）在哪里？
- 摘要的粒度如何选取（按天、按话题、按会话）？
- 如何处理摘要与原始细节不一致的情况？

## 易错点
1.  **摘要幻觉**：过度依赖 LLM 生成摘要，可能会捏造原文中没有的结论。务必强制要求摘要引用具体的 Message ID 或包含原文中的关键短语以供核查。
2.  **摘要滞后性**：实时生成摘要有延迟。如果用户刚说完一句话就提问，摘要尚未更新，此时必须依赖向量检索或短期记忆作为兜底，不能仅依赖摘要。
3.  **摘要粒度过粗**：试图将一整天的对话压缩成一段话，导致信息密度过大，检索时反而因为语义泛化而匹配不到具体的用户意图。

## 面试追问
1.  **关于摘要结构**：如果是为了供机器检索，你会选择生成自然语言摘要，还是生成结构化状态（如 JSON）？两者的优缺点和适用场景分别是什么？
2.  **关于检索回溯**：如果摘要和原文的 Embedding 空间距离不一致（即摘要很相关但召回的原文不相关），你会如何设计 Re-rank 策略来修正？
3.  **关于长文档**：在处理超长文档（如 100 页 PDF）时，如何设计层级摘要结构（章节摘要 -> 全文摘要），以平衡检索的精度和召回范围？
