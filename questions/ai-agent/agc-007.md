# 如何避免 Agent 在工具调用间「迷失」

明确停止条件与最大步数；维护任务清单与当前子目标；对每步输出要求结构化（JSON）；关键步骤强制验证（单元测试式检查、二次 LLM 审核）。

### 深度解析与控制机制

**1. 短期记忆与上下文管理**
- **滑动窗口**：Agent 很容易陷入无限循环或死胡同。必须维护一个 `System Counter`，强制设定 `MAX_STEPS`（通常为 5-10 步），超限即终止并返回兜底回复。
- **记忆压缩**：随着步数增加，Context 会爆炸。建议将早期的 Action-Observation 对总结为摘要，而不是保留原始 Token。

**2. 状态追踪与子目标**
- **TODO List 维护**：Planner 生成任务链 `[Task A, Task B, Task C]`，Executor 维护一个指针 `current_index`。每完成一步，指针后移，将已完成任务移至 `Done List`，确保 Agent 清晰知道「我在哪」和「接下来做什么」。

**3. 结构化输出与校验**
- 使用 Function Calling 或 JSON Mode 强制模型输出可被代码解析的指令。
- **自我纠错循环**：在执行关键操作（如 SQL 写入、邮件发送）前，插入一个验证层。

**4. 验证流程图**
```text
┌──────────────┐
│ LLM Output   │ (Proposed Action)
└──────┬───────┘
       ▼
┌──────────────┐    No    ┌──────────────┐
│ Syntax Check │ ────────▶│ Error Msg & │
│ (JSON/Func)  │          │ Retry Prompt │
└──────┬───────┘          └──────────────┘
       │ Yes
       ▼
┌──────────────┐    Fail  ┌──────────────┐
│Safety/Logic  │ ────────▶│ Human / Auto │
│  Validator   │ ◀───────│  Correction  │
│ (Guardrail)  │          └──────────────┘
└──────┬───────┘
       │ Pass
       ▼
┌──────────────┐
│ Tool Execute │
└──────────────┘
```

**5. 强制验证**
- **确定性校验**：对非 LLM 生成的逻辑（如 API 返回码 200）进行判断。
- **LLM 审核**：让另一个 LLM（或同一 LLM 的不同 Prompt）充当 Critic，检查当前输出是否回答了用户的问题，或工具返回结果是否有效。

### 边界情况补充
- **空结果死循环**：工具调用成功（HTTP 200）但返回空列表（如搜索无结果），Agent 可能误判为未完成而不断更换关键词重试。需校验「空结果也是一种有效结果」。
- **工具参数冲突**：当多个工具的参数命名相同但含义不同（如 `id`），在长上下文中 Agent 可能混淆参数归属，导致参数传错对象。需在 Tool Definition 中增加严格的上下文限定。
- **时间窗口失效**：某些工具依赖的时间戳（如 Token 有效期）在多步执行后过期，导致中途报错。需在执行层捕获 401/403 错误并触发重新鉴权流程，而非报错停止。

### 实战补充

**实战案例**：在开发客服 Agent 时，曾出现死循环：查询订单失败 -> 重试查询 -> 再次失败。原因是 Agent 没有意识到是「订单号格式错误」而非「网络波动」。引入「模式识别校验」后，检测到连续两次相同参数的失败，立即切断循环并提示用户检查输入，将无效步数从 20+ 降至 2 步。

**代码示例**：
```python
class AgentController:
    def __init__(self, max_steps=10):
        self.max_steps = max_steps
        self.history = []
        self.loop_detector = {}  # 追踪重复行为

    def run(self, query):
        for step in range(self.max_steps):
            action = llm.decide(query, self.history)
            
            # 边界校验：防止格式错误导致中断
            if not action.is_valid():
                return self.history.append("Error: Invalid JSON format")
            
            # 边界校验：防止死循环
            action_sig = f"{action.tool}:{action.args}"
            if self.loop_detector.get(action_sig, 0) > 2:
                return "Error: Stuck in a loop, please check input."
            self.loop_detector[action_sig] = self.loop_detector.get(action_sig, 0) + 1

            result = tools.execute(action)
            self.history.append((action, result))
            
            if result.is_final_answer:
                return result
```

## 面试追问
1. 如果 Agent 在执行过程中修改了系统环境（如删除了文件），导致后续步骤无法回滚，如何设计「沙箱」机制来隔离风险？
2. 在多 Agent 协作场景下，如果某个 Agent 彻底失联（挂起），主控 Agent 应如何判定超时并进行任务转移？

## 易错点
1. **过度依赖计数器**：仅用 `MAX_STEPS` 作为停止条件会导致复杂任务被迫中断。应结合「任务完成度检测」作为主要停止条件，步数仅作为兜底熔断。
2. **忽略无效工具调用**：Agent 可能调用了一个语法正确但逻辑无意义的工具（如查询参数为空），且工具返回了成功。必须在工具内部实现业务逻辑层面的参数校验。
