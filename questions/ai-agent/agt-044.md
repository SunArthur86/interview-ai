# 三大协作模式

协作模式决定了 Agent 之间的控制流与信息流向，常见三种模式：中心化、流水线、民主讨论。

### 1. 中心化模式
*   **结构**：一个 Manager 负责任务分解、分配与汇总；多个 Worker 执行子任务。
*   **优点**：决策路径清晰，易于审计和权限控制。
*   **缺点**：Manager 是单点瓶颈，规划错误会全局放大。

### 2. 流水线模式
*   **结构**：Agent A -> B -> C，上游输出作为下游输入。
*   **优点**：适合固定 SOP（如数据处理），易于分段测试。
*   **缺点**：错误会逐级传递，难以处理大范围回溯。

### 3. 民主协作模式
*   **结构**：多个对等 Agent 多轮发言，可能由协调者维持流程。
*   **优点**：适合头脑风暴、多角度审稿。
*   **缺点**：易空转，需设置硬终止条件（最大轮数、Token 预算）。

### 代码逻辑（示意）
```python
class BossWorker:
    def run(self, task):
        subtasks = self.boss.plan(task)
        return {name: self.workers[name](st) for name, st in subtasks}

class Pipeline:
    def run(self, input_data):
        for stage in self.stages:
            input_data = stage(input_data)
        return input_data
```

### 实战案例
在构建内容生成平台时，初期使用流水线模式（写手->编辑->发布），常因编辑 Agent 修改标题导致发布 Agent 的 URL 生成规则报错。后改为中心化模式，由 Manager Agent 统一校验数据格式后再分发给发布 Agent，彻底解决了格式冲突导致的流程中断。

### 代码示例（流水线异常处理）
```python
class RobustPipeline:
    def run(self, initial_data):
        context = initial_data
        history = [] # 用于回溯
        
        for stage in self.stages:
            try:
                context = stage(context)
                history.append(context)
            except ValidationError as e:
                # 简单回溯机制：记录错误并停止，或返回上一阶段
                print(f"Error at {stage.__name__}: {e}")
                return {"status": "failed", "last_checkpoint": history[-2]}
        return context
```

### 对比表格
| 模式 | 控制权 | 容错能力 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **中心化** | 集中于 Manager | 高（可重试单个 Worker） | 任务拆解明确的复杂项目 |
| **流水线** | 顺序流转 | 低（上游错误下游全挂） | 数据清洗、转码等 SOP |
| **民主** | 去中心化/轮流 | 中（可通过投票过滤） | 创意策划、辩论、决策辅助 |

## 常见考点
1. **扩展性问题**：在中心化模式中，如果 Worker 数量激增，Manager 如何避免被压垮？（答：分而治之，增加中间层级 Manager，或采用队列缓冲）。
2. **回溯策略**：流水线模式下，下游 Agent 发现上游数据错误，如何优雅处理？（答：设计「驳回」通道，将错误详情返回给上游或回滚到特定检查点 Checkpoint）。
3. **共识达成**：民主模式中如何防止「群体思维」？（答：引入 Devil's Advocate（红队）角色，强制要求提出反对意见，或采用加权投票机制）。
