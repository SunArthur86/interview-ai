# Agent 和 Prompt Chain 有什么本质区别

Agent 和 Prompt Chain 的本质区别在于 **「控制流的确定性」** 和 **「交互闭环」**。

1. **Prompt Chain (链式调用)**
   *   **本质**：静态的工程编排。开发者预先定义好了 A -> B -> C 的调用顺序和数据流转。
   *   **适用场景**：流程固定、逻辑清晰的任务（如：翻译 -> 摘要 -> 风格润色）。
   *   **局限性**：一旦某一步输出不符合预期，链很难自动回溯或调整路径。

2. **Agent (智能体)**
   *   **本质**：动态的自主决策。LLM 在每一步根据「观察」决定下一步动作，路径是运行时生成的树状或网状结构。
   *   **适用场景**：目标明确但路径模糊、需要处理未知错误的任务（如：查资料 -> 发现缺失关键词 -> 重新搜索 -> 整合）。
   *   **优势**：具备试错、调整和自我纠错的能力。

**拓扑结构对比图：**
```text
  Prompt Chain (Linear/Pipeline):      Agent (Dynamic Graph):

  ┌───┐    ┌───┐    ┌───┐             ┌───────┐
  │ A │───▶│ B │───▶│ C │───▶ Result  │ Start │
  └───┘    └───┘    └───┘             └───┬───┘
   (Fixed Path)                        │
                                       ▼
                                 ┌─────────┐   ┌─────────┐
                                 │ Choice 1│   │ Choice 2│ (根据环境反馈选择)
                                 └────┬────┘   └────┬────┘
                                      │             │
                                      ▼             ▼
                                 ┌─────────┐   ┌─────────┐
                                 │Observ-  │   │Observ-  │
                                 │ation    │   │ation    │
                                 └────┬────┘   └────┬────┘
                                      │             │
                                      └──────┬──────┘
                                             ▼
                                         (Result)
```

**实战案例：**
在做客服工单分类时，我们先用 Chain 提取关键词和意图，准确率极高且稳定（99%）。但在处理“客户退货退款”这类需要查询订单状态、计算时间、判断策略的复杂任务时，Chain 会因为中间某一步 API 失败直接报错中断。改用 Agent 后，它能自动识别 API 超时并重试，甚至在缺少订单号时自主追问用户，任务完成率提升了 40%。

**代码示例（LangChain 概念对比）：**
```python
# Prompt Chain: 硬编码顺序，确定性高
chain = (
    prompt_1 | model_1 | parser_1 |  # Step A: 提取信息
    prompt_2 | model_2 | parser_2    # Step B: 基于A的结果翻译
)
chain.invoke({"text": "Hello"})

# Agent: 运行时决策，动态路由
agent = Agent(
    tools=[search_tool, db_tool], 
    model=llm,
    type="structured-chat-zero-shot"
)
# LLM 自主决定调用 search_tool 还是 db_tool，或者直接回答
agent.invoke({"input": "查询最新新闻"})
```

## 面试追问
1. **成本与性能**：Agent 的动态性意味着多次 LLM 调用，成本较高。在实际工程中，如何将 Chain 的高效与 Agent 的灵活性结合（例如 DAG 优化）？
2. **调试难度**：Chain 的每个节点输入输出是确定的，易于调试；Agent 的路径是随机的。生产环境如何监控和 Debug Agent 的“思维链”？
3. **死循环风险**：Agent 在动态决策时可能会陷入“工具调用死循环”（如不断搜索但找不到结果）。如何设计安全护栏来终止这种循环？

## 易错点
1. **滥用 Agent**：对于逻辑完全确定的简单任务（如数据清洗、格式转换），使用 Agent 反而会引入不必要的延迟和不确定性，应优先使用 Chain 或脚本。
2. **忽视 Chain 的路由能力**：实际上 Chain 也可以通过 `RouterChain` 实现条件分支，但这种分支是基于显式规则或分类模型，而不是 Agent 那种基于自然语言推理的隐式动态路由，两者不要混淆。
