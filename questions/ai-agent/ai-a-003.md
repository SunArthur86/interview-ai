# Plan-and-Execute Agent架构是什么?相比ReAct有什么优势

- **ReAct = Reasoning + Acting**

让LLM交替进行推理和行动,使用工具后观察结果.

- **工作流程:**
```
**用户：** 北京明天天气如何?

**Thought：** 我需要查询北京的天气预报
**Action：** weather_api(city="北京", date="tomorrow")
**Observation：** 晴, 25°C, 东南风3级

**Thought：** 我拿到了天气信息,可以回答了
**Action：** finish(answer="北京明天晴,25°C,东南风3级")
```

- **优势:**
1. **减少幻觉** - 基于工具返回的真实数据
2. **可追溯** - 推理链可审查
3. **可纠错** - 工具失败后可重试或换方案

- **局限:**
- Token消耗大(每次都带完整推理历史)
- 延迟高(多轮工具调用)
- 需要精心设计工具描述

- **提示词工程细节：**
  - 使用 `Thought:` 前缀强制模型显式推理。
  - 限制 `Action Input` 为 JSON 格式以便解析。
  - 设置 `max_iterations` 防止无限循环。

### 架构图：ReAct 循环流程
```text
┌─────────────┐
│   用户 Query │
└──────┬──────┘
       ▼
┌───────────────────────────────────────┐
│           LLM 核心                     │
│  ┌─────────┐      ┌───────────────┐   │
│  │ Thought │ ───▶ │   Action      │   │
│  │ (推理)   │      │ (调用工具名)   │   │
│  └─────────┘      └───────┬───────┘   │
└────────────────────────────┼───────────┘
                             │
                             ▼
                    ┌────────────────┐
                    │   Tool / Env   │
                    │  (执行工具/环境) │
                    └────────┬───────┘
                             │
                             ▼
                    ┌────────────────┐
                    │  Observation   │
                    │  (工具返回结果)  │
                    └────────┬───────┘
                             │
                      (反馈给 LLM 继续循环)
```

### ## 常见考点
1. **Few-shot Prompting**：在ReAct Prompt中提供少量示例对性能的影响。
2. **错误处理**：当工具抛出异常时，如何引导模型进行自我修正（Self-Correction）？
3. **Token优化**：如何通过滑动窗口或摘要机制缓解上下文过长的问题？
