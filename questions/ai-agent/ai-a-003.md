# Plan-and-Execute Agent架构是什么?相比ReAct有什么优势

- **ReAct = Reasoning + Acting**

让LLM交替进行推理和行动,使用工具后观察结果.

- **工作流程:**
```
**用户：** 北京明天天气如何?

**Thought：** 我需要查询北京的天气预报
**Action：** weather_api(city="北京", date="tomorrow")
**Observation：** 晴, 25°C, 东南风3级

**Thought：** 我拿到了天气信息,可以回答了
**Action：** finish(answer="北京明天晴,25°C,东南风3级")
```

- **优势:**
1. **减少幻觉** - 基于工具返回的真实数据
2. **可追溯** - 推理链可审查
3. **可纠错** - 工具失败后可重试或换方案

- **局限:**
- Token消耗大(每次都带完整推理历史)
- 延迟高(多轮工具调用)
- 需要精心设计工具描述

- **提示词工程细节：**
  - 使用 `Thought:` 前缀强制模型显式推理。
  - 限制 `Action Input` 为 JSON 格式以便解析。
  - 设置 `max_iterations` 防止无限循环。

- **实战案例**：在企业内部知识库问答中，ReAct模式因为每次都要带上长上下文，导致查询慢且经常超时。我们改用摘要机制，每3步对历史Observation做一次压缩，将Token消耗降低了40%。

- **代码示例**：
```python
# 强制 JSON 输出以提高工具调用的稳定性
system_prompt = """
You are a helpful agent. 
Respond in JSON format with keys: 'thought', 'action', 'action_input'.
"""

response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "system", "content": system_prompt}, ...],
    response_format={"type": "json_object"}
)
```
