# Agent的安全权限如何设计?不同敏感级别的操作如何分级管理

- **操作安全分级:**

| 级别 | 操作类型 | 示例 | 控制策略 |
|------|---------|------|---------|
| L0 安全 | 只读操作 | 读取文件、搜索 | 自动执行 |
| L1 低风险 | 可逆写操作 | 写入临时文件 | 自动执行+日志 |
| L2 中风险 | 不可逆写操作 | 修改文件、git commit | **需确认** |
| L3 高风险 | 系统级操作 | 删除文件、安装包 | **需确认+审计** |
| L4 极高风险 | 外部影响 | 发送邮件、支付、部署 | **需双重确认** |

- **审批机制设计:**

```python
class ToolExecutor:
    def __init__(self, auto_approve_levels={0, 1}):
        self.auto_approve = auto_approve_levels

    async def execute(self, tool, params):
        level = tool.security_level
        if level in self.auto_approve:
            return await tool.run(params)
        else: 
            approved = await self.ask_user(
                f"工具 {tool.name} 需要审批\n"
                f"参数：{params}\n"
                f"风险级别：L{level}\n"
                f"是否允许?"
            )
            if approved:
                return await tool.run(params)
            return {"error": "用户拒绝执行"}
```

- **安全模式:**
- **suggest模式:** 所有写操作都需确认
- **auto模式:** L0-L2自动,L3+需确认
- **yolo模式:** 全部自动(仅限安全沙箱)

- **最佳实践:** 默认suggest模式,用户信任后逐步提升自动化级别

- **实战案例**: 曾有Agent在处理“清理日志”任务时，错误地将`/var/log`识别为目标，导致系统崩溃。引入“路径白名单”机制，并强制所有文件操作工具必须校验路径前缀，防止越界访问。

- **代码示例**:
```python
import os

# 简单的路径白名单校验
ALLOWED_BASE_PATH = "/safe_workspace/"

def safe_write_file(filepath: str, content: str):
    full_path = os.path.abspath(filepath)
    if not full_path.startswith(os.path.abspath(ALLOWED_BASE_PATH)):
        raise PermissionError(f"Access denied: Path {filepath} is outside allowed workspace")
    
    with open(full_path, 'w') as f:
        f.write(content)
    return {"status": "success", "path": full_path}
```

- **权限模型对比**:
| 模型 | 核心思想 | 适用场景 | Agent集成难度 |
|------|---------|---------|--------------|
| **ACL (访问控制列表)** | 资源绑定用户/组权限 | 文件系统、简单API | 低 (直接适配文件操作) |
| **RBAC (基于角色)** | 用户分配角色，角色绑定权限 | 企业内网、多租户系统 | 中 (需维护Agent的Session Role) |
| **ABAC (基于属性)** | 基于用户属性、环境、资源动态决策 | 云原生、微服务、复杂动态权限 | 高 (需复杂的策略引擎判断) |
