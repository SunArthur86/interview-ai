# 规划和执行要不要拆开两个模型

视任务而定。Planner-Executor 拆分可提升可控性（强模型规划、快模型执行）；单模型端到端更简单但易在长链路漂移。可混合：规划用强模型，执行层做确定性校验。

### 深度解析与架构细节

**1. 拆分架构的核心优势**
- **性能优化**：Planner（规划器）通常需要强大的逻辑推理能力，适合使用高参数量的慢模型（如 GPT-4）；Executor（执行器）侧重格式化输出和具体工具调用，适合使用低参数量、低延迟、低成本的快模型（如 GPT-3.5-turbo 或 Llama-7B）。
- **稳定性增强**：长上下文窗口容易导致模型遗忘。拆分后，Planner 仅输出宏观计划，Executor 只需关注当前子任务，显著减小了单次推理的上下文压力。

**2. 架构流图**
```text
┌─────────────┐       Plan (JSON)       ┌───────────────┐
│   User      │ ──────────────────────▶ │   Planner     │
│   Query     │                         │ (LLM - Strong)│
└─────────────┘                         └───────┬───────┘
                                                │ Step 1
                                                ▼
                                        ┌───────────────┐
                                        │   Executor    │ ◀───┐
                                        │ (LLM - Fast)  │     │ Feedback
                                        └───────┬───────┘     │ (Failure/Refine)
                                                │ Tool Call   │
                                                ▼            │
                                        ┌───────────────┐     │
                                        │  Environment  │ ────┘
                                        │ (API / DB)    │
                                        └───────────────┘
```

**3. 单模型端到端**
- 原理：LLM 在同一个 Context 内完成「思考-行动-观察」循环。
- 缺陷：在多步推理中容易发生「雪花漂移」，即随着对话轮次增加，模型对初始意图的理解逐渐偏离。

**4. 混合模式**
- Planner 生成计划后，Executor 并非只能被动执行。可以在执行层引入**硬编码逻辑**或**短代码片段**来处理高确定性任务（如数学计算、正则匹配），仅将需要模糊匹配的任务交回 LLM。

### 实战补充

**实战案例**：在电商数据分析场景中，单模型在做 SQL 生成时，容易把简单的字段拼写错误（如 `user_id` 拼成 `usr_id`）带到最终查询中。拆分后，Planner 负责生成 Schema 选型，Executor 负责生成 SQL 语法，代码层直接通过反射校验字段名，拦截了 90% 的拼写错误，避免了因 Schema 错误导致的重试成本。

**代码示例**：
```python
# Planner Output (Pydantic Model)
class Plan(BaseModel):
    steps: list[str]
    current_step: int = 0
    thought: str

# Executor Logic with Hard-coded Guardrail
if plan.current_step == "generate_sql":
    sql = llm_executor.generate_sql(query)
    # 硬编码校验：防止删库/改表操作
    if any(keyword in sql.lower() for keyword in ["drop", "delete", "truncate"]):
        return ErrorResponse("Dangerous operation blocked by Guardrail")
    return db.run(sql)
```

**架构对比**

| 特性 | 单模型端到端 | Planner-Executor 拆分 |
| :--- | :--- | :--- |
| **实现复杂度** | 低（仅需单 Prompt） | 高（需维护状态传递与接口） |
| **Token 成本** | 高（全量历史重复消费） | 低（Executor 仅关注当前上下文） |
| **推理深度** | 易发散，长链路漂移严重 | 深度可控，逻辑链清晰 |
| **容错性** | 难以精确定位错误环节 | 可针对 Executor 单独重试，成本低 |
| **适用场景** | 简单问答、单步任务 | 复杂任务编排、多工具协同 |

## 常见考点
1.  **Planner 如何处理计划变更？**（考点：重规划机制 Re-planning，触发条件通常基于 Executor 返回的错误码或观察到的环境状态不匹配）。
2.  **如何避免 Planner 和 Executor 之间的指令传递偏差？**（考点：Pydantic/JSON Schema 强约束，Prompt 中明确 Output Format）。
3.  **拆分后的系统整体 Latency 如何优化？**（考点：Executor 并行化，如果子任务间无依赖，可并发调用多个 Executor 实例）。
