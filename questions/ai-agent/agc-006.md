# 规划和执行要不要拆开两个模型

视任务而定。Planner-Executor 拆分可提升可控性（强模型规划、快模型执行）；单模型端到端更简单但易在长链路漂移。可混合：规划用强模型，执行层做确定性校验。

### 深度解析与架构细节

**1. 拆分架构的核心优势**
- **性能优化**：Planner（规划器）通常需要强大的逻辑推理能力，适合使用高参数量的慢模型（如 GPT-4）；Executor（执行器）侧重格式化输出和具体工具调用，适合使用低参数量、低延迟、低成本的快模型（如 GPT-3.5-turbo 或 Llama-7B）。
- **稳定性增强**：长上下文窗口容易导致模型遗忘。拆分后，Planner 仅输出宏观计划，Executor 只需关注当前子任务，显著减小了单次推理的上下文压力。

**2. 架构流图**
```text
┌─────────────┐       Plan (JSON)       ┌───────────────┐
│   User      │ ──────────────────────▶ │   Planner     │
│   Query     │                         │ (LLM - Strong)│
└─────────────┘                         └───────┬───────┘
                                                │ Step 1
                                                ▼
                                        ┌───────────────┐
                                        │   Executor    │ ◀───┐
                                        │ (LLM - Fast)  │     │ Feedback
                                        └───────┬───────┘     │ (Failure/Refine)
                                                │ Tool Call   │
                                                ▼            │
                                        ┌───────────────┐     │
                                        │  Environment  │ ────┘
                                        │ (API / DB)    │
                                        └───────────────┘
```

**3. 单模型端到端**
- 原理：LLM 在同一个 Context 内完成「思考-行动-观察」循环。
- 缺陷：在多步推理中容易发生「雪花漂移」，即随着对话轮次增加，模型对初始意图的理解逐渐偏离。

**4. 混合模式**
- Planner 生成计划后，Executor 并非只能被动执行。可以在执行层引入**硬编码逻辑**或**短代码片段**来处理高确定性任务（如数学计算、正则匹配），仅将需要模糊匹配的任务交回 LLM。

### 边界情况补充
- **空计划处理**：当 Planner 判断用户意图包含恶意或无法执行时，应能直接拒绝而不生成无效计划。
- **上下文溢出**：当 Executor 的执行步骤极大（如超过 50 步）时，即便拆分也可能导致累积的 Prompt 超出上限，需具备「中间状态快照」能力，将历史执行结果压缩为摘要传入。
- **工具级联失败**：当连续多个工具调用失败（如 API 宕机），Executor 应具备熔断机制，停止执行并通知 Planner 重新规划或报错，而不是无限重试。

### 实战补充

**实战案例**：在电商数据分析场景中，单模型在做 SQL 生成时，容易把简单的字段拼写错误（如 `user_id` 拼成 `usr_id`）带到最终查询中。拆分后，Planner 负责生成 Schema 选型，Executor 负责生成 SQL 语法，代码层直接通过反射校验字段名，拦截了 90% 的拼写错误，避免了因 Schema 错误导致的重试成本。

**代码示例**：
```python
# Planner Output (Pydantic Model)
class Plan(BaseModel):
    steps: list[str]
    current_step: int = 0
    thought: str

# Executor Logic with Hard-coded Guardrail
if plan.current_step == "generate_sql":
    sql = llm_executor.generate_sql(schema=context.schema)
    # 边界校验：确保生成的字段在 Schema 中真实存在
    for col in extract_columns(sql):
        if col not in context.schema:
             raise ToolExecutionError(f"Invalid column: {col}")
```

## 面试追问
1. 如果 Planner 和 Executor 使用的是不同厂商的模型（如一个用 GPT-4，一个用 Claude），Prompt 风格不一致导致 Executor 解析失败，你会怎么解？
2. 在需要严格顺序执行的任务中（如转账），如果 Executor 执行第 N 步失败了，Planner 如何感知并调整后续计划？

## 易错点
1. **过度规划**：认为拆分后就万事大吉，忽视了 Planner 本身可能产生幻觉，导致 Executor 根本无法执行（如调用不存在的函数）。必须在 Planner 输出端增加 Schema 级别的校验。
2. **状态同步延迟**：忽略了 Planner 生成的是静态计划，而 Executor 执行的是动态环境。若环境状态在执行期间发生剧烈变化（如库存耗尽），静态计划往往失效，需引入动态重规划机制。

## 记忆要点

- 拆分优势：Planner用强模型做逻辑，Executor用快模型做执行。
- 单模型端到端：简单但易在长链路中发生意图漂移。
- 混合模式：规划层用LLM，执行层引入硬编码校验。
- 拆分收益：降低单次推理上下文压力，提升稳定性。
- 适用场景：复杂任务拆分，简单任务端到端。
