# LATS 相比单次 ReAct 多在哪里成本?换来什么收益

会。频繁重规划可能让执行轨迹不稳定。缓解方式包括：限制重规划次数、局部重规划优先、在 state 中保留已验证事实、对计划变更加一致性检查（新旧计划差异说明）。

伪代码示例：
```python
if needs_replan(state):
    plan = replanner(task, plan, state)
    if error or inconsistency:
        new_plan = LLM_replanner(original_task, old_plan, state, error_trace)
        continue with new_plan
```

**补充细节**：
1. **状态一致性维护**：在 Re-planning 时，必须将“已完成步骤”产生的 `Observations` 转化为不可变的 `Facts` 传递给 LLM，防止模型推翻已验证的结论，减少回环。
2. **触发条件控制**：不要因为单一工具报错就立即重规划，应引入重试机制或回退策略；只有当连续失败或目标达成不可能时才触发 Re-planning。
3. **记忆/约束注入**：在请求 LLM 生成新计划时，显式注入 `constraint: keep_valid_steps`，并 penalize 改变已完成子计划的 Token 生成概率。

**实战案例**：
在自动化运维排错 Agent 中，模型曾因一次网络抖动反复重置整个排查路径（从重启服务回到检查配置），导致死循环。加入“已确认事实”黑名单后，模型跳过已验证正常的步骤，直接定位到深层端口占用问题。

**状态流转图**：
```text
   ┌──────────────┐     Execute      ┌──────────────┐
   │  Current     │ ──────────────> │   Feedback   │
   │   Plan       │                  │ (Observation)│
   └──────┬───────┘                  └──────┬───────┘
          │                                 │
          │ Success/Fail?                   │
          ▼                                 │
   ┌──────────────┐                         │

          │ Critical Failure /
          │ Plan Obsolete?
          ▼
   ┌──────────────┐
```

**对比表格**：

| 维度 | 单次 ReAct | LATS (Language Agent Tree Search) |
| :--- | :--- | :--- |
| **成本来源** | 仅线性探索一条路径 | 构建搜索树，生成并评估多条分支路径，导致大量并行 Token 消耗 |
| **核心收益** | 快速响应，适合简单任务 | 通过探索多步可能性，显著提升复杂推理任务的最终成功率 |
| **计算模式** | 串行迭代 | 树搜索（DFS/BFS），需环境模拟或实际执行验证 |
| **容错能力** | 低，单点失败即任务失败 | 高，可通过剪枝和回溯规避错误路径 |

**边界情况**：
1. **搜索空间爆炸**：在分支因子过大（如每个步骤有 10 种可能的工具选择）的场景下，LATS 的指数级成本增长会导致系统不可用。
2. **评估器幻觉**：如果 LATS 依赖 LLM 作为评估器对节点进行打分，评估器的幻觉会导致错误的路径被优先探索，浪费算力且结果更差。
3. **状态隔离成本**：在真实环境（如数据库）中模拟多条分支的执行需要昂贵的事务回滚或沙箱隔离，否则分支间的状态会互相污染。

## 易错点
1. **混淆重规划与树搜索**：误以为 LATS 仅仅是简单的“试错重跑”，忽略了其核心在于**同时维护多条路径**并进行比较，而非线性地失败重试。
2. **忽视探索利用率**：认为 LATS 仅仅是为了找到正确答案，实际上 LATS 的搜索树数据本身也是重要的资产，可用于离线训练或策略蒸馏。
