# 上下文窗口越来越大,还需要外部记忆吗

需要。长窗口 ≠ 低成本，也不等于可检索、可治理、可遗忘。外部记忆解决跨会话持久化、结构化权限、版本与溯源；窗口内更适合热工作集。

**补充细节**：
1. **计算成本与延迟**：长窗口推理成本随序列长度呈平方级增长（Attention机制），且推理首字延迟（TTFT）显著增加。外部记忆通过向量检索或键值查询实现亚线性复杂度，大幅降低实时交互成本。
2. **检索精度**：在 1M+ token 的上下文中，模型可能会出现“迷失中间”现象，难以精准定位具体信息。外部记忆配合 RAG（检索增强生成）可实现精准召回。
3. **治理与合规**：长窗口是“黑盒”，难以删除特定用户的敏感数据（被遗忘权）。外部记忆（如数据库）支持细粒度的权限控制（ACL）、版本管理和数据删除。

## 常见考点
- **追问 1**：如何设计“热”数据（Context）和“冷”数据的切换策略？（答案：基于时间窗口、重要性评分或重写机制）
- **追问 2**：长上下文场景下如何做 RAG？还需要 RAG 吗？（答案：需要，作为长上下文的补充，用于验证和更新实时信息）
- **追问 3**：混合检索中如何平衡精确匹配和语义匹配？（答案：使用 Reciprocal Rank Fusion (RRF) 或加权打分）
