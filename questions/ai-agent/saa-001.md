# ReAct框架的原理是什么？

ReAct（Reasoning + Acting）让LLM交替进行推理和行动。这种范式结合了思维链和决策执行能力。

**循环流程**：
1. **Thought**：LLM分析当前状态，推导下一步行动
2. **Action**：调用外部工具（搜索、计算器、API等）
3. **Observation**：获取工具返回的执行结果
4. 重复上述步骤，直到得出最终答案

**架构流程图**：
```text
┌─────────────┐
│   用户问题   │
└──────┬──────┘
       ▼
┌─────────────┐     ┌──────────────┐
│   Thought   │────▶│    Action    │────▶ 外部环境(API)
│  (LLM推理)   │     │  (调用工具)    │     (执行查询/计算)
└──────┬──────┘     └──────┬───────┘     └──────┬───────┘
       ▲                   ▼                    │
       │           ┌──────────────┐             │
       │           │ Observation  │◀────────────┘
       │           │  (获取结果)   │
       └───────────┴──────┬───────┘
                       ▼
                 (是否完成?) ──No──┐
                       │ Yes      │
                       ▼          │
                ┌─────────────┐   │
                │  Final Answer◀──┘
                └─────────────┘
```

**Prompt 模板示例**：
```
Question: 北京今天的天气怎么样？
Thought 1: 我需要查询北京的实时天气。
Action 1: Search["北京天气"]
Observation 1: 北京今天晴，气温25°C。
Thought 2: 已经获取到信息，现在可以回答用户了。
Action 2: Finish["北京今天晴，25°C"]
```

**核心优势**：
- **可解释性强**：推理过程（Thought）完全透明，便于调试
- **纠错能力**：如果工具返回错误，LLM可以在下一步Thought中修正策略
- **泛化性**：无需针对特定任务微调模型

**局限性**：
- **线性陷阱**：单线程执行，无法并行处理任务
- **幻觉累积**：中间某一步推理错误可能导致后续一连串错误
- **上下文消耗**：长任务会快速占用Token窗口

**对比延伸**：
- **Plan-and-Solve**：先分解问题再逐步解决，适合数学推理
- **ReAct**：边思考边行动，适合多跳问答和交互式任务

## 常见考点
1. **如何防止死循环？**
   - 设置最大迭代次数
   - 在Prompt中加入"如果重复了同样的动作，请停止"
2. **ReAct 和 CoT (Chain of Thought) 的区别？**
   - CoT 仅是推理过程，不涉及与外部环境交互
   - ReAct 在 CoT 基础上增加了 Action 和 Observation 步骤
3. **如何优化工具调用的准确率？**
   - 优化工具的 Description，让 LLM 理解工具的使用场景
   - 增加少样本示例
