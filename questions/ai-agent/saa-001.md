# ReAct框架的原理是什么？

ReAct（Reasoning + Acting）让LLM交替进行推理和行动。这种范式结合了思维链和决策执行能力。

**循环流程**：
1. **Thought**：LLM分析当前状态，推导下一步行动
2. **Action**：调用外部工具（搜索、计算器、API等）
3. **Observation**：获取工具返回的执行结果
4. 重复上述步骤，直到得出最终答案

**架构流程图**：
```text
┌─────────────┐
│   用户问题   │
└──────┬──────┘
       ▼
┌─────────────┐     ┌──────────────┐
│   Thought   │────▶│    Action    │────▶ 外部环境(API)
│  (LLM推理)   │     │  (调用工具)    │     (执行查询/计算)
└──────┬──────┘     └──────┬───────┘     └──────┬───────┘
       ▲                   ▼                    │
       │           ┌──────────────┐             │
       │           │ Observation  │◀────────────┘
       │           │  (获取结果)   │
       └───────────┴──────┬───────┘
                       ▼
                 (是否完成?) ──No──┐
                       │ Yes      │
                       ▼          │
                ┌─────────────┐   │
                │  Final Answer◀──┘
                └─────────────┘
```

**Prompt 模板示例**：
```
Question: 北京今天的天气怎么样？
Thought 1: 我需要查询北京的实时天气。
Action 1: Search["北京天气"]
Observation 1: 北京今天晴，气温25°C。
Thought 2: 已经获取到信息，现在可以回答用户了。
Action 2: Finish["北京今天晴，25°C"]
```

**核心优势**：
- **可解释性强**：推理过程（Thought）完全透明，便于调试
- **纠错能力**：如果工具返回错误，LLM可以在下一步Thought中修正策略
- **泛化性**：无需针对特定任务微调模型

**局限性**：
- **线性陷阱**：单线程执行，无法并行处理任务
- **幻觉累积**：中间某一步推理错误可能导致后续一连串错误
- **上下文消耗**：长任务会快速占用Token窗口

- **边界情况与极端场景**
- **死循环风险**: 若工具返回 `null` 或错误格式，LLM 可能会不断重试相同的 Action，导致 Token 耗尽。需在代码层设置 `max_steps`（如最大 10 步）硬中断。
- **非确定性工具**: 对于返回随机结果的工具（如“获取随机用户”），ReAct 的 Thought 可能无法收敛，认为每次结果不一致是错误，从而陷入无限修正循环。
- **工具输出截断**: 当 Observation 内容极长（如长代码文件），超过上下文窗口限制被截断时，LLM 可能会因为信息缺失而产生幻觉，需在 Prompt 中强制要求“若输出截断，请分批读取”。

- **面试追问**
1. **追问 1**: ReAct 依赖 Prompt 中的 Examples 引导，如果任务极其复杂，Few-Shot Examples 很长，如何优化上下文占用？
2. **追问 2**: ReAct 的行动是串行的，如果我要并行查询三个不相关的 API，如何在 ReAct 框架内改进或扩展？
3. **追问 3**: 如何在 ReAct 中引入“记忆”机制，让 Agent 记住之前的错误并在未来任务中避免？

- **易错点**
1. **易错点**: 认为 Thought 越详细越好。过于冗长的 Thought 会消耗大量 Token 且增加延迟，生产中通常引导 LLM 输出简洁的“关键词式”推理。
2. **易错点**: 忽略 Action 参数的类型校验。LLM 生成的 JSON 参数可能格式错误（如字符串未加引号），代码层需有 try-catch 修复逻辑，否则会导致整个流程崩溃。

## 记忆要点

- 核心循环：Thought(推理)→Action(行动)→Observation(观察)，直到得出答案。
- 优势：推理过程透明可解释，具备纠错能力，无需微调。
- 局限：线性执行无法并行，幻觉累积，长任务消耗Token多。
- 防坑：设置max_steps防止死循环，代码层校验Action参数格式。
- 适用：简单动态任务，依赖工具调用和逐步推理的场景。
