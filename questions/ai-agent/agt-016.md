# 模块综合面试题

### 模块综合面试题

**Q13：请用一句话解释 ReAct。**
A：让模型交替输出推理与可执行行动，并用工具返回的 Observation 闭环纠错，从而把推理接地到外部环境。

**实战案例**：在构建故障排查 Agent 时，ReAct 成功根据日志报错自主决定执行 `grep` 查找代码，而非一直询问用户，实现了从“问答”到“操作”的转变。

**Q14：ReAct 轨迹里，哪一部分必须由系统生成？为什么？**
A：Observation 必须由工具/环境生成，防止模型伪造证据导致「看似合理但错误」的答案。

**Q15：Plan-and-Execute 的最大风险是什么？如何缓解？**
A：风险是错误计划污染全局；缓解是可验证子步骤、重规划、强 Planner 约束输出与执行期监控。

**实战案例**：在编写复杂的数据分析报告时，前期生成的 SQL 计划字段名错误，导致后续所有图表生成失败。通过增加中间步骤的“人工确认节点”才得以止损。

**代码示例 (Python/LangChain)**：
```python
from langchain.agents import AgentExecutor, create_react_agent
from langchain.tools import Tool

def search_tool(query: str) -> str:
    return "Observation: Found 3 related documents."

tools = [Tool(name="Search", func=search_tool, description="useful for searching")]
agent = create_react_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True, max_iterations=5)
```

**Q16：Reflexion 的关键产出是什么？它如何提升下一轮？**
A：关键是高质量反思文本；它作为记忆进入下一轮提示，指导 Actor 改变策略而非重复错误。

**Q17：LATS 与 ReAct 在「探索能力」上如何对比？**
A：ReAct 通常主路径贪心推进；LATS/MCTS 通过多分支搜索与价值回传更系统探索决策空间（成本更高）。

**对比表格**：
| 维度 | ReAct | LATS (Language Agent Tree Search) |
| :--- | :--- | :--- |
| **搜索策略** | 线性/贪心 | 树搜索 (MCTS) + 多分支 |
| **纠错能力** | 仅在当前路径回溯 | 可回溯并探索备选分支 |
| **推理成本** | 低 (单步推进) | 高 (需多次评估节点价值) |
| **适用场景** | 任务简单、逻辑明确 | 任务复杂、需多步决策优化 |

**Q18：LangChain AgentExecutor 中为什么要限制 max_iterations？**
A：防止工具循环、解析失败导致的无限循环，并控制成本与延迟。

**实战案例**：某客服 Agent 因正则匹配错误，陷入了 `Search -> 'No Info' -> Search Again` 的死循环，单次对话消耗了 50 美元 Token。强制设置 `max_iterations=10` 后解决了问题。

**Q19：描述 LangGraph 的条件边解决的业务问题。**
A：根据中间状态动态路由，例如失败重试、需要人工审核、不同客户等级走不同流程。

**Q20：什么时候更建议「单 Agent + 强工具」，而不是多 Agent？**
A：任务边界清晰、无需组织化分工、协调成本可能大于收益时；或延迟/成本敏感场景。

**Q21：多 Agent 系统的「评估器」可以有哪些实现？**
A：单元测试/静态规则、更强模型评审、人工审核节点、外部检索核对事实等。

**Q22：ReAct Prompt 为什么要给 few-shot 示例？**
A：提升模型对固定格式的遵从度，降低解析失败率，稳定工具调用。

**实战案例**：在升级 LLM 到新版本后，模型开始偶尔输出 `Action: <search>`（带尖括号），导致解析器无法识别。加入 3 个严格的 Few-shot 示例后，格式稳定性恢复到 99.9%。

**Q23：Re-planning 与 Reflexion 都「改正错误」，区别是什么？**
A：Re-planning 偏计划结构的调整（下一步怎么走）；Reflexion 偏策略/经验的语言化总结并跨轮复用。

**Q24：LangGraph 相比普通脚本编排的核心收益？**
A：把流程显式化为图，分支/回环/人机节点一等公民，更易维护与观测。

**Q25：如果工具返回噪声很大，ReAct 可能出什么问题？怎么改进？**
A：模型可能被噪声误导；改进包括工具侧清洗/结构化输出、二次检索、在 Thought 里强制引用证据片段、增加校验工具。

**Q26：你如何为一个企业场景选择 ReAct vs Plan-and-Execute？**
A：看任务是否强流程、是否需要可审计计划书、是否允许前期规划成本；需要强工具交互与动态环境用 ReAct，强分解与多步骤交付用 Plan-and-Execute。

**对比表格**：
| 维度 | ReAct | Plan-and-Execute |
| :--- | :--- | :--- |
| **执行模式** | 边想边做 | 先规划后执行 |
| **灵活性** | 极高，适应动态变化 | 较低，依赖初始计划 |
| **可解释性** | 过程透明，但碎片化 | 输出完整计划，可审计性强 |
| **典型场景** | API 调试、网页浏览 | 复杂项目拆解、代码生成 |

**Q27：为什么说「工具描述」是 Agent 的接口设计？**
A：LLM 靠描述做路由与填参；描述就是人机接口，直接影响成功率。

**代码示例 (Tool Description)**：
```python
research_tool = Tool(
    name="SecFilings",
    func=get_sec_data,
    description="获取上市公司 10-K 或 10-Q 报告数据。输入应为股票代码（如 AAPL）和年份。"
)
```
