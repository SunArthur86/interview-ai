# Rerank(重排序)在 RAG 中解决什么问题?如何选择重排序模型

- **Rerank** 是在初步检索后对候选结果进行精细排序的步骤，位于向量检索（召回）与 LLM 生成（阅读）之间，是提升 RAG 精度的关键组件。

- **解决的问题:**
  - **向量检索的语义鸿沟**：基于 ANN（近似最近邻）的向量检索追求速度，主要通过余弦相似度计算，容易在 top-k 中混入语义相似但实质不相关的结果（如关键词匹配但语境不符）。
  - **信息密度低**：如果不重排，大量无关 token 占据 LLM 上下文窗口，导致 "Lost in the Middle" 现象（忽略中间信息）或增加幻觉概率。

- **Rerank 工作原理与架构:**
  1. **粗排**：向量检索快速从亿级库中召回 top-50 或 top-100 候选。
  2. **精排**：Rerank 模型对 Query 和每个 Doc 进行深度交互计算。
  3. **截断**：按相关性分数降序排列，仅保留 top-5 或 top-10 送入 LLM。

```text
用户查询:
   │
   ▼
┌─────────────┐     Top-100 (粗排)      ┌──────────────┐
│   向量检索    │ ──────────────────────>│   Reranker   │
│ (Bi-encoder) │                        │(Cross-encoder)│
└─────────────┘                        └──────┬───────┘
                                              │
                                     Score & Sort
                                              │
                                              ▼
                                     Top-5 (精排)
                                              │
                                              ▼
                                      ┌──────────────┐
                                      │ LLM 生成答案  │
                                      └──────────────┘
```

- **为什么 Cross-encoder 更准 (原理差异):**
  - **Bi-encoder (双塔)**：Query 和 Doc 分别通过独立编码器生成向量，仅计算向量点积。**优点**：可预先存向量库，检索极快；**缺点**：交互发生在最后一层，无法捕获细粒度特征对齐（如否定词、特定修饰语）。
  - **Cross-encoder (交叉塔)**：将 `[CLS] Query [SEP] Doc [SEP]` 拼接后整体送入 Transformer。**优点**：全层注意力机制允许 Query 的每个词与 Doc 的每个词深度交互（Self-Attention），理解复杂的语义逻辑；**缺点**：无法预计算，必须对每个候选实时推理，计算量大（耗时通常是检索的 10-100 倍），因此只能处理少量候选。

- **选型指南:**
  - **Cohere Rerank v3**：商用 SOTA（State of the Art），支持多语言，处理长文本能力强，但需付费且数据出境。
  - **BGE-reranker-large (BAAI)**：开源界首选，中文效果极佳，基于 BERT 架构微调，支持本地部署保障数据隐私。
  - **Jina Reranker / Colbert**：轻量级或基于 Late Interaction 的方案，适合对延迟极度敏感的场景。
  - **Llama-based Rerankers**：如 RankGPT，利用 7B+ 模型生成式重排，效果上限高但成本极贵，适合离线处理。

- **参数与边界条件:**
  - **Top-N (Recall Size)**：通常取 50-100。太少可能漏掉正确文档，太大增加重排延迟。
  - **Top-K (Rerank Size)**：通常取 5-10。取决于 LLM 的上下文窗口大小和任务对噪声的敏感度。

## 常见考点
1. **性能权衡**：Rerank 增加了端到端延迟（通常 +50ms~200ms），有哪些优化手段？（答：缓存、模型量化、Async 调用、仅对复杂 Query 触发）。
2. **多路召回融合**：如果有向量检索和关键词检索（BM25）两路结果，Rerank 如何处理？（答：将两路结果合并去重后一起送入 Rerank，由模型统一打分排序）。
3. **长文本处理**：如果 Document 超过 Rerank 模型的 max_length（如 512）怎么办？（答：截断、滑动窗口打分取最高分、或只打分 Chunk 汇总）。
