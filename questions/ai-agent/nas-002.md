# RAGFlow、Dify、FastGPT 等开源 RAG 框架有什么区别?如何选型

- **主流开源 RAG 框架对比**

| 框架 | 定位 | 核心优势 | 适用场景 | 底层架构倾向 |
|------|------|----------|----------|--------------|
| **RAGFlow** | 深度文档理解 RAG | 基于 DeepDoc 的精准文档解析(表格/图片/布局) | 复杂文档(PDF/合同/论文) | 重解析，轻编排 |
| **Dify** | LLM 应用开发平台 | 可视化工作流 + Agent + RAG 全栈 | 快速构建 AI 应用 | 平台化，SaaS 化 |
| **FastGPT** | 知识库问答 | 工作流编排 + 函数调用 | 企业知识库客服 | 基于 Node.js/HTTP 服务 |
| **LangChain** | LLM 开发框架 | 最灵活的组件化设计 | 定制化 RAG 开发 | Python/JS SDK 库 |
| **LlamaIndex** | 数据框架 | 专注数据连接和索引 | 企业数据湖 RAG | 数据为中心的索引 |

- **技术架构深度对比**
```text
┌──────────────────────┐  ┌───────────────────────┐  ┌──────────────────────┐
│    LangChain (SDK)   │  │    Dify (Platform)    │  │   RAGFlow (Focus)    │
├──────────────────────┤  ├───────────────────────┤  ├──────────────────────┤
│ • Chains / Runnables │  │ • Visual Workflow     │  │ • DeepDoc (OCR/Layout)│
│ • Integrations (100+)│  │ • RAG Pipeline UI     │  │ • Hybrid Search       │
│ • Memory management  │  │ • Model/Key Mgmt      │  │ • Knowledge Graph     │
└──────────┬───────────┘  └───────────┬───────────┘  └──────────┬───────────┘
           │                          │                         │
           ▼                          ▼                         ▼
    High Code Complexity         Low Code / Ops           High Data Accuracy
```

- **选型决策**

1. **文档复杂度高**(PDF 表格、扫描件)→ **RAGFlow**
- DeepDoc 引擎支持布局识别、表格还原、OCR
- 切片质量显著高于朴素按字数切分

2. **快速构建产品**(工作流 + 多 Agent)→ **Dify**
- 拖拽式工作流设计
- 内置 prompt 模板、模型管理、监控

3. **需要深度定制**(自定义检索逻辑)→ **LangChain**
- 最灵活,但需要更多开发工作
- 适合有工程团队的企业

4. **企业知识库**(客服/FAQ)→ **FastGPT**
- 开箱即用的知识库管理
- 支持函数调用增强

- **边界情况与极端场景**
- **混合检索精度**: 在处理专有名词或稀有关键词时，Dify 默认的向量检索可能失效，需强制开启“混合检索（向量+关键词）”并调整权重（RRF算法），否则命中率极低。
- **LangChain 的版本兼容性**: LangChain 更新极快，Deprecated 严重。项目锁定 `langchain-core` 和 `langchain-community` 版本至关重要，否则升级依赖会导致原有 Pipeline 崩溃。
- **多模态支持**: RAGFlow 在处理包含大量图片的文档时，对显存（GPU）要求较高（用于 OCR 模型推理），若无 GPU 加速，解析速度会下降 10 倍以上。

- **面试追问**
1. **追问 1**: Dify 这类平台虽然上手快，但如何解决在复杂业务逻辑中“可视化编排无法表达的代码逻辑”问题？
2. **追问 2**: 如果要在 LangChain 和 LlamaIndex 中二选一作为底层基座，从“数据索引”和“查询路由”的角度分别怎么选？
3. **追问 3**: RAGFlow 的 DeepDoc 解析速度慢，如何在不牺牲精度的前提下优化吞吐量（如引入缓存队列）？

- **易错点**
1. **易错点**: 认为 Dify 可以完全替代开发。实际上 Dify 主要解决“最后一公里”的应用编排，底层数据清洗、ETL 管道仍需外部代码支持。
2. **易错点**: 混淆“知识库管理平台”和“应用开发框架”。FastGPT 更偏向于成品级的问答系统搭建，而 LangChain 是代码级的库，两者的维护成本不在一个维度。
