# 你如何为一个企业场景选择 ReAct vs Plan-and-Execute

**选择核心逻辑**：依据任务的**确定性**、**流程复杂度**与**交互成本**进行权衡。

**详细增强**：
1. **ReAct (Reason + Act)**：适合**动态环境**或**未知路径**的任务。特点是“边走边看”，交互频繁。场景：如客服问答、调试代码、实时信息查询。优势是灵活，劣势是容易迷失且过程不可审计。
2. **Plan-and-Execute (Plan then Solve)**：适合**目标明确**、**步骤可分解**的任务。特点是“先谋后动”，先生成完整计划表，再按步骤执行。场景：如多模块数据分析报告生成、复杂旅行规划。优势是全局视野强、过程可审计，劣势是前期规划 Token 消耗大，且一旦计划错误需全盘重推。

**决策流程图**：
```text
        Start
          |
   环境是否高度动态? (需实时反馈)
   /           \
 Yes           No
  |             |
 ReAct      步骤是否可预判?
             /    \
          Yes     No
           |       |
 Plan-Exe  ReAct
 (需可审计性?) (兜底)
```

**实战案例**：
在开发“电商数据分析 Agent”时，初期使用 ReAct，结果模型在生成图表、计算指标时频繁迷失，经常计算完销售额又忘记录入时间范围，导致反复重试。改为 Plan-and-Execute 后，第一步先让模型列出完整计划（1.筛选数据 -> 2.计算GMV -> 3.生成趋势图 -> 4.输出结论），再逐条执行。这不仅让过程可被人类审计，还复用了同一个“筛选数据”的变量，节省了重复查询的成本。

**代码示例 (Python)**：
```python
# Plan-and-Execute 模式伪代码
def plan_and_execute_agent(goal):
    # Stage 1: Planner
    plan = llm.generate(f"Create a step-by-step plan for: {goal}")
    steps = parse_steps(plan)
    
    # Stage 2: Executor
    context = {}
    for step in steps:
        result = execute_step(step, context)
        context[step.id] = result # 传递上下文
        if result.status == 'fail':
            break # 计划失败需人工介入或全盘重推
    return context
```

**对比表格**：
| 维度 | ReAct | Plan-and-Execute |
| :--- | :--- | :--- |
| **路径规划** | 动态生成，每一步重新思考 | 静态生成，预先列出所有步骤 |
| **容错性** | 单步失败易修正，易迷失方向 | 单步失败往往导致整体计划崩溃 |
| **Token 成本** | 不可控 (取决于推理循环次数) | 前期高 (生成计划)，后期线性增长 |
| **适用性** | 探索性、开放式任务 | 逻辑严密、流程标准化任务 |

## 常见考点
1. **混合模式**：是否存在 ReAct 和 Plan-and-Execute 混合使用的场景？
2. **执行中断**：Plan-and-Execute 在执行过程中遇到不可抗力如何处理？
3. **成本分析**：在长上下文任务下，哪种模式的 Token 消耗更低？
