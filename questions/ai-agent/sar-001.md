# GraphRAG是什么？它和传统RAG有什么区别？

🎯 本质：GraphRAG是微软提出的RAG增强方案，通过构建知识图谱来捕获文档间的实体关系，解决传统RAG只能回答局部问题、无法做全局推理的局限。

🧒 类比：传统RAG像在一堆散落的文件里按关键词搜索（找到相关文件但看不到文件之间的联系）；GraphRAG像先画了一张关系图（谁认识谁、谁在哪工作），然后基于关系图回答问题。

📊 GraphRAG vs 传统RAG：
| 特性 | 传统RAG | GraphRAG |
|------|---------------|----------|
| 数据结构 | 向量索引 | 知识图谱+向量 |
| 擅长 | 具体事实查询 | 全局/关系推理 |
| 示例 | "X的定义是什么" | "整个文档集的主要主题" |
| 构建 | 快(切分+向量化) | 慢(LLM提取实体关系) |
| 更新 | 增量容易 | 需重建社区 |

GraphRAG工作流程：
1. Indexing阶段
  - 用LLM从文档中提取实体和关系
  - 构建知识图谱（节点=实体，边=关系）
  - 用Leiden算法进行社区检测（将图分为主题社区）
  - 为每个社区生成摘要（LLM总结）

2. Query阶段
  - 全局查询：从社区摘要中汇总答案
  - 局部查询：从相关实体/关系出发回答
  - 混合查询：结合两者

优势：能回答跨文档的全局性问题、能做多跳推理、答案更有结构
劣势：索引构建慢且贵（大量LLM调用）、增量更新困难、存储成本高

适用场景：法律文档分析、学术论文综述、企业知识库全局分析

## 常见考点
1. **Leiden 算法的作用**：为什么在 GraphRAG 中需要社区检测？为了解决“总结整个数据集”这类全局性问题，直接遍历所有节点计算量太大，社区摘要提供了中间层级的抽象。
2. **实体提取的 Prompt 技巧**：如何保证 LLM 提取的实体和关系标准化？（使用严格的 Schema 定义和 Enum 类型约束）。
3. **混合检索策略**：GraphRAG 通常不只用图谱检索，而是“图谱 + 向量”混合检索，如何平衡两者的权重？（通常使用 RRF 或 LLM 自行判断）。
4. **增量更新难题**：如果新增了一个文档，是否需要全量重建图谱？目前的 GraphRAG 实现大多需要全量或分区重建，这是其生产落地的主要瓶颈。
