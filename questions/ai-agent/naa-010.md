# 如何全面评估一个 Agent 系统的质量?有哪些专门的 Agent 基准

- **Agent 评估的多维度框架**

- **1. 任务完成率**
- 最基本的指标:任务是否正确完成 (Success Rate)
- 评测方式:人工标注 / 规则判定 / LLM-as-Judge
- 细分:最终结果 vs 关键子步骤达成情况

- **2. 效率指标**
- **步数效率**:完成任务用了多少步(越少越好)
- **Token 消耗**:完成任务用了多少 token (Cost)
- **时间延迟**:首 token 延迟 + 总执行时间
- **成本效率**:每完成任务的成本($)

- **3. 过程指标**
- **工具调用准确率**:选对工具 / 传对参数
- **规划合理性**:步骤是否逻辑通顺,有无冗余步骤
- **错误恢复率**:遇到错误后能否自动恢复

- **4. 安全指标**
- **越狱成功率**:对抗性 prompt 测试
- **信息泄露率**:是否泄露 system prompt / 敏感数据
- **权限越界率**:是否执行了未授权操作

- **评估流程 ASCII 图**

- **专门 Agent 基准**

| 基准 | 评测维度 | 任务类型 |
|------|---------|---------|
| **AgentBench** | 综合能力 | 操作系统/数据库/游戏/网页 |
| **GAIA** | 通用 AI 助手 | 多步推理 + 工具使用 (高难度) |
| **SWE-bench** | 代码修复 | GitHub PR 修复 |
| **WebArena** | 网页操作 | 真实网站交互 |
| **ToolBench** | 工具调用 | 16000+ API 调用 |
| **τ-bench** | 多轮对话 | 客服/预订场景 (动态环境) |

- **生产级评估最佳实践**
1. 建立 100-500 条 golden test set (覆盖核心路径与边缘案例)
2. 每次迭代跑回归测试 (CI/CD 集成)
3. 持续从线上日志收集 bad case (用于数据飞轮)
4. 用 LLM-as-Judge 做批量评估(配合人工抽检,评估一致性 Kappa 系数 > 0.8)

- **实战案例**
- 在某电商客服 Agent 上线初期，仅关注“任务解决率”，导致 Agent 在无效问题上死循环，Token 成本爆炸。引入“步数效率”和“规划合理性”评估后，优化了 Stop 策略，Token 成本降低 40% 且用户体验未受损。

- **评估方法对比表**

| 方法 | 优点 | 缺点 | 适用阶段 |
| :--- | :--- | :--- | :--- |
| **Golden Set + 规则** | 绝对客观，执行快 | 覆盖面窄，维护成本高 | 回归测试 (CI/CD) |
| **Human Evaluation** | 最准确，理解细微差别 | 慢，贵，一致性难保证 | 上线前的最终验收 |
| **LLM-as-Judge** | 速度快，可解释，覆盖广 | 可能存在偏好/幻觉 | 日常开发迭代、大规模评测 |

- **## 常见考点**
- LLM-as-Judge 的主要偏差有哪些，如何消除？
- 如何设计测试集来测试 Agent 的泛化能力？
- 环境依赖（如 API 变动）如何解耦，以保证测试的稳定性？
