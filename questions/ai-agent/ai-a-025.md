# Agent如何做多步推理规划?Forward Planning vs Backward Planning的区别

- **Forward Planning(前向规划):**
- 从当前状态出发,向目标推进
- 「我现在有什么 → 下一步做什么 → 最终达到目标」
- **机制**：基于状态空间的前向搜索（如BFS, A*），每一步评估Precondition是否满足。
- 适合:步骤较少、状态转移明确的任务（如机器人导航）

- **Backward Planning(反向规划):**
- 从目标倒推需要的步骤
- 「要达到X → 需要先完成Y → 需要先完成Z」
- **机制**：基于目标分解，通过Means-Ends Analysis找到差距。
- 适合:目标复杂但路径需要推理的任务（如项目拆解）

```text
       Forward vs Backward 思维路径对比

   Forward Planning (数据驱动):      Backward Planning (目标驱动):

   [当前状态]                         [最终目标]
      |                                  ^
      v                                  |
   [Action A] --> [State B]         [Sub-Goal 2] (需先达成)
      |                                  ^
      v                                  |
   [Action B] --> [State C] ...     [Sub-Goal 1] (需先达成)
                                         |
   [最终目标] (自然到达)             [当前状态] (起步)
```

- **对比:**

| | Forward | Backward |
|--|---------|----------|
| 起点 | 当前状态 | 目标状态 |
| 方向 | 向前推 | 向后推 |
| 搜索空间 | 较大（易发散） | 较聚焦（子目标约束） |
| 适用 | 简单线性、状态明确 | **复杂多步、逻辑依赖** |
| 效率 | 快 | 需更多推理能力 |
| 示例 | 「写完代码→测试→部署」 | 「要部署→需要测试→需要代码」 |

- **混合策略:**
1. 先Backward:分解目标到子目标
2. 再Forward:按依赖顺序执行
3. 执行中动态调整

- **LLM规划prompt设计:**
```
目标：{goal}
当前状态：{state}
可用工具：{tools}

请制定详细计划：
1. 倒推实现目标需要哪些步骤
2. 标注每个步骤的依赖关系
3. 标注每步使用的工具
```

- **实战案例**: 在“一键发布APP到App Store”的复杂Agent任务中，单纯Forward Planning常因构建证书失败而卡住中途。采用Backward Planning后，Agent先列出发布所需的关键资源（证书、描述文件、构建包），发现本地没有证书，倒推生成“创建CSR->申请证书->下载”的子任务链，确保前置条件完备后再开始构建，成功率大幅提升。

- **代码示例 (Python - 目标分解):**
```python
def backward_planner(goal, current_state, actions):
    plan = []
    stack = [goal]
    
    while stack:
        desired_state = stack.pop()
        
        # 寻找能产生该状态的Action
        relevant_action = find_action_precondition(desired_state, actions)
        
        if relevant_action:
            # 检查前置条件是否满足
            for precond in relevant_action.preconditions:
                if precond not in current_state:
                    stack.append(precond) # 将前置条件作为新的子目标
            
            plan.insert(0, relevant_action) # 添加到计划头部
    return plan
```

## 易错点
1. **状态描述的不一致性**：在 Forward Planning 中，LLM 对“当前状态”的理解可能会随着步骤增加而产生漂移，导致后续动作基于错误的假设执行。需要引入外部状态验证机制。
2. **Backward Planning 的不可行性**：反向规划假设目标状态是可达的，如果目标本身定义模糊（如“写一篇好文章”）或工具集中不存在能产生目标状态的原子动作，Backward Planning 会无限递归或失败。

## 面试追问
1. 当 Agent 执行过程中遇到突发错误（如 API 调用失败），导致无法按原计划进行，应该如何在 Forward 和 Backward 模式之间进行动态切换或重新规划？（Re-planning 策略）
2. 在缺乏明确的 Precondition（前置条件）定义的开域环境中（如让 Agent 去订票），如何让 LLM 准确地推断出完成目标所需的隐式状态？
3. 结合 RAG（检索增强生成），如何利用检索到的知识来辅助 Backward Planning，避免生成不可执行的子目标？

## 记忆要点

- Forward Planning从当前状态向目标推，适合简单线性任务；Backward从目标倒推，适合复杂任务。
- Forward易发散，Backward聚焦子目标，混合策略是先Backward分解再Forward执行。
- Backward核心：Means-Ends Analysis，找差距，将前置条件转为新子目标。
- 风险：Forward状态易漂移，Backward若目标不可达或工具缺失会无限递归。
